## Purpose

This note provides a practical workflow for approaching a single CTF, HTB, TryHackMe, Proving Grounds, OSCP-style, or course lab machine.

Use this note when starting a new box.

The goal is to keep the process structured, avoid missing obvious checks, and make sure the final attack path is documented clearly.

---

## Core Principle

A CTF box should be treated as a small investigation.

Do not jump straight from scan results to exploitation. Work through the box in stages:

```text
Prepare workspace
↓
Scan
↓
Enumerate
↓
Identify findings
↓
Test attack paths
↓
Gain initial access
↓
Re-enumerate
↓
Escalate privileges
↓
Capture proof
↓
Write up lessons learned
```

Enumeration should happen repeatedly. Every new credential, hostname, service, file, or user should trigger re-enumeration.

## 1. Create the Box Workspace

Create a dedicated workspace for the machine before starting.

Recommended structure:

```
Box Name - IP Address
├── 00 - Overview
├── 01 - Scans
├── 02 - Enumeration
├── 03 - Findings
├── 04 - Exploitation
├── 05 - Initial Access
├── 06 - Privilege Escalation
├── 07 - Loot
├── 08 - Scripts & Payloads
├── 09 - Screenshots & Evidence
├── 10 - Writeup
└── 99 - Scratchpad
```

Example:

```

```

## 8. Research and Prioritise Attack Paths

Before exploiting, understand the likely weakness.

### Check

- [ ]  Service versions researched
- [ ]  Application/CMS versions researched
- [ ]  Default credentials tested
- [ ]  Known exploits reviewed
- [ ]  Exploit requirements understood
- [ ]  Manual exploitation options considered
- [ ]  Credential-based access considered
- [ ]  Password reuse considered
- [ ]  File upload abuse considered
- [ ]  Injection points considered
- [ ]  Misconfigurations considered
- [ ]  Attack paths prioritised

### Prioritisation

Favour:

1. Confirmed credentials
2. Misconfigurations
3. Clear version-specific vulnerabilities
4. Manual exploit paths
5. Public exploit code
6. Noisy or unstable exploits

In labs, the intended route is often simpler than it first appears. Check simple misconfigurations before chasing complex kernel exploits or obscure CVEs.

## 16. Final Writeup

The final writeup should explain the box clearly from start to finish.

Suggested notes:

```
10.01 - Attack Chain
10.02 - Final Writeup
10.03 - Lessons Learned
```

### Attack Chain Format

```
1. Scanned target and identified exposed services
2. Enumerated service/application and found weakness
3. Exploited weakness to gain initial access
4. Enumerated local system
5. Identified privilege escalation path
6. Escalated to root/Admin/SYSTEM
7. Captured proof
```

### Include

- Target summary
- Open ports
- Key enumeration findings
- Confirmed vulnerability
- Initial access method
- Privilege escalation method
- Credentials used or discovered
- Flags/proof
- Dead ends
- Lessons learned
- Links to reusable knowledge notes

### Check

- [ ]  Final attack path written
- [ ]  Commands/results referenced where useful
- [ ]  Screenshots/evidence linked
- [ ]  Dead ends summarised
- [ ]  Lessons learned recorded
- [ ]  Useful techniques linked to main vault notes

## 19. Common Mistakes to Avoid

- [ ]  Starting exploitation before basic enumeration is complete
- [ ]  Not running a full port scan
- [ ]  Ignoring UDP entirely
- [ ]  Forgetting to check default credentials
- [ ]  Forgetting to check anonymous access
- [ ]  Missing virtual hosts
- [ ]  Ignoring source code comments
- [ ]  Not checking web directories with extensions
- [ ]  Not recording credentials immediately
- [ ]  Not testing password reuse
- [ ]  Not re-enumerating after finding credentials
- [ ]  Not re-enumerating after getting a shell
- [ ]  Overlooking local-only services
- [ ]  Overlooking simple file permissions
- [ ]  Trusting automated tools without manual verification
- [ ]  Losing track of failed paths
- [ ]  Not documenting the working path as soon as it works
