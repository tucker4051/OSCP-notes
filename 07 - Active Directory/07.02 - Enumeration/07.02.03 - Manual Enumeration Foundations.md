## Overview

Manual Active Directory enumeration is the process of gathering domain information using built-in Windows tools, PowerShell, LDAP, and lightweight scripts before relying on larger automated tools.

The goal is to understand:

```text
who exists in the domain
which groups exist
who belongs to privileged groups
which users may be useful targets
how groups are nested
which domain controller is authoritative
how LDAP queries work
how PowerView automates the same concepts
```

This note focuses on two approaches:

```text
1. Manual enumeration with built-in Windows tools and custom PowerShell/LDAP scripts
2. Faster enumeration with PowerView
```

> [!important]
> Repeat enumeration after every new user, host, or privilege level is obtained. Active Directory permissions are perspective-dependent.

---

## Scenario

Domain:

```text
corp.com
```

Initial user:

```text
stephanie
```

Initial access:

```text
RDP to a Windows 11 domain-joined client
```

Example client:

```text
CLIENT75
192.168.50.75
```

Initial user context:

```text
corp.com\stephanie
low-privileged domain user
not local administrator
```

Goal:

```text
Enumerate the full domain.
Identify users, groups, privileged accounts, nested groups, and possible paths toward domain admin.
```

---

# Enumeration Mindset

## Rinse and Repeat

Enumeration is not a one-time task.

Repeat enumeration whenever you obtain:

```text
new domain credentials
a new shell
access to a new workstation
access to a server
local administrator rights
a privileged token
access to a new subnet
```

Why?

```text
Different users see different resources.
Different hosts have different local access.
Individual users may have unique permissions.
Nested groups may expose hidden access paths.
A low-privileged account may still have special rights somewhere.
```

Example:

```text
If you compromise another low-privileged user, do not dismiss it.
Re-run enumeration from that user's context.
```

---

# Access Method Recommendation

Use RDP where possible when you have valid domain credentials.

Example:

```bash
xfreerdp /u:stephanie /d:corp.com /v:192.168.50.75
```

Why RDP is preferred:

```text
interactive desktop session
domain context behaves normally
avoids many Kerberos double-hop issues
domain enumeration tools work more reliably
```

> [!warning]
> PowerShell Remoting or WinRM sessions may hit the Kerberos Double Hop problem, causing domain enumeration tools to fail or return incomplete results.

# Phase 2 – Understanding LDAP Enumeration

## What LDAP Does

Active Directory enumeration relies heavily on LDAP.

LDAP is used when querying AD objects such as:

```text
users
groups
computers
organizational units
printers
service accounts
group policy objects
```

LDAP is the protocol used to communicate with directory services such as Active Directory.

---

## ADSI

PowerShell can use ADSI to interact with Active Directory.

ADSI means:

```text
Active Directory Services Interface
```

It provides a way to access AD through LDAP using built-in Windows/.NET functionality.

This is useful because:

```text
no RSAT required
no Get-ADUser module required
works from normal domain-joined clients
works with low-privileged domain users
uses normal LDAP queries
```

---

## LDAP Path Format

LDAP paths use this general format:

```text
LDAP://HostName[:PortNumber][/DistinguishedName]
```

Example:

```text
LDAP://DC1.corp.com/DC=corp,DC=com
```

Components:

| Component | Meaning |
|---|---|
| `LDAP://` | LDAP provider prefix |
| `DC1.corp.com` | Domain controller or PDC hostname |
| `DC=corp,DC=com` | Distinguished Name of the domain |

---

## Distinguished Name

A Distinguished Name uniquely identifies an AD object.

Example user DN:

```text
CN=Stephanie,CN=Users,DC=corp,DC=com
```

Read it from right to left:

```text
DC=corp,DC=com
↓
CN=Users
↓
CN=Stephanie
```

Meaning:

| Component | Meaning |
|---|---|
| `DC=corp` | Domain component: `corp` |
| `DC=com` | Domain component: `com` |
| `CN=Users` | Users container |
| `CN=Stephanie` | User object common name |

> [!note]
> In a Distinguished Name, `DC` means Domain Component, not Domain Controller.

# Phase 3 – Building a Manual LDAP Enumeration Script

This section builds the script step by step.

The goal is to create a reusable function:

```powershell
LDAPSearch -LDAPQuery "<LDAP filter>"
```

This function will:

```text
find the current domain
find the PDC
find the domain DN
build the LDAP path
create a DirectoryEntry
create a DirectorySearcher
run an LDAP query
return matching AD objects
```

---

## Step 1 – Get Current Domain

PowerShell:

```powershell
[System.DirectoryServices.ActiveDirectory.Domain]::GetCurrentDomain()
```

Example output:

```text
Forest                  : corp.com
DomainControllers       : {DC1.corp.com}
PdcRoleOwner            : DC1.corp.com
Name                    : corp.com
```

Why this matters:

```text
The domain object tells us which domain we are in.
It also identifies the PDC role owner.
```

---

## Step 2 – Store the Domain Object

```powershell
$domainObj = [System.DirectoryServices.ActiveDirectory.Domain]::GetCurrentDomain()
$domainObj
```

This stores the domain information in:

```powershell
$domainObj
```

---

## Step 3 – Extract the PDC

```powershell
$domainObj = [System.DirectoryServices.ActiveDirectory.Domain]::GetCurrentDomain()
$PDC = $domainObj.PdcRoleOwner.Name
$PDC
```

Example output:

```text
DC1.corp.com
```

Why use the PDC?

```text
There can be multiple domain controllers.
The PDC is a reliable target for current domain information.
```

---

## Step 4 – Get the Domain Distinguished Name

Use ADSI:

```powershell
([adsi]'').distinguishedName
```

Example output:

```text
DC=corp,DC=com
```

Store it:

```powershell
$DN = ([adsi]'').distinguishedName
$DN
```

---

## Step 5 – Build the LDAP Path

```powershell
$PDC = [System.DirectoryServices.ActiveDirectory.Domain]::GetCurrentDomain().PdcRoleOwner.Name
$DN = ([adsi]'').distinguishedName
$LDAP = "LDAP://$PDC/$DN"
$LDAP
```

Expected output:

```text
LDAP://DC1.corp.com/DC=corp,DC=com
```

This is the LDAP search root.

---

## Step 6 – Create DirectoryEntry

```powershell
$direntry = New-Object System.DirectoryServices.DirectoryEntry($LDAP)
```

What this means:

```text
DirectoryEntry represents the root LDAP object we want to search from.
```

---

## Step 7 – Create DirectorySearcher

```powershell
$dirsearcher = New-Object System.DirectoryServices.DirectorySearcher($direntry)
```

What this means:

```text
DirectorySearcher performs LDAP queries against the DirectoryEntry search root.
```

---

## Step 8 – Run an Unfiltered Search

```powershell
$dirsearcher.FindAll()
```

This returns many AD objects.

Example paths:

```text
LDAP://DC1.corp.com/DC=corp,DC=com
LDAP://DC1.corp.com/CN=Users,DC=corp,DC=com
LDAP://DC1.corp.com/CN=Computers,DC=corp,DC=com
LDAP://DC1.corp.com/OU=Domain Controllers,DC=corp,DC=com
```

> [!warning]
> Unfiltered LDAP searches can produce a lot of output. Add filters as soon as possible.

# Phase 4 – LDAP Filters

LDAP filters limit what objects are returned.

Example structure:

```text
(attribute=value)
```

Examples:

```text
(samAccountType=805306368)
(objectClass=group)
(objectCategory=computer)
(cn=Sales Department)
```

Compound filter:

```text
(&(objectCategory=group)(cn=Sales Department))
```

Meaning:

```text
Return objects where:
objectCategory is group
AND
cn is Sales Department
```

---

## Common LDAP Filters

| Goal | LDAP Filter |
|---|---|
| Users | `(samAccountType=805306368)` |
| Groups | `(objectClass=group)` |
| Groups, cleaner | `(objectCategory=group)` |
| Computers | `(objectCategory=computer)` |
| Specific user by name | `(name=jeffadmin)` |
| Specific group by CN | `(&(objectCategory=group)(cn=Sales Department))` |
| Wildcard group search | `(&(objectCategory=group)(cn=Development Department*))` |
| Domain Admins group | `(&(objectCategory=group)(cn=Domain Admins))` |

---

## samAccountType Values

Useful values:

| Object Type | Decimal Value |
|---|---:|
| Normal user account | `805306368` |
| Computer account | `805306369` |
| Security group | `268435456` |

For users:

```powershell
$dirsearcher.filter = "samAccountType=805306368"
```

> [!note]
> LDAP filters are usually written with parentheses when passed into the final function, for example `(samAccountType=805306368)`.

# Phase 6 – Create a Reusable LDAPSearch Function

Rather than editing the script for every query, create a function.

Save as:

```text
function.ps1
```

Function:

```powershell
function LDAPSearch {
    param (
        [string]$LDAPQuery
    )

    $PDC = [System.DirectoryServices.ActiveDirectory.Domain]::GetCurrentDomain().PdcRoleOwner.Name
    $DistinguishedName = ([adsi]'').distinguishedName

    $DirectoryEntry = New-Object System.DirectoryServices.DirectoryEntry("LDAP://$PDC/$DistinguishedName")

    $DirectorySearcher = New-Object System.DirectoryServices.DirectorySearcher($DirectoryEntry, $LDAPQuery)

    return $DirectorySearcher.FindAll()
}
```

---

## Import the Function

From PowerShell:

```powershell
Import-Module .\function.ps1
```

If script execution is blocked:

```powershell
powershell -ep bypass
```

Then import again:

```powershell
Import-Module .\function.ps1
```

# Phase 8 – Understanding Nested Groups

Nested groups are groups that are members of other groups.

Example:

```text
Sales Department
├── Development Department
├── pete
└── stephanie
```

Then:

```text
Development Department
├── Management Department
├── pete
└── dave
```

Then:

```text
Management Department
└── jen
```

This means `jen` is directly in:

```text
Management Department
```

but indirectly inherits membership through:

```text
Development Department
Sales Department
```

Why this matters:

```text
nested groups can grant indirect access
net.exe may miss group-in-group relationships
users may have more access than expected
misconfigured nesting can create privilege paths
```

---

```powershell
function LDAPSearch {
    param (
        [string]$LDAPQuery
    )

    $PDC = [System.DirectoryServices.ActiveDirectory.Domain]::GetCurrentDomain().PdcRoleOwner.Name
    $DistinguishedName = ([adsi]'').distinguishedName

    $DirectoryEntry = New-Object System.DirectoryServices.DirectoryEntry("LDAP://$PDC/$DistinguishedName")

    $DirectorySearcher = New-Object System.DirectoryServices.DirectorySearcher($DirectoryEntry, $LDAPQuery)

    return $DirectorySearcher.FindAll()
}
```

## LDAPSearch Queries

Users:

```powershell
LDAPSearch -LDAPQuery "(samAccountType=805306368)"
```

Groups:

```powershell
LDAPSearch -LDAPQuery "(objectCategory=group)"
```

Computers:

```powershell
LDAPSearch -LDAPQuery "(objectCategory=computer)"
```

Specific user:

```powershell
LDAPSearch -LDAPQuery "(name=jeffadmin)"
```

Specific group:

```powershell
LDAPSearch -LDAPQuery "(&(objectCategory=group)(cn=Sales Department))"
```

Print group members:

```powershell
$group = LDAPSearch -LDAPQuery "(&(objectCategory=group)(cn=Sales Department))"
$group.Properties.member
```

# Troubleshooting

## PowerShell Script Will Not Run

Use execution policy bypass for the process:

```powershell
powershell -ep bypass
```

Then import or run the script again:

```powershell
Import-Module .\function.ps1
```

---

## Get-ADUser Is Missing

This is normal on most workstations.

Reason:

```text
Get-ADUser is part of RSAT / ActiveDirectory module.
RSAT is usually installed on domain controllers or admin workstations, not normal clients.
```

Use instead:

```text
ADSI
System.DirectoryServices.DirectorySearcher
PowerView
```

---

## LDAPSearch Returns Too Much Output

Use a narrower LDAP filter.

Bad:

```powershell
LDAPSearch -LDAPQuery "(objectClass=*)"
```

Better:

```powershell
LDAPSearch -LDAPQuery "(samAccountType=805306368)"
```

or:

```powershell
LDAPSearch -LDAPQuery "(&(objectCategory=group)(cn=Sales Department))"
```

---

## Group Output Is Hard to Read

Store one group in a variable and print only `member`.

```powershell
$group = LDAPSearch -LDAPQuery "(&(objectCategory=group)(cn=Sales Department))"
$group.Properties.member
```

---

## net.exe Misses Nested Groups

Use LDAP or PowerView.

Manual LDAP:

```powershell
$group = LDAPSearch -LDAPQuery "(&(objectCategory=group)(cn=Sales Department))"
$group.Properties.member
```

PowerView:

```powershell
Get-NetGroup "Sales Department" | select member
```

---

## Enumeration Fails Over WinRM

Possible cause:

```text
Kerberos Double Hop
```

Use RDP where possible when running domain enumeration tools from a Windows client.

# Key Takeaways

- Start with simple built-in tools like `net user /domain` and `net group /domain`.
- Use RDP where possible to avoid Kerberos double-hop issues.
- `net.exe` is quick but limited.
- LDAP is the core protocol used to query AD.
- ADSI and `.NET` classes allow LDAP enumeration without RSAT.
- Build the LDAP path from the PDC and domain Distinguished Name.
- `DirectoryEntry` defines where the search starts.
- `DirectorySearcher` performs the LDAP query.
- LDAP filters control what objects are returned.
- `LDAPSearch` makes manual enumeration reusable and flexible.
- Nested groups are important and may be missed by `net.exe`.
- PowerView automates many LDAP enumeration tasks.
- Understanding manual LDAP makes PowerView output easier to interpret.
- Repeat enumeration every time your user, host, or privilege context changes.
