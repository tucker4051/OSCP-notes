## Overview

**SharpHound** is the data collector for BloodHound.

**BloodHound** is the graph-based analysis tool used to visualize Active Directory relationships and identify attack paths.

Together, they help map relationships such as:

```text
users
groups
computers
sessions
local admin rights
ACLs
GPO links
SPNs
domain trusts
remote access rights
certificate services relationships
paths to high-value targets
```

SharpHound collects the data.

BloodHound analyzes and visualizes it.

```text
SharpHound
↓
Collect AD data
↓
ZIP / JSON output
↓
BloodHound
↓
Graph analysis
↓
Attack path discovery
```

> [!important]
> BloodHound is most useful when combined with manual enumeration. Manual enumeration gives context; BloodHound helps visualize relationships that are easy to miss.

# Why Use BloodHound

Manual enumeration can answer questions like:

```text
Who are the Domain Admins?
Which computers exist?
Where do I have local admin?
Which users are logged on?
Which groups are nested?
Which objects have dangerous ACLs?
```

BloodHound helps answer relationship questions:

```text
How does this user reach Domain Admin?
Which path is shortest?
Which host should I prioritize?
Which user sessions matter?
Which ACLs create escalation paths?
Which owned principals create new routes?
```

Example manual finding:

```text
stephanie has local admin on CLIENT74.
jeffadmin has a session on CLIENT74.
jeffadmin is a Domain Admin.
```

BloodHound can show this as a graph:

```text
stephanie
  └── AdminTo
      └── CLIENT74
          └── HasSession
              └── jeffadmin
                  └── MemberOf
                      └── Domain Admins
```

# Collection Strategy

## Snapshot Collection

A snapshot collection captures the domain at one point in time.

Use it to gather:

```text
users
groups
computers
ACLs
GPOs
SPNs
trusts
local admin relationships
sessions visible at collection time
```

Example:

```powershell
Invoke-BloodHound -CollectionMethod All
```

or with the current executable-style collector:

```powershell
.\SharpHound.exe --CollectionMethods All
```

Snapshot collections are useful for:

```text
initial domain mapping
identifying obvious attack paths
importing into BloodHound for graph analysis
```

---

## Loop Collection

Loop collection repeatedly collects selected data over time.

This is especially useful for session data because users log on and off throughout the day.

Current-style example:

```powershell
.\SharpHound.exe --CollectionMethods Session --Loop
```

Why loop sessions?

```text
privileged users may log on after your first snapshot
server admins may only appear briefly
help desk users may move between machines
short-lived sessions can create important attack paths
```

Use loop collection when:

```text
the engagement allows it
you need better session coverage
you are waiting for privileged user activity
you want to enrich an initial BloodHound graph
```

# Collection Method Overview

Common SharpHound collection methods:

| Method | Purpose |
|---|---|
| `Default` | Common baseline collection |
| `All` | Broad collection of most available data |
| `Group` | Security group memberships |
| `ACL` | Abusable AD object permissions |
| `Trusts` | Domain trust relationships |
| `Container` | OU tree and GPO links |
| `LocalAdmin` | Local Administrators group membership |
| `RDP` | Remote Desktop Users group membership |
| `DCOM` | Distributed COM Users membership |
| `PSRemote` | Remote Management group membership |
| `Session` | User session collection |
| `LoggedOn` | Privileged logged-on collection method |
| `ObjectProps` | Useful object properties such as timestamps |
| `SPNTargets` | SPN target relationships |
| `DCOnly` | Domain-controller-only collection |
| `ComputerOnly` | Computer-focused collection |

> [!tip]
> `All` is useful in labs. In real assessments, choose collection methods based on scope, stealth, timing, and detection risk.

# BloodHound Core Concepts

## Nodes

Nodes are objects in the graph.

Examples:

```text
users
groups
computers
domains
OUs
GPOs
certificate authorities
sessions
```

Example nodes:

```text
STEPHANIE@CORP.COM
CLIENT74.CORP.COM
DOMAIN ADMINS@CORP.COM
JEFFADMIN@CORP.COM
```

---

## Edges

Edges are relationships between nodes.

Examples:

```text
MemberOf
AdminTo
HasSession
CanRDP
CanPSRemote
GenericAll
GenericWrite
WriteDacl
Owns
AddMember
ForceChangePassword
Contains
GpLink
```

Edges answer:

```text
How are these objects connected?
What can this principal do?
What path exists from current access to target access?
```

---

## Paths

A path is a chain of nodes and edges.

Example:

```text
STEPHANIE@CORP.COM
  └── AdminTo
      └── CLIENT74.CORP.COM
          └── HasSession
              └── JEFFADMIN@CORP.COM
                  └── MemberOf
                      └── DOMAIN ADMINS@CORP.COM
```

Interpretation:

```text
stephanie can administer CLIENT74.
jeffadmin has a session on CLIENT74.
jeffadmin is a Domain Admin.
CLIENT74 is a high-value target.
```

# Marking Owned Principals

Owned principals represent objects you control.

Mark objects as owned when you have:

```text
valid credentials
interactive shell
local admin access
control over a computer
control over a user
control over a group
reliable ability to execute as that principal
```

Examples:

```text
stephanie user
CLIENT75 computer
a compromised service account
a controlled workstation
```

Older UI:

```text
Right-click node
↓
Mark User as Owned
```

or:

```text
Right-click node
↓
Mark Computer as Owned
```

Current UI may use:

```text
node details
object menu
mark as owned / owned toggle
```

> [!important]
> Mark every object you genuinely control. BloodHound pathfinding is much more useful when owned principals are accurate.

# Important Edge Types

## MemberOf

Meaning:

```text
User, group, or computer is a member of another group.
```

Example:

```text
jeffadmin MemberOf Domain Admins
```

Impact:

```text
jeffadmin inherits Domain Admin privileges.
```

---

## AdminTo

Meaning:

```text
A principal has local administrator rights on a computer.
```

Example:

```text
stephanie AdminTo CLIENT74
```

Impact:

```text
stephanie may be able to execute code on CLIENT74 or access admin-only resources.
```

---

## HasSession

Meaning:

```text
A user has a session on a computer.
```

Example:

```text
CLIENT74 HasSession jeffadmin
```

Impact:

```text
Credential material for jeffadmin may exist on CLIENT74.
```

---

## GenericAll

Meaning:

```text
Full control over the target object.
```

Possible impact:

```text
take over user
modify group membership
change attributes
abuse object depending on target type
```

---

## GenericWrite

Meaning:

```text
Can write certain attributes on the target object.
```

Possible impact:

```text
modify SPNs
modify logon scripts
change object attributes
create follow-on abuse paths
```

---

## WriteDacl

Meaning:

```text
Can modify the target object's ACL.
```

Possible impact:

```text
grant yourself stronger rights
create persistence
escalate privileges
```

---

## WriteOwner

Meaning:

```text
Can change ownership of the object.
```

Possible impact:

```text
take ownership
modify DACL
grant yourself control
```

---

## ForceChangePassword

Meaning:

```text
Can reset a user's password.
```

Possible impact:

```text
take over target account
```

---

## CanRDP

Meaning:

```text
Principal can RDP to the computer.
```

Possible impact:

```text
interactive logon
credential exposure
lateral movement
```

---

## CanPSRemote

Meaning:

```text
Principal can use PowerShell Remoting to the computer.
```

Possible impact:

```text
remote command execution through WinRM
```


# OpSec and Detection Notes

SharpHound can generate noticeable activity.

Potential telemetry:

```text
LDAP queries to domain controllers
SMB connections to many hosts
Remote Registry access
Service Control Manager access
NetSessionEnum calls
local group enumeration
session enumeration
large numbers of computer touches
PowerShell script execution
SharpHound process execution
ZIP/JSON output creation
```

Potential detections:

```text
SharpHound executable or PowerShell script on disk
PowerShell importing SharpHound.ps1
Invoke-BloodHound command line
many LDAP queries from one workstation
connections to many hosts in short time
Remote Registry queries
enumeration of local admin groups
session enumeration events
known SharpHound indicators
```

Noise reduction options:

```text
DCOnly collection
ComputerFile with scoped host list
Session-only loop during expected admin activity
Stealth option where appropriate
Throttle and Jitter
targeted collection methods
avoid repeated All collections
```

> [!warning]
> Do not assume SharpHound is stealthy. Use it when allowed by scope and engagement rules.

# Quick Command Reference

## Import PowerShell SharpHound

```powershell
powershell -ep bypass
Import-Module .\SharpHound.ps1
```

## SharpHound Help

```powershell
Get-Help Invoke-BloodHound
```

## Run Older PowerShell Collection

```powershell
Invoke-BloodHound -CollectionMethod All -OutputDirectory C:\Users\<user>\Desktop\ -OutputPrefix "<domain> audit"
```

## Run Current Executable Collection

```powershell
.\SharpHound.exe --CollectionMethods All
```

## Output Directory and Prefix

```powershell
.\SharpHound.exe --CollectionMethods All --OutputDirectory C:\Users\<user>\Desktop\ --OutputPrefix "<domain> audit"
```

## Session Loop

```powershell
.\SharpHound.exe --CollectionMethods Session --Loop --LoopDuration 02:00:00
```

## DC-Only Collection

```powershell
.\SharpHound.exe --CollectionMethods DCOnly
```

```bash
sudo neo4j start
```

```bash
bloodhound-start
```

# Key Takeaways

- SharpHound collects Active Directory data for BloodHound.
- BloodHound visualizes AD relationships as nodes and edges.
- BloodHound can reveal attack paths missed during manual enumeration.
- SharpHound uses techniques similar to PowerView, LDAP queries, Windows APIs, and Remote Registry.
- Collection quality determines graph quality.
- `All` collection is useful in labs, but can be noisy.
- Session data is time-sensitive and may require loop collection.
- Mark owned principals before running owned-principal pathfinding.
- Inspect edges carefully; edges explain why a path exists.
- Validate important BloodHound paths manually.
- Older BloodHound labs may reference Neo4j and the desktop app.
- Current BloodHound CE uses a web-style workflow and file ingest through the UI/API.
- BloodHound is a decision-support tool, not a substitute for operator judgment.
- Re-run collection after every new credential, host, or privilege change.
