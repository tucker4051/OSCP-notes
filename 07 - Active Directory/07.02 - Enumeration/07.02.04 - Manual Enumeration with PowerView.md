## Overview

`PowerView.ps1` is a PowerShell-based Active Directory enumeration tool from PowerSploit.

It automates many common LDAP and .NET enumeration techniques and makes it easier to query Active Directory objects, relationships, permissions, sessions, and potential attack paths.

PowerView is useful for enumerating:

```text
domain information
users
groups
group membership
nested groups
computers
operating systems
local administrator access
logged-on users
sessions
potential credential exposure locations
attack path relationships
```

> [!important]
> PowerView is not magic. It automates LDAP, Windows API, and .NET enumeration techniques. Understanding the manual process helps you interpret results, troubleshoot failures, and avoid relying blindly on tool output.

---

## Why Use PowerView

Manual LDAP enumeration works, but it can become repetitive.

PowerView helps by providing commands such as:

```text
Get-NetDomain
Get-NetUser
Get-NetGroup
Get-NetComputer
Find-LocalAdminAccess
Get-NetSession
```

Compared to `net.exe`, PowerView can:

```text
display more object attributes
query users, groups, computers, and other AD objects
show nested group relationships
filter and format results easily
return data that can be piped into other commands
enumerate computer operating systems
check local admin access across hosts
attempt logged-on user and session enumeration
speed up domain mapping
```

# Enumeration Mindset

At this stage, the goal is not just to list objects.

The goal is to understand relationships between objects.

Important relationships include:

```text
which users are in which groups
which groups are nested inside other groups
which users are logged on to which computers
which computers the current user can administer
which users may have credentials cached on reachable hosts
which systems are likely servers, workstations, or domain controllers
```

These relationships often become attack paths.

Example:

```text
Current user has local admin on CLIENT74.
A privileged user is logged on to CLIENT74.
Credentials may be cached in memory.
This creates a possible path to that privileged user.
```

> [!important]
> In real assessments, do not immediately chase every tempting path. Record it, continue planned enumeration, then prioritize based on scope, goals, risk, and reliability.

# Enumerating Local Administrator Access

## Why This Matters

Finding where the current user has local administrator rights is a major step in AD enumeration.

If the current user is local admin on a host, you may be able to:

```text
log in interactively
execute commands remotely
dump local credential material
inspect running processes
access administrative shares
search for cached credentials
pivot to users logged on to that system
```

This can create a chained compromise.

Example chain:

```text
stephanie has local admin on CLIENT74
jeffadmin is logged on to CLIENT74
jeffadmin is a privileged user
CLIENT74 becomes a high-priority target
```

---

## Find Local Admin Access with PowerView

Run:

```powershell
Find-LocalAdminAccess
```

Example output:

```text
client74.corp.com
```

Interpretation:

```text
The current user has local administrator access on client74.corp.com.
```

In this example:

```text
corp.com\stephanie has local admin rights on CLIENT74.
```

---

## How Find-LocalAdminAccess Works

`Find-LocalAdminAccess` checks whether the current user has administrative access on domain computers.

It uses Windows Service Control Manager access checks.

PowerView attempts to connect to the target Service Control Manager using:

```text
OpenServiceW
SC_MANAGER_ALL_ACCESS
```

If the access check succeeds, PowerView concludes the current user has local administrative privileges on that host.

---

## Why This Is Useful

This identifies systems where your current credential has elevated local control.

Prioritize these hosts for:

```text
session enumeration
credential exposure checks
local admin validation
share access checks
lateral movement planning
```

> [!important]
> Local admin access on a workstation is often more useful if a higher-privileged domain user is logged on there.

# Logged-On User Enumeration

## Why Logged-On Users Matter

When a user logs on to a computer, credential material may exist on that machine.

Depending on logon type, protections, and configuration, the host may contain:

```text
Kerberos tickets
NTLM hashes
cached credentials
access tokens
DPAPI material
session artifacts
```

This matters because:

```text
a low-privileged user with local admin on a host may be able to compromise a higher-privileged user logged on to that host
```

Example:

```text
jeffadmin logged on to CLIENT74
stephanie has local admin on CLIENT74
CLIENT74 may contain useful credential material for jeffadmin
```

---

## Session Enumeration APIs

Historically, Windows APIs used for logged-on user and session enumeration include:

```text
NetWkstaUserEnum
NetSessionEnum
```

High-level difference:

| API | Purpose | Privilege Notes |
|---|---|---|
| `NetWkstaUserEnum` | Enumerates users logged on to a workstation | Requires administrative privileges |
| `NetSessionEnum` | Enumerates sessions established with a server | Some levels may work without admin, but modern defaults often restrict access |

PowerView’s `Get-NetSession` uses these APIs under the hood.

```text

```

## Example Successful Output

```powershell

```

# Why Get-NetSession May Fail

## Access Denied

Modern Windows versions often restrict remote session enumeration.

Example:

```text
VERBOSE: [Get-NetSession] Error: Access is denied
```

This can happen because of permissions on the registry/security configuration backing session enumeration.

---

## NetSessionEnum Query Levels

`NetSessionEnum` supports multiple information levels:

```text
0
1
2
10
502
```

General notes:

| Level | Notes |
|---|---|
| `0` | Returns only the client computer name |
| `1` | More detail, generally requires admin |
| `2` | More detail, generally requires admin |
| `10` | Returns computer and username information; commonly used by PowerView |
| `502` | More detailed session information |

PowerView commonly uses query level `10`.

---

## Modern Windows Restrictions

Permissions related to session enumeration are influenced by:

```text
HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\LanmanServer\DefaultSecurity
```

Relevant key area:

```text
SrvsvcSessionInfo
```

On newer Windows versions, regular authenticated users may not be able to remotely enumerate this information.

This affects:

```text
Windows 11
newer Windows Server versions
some Windows 10 builds
hardened environments
```

---

# Checking Local Registry Permissions

On the current Windows host, inspect permissions with:

```powershell
Get-Acl -Path HKLM:SYSTEM\CurrentControlSet\Services\LanmanServer\DefaultSecurity\ | fl
```

Example output:

```text
Owner  : NT AUTHORITY\SYSTEM
Group  : NT AUTHORITY\SYSTEM
Access : BUILTIN\Users Allow ReadKey
         BUILTIN\Administrators Allow FullControl
         NT AUTHORITY\SYSTEM Allow FullControl
         CREATOR OWNER Allow FullControl
         APPLICATION PACKAGE AUTHORITY\ALL APPLICATION PACKAGES Allow ReadKey
```

Important interpretation:

```text
Local ReadKey permissions do not necessarily mean remote NetSessionEnum access is allowed.
```

Built-in local groups and capability SIDs do not necessarily grant usable remote session enumeration rights to a low-privileged domain user.

---

# Practical Takeaway for Get-NetSession

Use `Get-NetSession`, but do not fully trust silence.

No output may mean:

```text
no active sessions
access denied
API restrictions
firewall restrictions
modern OS hardening
insufficient privileges
```

Always check with:

```powershell
Get-NetSession -ComputerName <hostname> -Verbose
```

If it fails, use another method.

## PsLoggedOn Dependency

Remote Registry notes:

```text

may stop after inactivity and restart on demand
```

If Remote Registry is disabled, PsLoggedOn may fail or return incomplete information.

# Chained Compromise

A chained compromise means improving access through multiple accounts or systems to reach a goal.

Example:

```text
Initial user:
stephanie

Finding:
stephanie has local admin on CLIENT74

Finding:
jeffadmin is logged on to CLIENT74

Possible result:
credential access to jeffadmin

Possible final impact:
higher privilege or domain admin path
```

Chained compromise can also target:

```text
service accounts
database admins
file server admins
backup operators
application admins
local admins on sensitive servers
```

# PowerView vs PsLoggedOn

| Tool | Strength | Limitation |
|---|---|---|
| `Get-NetSession` | Built into PowerView workflow; fast if permitted | Often blocked by modern permissions |
| `PsLoggedOn` | Can find local logons via Remote Registry | Requires Remote Registry service |
| `Find-LocalAdminAccess` | Finds hosts where current user is local admin | Does not show who is logged on |
| `Get-NetComputer` | Identifies OS/build and host roles | Does not show active sessions |

Use them together.

# PowerView vs Manual LDAP

## Manual LDAP Function

## Missing or Empty Results

Possible reasons:

```text
not domain joined
not authenticated as a domain user
network cannot reach domain controller
PowerView import failed
LDAP blocked
Kerberos double-hop issue from WinRM session
permissions restrictions
```

Prefer running enumeration from an RDP session when possible.

---

## Get-NetSession Returns No Output

Try verbose mode:

```powershell
Get-NetSession -ComputerName <hostname> -Verbose
```

If you see:

```text
Access is denied
```

then the issue is permissions or OS hardening, not necessarily absence of sessions.

Try PsLoggedOn:

```powershell
.\PsLoggedon.exe \\<hostname>
```

---

## PsLoggedOn Returns No Local Users

Possible meanings:

```text
no one is logged on locally
Remote Registry is disabled
access is blocked
tool cannot query the host
```

Interpret carefully and corroborate with other data.

# Key Takeaways

- PowerView automates many manual AD enumeration tasks.
- `Import-Module .\PowerView.ps1` loads PowerView into memory.
- `Get-NetDomain` shows domain and PDC information.
- `Get-NetUser` enumerates user objects and attributes.
- Use `select` to reduce noisy output.
- `Get-NetUser | select cn,pwdlastset,lastlogon` helps identify stale or interesting users.
- `Get-NetGroup | select cn` lists groups.
- `Get-NetGroup "<group>" | select member` reveals direct members and nested groups.
- `Get-NetComputer` enumerates computer objects.
- `Get-NetComputer | select dnshostname,operatingsystem,operatingsystemversion` maps hosts and OS builds.
- OS version matters because some enumeration APIs behave differently on modern Windows.
- `Find-LocalAdminAccess` identifies hosts where the current user has local admin rights.
- Local admin access becomes more valuable when privileged users are logged on to the same host.
- `Get-NetSession` can enumerate sessions, but often fails on modern systems due to permissions.
- Use `-Verbose` with `Get-NetSession` to distinguish no output from access denied.
- `PsLoggedOn` can identify locally logged-on users when Remote Registry is available.
- Session enumeration results should be interpreted carefully.
- Chained compromise often comes from combining multiple findings, not from one command.
- PowerView is faster than manual LDAP, but manual LDAP knowledge helps interpret and troubleshoot results.
- Re-run enumeration after every new credential, host, or privilege change.
