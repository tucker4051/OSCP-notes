## Overview

Further manual Active Directory enumeration focuses on the relationships and weak points that are easy to miss after basic user, group, and computer enumeration.

This note covers:

```text
Service Principal Names

object permissions

interesting AD rights
Get-ObjectAcl
SID conversion
GenericAll abuse example

Group Policy Preferences cpassword discovery

```

The goal is to move from “what objects exist?” to:

```text
which accounts run services
```

# Section 1 – Enumeration Through Service Principal Names

## What Is a Service Account?

Applications and services must run under a security context.

That context may be:

```text
LocalSystem
LocalService
NetworkService
domain user account
managed service account
group managed service account
```

Simple Windows services may use built-in service accounts.

More complex domain-integrated services often use domain accounts.

Examples:

```text
IIS application pools
MS SQL services
Exchange services
backup services
web applications
scheduled service jobs
```

These accounts are interesting because they may have more privileges than normal users.

---

## Why Service Accounts Matter

Service accounts often have access to:

```text
servers
databases
file shares
application secrets
network resources
backend systems
scheduled tasks
service control permissions
```

They may also have weaker operational security:

```text
passwords rarely changed
passwords set to never expire
passwords reused
older passwords
over-permissioned access
SPNs registered in AD
```

> [!important]
> Service accounts may not be Domain Admins, but they can still provide valuable access to servers, databases, shares, or applications.

---

## What Is an SPN?

A **Service Principal Name (SPN)** is a unique identifier that maps a service instance to a service account in Active Directory.

SPNs allow Kerberos authentication to a service.

Example SPNs:

```text
HTTP/web04.corp.com
HTTP/web04
HTTP/web04.corp.com:80
MSSQLSvc/sql01.corp.com:1433
CIFS/files04.corp.com
```

SPNs can reveal:

```text
service type
hostname
port
service account
server role
potential Kerberoast targets
```

---

## Common SPN Service Classes

| SPN Prefix | Likely Service |
|---|---|
| `HTTP/` | Web service, IIS, web app |
| `MSSQLSvc/` | Microsoft SQL Server |
| `CIFS/` | SMB/file sharing |
| `HOST/` | Host service |
| `LDAP/` | LDAP service |
| `TERMSRV/` | RDP/Terminal Services |
| `WSMAN/` | WinRM |
| `Exchange/` | Microsoft Exchange |
| `FIMService/` | Forefront Identity Manager |
| `AGPMServer/` | Advanced Group Policy Management |

# Enumerating SPNs with setspn.exe

`setspn.exe` is installed on Windows by default.

Use it to list SPNs for a specific account.

Syntax:

```cmd
setspn -L <account>
```

Example:

```cmd
setspn -L iis_service
```

Example output:

```text
Registered ServicePrincipalNames for CN=iis_service,CN=Users,DC=corp,DC=com:
        HTTP/web04.corp.com
        HTTP/web04
        HTTP/web04.corp.com:80
```

Interpretation:

```text
iis_service is linked to HTTP services.
The service appears to run on web04.corp.com.
The service likely listens on TCP/80.
```

---

## Resolve Hostname from SPN

Use `nslookup` to resolve the hostname found in the SPN.

```powershell
nslookup.exe web04.corp.com
```

Example output:

```text
Server:  UnKnown
Address:  192.168.50.70

Name:    web04.corp.com
Address:  192.168.50.72
```

Interpretation:

```text
web04.corp.com resolves to 192.168.50.72.
The SPN points to an internal web server.
```

---

# Section 2 – Enumerating Object Permissions

## What Are Object Permissions?

Active Directory objects have permissions.

These permissions define who can read, modify, control, or administer an object.

Objects include:

```text
users
groups
computers
OUs
GPOs
shares
service accounts
```

Permissions are stored in an ACL.

---

## ACL and ACE Basics

An **Access Control List (ACL)** is a list of access rules.

Each rule is an **Access Control Entry (ACE)**.

Each ACE defines:

```text
who the rule applies to
whether access is allowed or denied
what rights are granted or denied
whether the permission is inherited
```

Simplified flow:

```text
User requests access to object.
User presents access token.
Object checks token against ACL.
If ACE allows required access, access is granted.
If not, access is denied.
```

---

## Why Object Permissions Matter

Weak AD object permissions can create privilege escalation paths.

Examples:

```text
user can reset another user's password
user can modify group membership
user can add themselves to a privileged group
user can change ownership of an object
user can edit the ACL on an object
user can modify attributes on a service account
```

> [!important]
> Object permission abuse is one of the most important AD privilege escalation areas. Small misconfigurations can create major attack paths.

---

# Interesting AD Rights

From an attacker or penetration tester perspective, these rights are especially important:

| Permission | Meaning |
|---|---|
| `GenericAll` | Full control over the object |
| `GenericWrite` | Edit certain attributes on the object |
| `WriteOwner` | Change ownership of the object |
| `WriteDACL` | Modify ACEs applied to the object |
| `AllExtendedRights` | Extended rights such as password reset |
| `ForceChangePassword` | Reset or change password for the object |
| `Self` / `Self-Membership` | Add self to a group in some contexts |

---

## Practical Meaning of Dangerous Rights

| Right | Possible Impact |
|---|---|
| `GenericAll` on group | Add yourself or another user to the group |
| `GenericAll` on user | Reset password, modify attributes, take over account |
| `GenericWrite` on user | Modify attributes, potentially set SPN or other useful values |
| `WriteDACL` | Grant yourself stronger rights |
| `WriteOwner` | Take ownership, then modify DACL |
| `ForceChangePassword` | Reset another user's password |
| `AllExtendedRights` | May include password reset or replication-related extended rights |
| `Self-Membership` | Add self to a group where allowed |

---

# Enumerating ACLs with Get-ObjectAcl

PowerView function:

```powershell

```

Enumerate ACLs for a specific identity:

```powershell

```

Example:

```powershell

```

Interesting fields:

```text

```

---

## Understanding Get-ObjectAcl Output

Example fields:

```text
ObjectDN               : CN=stephanie,CN=Users,DC=corp,DC=com
ObjectSID              : S-1-5-21-1987370270-658905905-1781884369-1104
ActiveDirectoryRights  : ReadProperty
SecurityIdentifier     : S-1-5-21-1987370270-658905905-1781884369-553
```

Meaning:

| Field | Meaning |
|---|---|
| `ObjectDN` | Object being enumerated |
| `ObjectSID` | SID of the target object |
| `ActiveDirectoryRights` | Permission granted or denied |
| `SecurityIdentifier` | Principal receiving the permission |

In plain English:

```text
SecurityIdentifier has ActiveDirectoryRights over ObjectDN.
```

# Converting SIDs to Names

SIDs are hard to read.

PowerView can convert them:

```powershell
Convert-SidToName <SID>
```

Example:

```powershell
Convert-SidToName S-1-5-21-1987370270-658905905-1781884369-1104
```

Output:

```text
CORP\stephanie
```

Convert another SID:

```powershell
Convert-SidToName S-1-5-21-1987370270-658905905-1781884369-553
```

Output:

```text
CORP\RAS and IAS Servers
```

Interpretation:

```text
The RAS and IAS Servers group has the listed permission on stephanie.
```

# Filtering for Dangerous Rights

The full ACL output is noisy.

Filter for a specific right, such as `GenericAll`.

Example target:

```text

```

Command:

```powershell

? {$_.ActiveDirectoryRights -eq "GenericAll"} |
select SecurityIdentifier,ActiveDirectoryRights
```

# SYSVOL Enumeration

## What Is SYSVOL?

`SYSVOL` is a domain share hosted on domain controllers.

It stores domain-wide files such as:

```text
Group Policy data
logon scripts
policy templates
domain scripts
configuration artifacts
```

Default path on the domain controller:

```text
%SystemRoot%\SYSVOL\Sysvol\<domain-name>
```

UNC path example:

```text
\\dc1.corp.com\SYSVOL\corp.com\
```

Every domain user usually has read access.

---

# Group Policy Preferences cpassword

## Finding an Old Policy File

Example file:

```text
\\dc1.corp.com\sysvol\corp.com\Policies\oldpolicy\old-policy-backup.xml
```

Read it:

```powershell
cat \\dc1.corp.com\sysvol\corp.com\Policies\oldpolicy\old-policy-backup.xml
```

Example content:

```xml
<?xml version="1.0" encoding="utf-8"?>
<Groups clsid="{3125E937-EB16-4b4c-9934-544FC6D24D26}">
  <User clsid="{DF5F1855-51E5-4d24-8B1A-D9BDE98BA1D1}"
        name="Administrator (built-in)"
        image="2"
        changed="2012-05-03 11:45:20"
        uid="{253F4D90-150A-4EFB-BCC8-6E894A9105F7}">
    <Properties
        action="U"
        newName=""
        fullName="admin"
        description="Change local admin"
        cpassword="+bsY0V3d4/KgX3VJdO/vyepPfAN1zMFTiQDApgR92JE"
        changeLogon="0"
        noChange="0"
        neverExpires="0"
        acctDisabled="0"
        userName="Administrator (built-in)"
        expires="2016-02-10" />

</Groups>
```

Interesting field:

```text
cpassword
```

Value:

```text
+bsY0V3d4/KgX3VJdO/vyepPfAN1zMFTiQDApgR92JE
```

---

## Why cpassword Matters

Historically, administrators used Group Policy Preferences to set local passwords.

These passwords were stored encrypted in XML files as:

```text
cpassword
```

Although encrypted with AES-256, the key was publicly documented.

This means GPP `cpassword` values can be decrypted.

---

## Decrypt cpassword with gpp-decrypt

On Kali:

```bash
gpp-decrypt "+bsY0V3d4/KgX3VJdO/vyepPfAN1zMFTiQDApgR92JE"
```

Example output:

```text
P@$$w0rd
```

Document:

```text
Source file
Username referenced
Decrypted password
Possible scope
Whether it appears old or current
```

Example note:

```text
Found old GPP policy backup:
\\dc1.corp.com\sysvol\corp.com\Policies\oldpolicy\old-policy-backup.xml

User referenced:
Administrator (built-in)

cpassword decrypted to:
P@$$w0rd

Potential impact:
possible local Administrator password pattern or reused local admin password
```

> [!warning]
> Treat decrypted passwords carefully. Validate only within scope and record where they were found.

---

## Useful PowerShell Search Commands

List recursively:

```powershell
ls \\<host>\<share> -Recurse
```

Find text files:

```powershell
ls \\<host>\<share> -Recurse -Include *.txt,*.xml,*.config,*.ini
```

Search for password strings:

```powershell
Select-String -Path "\\<host>\<share>\*" -Pattern "password","passwd","pwd","credential","secret" -Recurse
```

Search XML files:

```powershell
ls \\<host>\<share> -Recurse -Include *.xml
```

Search config files:

```powershell
ls \\<host>\<share> -Recurse -Include *.config,*.ini
```

> [!note]
> Large recursive searches can be noisy and slow. Start targeted, especially on production assessments.


# Common Findings and Interpretation

| Finding | Interpretation |
|---|---|
| `iis_service` has `HTTP/web04` SPNs | Service account for web server |
| `web04.corp.com` resolves internally | Web server target identified through AD |
| `stephanie` has `GenericAll` over `Management Department` | Dangerous ACL misconfiguration |
| `Management Department` contains `jen` | Group membership may matter for access path |
| `SYSVOL` contains `oldpolicy` | Old GPO artifacts may contain secrets |
| XML contains `cpassword` | GPP password can likely be decrypted |
| `gpp-decrypt` returns `P@$$w0rd` | Possible local Administrator password |
| `docshare` contains `start-email.txt` | Documentation may expose credentials |
| Email contains `HenchmanPutridBonbon11` | Possible password candidate and pattern clue |

# Troubleshooting

## setspn Shows No Results

Possible reasons:

```text
account has no SPNs
wrong account name
querying wrong domain
insufficient domain connectivity
```

Try PowerView:

```powershell
Get-NetUser -SPN | select samaccountname,serviceprincipalname
```

---

## Get-NetUser -SPN Shows Only krbtgt

Possible reasons:

```text
few domain service accounts exist
SPNs are registered on computer accounts instead
service accounts are named unexpectedly
domain is small
```

Continue with computer and service enumeration.

---

## Get-ObjectAcl Output Is Overwhelming

Filter for specific rights:

```powershell
Get-ObjectAcl -Identity "<object>" |
? {$_.ActiveDirectoryRights -eq "GenericAll"} |
select SecurityIdentifier,ActiveDirectoryRights
```

Also check:

```text
GenericWrite
WriteDACL
WriteOwner
AllExtendedRights
ForceChangePassword
```

---

## SIDs Are Hard to Read

Convert them:

```powershell
Convert-SidToName <SID>
```

For multiple SIDs:

```powershell
"<SID1>","<SID2>" | Convert-SidToName
```

---

## Find-DomainShare Takes Time

This is normal.

It may query multiple systems and shares.

Use:

```powershell
Find-DomainShare -CheckShareAccess
```

to reduce output to shares your current user can access.

# Key Takeaways

- SPNs map services to service accounts in Active Directory.
- `setspn -L <account>` lists SPNs for a specific account.
- `Get-NetUser -SPN` lists domain users with SPNs.
- SPNs reveal services, hosts, ports, and possible Kerberoast targets.
- Service accounts often have more access than normal users.
- AD object permissions are controlled by ACLs made of ACEs.
- `Get-ObjectAcl` enumerates permissions on AD objects.
- `Convert-SidToName` makes ACL output readable.
- Dangerous rights include `GenericAll`, `GenericWrite`, `WriteDACL`, `WriteOwner`, `AllExtendedRights`, and `ForceChangePassword`.
- `GenericAll` over a group can allow adding users to that group.
- Always clean up intentional group membership changes.
- `Find-DomainShare` identifies shares across the domain.
- `SYSVOL` is readable by domain users and may contain old policy artifacts.
- GPP `cpassword` values can be decrypted with `gpp-decrypt`.
- Custom shares often contain documentation, scripts, configs, backups, and credentials.
- Stale passwords are still useful for pattern analysis.
- Further enumeration should connect findings into possible attack paths.
