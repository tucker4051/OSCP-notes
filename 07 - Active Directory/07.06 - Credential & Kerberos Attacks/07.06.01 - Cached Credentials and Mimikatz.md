## Overview

Windows domain authentication uses cached credential material and Kerberos tickets to support single sign-on.

In Active Directory environments, this material may be stored in memory inside:

```text
LSASS
Local Security Authority Subsystem Service
```

If an attacker or penetration tester gains local administrator or SYSTEM-level access to a Windows host, they may be able to extract:

```text
NTLM hashes
SHA1 hashes
DPAPI keys
Kerberos TGTs
Kerberos TGS tickets
cleartext passwords in older or weakened configurations
certificate private keys in some cases
```

The most well-known tool for this is:

```text
Mimikatz
```

> [!important]
> Cached credential extraction requires high privileges on the target host. In practice, this usually means local administrator or SYSTEM access.

---

# Why Cached Credentials Matter

Microsoft’s Kerberos implementation supports single sign-on.

This means a user can authenticate once and then access multiple domain resources without repeatedly typing their password.

To make this work, Windows stores authentication material that can be reused to request or present Kerberos tickets.

This material may include:

```text
password-derived hashes
Kerberos tickets
session keys
DPAPI material
credential manager data
```

If extracted, this material may allow:

```text
offline password cracking
pass-the-hash
pass-the-ticket
access to network resources
lateral movement
privilege escalation
domain compromise paths
```

# LSASS

## What Is LSASS?

`lsass.exe` is the Local Security Authority Subsystem Service.

It handles security operations such as:

```text
user logon
password verification
Kerberos authentication
NTLM authentication
access token creation
credential caching
security policy enforcement
```

LSASS runs as:

```text
SYSTEM
```

Because of this, reading LSASS memory generally requires:

```text
local administrator
SYSTEM
SeDebugPrivilege
```

---

## Why LSASS Is Protected

Credential material inside LSASS is sensitive.

Modern Windows systems may use protections such as:

```text
LSA Protection
RunAsPPL
Credential Guard
EDR memory access detection
AV signatures
protected process controls
```

These protections are designed to prevent tools from reading LSASS memory.

> [!note]
> In lab environments, these protections may be disabled or simplified. In real environments, LSASS access is heavily monitored and often blocked.

---

# Required Privileges

To extract cached credentials from LSASS, you usually need:

```text
local administrator rights
or
SYSTEM privileges
```

Why:

```text
LSASS is owned by SYSTEM.
Credential material is stored in LSASS memory.
Processes running as normal users cannot read LSASS memory.
```

Common path:

```text
low-privileged domain user
↓
local privilege escalation or local admin access
↓
LSASS access
↓
credential extraction
↓
lateral movement or privilege escalation
```

# Mimikatz

## What Is Mimikatz?

`Mimikatz` is a post-exploitation tool that can interact with Windows authentication material.

It can extract or interact with:

```text
NTLM hashes
Kerberos tickets
cleartext passwords in certain configurations
DPAPI material
LSA secrets
cached logons
certificates
private keys
```

Common modules:

```text
privilege
sekurlsa
kerberos
lsadump
crypto
```

---

## Operational Caveat

Mimikatz is heavily detected.

In real assessments, running the standalone binary may trigger:

```text
antivirus alerts
EDR detections
process command-line detections
memory scanning detections
known signature detections
LSASS access alerts
```

Alternative approaches often used in controlled assessments include:

```text
memory-only execution
LSASS dump collection for offline analysis
trusted tooling allowed by rules of engagement
EDR-safe testing procedures
```

> [!warning]
> Use credential extraction tooling only in authorized environments and within scope. LSASS access is sensitive and commonly logged.

# WDigest and Cleartext Passwords

On older systems, such as Windows 7, or systems where WDigest has been manually enabled, Mimikatz may reveal cleartext passwords.

Example:

```text
wdigest :
 * Username : user
 * Domain   : DOMAIN
 * Password : cleartext-password
```

Modern systems usually show:

```text
Password : (null)
```

> [!warning]
> If WDigest cleartext storage is enabled, this is a serious defensive weakness.

# Kerberos Tickets in LSASS

Kerberos uses tickets.

Important ticket types:

```text
TGT
Ticket Granting Ticket

TGS
Ticket Granting Service ticket
```

## TGT

A TGT is used to request service tickets.

Plain English:

```text
The TGT proves the user has authenticated to the domain.
It can be used to request access to services.
```

## TGS

A TGS is used to access a specific service.

Plain English:

```text
The TGS is scoped to one service, such as CIFS on a file server.
```

---

# Creating a Cached Service Ticket

Accessing a network service can create and cache a service ticket.

Example: list an SMB share on `WEB04`.

```powershell
dir \\web04.corp.com\backup
```

Example output:

```text
Directory: \\web04.corp.com\backup

Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
-a----         9/13/2022   2:52 AM              0 backup_schemata.txt
```

This creates a service ticket for:

```text
cifs/web04.corp.com
```


---

# TGT vs TGS

| Ticket | Purpose | Scope |
|---|---|---|
| TGT | Used to request service tickets | Broad, domain authentication |
| TGS | Used to access a specific service | Specific service and host |

Practical meaning:

```text
A stolen TGS may allow access only to the specific service it targets.
A stolen TGT may allow requesting new TGS tickets for other services.
```

# AD CS and Certificate Private Keys

Active Directory Certificate Services can provide PKI services in a domain.

AD CS may be used for:

```text
HTTPS certificates
user certificates
smart card authentication
machine certificates
service certificates
```

A certificate may be configured with a:

```text
non-exportable private key
```

This means the private key is not normally exportable through the GUI or standard tools.

However, Mimikatz includes crypto-related modules that may interact with CryptoAPI or CNG.

Relevant commands from the notes:

```text
crypto::capi
crypto::cng
```

Purpose:

```text
patch CryptoAPI or KeyIso behavior so private keys marked non-exportable may become exportable
```

> [!warning]
> Certificate private key extraction is sensitive and should only be performed with explicit authorization.

# Defensive Controls

## LSA Protection

A defensive technique is to enable additional LSA protection.

This helps prevent unauthorized memory reads from LSASS.

Commonly referenced protection:

```text
RunAsPPL
Protected Process Light
```

Effect:

```text
harder for tools to read LSASS memory
reduces basic Mimikatz credential dumping success
raises bar for attackers
```

---

## Credential Guard

Credential Guard helps protect credential material using virtualization-based security.

Effect:

```text
limits exposure of secrets in normal LSASS memory
reduces pass-the-hash and credential theft opportunities
```

---

## Reduce Credential Exposure

Defensive practices:

```text
avoid logging privileged users into workstations
use separate admin workstations
use tiered administration
limit local administrator rights
enable LSA Protection
enable Credential Guard where possible
disable WDigest cleartext credential storage
monitor LSASS access
restrict debug privileges
use EDR/AV controls
rotate credentials after suspected exposure
```

# Common Findings and Interpretation

| Finding | Meaning |
|---|---|
| `Privilege '20' OK` | SeDebugPrivilege enabled successfully |
| NTLM hash present | Potential pass-the-hash or cracking candidate |
| SHA1 hash present | Additional password-derived material |
| DPAPI value present | May support DPAPI secret decryption workflows |
| `Password : (null)` under WDigest | Cleartext password not stored for that package |
| Cleartext under WDigest | Weak/legacy configuration |
| TGS for `cifs/web04.corp.com` | User accessed SMB on WEB04 |
| TGT for `krbtgt/CORP.COM` | User has domain TGT cached |
| Multiple users in output | Multiple logon sessions exist on the host |
| Service session for another user | Possible service account credential exposure |

# Key Takeaways

- LSASS stores authentication material used by Windows and AD.
- Reading LSASS usually requires local administrator or SYSTEM privileges.
- Mimikatz can extract credential material from LSASS.
- `privilege::debug` enables SeDebugPrivilege.
- `sekurlsa::logonpasswords` dumps logon session credential material.
- Extracted material may include NTLM hashes, SHA1 hashes, DPAPI values, and sometimes cleartext passwords.
- WDigest cleartext passwords are usually disabled on modern Windows.
- Kerberos TGTs and TGS tickets are stored in LSASS.
- Accessing a network service can create a cached TGS.
- `sekurlsa::tickets` lists Kerberos tickets in memory.
- A TGS is service-specific.
- A TGT can be used to request additional service tickets.
- Mimikatz can also interact with certificate private key export protections through crypto modules.
- LSA Protection and Credential Guard reduce credential theft risk.
- Credential extraction is highly sensitive and heavily monitored.
- In AD attack paths, local admin access plus privileged user sessions can lead to major escalation opportunities.
