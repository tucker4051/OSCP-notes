## Overview

**Kerberoasting** is an Active Directory password attack that targets service accounts with registered **Service Principal Names**.

If a domain user account has an SPN, any authenticated domain user can request a Kerberos service ticket for that SPN. The ticket is encrypted using the service account’s password hash.

If the service account uses a weak password, the ticket can be cracked offline to recover the plaintext password.

```text
Authenticated domain user
↓
Request TGS for service account SPN
↓
Receive TGS-REP hash
↓
Crack offline with Hashcat
↓
Recover service account password
```

> [!important]
> Kerberoasting does not require access to the target service itself. The domain controller does not check whether the requesting user is allowed to access the service before issuing the service ticket.

---

# Kerberos Refresher

When a user wants to access a service registered with an SPN:

```text
1. User authenticates to the domain and receives a TGT.
2. User requests a service ticket for the SPN.
3. Domain controller returns a TGS.
4. User presents the TGS to the service.
5. Service decrypts and validates the ticket.
```

The service ticket is encrypted with the password hash of the account that owns the SPN.

This is what makes Kerberoasting possible.

---

# Why Kerberoasting Works

The domain controller allows authenticated users to request service tickets for SPNs.

The important detail:

```text
The DC issues the service ticket before the service itself checks authorization.
```

Meaning:

```text
you can request a TGS for an SPN
even if
you do not have permission to access that service
```

The returned ticket contains encrypted material that can be attacked offline.

---

# Kerberoasting Target Requirements

A good Kerberoasting target is usually:

```text
a domain user account
with one or more SPNs
using a weak or guessable password
```

Examples:

```text
iis_service
sql_svc
svc_web
svc_backup
svc_mssql
exchange_svc
```

Less useful targets:

```text
computer accounts
managed service accounts
group managed service accounts
krbtgt
```

Why:

```text
computer accounts and managed service accounts usually have long random passwords
gMSA passwords are complex and automatically managed
krbtgt has a strong machine-generated secret in normal environments
```

> [!tip]
> Kerberoasting is usually most effective against user-based service accounts with manually set passwords.

---

# Service Principal Names

An SPN maps a service instance to an account.

Example:

```text
HTTP/web04.corp.com:80
```

This suggests:

```text
service type: HTTP
host: web04.corp.com
port: 80
```

Common SPN prefixes:

```text
HTTP
MSSQLSvc
CIFS
HOST
LDAP
TERMSRV
WSMAN
Exchange
```

---

# Attack Flow

```text
1. Authenticate as any domain user.
2. Enumerate user accounts with SPNs.
3. Request TGS tickets for SPN accounts.
4. Save TGS-REP hashes.
5. Crack hashes offline.
6. Validate recovered credentials.
7. Re-enumerate as the service account.
```

# Transfer Hash to Kali

Copy the output file to Kali:

```text
hashes.kerberoast
```

View the hash:

```bash
cat hashes.kerberoast
```

Example hash prefix:

```text
$krb5tgs$23$*iis_service$corp.com$HTTP/web04.corp.com:80@corp.com*
```

Meaning:

```text
Kerberos TGS hash
etype 23
target user: iis_service
domain: corp.com
SPN: HTTP/web04.corp.com:80
```

---

# Hashcat Mode

For Kerberos TGS-REP etype 23 hashes, use Hashcat mode:

```text
13100
```

Check with:

```bash
hashcat --help | grep -i "Kerberos"
```

Relevant output:

```text
13100 | Kerberos 5, etype 23, TGS-REP | Network Protocol
```

Other Kerberos TGS modes may appear:

```text
19600 | Kerberos 5, etype 17, TGS-REP
19700 | Kerberos 5, etype 18, TGS-REP
```

But for the shown `$krb5tgs$23$` hash:

```text
use 13100
```

---

# Crack TGS-REP Hash with Hashcat

```bash
sudo hashcat -m 13100 hashes.kerberoast /usr/share/wordlists/rockyou.txt -r /usr/share/hashcat/rules/best64.rule --force
```

Options:

| Option | Meaning |
|---|---|
| `-m 13100` | Kerberos 5 TGS-REP etype 23 mode |
| `hashes.kerberoast` | Hash file |
| `rockyou.txt` | Wordlist |
| `-r best64.rule` | Hashcat rule |
| `--force` | Force execution in lab VM |

Example cracked result:

```text
iis_service:Strawberry1
```

Finding:

```text
User: iis_service
Password: Strawberry1
SPN: HTTP/web04.corp.com:80
```

# Kerberos Clock Skew Error

If Impacket returns:

```text
KRB_AP_ERR_SKEW (Clock skew too great)
```

Meaning:

```text
Kali and the domain controller have too much time drift.
Kerberos is time-sensitive.
```

Options:

```text
sync time carefully
query DC time
use faketime for the affected Impacket process
```

> [!tip]
> Avoid changing your whole system clock unless necessary. Using `faketime` can be cleaner for one-off Kerberos tooling.

# Targeted Kerberoasting

## Concept

If no useful SPNs exist, but you have dangerous permissions over a user, you may be able to create a roastable condition.

Relevant permissions:

```text
GenericWrite
GenericAll
```

Targeted Kerberoasting flow:

```text
1. Identify user where you have GenericWrite or GenericAll.
2. Add an SPN to that user.
3. Request a TGS for the new SPN.
4. Crack the TGS hash offline.
5. Remove the SPN.
```

Why this works:

```text
Kerberoasting depends on the target user having an SPN.
If you can write an SPN to the user object, you can create a kerberoastable target.
```

> [!warning]
> Always remove the SPN after obtaining the hash. Leaving a new SPN can introduce risk and create obvious forensic evidence.

---

## Targeted Kerberoasting vs Password Reset

If you control a user object, one option is:

```text
reset the user's password
```

But this may:

```text
lock out the real user
cause operational disruption
trigger alerts
make the compromise obvious
```

Targeted Kerberoasting may avoid immediate password reset, but it still modifies AD and is detectable.

# Common Findings and Interpretation

| Finding | Meaning |
|---|---|
| User has SPN | User may be Kerberoastable |
| `HTTP/web04.corp.com:80` | Web service on WEB04 |
| `Supported ETypes: RC4_HMAC_DEFAULT` | Hashcat mode likely `13100` for `$krb5tgs$23$` |
| `$krb5tgs$23$` | Kerberos TGS etype 23 hash |
| `PwdLastSet` is old | Password may be weaker or stale |
| Cracked password | Service account password recovered |
| Target is gMSA/computer account | Cracking is likely infeasible |
| GenericWrite/GenericAll over user | Targeted Kerberoasting may be possible |

# Troubleshooting

## Rubeus Finds No Kerberoastable Users

Possible reasons:

```text
no user accounts have SPNs
only computer/gMSA accounts have SPNs
account is disabled
running outside domain context
LDAP/DC connectivity issue
```

Check SPNs manually:

```powershell
Get-NetUser -SPN | select samaccountname,serviceprincipalname
```

or:

```cmd
setspn -L <account>
```

---

## Hashcat Mode Fails

Check the hash prefix.

For:

```text
$krb5tgs$23$
```

Use:

```text
-m 13100
```

For AES hashes:

```text
etype 17 or etype 18 may require different Hashcat modes
```

Check:

```bash
hashcat --help | grep -i "Kerberos"
```

---

## Hashcat Does Not Crack the Hash

Possible reasons:

```text
password not in wordlist
rules not sufficient
password is long and complex
target is managed service account
target is computer account
hash copied incorrectly
wrong hashcat mode
```

Try:

```text
better wordlists
custom rules
organization-specific password patterns
passwords found during share hunting
```

---

## Impacket Clock Skew Error

Error:

```text
KRB_AP_ERR_SKEW (Clock skew too great)
```

Cause:

```text
Kali and DC time differ too much.
```

Fix:

```text
sync time with DC
or
use faketime for the Impacket command
```

# Detection and Defensive Notes

## Detection Opportunities

Kerberoasting may generate:

```text
TGS requests for SPN accounts
unusual volume of TGS-REQ traffic
requests for RC4 tickets
requests from unexpected workstations
Rubeus execution
impacket-GetUserSPNs activity
LDAP queries for servicePrincipalName
offline cracking activity on attacker host
```

Potential indicators:

```text
many TGS requests in a short period
TGS requests for many SPNs from one user
TGS requests using weak encryption types
service ticket requests from low-privileged users
new SPN added to a user account
SPN removed shortly after TGS request
```

---

## Defensive Controls

Reduce Kerberoasting risk by:

```text
using gMSA or managed service accounts
using long random service account passwords
rotating service account passwords
removing unnecessary SPNs
auditing user accounts with SPNs
disabling RC4 where possible
monitoring TGS requests
monitoring SPN changes
reviewing GenericWrite and GenericAll permissions over users
limiting service account privileges
avoiding privileged service accounts where possible
```

# Kerberoasting vs AS-REP Roasting

| Attack | Target Condition | Hash Type | Hashcat Mode |
|---|---|---|---|
| AS-REP Roasting | User does not require Kerberos pre-auth | `$krb5asrep$` | `18200` |
| Kerberoasting | User has SPN | `$krb5tgs$23$` | `13100` |

Plain English:

```text
AS-REP Roasting abuses missing pre-authentication.
Kerberoasting abuses service tickets for SPN accounts.
```

---

# Key Takeaways

- Kerberoasting targets domain user accounts with SPNs.
- Any authenticated domain user can request a TGS for a known SPN.
- The TGS is encrypted with the service account’s password hash.
- If the service account password is weak, the TGS hash can be cracked offline.
- Rubeus can perform Kerberoasting from Windows.
- `impacket-GetUserSPNs` can perform Kerberoasting from Kali.
- Hashcat mode for `$krb5tgs$23$` TGS-REP hashes is `13100`.
- User-based service accounts are better targets than computer accounts or gMSAs.
- Cracking gMSA, managed service account, computer account, or krbtgt passwords is usually infeasible.
- Targeted Kerberoasting may be possible with `GenericWrite` or `GenericAll` over a user.
- Remove any SPN added during targeted Kerberoasting.
- Re-enumerate as the cracked service account after validation.
