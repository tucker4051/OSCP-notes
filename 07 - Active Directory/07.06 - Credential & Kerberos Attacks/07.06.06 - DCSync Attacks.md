## Overview

**DCSync** is an Active Directory credential extraction technique that abuses domain replication.

Instead of dumping credentials directly from a domain controller’s disk, DCSync impersonates a domain controller and requests replicated credential data from another domain controller.

If successful, DCSync can retrieve credential material for domain users, including:

```text
NTLM hashes
LM hashes
Kerberos AES keys
Kerberos DES keys
password metadata
account metadata
```

> [!important]
> DCSync is extremely powerful because it can retrieve credentials for any domain user, including `Administrator`, without interactive logon to the domain controller.

---

# Why DCSync Works

Production domains often use multiple domain controllers for redundancy.

Domain controllers synchronize directory data using:

```text
Directory Replication Service
DRS Remote Protocol
DRSUAPI
```

A domain controller can request updates for objects such as:

```text
users
groups
computers
password hashes
directory attributes
```

The key issue:

```text
The receiving domain controller does not only check whether the request came from a known domain controller.
It checks whether the requesting SID has the required replication rights.
```

If a user has the required replication rights, that user can request replicated credential data.

---

# Required Rights

To perform DCSync, an account needs replication-related rights.

Required rights:

```text
Replicating Directory Changes
Replicating Directory Changes All
Replicating Directory Changes in Filtered Set
```

By default, these rights are assigned to:

```text
Domain Admins
Enterprise Admins
Administrators
```

Possible DCSync-capable accounts:

```text
Domain Admin user
Enterprise Admin user
built-in Administrator
user delegated replication rights
compromised account with required ACEs
```

> [!warning]
> DCSync usually means the attacker already has very high domain privileges or has found a dangerous delegation/misconfiguration.

---

# DCSync Attack Flow

```text
1. Obtain credentials for a DCSync-capable account.
2. Authenticate to the domain.
3. Request replication data from a domain controller.
4. Extract target user credential material.
5. Crack or reuse the hashes/keys.
```

Diagram:

```text
Attacker-controlled account
  |
  | DRSUAPI / replication request
  v
Domain Controller
  |
  | replicated credential data
  v
Attacker
```

# DCSync the Domain Administrator

DCSync can target any domain account if the requesting user has the required rights.

Example:

```text
lsadump::dcsync /user:corp\Administrator
```

Example relevant output:

```text
Credentials:
  Hash NTLM: 2892d26cdf84d7a70e2eb3b9f05c425e
```

Interpretation:

```text
The NTLM hash for the domain Administrator account was retrieved.
```

> [!important]
> DCSync against high-value users such as `Administrator`, `krbtgt`, or Domain Admins represents severe domain compromise.

# When to Use DCSync

Use DCSync when you have:

```text
Domain Admin credentials
Enterprise Admin credentials
Administrator credentials
delegated replication rights
control over an account with replication rights
```

Useful goals:

```text
retrieve a specific user's NTLM hash
retrieve Domain Admin hashes
retrieve krbtgt hash for Golden Ticket workflows
retrieve service account hashes
collect Kerberos keys
support pass-the-hash or ticket-based attacks
```

# High-Value DCSync Targets

Prioritize:

```text
Administrator
krbtgt
Domain Admin users
Enterprise Admin users
service accounts
backup/admin accounts
users with high-value access
```

## krbtgt

The `krbtgt` account is especially sensitive.

Why:

```text
The krbtgt hash can be used to forge Golden Tickets.
Golden Tickets can provide long-term domain Kerberos access.
```

> [!warning]
> DCSync of `krbtgt` is a major domain compromise event.

---

# DCSync vs LSASS Dumping

| Technique | Target | Requires |
|---|---|---|
| LSASS dumping | Local machine sessions | Local admin/SYSTEM on host |
| DCSync | Domain controller replication data | Replication rights in AD |

Plain English:

```text
LSASS dumping extracts credentials cached on one host.
DCSync asks a domain controller for replicated domain credentials.
```

DCSync can retrieve accounts that are not logged on to your current machine.

---

# DCSync vs NTDS.dit Dumping

| Technique | Access Pattern | Notes |
|---|---|---|
| NTDS.dit dump | Extract database from DC | Often requires DC file access or backup abuse |
| DCSync | Replication request over DRSUAPI | Does not require copying NTDS.dit directly |

DCSync is powerful because:

```text
it uses legitimate domain replication functionality
it can be run remotely
it can target specific users
```

# Troubleshooting

## Mimikatz DCSync Fails

Possible causes:

```text
user lacks replication rights
wrong domain
wrong username format
network cannot reach DC
tool blocked by AV/EDR
Kerberos/authentication issue
```

Check:

```text
current user is Domain Admin or equivalent
domain name is correct
DC is reachable
time is synchronized
```

---

## secretsdump Fails with Access Denied

Likely cause:

```text
provided account does not have required replication rights
```

Confirm account membership:

```cmd
net user <user> /domain
```

or:

```powershell
Get-NetUser <user> | select memberof
```

---

## Shell Breaks Password with `!`

If the password contains `!`, Bash history expansion may interfere.

Use single quotes:

```bash
'PasswordWith!'
```

or escape the exclamation mark:

```bash
PasswordWith\!
```

---

## Kerberos / Time Errors

Kerberos is time-sensitive.

If tools return clock-related errors:

```text
sync time with the domain controller
or
use faketime for the command
```

# Detection and Defensive Notes

## Detection Opportunities

DCSync can generate:

```text
Directory Replication Service requests
DRSUAPI activity from non-domain-controller systems
replication requests for sensitive accounts
replication activity using user accounts
unusual access to NTDS secrets
Mimikatz lsadump::dcsync usage
secretsdump DRSUAPI method usage
```

Potential indicators:

```text
replication request from workstation
replication request from non-DC IP
account requesting replication that is not a DC
access to krbtgt credentials
Event ID 4662 with replication rights GUIDs
unexpected DRSGetNCChanges calls
```

---

## Defensive Controls

Reduce DCSync risk by:

```text
limiting Domain Admin membership
monitoring replication rights
auditing users with Replicating Directory Changes rights
removing unnecessary delegated replication permissions
monitoring DRSUAPI from non-DC systems
tiering administration
protecting privileged credentials
using separate admin workstations
monitoring for Mimikatz and secretsdump behavior
regularly reviewing ACLs on the domain root
```

High-value rights to audit:

```text
Replicating Directory Changes
Replicating Directory Changes All
Replicating Directory Changes in Filtered Set
```

# Key Takeaways

- DCSync abuses Active Directory replication.
- It impersonates a domain controller from a permissions perspective.
- It does not require interactive logon to a domain controller.
- It requires replication rights.
- Domain Admins, Enterprise Admins, and Administrators have these rights by default.
- Mimikatz can perform DCSync from Windows with `lsadump::dcsync`.
- Impacket `secretsdump` can perform DCSync from Linux.
- DCSync can retrieve NTLM hashes and Kerberos keys.
- DCSync can target any domain user if the operator has required rights.
- DCSync of `Administrator` or `krbtgt` indicates severe domain compromise.
- Hashcat mode `1000` cracks NTLM hashes.
- Monitor replication requests from non-domain-controller systems.
- Audit replication rights regularly.
