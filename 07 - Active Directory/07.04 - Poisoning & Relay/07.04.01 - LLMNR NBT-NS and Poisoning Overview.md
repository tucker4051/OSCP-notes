## Purpose

LLMNR and NBT-NS poisoning abuses local name resolution protocols to capture authentication attempts from Windows systems.

Use this section when working in an internal Windows or Active Directory environment where hosts may attempt to resolve unknown names on the local network.

This technique can help capture Net-NTLMv2 challenge-response hashes for offline cracking or further credential-based attacks.

---

## Core Idea

When Windows cannot resolve a hostname through DNS, it may fall back to local name resolution protocols such as LLMNR or NBT-NS.

An attacker on the same network segment can respond to those name resolution requests and impersonate the requested host.

If a victim then attempts to authenticate to the attacker-controlled listener, authentication material may be captured.

The simplified flow is:

1. Victim tries to access an unresolved hostname.
2. DNS resolution fails.
3. Victim broadcasts or multicasts a name resolution request.
4. Attacker responds and claims to be that host.
5. Victim attempts authentication.
6. Attacker captures Net-NTLMv2 material.
7. Captured hash may be cracked or used in related attacks where applicable.

---

## Protocols

|Protocol|Purpose|
|---|---|
|LLMNR|Link-Local Multicast Name Resolution|
|NBT-NS|NetBIOS Name Service|
|mDNS|Multicast DNS, sometimes relevant in mixed environments|

These protocols are intended to help systems resolve names locally when DNS does not provide an answer.

In many enterprise environments, this fallback behaviour creates an opportunity for poisoning attacks.

## When This Technique Is Useful

LLMNR/NBT-NS poisoning is useful when:

- [ ]  you are on the same network segment as Windows clients
- [ ]  clients use fallback name resolution
- [ ]  users or services may authenticate to unknown hostnames
- [ ]  Net-NTLMv2 capture is useful
- [ ]  password cracking is in scope
- [ ]  relay paths are being assessed
- [ ]  controls such as SMB signing are being reviewed

---

## Typical Attack Flow

1. Gain access to an internal network or host.
2. Confirm network position.
3. Start poisoning/listening tool.
4. Wait for name resolution requests.
5. Capture authentication attempts.
6. Save captured Net-NTLMv2 hashes.
7. Crack hashes offline where appropriate.
8. Test recovered credentials carefully.
9. Re-enumerate using valid credentials.
10. Document source, timing, and captured material.

---

## Captured Material

The common output is a Net-NTLMv2 challenge-response hash.

This is not the same as a plaintext password.

Possible next steps:

- offline cracking
- password reuse testing if cracked
- identifying weak passwords
- identifying users and hostnames
- supporting relay testing where applicable

## Windows-Based Poisoning

Windows-based poisoning can be useful when operating from a compromised Windows host or when tooling needs to run inside the target environment.

Use this when:

- operating from a Windows pivot host
- internal network traffic is only visible from that host

## Important Conditions

This technique depends on network and configuration conditions.

Check:

- [ ]  Are LLMNR/NBT-NS enabled?
- [ ]  Are you on the right network segment?
- [ ]  Can you see broadcast/multicast traffic?
- [ ]  Are clients attempting unresolved name lookups?
- [ ]  Are users or services likely to authenticate?
- [ ]  Is SMB signing enabled?
- [ ]  Are relay protections in place?
- [ ]  Is endpoint protection likely to flag tooling?
- [ ]  Is cracking feasible?

---

## Common Defensive Controls

Controls that reduce or block this technique include:

- disabling LLMNR
- disabling NBT-NS
- enforcing SMB signing
- enforcing LDAP signing
- enabling EPA/channel binding where relevant
- restricting NTLM usage
- using strong passwords
- monitoring for poisoning tools
- endpoint protection
- network segmentation

Related note:

- [[07.05.01 - AD Security Controls Overview]]

---

## Common Findings

Useful information gathered from poisoning may include:

- usernames
- hostnames
- domain names
- Net-NTLMv2 hashes
- service account activity
- naming mistakes
- internal naming patterns
- credential reuse opportunities
- weak password evidence if cracking succeeds

## Common Mistakes

- [ ]  Confusing Net-NTLMv2 with NTLM hashes
- [ ]  Assuming captured hashes are immediately usable as plaintext passwords
- [ ]  Forgetting to crack captured hashes offline
- [ ]  Running poisoning from the wrong network segment
- [ ]  Ignoring SMB signing and relay protections
- [ ]  Not documenting captured usernames and hostnames
- [ ]  Not testing cracked credentials carefully
- [ ]  Ignoring account lockout policy when testing recovered passwords
- [ ]  Leaving tools running without understanding network impact
