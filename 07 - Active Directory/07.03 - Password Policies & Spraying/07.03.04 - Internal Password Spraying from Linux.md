## Overview

Once we have built a clean target user list, the next step is to perform a **password spraying attack.**

Password spraying is one of the most effective ways to gain initial domain credentials during an internal assessment, but it must be done carefully. This is not a brute-force attack against one user. Instead, it is a controlled attempt of **one password against many users**, designed to reduce the risk of account lockouts.

Think of it like trying **one master key pattern** across many doors rather than smashing one lock repeatedly.

---

# Core Principle

A safe password spray depends on two things:

- a **valid target user list**
- an understanding of the **password policy**

Before spraying, you should already know or have estimated:

- lockout threshold
- lockout reset timer
- minimum password length
- complexity requirements

If you do not know the policy, be conservative.

---

# Password Spraying from a Linux Host

There are several ways to perform an internal password spraying attack from Linux.

Common options include:

- `rpcclient`
- `kerbrute`
- `crackmapexec`

Each has strengths and tradeoffs.

---

# Using rpcclient

`rpcclient` is a solid choice for spraying from Linux.

One thing to understand is that `rpcclient` does **not** return a giant obvious “SUCCESS” banner. Instead, a valid login is indicated by output containing:

- `Authority Name`

So the trick is to filter the output to show only successful attempts.

---

## Bash One-Liner with rpcclient

```bash
for u in $(cat valid_users.txt); do rpcclient -U "$u%Welcome1" -c "getusername;quit" 172.16.5.5 | grep Authority; done
```

## Example output

```text
Account Name: tjohnson, Authority Name: INLANEFREIGHT
Account Name: sgage, Authority Name: INLANEFREIGHT
```

## What this does

- reads each username from `valid_users.txt`
- tries the password `Welcome1`
- runs `getusername;quit` after login
- filters for `Authority`, which indicates a successful authentication

## Why this works

This is a lightweight way to test one password across a user list.

The `grep Authority` piece is important because it removes failed logons from the output and leaves only hits.

---

## Breakdown of the one-liner

```bash
for u in $(cat valid_users.txt)
```

Loops through each username in the file.

```bash
rpcclient -U "$u%Welcome1" -c "getusername;quit" 172.16.5.5
```

Attempts authentication using:

- username = current value of `$u`
- password = `Welcome1`

Then runs a simple command and exits.

```bash
| grep Authority
```

Keeps only successful output.

---

# Using Kerbrute

Kerbrute is another excellent option for password spraying.

It is fast, easy to use, and purpose-built for Kerberos-based enumeration and spraying.

---

## Kerbrute password spraying

```bash
kerbrute passwordspray -d inlanefreight.local --dc 172.16.5.5 valid_users.txt Welcome1
```

## Example output

```text
[+] VALID LOGIN: sgage@inlanefreight.local:Welcome1
[+] Done! Tested 57 logins (1 successes) in 0.172 seconds
```

## Why Kerbrute is useful

Kerbrute is efficient and gives very clear success output.

It is often faster and more convenient than building custom one-liners.

---

## Important caution with Kerbrute

Earlier username enumeration with Kerbrute is relatively safe because it checks whether users exist using Kerberos behavior.

But **password spraying with Kerbrute is different**.

Failed pre-authentication attempts:

- **do count** toward bad password attempts
- **can lead to account lockouts**

So even though Kerbrute feels efficient, it is still dangerous if used recklessly.

---

# Using CrackMapExec

CrackMapExec is one of the most versatile tools for spraying from Linux.

It accepts:

- a user file
- one password
- a target host

and attempts the password against all listed users.

---

## Spray with CrackMapExec

```bash
sudo crackmapexec smb 172.16.5.5 -u valid_users.txt -p Password123 | grep +
```

## Example output

```text
SMB  172.16.5.5  445  ACADEMY-EA-DC01  [+] INLANEFREIGHT.LOCAL\avazquez:Password123
```

## Why the `grep +` helps

CrackMapExec prints a lot of lines, especially with large user lists.

Filtering with:

```bash
grep +
```

helps you quickly isolate successful logins and avoid missing them in noisy output.

---

# Validating Discovered Credentials

Once you get a hit, validate it immediately before moving on.

This confirms:

- the password really works
- the account is still valid
- the login was not a false positive

---

## Validate with CrackMapExec

```bash
sudo crackmapexec smb 172.16.5.5 -u avazquez -p Password123
```

## Example output

```text
SMB  172.16.5.5  445  ACADEMY-EA-DC01  [+] INLANEFREIGHT.LOCAL\avazquez:Password123
```

## Why validation matters

Treat successful spray results like finding a key in the dark.

Before trying to open more doors with it, make sure the key is real.

# Local Administrator Password Reuse

Internal spraying is not limited to domain user accounts.

If you obtain:

- a local administrator password, or
- an NTLM hash from the local SAM

you can test whether that same credential is reused across many hosts.

This is very common in environments built from:

- gold images
- cloned workstation/server templates
- weak admin practices
- shared local admin standards

---

# Why Local Admin Reuse Matters

A reused local admin credential can turn one host compromise into many.

This is especially dangerous on:

- SQL servers
- Exchange servers
- web servers
- file servers
- jump boxes
- admin workstations

These systems are more likely to contain:

- high-value data
- privileged user sessions
- cached credentials
- lateral movement opportunities

---

# Credential Reuse Patterns to Watch For

Sometimes reuse is not exact, but follows a pattern.

Examples:

- Desktop local admin password: `$desktop%@admin123`
- Server local admin password: `$server%@admin123`

Or:

- Local account: `bsmith`
- Domain account: `bsmith`
- Admin account: `bsmith_adm`

If you recover one password, try to think in **families of reuse**, not just exact reuse.

This applies to:

- local accounts
- domain accounts
- trust relationships
- admin suffix accounts

---

# Local Admin Hash Spraying with CrackMapExec

If you only obtain the NTLM hash, you can still spray it.

This is effectively a **Pass-the-Hash spray**.

---

## Example command

```bash
sudo crackmapexec smb --local-auth 172.16.5.0/23 -u administrator -H 88ad09182de639ccc6579eb0849751cf | grep +
```

## Example output

```text
SMB  172.16.5.50   445  ACADEMY-EA-MX01  [+] ACADEMY-EA-MX01\administrator 88ad09182de639ccc6579eb0849751cf (Pwn3d!)
SMB  172.16.5.25   445  ACADEMY-EA-MS01  [+] ACADEMY-EA-MS01\administrator 88ad09182de639ccc6579eb0849751cf (Pwn3d!)
SMB  172.16.5.125  445  ACADEMY-EA-WEB0  [+] ACADEMY-EA-WEB0\administrator 88ad09182de639ccc6579eb0849751cf (Pwn3d!)
```

## What this tells us

The same local administrator hash worked on three systems.

That means:

- password reuse exists
- we likely have local admin on multiple hosts
- lateral movement is immediately possible

---

# Why `--local-auth` Is Critical

When spraying local admin credentials, always use:

```bash
--local-auth
```

## Why this matters

Without `--local-auth`, CrackMapExec may try authenticating against the **domain context** instead of the **local account database**.

That can be dangerous because it could:

- generate unnecessary domain failures
- hit lockout thresholds
- lock out built-in administrator accounts
- create a lot of noise

With `--local-auth`, the tool performs one attempt per host using the host’s local SAM.

This makes the spray safer and more targeted.

---

# What to Do After Local Admin Reuse Is Confirmed

Once you confirm reuse, the next step is to enumerate each compromised host for:

- logged-on users
- credential material
- shares
- admin tools
- RDP access
- service accounts
- SQL or Exchange connections
- cached tickets or hashes

A reused local admin password is rarely the end goal. It is usually a **bridge** to more privileged access.

---

# Noise and Stealth Considerations

This technique is effective, but it is also noisy.

It generates:

- authentication attempts across many systems
- SMB log entries
- possible EDR alerts
- lateral movement indicators

Because of that, this is **not ideal for stealth-heavy engagements**.

Use it when:

- speed matters more than stealth
- the rules of engagement allow it
- you have a clear reason for broad testing

Avoid it when:

- the assessment requires low noise
- the client is actively monitoring
- you already have quieter paths forward

---

# Defensive Recommendation: LAPS

A major recommendation when local admin reuse is found is:

- **Microsoft LAPS** (Local Administrator Password Solution)

## Why LAPS helps

LAPS:

- assigns a unique local admin password to each host
- stores it in Active Directory
- rotates it on a set interval

This breaks the chain of lateral movement caused by local admin password reuse.

Without LAPS, one compromised host often becomes many.

With LAPS, one compromised host tends to stay one compromised host.

# Practical Spray Workflow

A sensible Linux-based spray workflow often looks like this:

1. build a clean user list
2. confirm the password policy if possible
3. choose one carefully selected password
4. spray one attempt across the list
5. log every attempt
6. validate any hits immediately
7. wait according to the lockout reset timer
8. repeat only if justified

This keeps the attack measured rather than reckless.

---

# Key Takeaways

- Password spraying is one of the best foothold techniques in AD, but it must be controlled
- `rpcclient`, `kerbrute`, and `crackmapexec` are all viable Linux options
- Always validate discovered credentials immediately
- Local admin password reuse can be just as powerful as domain credential discovery
- `--local-auth` is essential when testing local accounts
- Broad local admin spraying is effective but noisy
- Findings like local admin reuse should lead to a LAPS recommendation
