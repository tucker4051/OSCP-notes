## Overview

During Linux privilege escalation, **credential hunting** is one of the highest-value activities. Credentials found on the local system can often be used to:

- switch to another local user
- escalate to root
- access databases
- connect to internal services
- move laterally to other hosts

Unlike exploit-based escalation, credential hunting is often quiet and reliable. It is also very common, because administrators, developers, and services frequently leave credentials behind in files, scripts, backups, shell history, or application configuration.

---

# Why Credential Hunting Matters

When enumerating a system, always note down any credentials or credential-like material you find. These may include:

- usernames and passwords
- database connection strings
- SSH private keys
- API keys
- tokens
- service account credentials
- password hashes
- backup copies of config files
- plaintext secrets in notes or text files

These may not always give immediate root access, but they can often unlock:

- another local account
- `sudo` access
- database access
- SSH access
- internal application access
- pivot opportunities

Credential hunting is like searching for spare keys in a building: you may not open the main vault directly, but you often find a door that leads closer to it.

---

# Common Places to Hunt for Credentials

## 1. Web Roots and Application Directories

The `/var` directory often contains web application data, including the web root for Apache, Nginx, or other services. Web applications frequently store database credentials in configuration files.

A classic example is WordPress:

```bash
grep 'DB_USER\|DB_PASSWORD' wp-config.php
```

Example output:

```php
define( 'DB_USER', 'wordpressuser' );
define( 'DB_PASSWORD', 'WPadmin123!' );
```

These credentials may be useful for:

- logging into MySQL or MariaDB
- re-using the password against local users
- accessing other services that share credentials

Other common application files to inspect:

- `wp-config.php`
- `.env`
- `config.php`
- `settings.php`
- `database.yml`
- `application.properties`
- `web.config`
- `db.php`

Useful paths include:

- `/var/www/`
- `/srv/www/`
- `/opt/`
- application install directories under user home folders

---

## 2. Configuration Files

Configuration files often contain service credentials, hardcoded usernames, internal hostnames, and authentication tokens.

Search broadly for config-like files:

```bash
find / ! -path "*/proc/*" -iname "*config*" -type f 2>/dev/null
```

Example output:

```text
/etc/ssh/ssh_config
/etc/ssh/sshd_config
/etc/python3/debian_config
/etc/kbd/config
...
```

This broad search returns a lot of noise, so once you identify interesting application directories, narrow it down with searches such as:

```bash
find /var /etc /opt /home -type f \( -name "*.conf" -o -name "*.config" -o -name "*.xml" -o -name "*.ini" -o -name "*.yaml" -o -name "*.yml" -o -name "*.json" \) 2>/dev/null
```

Then grep for likely secrets:

```bash
grep -RniE "pass|password|passwd|user|username|token|secret|key|db_" /var /etc /opt /home 2>/dev/null
```

Be especially alert for:

- database usernames and passwords
- internal URLs
- LDAP bind credentials
- cloud keys
- SSH paths
- service account details

---

## 3. Shell Scripts

Scripts often expose credentials in plaintext, especially backup and automation scripts.

Look for shell scripts:

```bash
find / -type f -name "*.sh" 2>/dev/null
```

Things to search for inside scripts:

- `mysql -u user -pPassword123`
- `ssh user@host`
- `scp`
- `rsync`
- `curl -u`
- `wget --user`
- exported secrets
- environment variables containing credentials

Scripts are especially useful because they often reveal both **credentials** and **operational context**.

---

## 4. Bash History and Other History Files

Users often leak credentials in their shell history without realising it.

Examples include:

- database login commands
- `su` attempts
- `sshpass`
- `mysql -p`
- API test commands
- copied secrets
- admin scripts being executed manually

Check history files such as:

```bash
cat ~/.bash_history
cat /home/*/.bash_history 2>/dev/null
```

Search for all history-like files:

```bash
find / -type f \( -name "*_hist" -o -name "*_history" -o -name ".bash_history" \) 2>/dev/null
```

These files can reveal:

- plaintext passwords
- reused admin credentials
- internal hosts
- backup paths
- temporary workarounds created by admins

---

## 5. Backup Files

Backup copies often survive long after secrets should have been removed.

Look for:

- `.bak`
- `.old`
- `.save`
- `.swp`
- `.tmp`
- `~`

Useful search example:

```bash
find / -type f \( -name "*.bak" -o -name "*.old" -o -name "*.save" -o -name "*~" \) 2>/dev/null
```

Backup files are useful because they may contain:

- older credentials
- commented-out secrets
- deprecated but still valid access paths
- full copies of sensitive config files

---

## 6. Database Files

Applications sometimes store credentials in SQLite or other local database files.

Look for:

- `.db`
- `.sqlite`
- `.sqlite3`

Example:

```bash
find / -type f \( -name "*.db" -o -name "*.sqlite" -o -name "*.sqlite3" \) 2>/dev/null
```

These may contain:

- local users
- password hashes
- API tokens
- application secrets
- session data

---

## 7. Mail and Spool Directories

Mailboxes and spool directories are often overlooked and can contain credentials, reset links, or sensitive operational detail.

Interesting locations include:

- `/var/mail/`
- `/var/spool/mail/`
- `/var/spool/`
- `/var/log/`

Possible contents:

- password reset messages
- deployment notifications
- cron output
- credentials sent over email
- helpdesk data
- internal instructions

# SSH Key Hunting

SSH private keys are among the most useful artifacts you can find.

Check your own SSH directory first:

```bash
ls ~/.ssh
```

Example:

```text
id_rsa  id_rsa.pub  known_hosts
```

Then search more broadly:

```bash
find /home /root -type f \( -name "id_rsa" -o -name "id_dsa" -o -name "*.pem" \) 2>/dev/null
```

Useful targets:

- `/home/*/.ssh/`
- `/root/.ssh/`
- backup folders
- developer working directories
- deployment directories

If you find an SSH private key, check whether it is:

- readable
- unencrypted
- accepted by SSH
- reusable on another host

---

## known_hosts

Whenever you find SSH keys, also inspect `known_hosts`.

This file lists hosts the user previously connected to and can point you toward:

- internal servers
- jump boxes
- backup servers
- development infrastructure
- cloud instances

Example:

```bash
cat ~/.ssh/known_hosts
```

This is valuable because a private key without a destination is less useful. `known_hosts` helps tell you where that key might work.

# Practical Workflow

A good credential hunting flow looks like this:

## 1. Search application directories first

These often yield the fastest wins.

```bash
find /var/www /opt -type f 2>/dev/null
```

## 2. Search configs and scripts

```bash
find / -type f \( -name "*.conf" -o -name "*.config" -o -name "*.xml" -o -name "*.sh" \) 2>/dev/null
```

## 3. Grep for common secret patterns

```bash
grep -RniE "pass|password|passwd|secret|token|key|user|username" /var /etc /opt /home 2>/dev/null
```

## 4. Check shell histories

```bash
cat ~/.bash_history
cat /home/*/.bash_history 2>/dev/null
```

## 5. Hunt for SSH keys

```bash
find /home /root -type f \( -name "id_rsa" -o -name "*.pem" \) 2>/dev/null
```

## 6. Check known_hosts

```bash
cat ~/.ssh/known_hosts
cat /home/*/.ssh/known_hosts 2>/dev/null
```

## 7. Try credential reuse carefully

If you find a password, test it against:

- other local users
- `su`
- `sudo`
- database services
- SSH
- internal apps

Password reuse is extremely common.

# What Credentials Can Lead To

Credential hunting can directly lead to:

## Local privilege escalation

Examples:

- switching to another user with `su`
- gaining `sudo` access
- accessing root-owned application credentials

## Service access

Examples:

- MySQL / MariaDB
- PostgreSQL
- Redis
- LDAP
- web admin panels

## Lateral movement

Examples:

- SSH to another host
- access to backup servers
- reuse against domain services
- cloud console or API access

## Additional evidence of compromise paths

Credentials found locally often strengthen report findings because they show how one weakness can chain into another.

# High-Value Searches

## Find config files

```bash
find / -type f \( -name "*.conf" -o -name "*.config" -o -name "*.xml" -o -name "*.ini" -o -name "*.yml" -o -name "*.yaml" -o -name "*.json" \) 2>/dev/null
```

## Find likely secrets

```bash
grep -RniE "pass|password|passwd|secret|token|api_key|db_user|db_pass|username" /var /etc /opt /home 2>/dev/null
```

## Find SSH keys

```bash
find /home /root -type f \( -name "id_rsa" -o -name "id_dsa" -o -name "*.pem" \) 2>/dev/null
```

## Find backups

```bash
find / -type f \( -name "*.bak" -o -name "*.old" -o -name "*.save" -o -name "*~" \) 2>/dev/null
```

## Find database files

```bash
find / -type f \( -name "*.db" -o -name "*.sqlite" -o -name "*.sqlite3" \) 2>/dev/null
```

## Find shell history

```bash
find / -type f -name ".bash_history" 2>/dev/null
```

---

# Key Takeaways

- Credential hunting is often one of the quickest and safest privilege escalation paths.
- Always inspect application directories, config files, scripts, history files, mail, backups, and SSH material.
- Web roots under `/var/www` are especially valuable.
- SSH private keys are high-impact finds, especially when paired with `known_hosts`.
- A credential does not need to be root’s password to be useful. It may still unlock another user, a service, or another host.
- Reuse is common. Always test discovered credentials methodically.
