## Purpose

Server-side attacks target backend processing, parsers, templates, request handling, deserialisation, internal network access, and server-side trust boundaries.

This section is currently a placeholder for future notes.

XXE may either live here or under `10.02 - Injection Attacks`, depending on how you prefer to browse the vault.

---

## Core Idea

Server-side attacks usually occur when the backend server processes attacker-controlled input in a dangerous way.

The affected component may be:

- XML parser
- template engine
- HTTP request handler
- backend service client
- cache layer
- deserialisation library
- internal URL fetcher
- file parser
- archive extractor

The impact is often server-side rather than browser-side.

## Server-Side Testing Flow

1. Identify backend functionality.
2. Identify parsers, templates, imports, fetchers, or processors.
3. Look for user-controlled input reaching backend components.
4. Test for safe behavioural differences.
5. Check for file read, internal access, or backend error messages.
6. Determine whether the issue can expose data or reach internal services.
7. Assess whether command execution or file write is possible.
8. Document impact and related attack path.

---

## Common Server-Side Impact

Server-side attacks may lead to:

- internal service access
- local file read
- backend request forgery
- credential disclosure
- source code disclosure
- request routing abuse
- cache poisoning
- command execution
- data extraction
- internal network discovery
