## Overview

HTTP requests begin with a **method (verb)** that tells the server what action should be performed.

Most developers primarily think about:

- GET → retrieve data
- POST → submit data

However, the HTTP protocol supports several additional verbs that can trigger different behaviour on the server.

If either:

- the web server is misconfigured
- the application only validates certain HTTP methods

an attacker may be able to **bypass authentication, filters, or security logic** by sending an unexpected HTTP method.

This is known as **HTTP Verb Tampering**.

Think of this like a building that only checks ID badges at the front door, but forgets to secure the side entrance.

---

# 1. Common HTTP Methods

HTTP supports multiple verbs beyond GET and POST.

| Verb | Purpose |
|------|--------|
| GET | Retrieve a resource |
| POST | Submit data to the server |
| HEAD | Same as GET but returns headers only |
| PUT | Upload or replace a resource |
| DELETE | Delete a resource |
| OPTIONS | Lists allowed HTTP methods |
| PATCH | Partially modify a resource |

Some of these verbs interact directly with server resources, meaning misconfiguration can lead to:

- file upload
- file deletion
- authentication bypass
- command injection bypass
- logic bypass

---

# 2. What is HTTP Verb Tampering

HTTP Verb Tampering occurs when an application:

- expects only certain methods
- only validates certain methods
- only protects certain methods
- inconsistently handles request parameters

Attackers can then send alternative HTTP verbs to:

- bypass authentication
- bypass validation
- bypass filtering
- reach unintended functionality

---

# 3. Two Main Causes

## 3.1 Insecure Server Configuration

Authentication rules may only apply to certain HTTP methods.

Example configuration:

```xml
<Limit GET POST>
    Require valid-user
</Limit>
```

This protects only:

- GET
- POST

If the server also accepts:

- HEAD
- PUT
- DELETE

an attacker may bypass authentication by using another method.

---

## 3.2 Insecure Application Code

Developers may apply validation to only one request method.

Example:

```php
$pattern = "/^[A-Za-z\s]+$/";

if(preg_match($pattern, $_GET["code"])) {
    $query = "Select * from ports where port_code like '%" . $_REQUEST["code"] . "%'";
}
```

Problem:

- validation checks only GET
- query uses $_REQUEST

$_REQUEST contains:

- GET parameters
- POST parameters
- COOKIE parameters

An attacker can:

- send malicious payload via POST
- bypass validation
- exploit the underlying vulnerability

---

# 4. Why HTTP Verb Tampering is Dangerous

Verb tampering can lead to:

- authentication bypass
- command injection bypass
- SQL injection bypass
- access control bypass
- unintended file operations
- business logic abuse

Because many developers assume:

"Only GET and POST will ever be used"

they may fail to secure alternative methods.

---

# 5. Identifying HTTP Verb Tampering Opportunities

Look for functionality that:

- requires authentication
- performs file operations
- uses security filters
- performs sensitive actions

Common targets:

- admin panels
- file upload endpoints
- reset functions
- API endpoints
- form handlers

---

# 6. Discovering Allowed HTTP Methods

Use OPTIONS request:

```bash
curl -i -X OPTIONS http://target
```

Example response:

```http
Allow: POST,OPTIONS,HEAD,GET
```

This tells us the server accepts:

- GET
- POST
- HEAD
- OPTIONS

HEAD is often enabled by default.

---

# 7. Case Study – Authentication Bypass via HEAD

## Scenario

File Manager application:

```text
/admin/reset.php
```

Reset functionality requires authentication.

Access attempt results in:

```http
401 Unauthorized
```

---

## Step 1 – Identify request method

Intercept request using Burp.

Observed method:

```http
GET /admin/reset.php
```

---

## Step 2 – Test alternative methods

Change request method in Burp:

### Change GET → POST

Still blocked.

Authentication applies to POST.

---

## Step 3 – Try HEAD request

HEAD behaves like GET but returns headers only.

Burp request:

```http
HEAD /admin/reset.php
```

Result:

- no authentication prompt
- no output
- action still executed

Result:

files deleted successfully.

Authentication bypass achieved.

---

# 8. Why HEAD Worked

Server configuration restricted only:

- GET
- POST

HEAD remained unprotected.

Even though HEAD returns no body, the application logic still executed the reset function.

This demonstrates how authentication logic may be applied inconsistently across HTTP methods.

---

# 9. Case Study – Filter Bypass via Method Switching

Application blocks malicious filenames.

Example attempt:

```text
test;
```

Response:

```text
Malicious Request Denied!
```

Security filter detects injection characters.

---

## Step 1 – intercept request

Captured request:

```http
POST /create-file
```

---

## Step 2 – change request method

Modify:

```http
POST → GET
```

Send request again.

Result:

file successfully created.

Security filter bypassed.

---

## Step 3 – confirm command injection

Payload:

```text
file1; touch file2;
```

After request:

- file1 created
- file2 created

Command injection successful.

Filter bypass achieved via verb tampering.

---

# 10. Why This Works

Application validation:

```php
if(preg_match($pattern, $_POST['filename']))
```

But processing logic uses:

```php
$_REQUEST['filename']
```

GET parameters bypass validation.

Security control only applied to POST.

# 14. Prevention

## Server-side protections

Restrict allowed methods explicitly.

Apache example:

```apache
<LimitExcept GET POST>
    Deny from all
</LimitExcept>
```

---

## Application protections

Ensure validation applies consistently across:

- GET
- POST
- PUT
- PATCH

Avoid mixing:

```php
$_GET
$_POST
$_REQUEST
```

Use explicit handling.

---

## Secure coding practice

Apply validation to all input sources.

Example:

```php
$input = $_POST['code'] ?? $_GET['code'] ?? null;
```

Validate once consistently.

---

# 15. Key Takeaways

- HTTP supports multiple verbs beyond GET and POST
- servers may accept methods developers did not intend to expose
- authentication logic may only apply to certain HTTP methods
- validation filters may only check specific request types
- attackers can switch request methods to bypass security controls
- testing alternative HTTP methods is quick and often overlooked
- always ensure consistent validation and access control across all HTTP verbs
