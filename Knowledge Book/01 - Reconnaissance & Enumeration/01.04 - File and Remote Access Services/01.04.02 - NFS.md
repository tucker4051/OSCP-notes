## Overview

NFS (Network File System) allows systems to share directories across the network.

Common risks:

- sensitive files exposed
- weak permissions
- UID/GID misconfiguration
- writable shares
- privilege escalation via no_root_squash

Often encountered in Linux environments.

---

## Default Ports

| Port | Protocol | Service |
|------|----------|--------|
| 111 | TCP/UDP | RPCbind |
| 2049 | TCP/UDP | NFS |

---

## Quick Workflow

1. Identify NFS service
2. enumerate available shares
3. mount shares locally
4. check permissions
5. search for sensitive files
6. check root squash configuration

---

## Nmap Enumeration

List RPC services and NFS info:

```bash
sudo nmap -sV -p111,2049 --script nfs* {IP}
```

Useful scripts:

| Script | Purpose |
|--------|--------|
| nfs-showmount | lists available shares |
| nfs-ls | lists share contents |
| nfs-statfs | filesystem info |

RPC service discovery:

```bash
sudo nmap -sV -p111 --script rpcinfo {IP}
```

Shows:
- RPC services
- ports in use
- service mappings

---

## Show Available Shares

```bash
showmount -e {IP}
```

Example output:

```
Export list for 10.10.10.10:
/home    *
/var/nfs *(rw)
```

---

## Mount NFS Share

Create local mount directory:

```bash
mkdir target-NFS
```

Mount share:

```bash
sudo mount -t nfs {IP}:/ ./target-NFS -o nolock
```

Navigate to mounted directory:

```bash
cd target-NFS
```

View structure:

```bash
tree .
```

---

## Enumerate File Permissions

List files:

```bash
ls -lah
```

List with usernames:

```bash
ls -l mnt/nfs/
```

List when permissions restrict access:

```bash
ls -l -lah mnt/nfs/
```

List using UID/GID values:

```bash
ls -n mnt/nfs/
```

Useful for identifying:

- file ownership mismatches
- writable files
- privilege escalation paths

---

## root_squash vs no_root_squash

NFS may restrict root privileges when mounting shares.

| Setting | Effect |
|--------|--------|
| root_squash | root mapped to low privilege user |
| no_root_squash | root retains full privileges |

If **no_root_squash** enabled:

- attacker can create files as root
- potential privilege escalation
- ability to modify scripts/binaries

Example risk:

```
backup.sh writable → privilege escalation vector
```
