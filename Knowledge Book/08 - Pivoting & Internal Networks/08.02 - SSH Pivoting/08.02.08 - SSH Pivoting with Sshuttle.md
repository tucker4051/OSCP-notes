## Overview

**sshuttle** is a Python-based pivoting tool that creates a transparent tunnel over SSH, allowing traffic to be routed through a pivot host **without needing proxychains or SOCKS configuration**.

It works by:

- creating temporary **iptables rules**
- intercepting traffic destined for specific subnets
- transparently routing that traffic through an SSH tunnel
- making the remote network appear locally reachable

Because of this behaviour, sshuttle is often referred to as:

> a “VPN over SSH”

---

# Why Use Sshuttle?

Traditional SSH pivoting often requires:

- dynamic port forwarding (`ssh -D`)
- proxychains configuration
- application proxy support

sshuttle simplifies this by automatically:

- modifying firewall rules
- routing traffic via SSH
- making the remote network behave like a local network
- removing the need for proxy-aware tools

This allows tools like:

- nmap
- rdp clients
- smb clients
- curl
- browsers

to work normally without proxy configuration.

# Scenario

Assume:

- attack host can SSH into Ubuntu pivot host
- pivot host can access internal network
- internal network includes:

```

```

Instead of creating a SOCKS proxy, we configure sshuttle to route traffic automatically.

# Basic sshuttle Syntax

```bash
sshuttle -r <user>@<pivot_host> <subnet>
```

Where:

- `-r` specifies SSH connection target
- `<subnet>` is the network we want routed through the pivot

---

# Run sshuttle Pivot

## Command

```bash
sudo sshuttle -r ubuntu@10.129.202.64 172.16.5.0/23 -v
```

## Breakdown

### `sudo`

Required because sshuttle modifies local firewall rules.

### `-r ubuntu@10.129.202.64`

SSH connection target:

- user: ubuntu
- pivot host IP: 10.129.202.64

### `172.16.5.0/23`

Subnet that should be routed through the pivot host.

### `-v`

Verbose output.

---

# What Happens Internally

sshuttle:

1. creates an SSH session to the pivot host
2. launches a Python process remotely
3. modifies local iptables rules
4. intercepts traffic destined for the target subnet
5. tunnels that traffic through SSH
6. sends it from the pivot host into the internal network

The result:

Your system behaves as if it has direct network access to the internal subnet.

# Traffic Flow

```text
Attack host tool --> iptables redirect
                    --> sshuttle client
                        --> SSH tunnel
                            --> pivot host
                                --> internal network
```

Applications do not need proxy configuration because the routing is handled at the OS level.

# Why sshuttle Feels Like a VPN

sshuttle behaves similarly to a VPN because:

- routes traffic at network layer
- transparently forwards packets
- allows normal tool usage
- avoids per-tool proxy configuration

Difference from a real VPN:

- uses SSH instead of VPN protocol
- typically TCP-focused
- requires SSH access to pivot host

# RDP Example

---

# Limitations

## SSH only

Cannot pivot via:

- TOR
- HTTPS proxy
- SOCKS proxy chains

## Limited UDP support

Primarily TCP-focused.

## Requires root privileges

iptables modification requires elevated permissions.

## SSH credentials required

Must authenticate to pivot host.

---

# Comparison with Other Pivot Methods

| Tool | Method | Requires proxy config |
|------|--------|----------------------|
| ssh -D | SOCKS proxy | yes |
| proxychains | proxy wrapper | yes |
| meterpreter socks | SOCKS proxy | yes |
| plink | SOCKS proxy | yes |
| sshuttle | transparent routing | no |

sshuttle is often the simplest method when SSH access is available.

# Typical Workflow

1. compromise pivot host with SSH access
2. identify internal network range
3. start sshuttle tunnel
4. use tools normally against internal hosts

Example:

```bash
sudo sshuttle -r ubuntu@10.129.202.64 172.16.5.0/23
```

Then:

```bash
nmap 172.16.5.19
xfreerdp 172.16.5.19
smbclient -L //172.16.5.19
```

# Mental Model

sshuttle makes the remote subnet appear locally reachable.

Instead of telling tools:

> “send traffic through this proxy”

sshuttle tells the OS:

> “send traffic for this network through this tunnel”

So tools behave as if they are directly connected to the remote network.

- No proxychains required.
- Automatically configures iptables rules.
- Makes remote networks appear locally reachable.

- Often the fastest way to pivot when SSH access is available.
