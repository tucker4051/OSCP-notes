## Overview

This note covers two closely related SSH pivoting techniques:

- **Local port forwarding**
- **Dynamic port forwarding with a SOCKS proxy**

The short version is:

- **Local port forwarding** is best when you already know the exact target service and port you want to reach.
- **Dynamic port forwarding** is best when you need a more flexible pivot into an internal network and want your tools to route traffic through a compromised host.

Think of it like this:

- **Local port forwarding** is like building a single private hallway to one locked room.
- **Dynamic port forwarding with SOCKS** is like getting access to the whole building’s internal corridor system, then sending your tools through it.

---

# Port Forwarding in Context

## What Port Forwarding Is

Port forwarding is a technique used to redirect traffic from one port to another.

It is commonly used to:

- reach internal services
- bypass firewall restrictions
- pivot through a compromised host
- interact with services that are only listening on localhost or an internal interface

Although port forwarding fundamentally works over **TCP**, it is often implemented through encapsulation mechanisms such as:

- **SSH**
- **SOCKS**

This is useful because it allows us to tunnel traffic through an already allowed or already compromised communication path.

---

# SSH Local Port Forwarding

## When to Use It

Use **local port forwarding** when:

- you know the exact target service you want
- you know the exact destination port
- you want to access that remote service as if it were local on your own machine

This is ideal for:

- web apps bound to localhost
- databases bound to localhost
- admin interfaces not exposed externally
- exploit development against internal-only services

---

## Scenario

Assume:

- your attack host is `10.10.15.x`
- you have compromised an Ubuntu pivot host at `10.129.202.64`
- the pivot host has SSH open
- a MySQL service exists on the pivot host but is only reachable locally

First, scan the pivot target:

```bash
nmap -sT -p22,3306 10.129.202.64
```

### Example Output

```text
Starting Nmap 7.92 ( https://nmap.org ) at 2022-02-24 12:12 EST
Nmap scan report for 10.129.202.64
Host is up (0.12s latency).

PORT     STATE  SERVICE
22/tcp   open   ssh
3306/tcp closed mysql
```

## Why MySQL Looks Closed

From your attack host, port `3306` appears closed because the MySQL service is likely:

- bound to `127.0.0.1`
- not listening on the external interface
- protected from direct remote access

This is very common.

---

## Forward MySQL to Your Local Machine

To access the remote MySQL service locally, use SSH local port forwarding:

```bash
ssh -L 1234:localhost:3306 ubuntu@10.129.202.64
```

### What `-L` Means

The format is:

```bash
ssh -L <local_port>:<target_host_from_pivot>:<target_port> <user>@<pivot>
```

So in this case:

- your local machine listens on `1234`
- traffic hitting `127.0.0.1:1234` is sent through SSH
- the pivot host forwards it to its own `localhost:3306`

### In Plain English

You are saying:

> “Whenever I connect to local port 1234 on my machine, send that traffic through SSH and connect it to MySQL on the pivot host’s localhost port 3306.”

---

## Confirm the Forward Is Working

### Using Netstat

```bash
netstat -antp | grep 1234
```

### Example Output

```text
tcp        0      0 127.0.0.1:1234          0.0.0.0:*               LISTEN      4034/ssh
tcp6       0      0 ::1:1234                :::*                    LISTEN      4034/ssh
```

This shows your local SSH client is listening on port `1234`.

---

### Using Nmap

```bash
nmap -v -sV -p1234 localhost
```

### Example Output

```text
PORT     STATE SERVICE VERSION
1234/tcp open  mysql   MySQL 8.0.28-0ubuntu0.20.04.3
```

Now the MySQL service appears local, even though it is actually remote.

---

## Forwarding Multiple Ports

You can forward more than one service in the same SSH session.

Example:

```bash
ssh -L 1234:localhost:3306 -L 8080:localhost:80 ubuntu@10.129.202.64
```

This creates:

- `localhost:1234` → remote `localhost:3306`
- `localhost:8080` → remote `localhost:80`

### Why this is useful

This is handy when the pivot host runs multiple internal-only services, for example:

- MySQL
- Apache
- admin interfaces
- APIs
- dashboards

---

# Identifying Pivot Opportunities

Once connected to the pivot host, inspect its interfaces.

```bash
ifconfig
```

### Example Output

```text
ens192: inet 10.129.202.64
ens224: inet 172.16.5.129
lo:     inet 127.0.0.1
```

## Why This Matters

This tells us the host is **dual-homed**:

- one interface faces us
- one interface faces another internal network

That makes it a **pivot point**.

### Example Interpretation

- `ens192` connects to the network we already reach
- `ens224` connects to an internal subnet we cannot directly route to
- `lo` is loopback

This means the pivot host can see systems that we cannot.

# Dynamic Port Forwarding with SSH

## When to Use It

Use **dynamic port forwarding** when:

- you do **not** know the exact port or service you need yet
- you want to scan or access many internal systems
- you want flexible proxy-based pivoting

This is especially useful when you discover a second internal subnet on the pivot host, such as:

- `172.16.5.0/23`

Your attack machine cannot reach this subnet directly because it has no route to it.

---

## What Dynamic Port Forwarding Does

Instead of forwarding just one fixed port, SSH creates a **SOCKS proxy listener** on your local machine.

Your tools can then send traffic to that SOCKS proxy, and SSH will relay the traffic through the pivot host into the internal network.

---

## Enable Dynamic Port Forwarding

```bash
ssh -D 9050 ubuntu@10.129.202.64
```

### What `-D` Means

The format is:

```bash
ssh -D <local_socks_port> <user>@<pivot>
```

This tells SSH to:

- create a SOCKS proxy on your local machine at `127.0.0.1:9050`
- relay traffic through the SSH session
- let client applications use that proxy to reach internal hosts

---

# SOCKS Tunnelling

## What SOCKS Is

**SOCKS** stands for **Socket Secure**.

It is a proxy protocol that allows a client to send traffic through a proxy server to reach other systems.

### Why it matters for pivoting

SOCKS is useful because:

- it can tunnel traffic through an allowed path
- it helps bypass routing restrictions
- it makes your tools behave as though they are “inside” the internal network
- the target sees the **pivot host’s IP**, not your attack box

---

## SOCKS4 vs SOCKS5

There are two common SOCKS versions:

| Version | Notes |
|---|---|
| SOCKS4 | No authentication, no UDP support |
| SOCKS5 | Supports authentication and UDP |

In many SSH pivoting cases, you will see **SOCKS4** used with proxychains for simple TCP pivoting.

---

# Using Proxychains with SSH SOCKS

## Why Proxychains Is Needed

Most tools do not automatically know to use your new SOCKS proxy.

`proxychains` forces supported tools to send their TCP traffic through a proxy such as:

- SOCKS4
- SOCKS5
- HTTP proxy
- HTTPS proxy
- TOR

This lets tools like `nmap`, `xfreerdp`, and `msfconsole` pivot through the SSH tunnel.

---

## Proxychains Configuration

Edit `/etc/proxychains.conf` and ensure the SOCKS proxy is configured:

```bash
tail -4 /etc/proxychains.conf
```

### Example Output

```text
# meanwile
# defaults set to "tor"
socks4 127.0.0.1 9050
```

### What this means

Proxychains will send traffic to:

- `127.0.0.1`
- port `9050`
- using the SOCKS4 protocol

That local port is the SSH SOCKS listener created by `ssh -D 9050`.

---

# Scanning Through the SOCKS Tunnel

## Host Discovery Example

You can attempt to probe an internal range:

```bash
proxychains nmap -v -sn 172.16.5.1-200
```

### Important Note

This may be unreliable depending on:

- host firewall behaviour
- protocol used for discovery
- how proxychains handles the scan
- whether the targets respond to the probes used

---

## Why Full TCP Connect Scans Are Needed

When using `proxychains`, you should use:

- **`-sT`** full TCP connect scans

Not:

- `-sS` SYN scans
- raw packet scans
- half-open scan types

### Why

Proxychains works at the TCP connection level. It cannot properly handle partial/raw packet behaviour.

So if you try SYN scans or other low-level scan types through proxychains, you may get:

- incorrect results
- broken scans
- false negatives

---

## Important Nmap Guidance

Use:

```bash
proxychains nmap -v -Pn -sT 172.16.5.19
```

### What each option means

- `proxychains` → forces traffic through the SOCKS proxy
- `-Pn` → skip host discovery, assume host is up
- `-sT` → full TCP connect scan

### Why `-Pn` matters

Windows firewalls often block ICMP echo requests by default, so ping-based host discovery may fail even when the host is up.

---

## Example Scan Through the Pivot

```bash
proxychains nmap -v -Pn -sT 172.16.5.19
```

### Example Findings

```text
Discovered open port 445/tcp on 172.16.5.19
Discovered open port 135/tcp on 172.16.5.19
Discovered open port 3389/tcp on 172.16.5.19
Discovered open port 139/tcp on 172.16.5.19
```

## What this tells us

The internal target appears to be a Windows host exposing:

- SMB (`445`, `139`)
- RPC (`135`)
- RDP (`3389`)

This is enough to justify deeper enumeration or direct access attempts.

# Using Metasploit Through the Tunnel

You can also run Metasploit through proxychains:

```bash
proxychains msfconsole
```

Then use auxiliary modules such as the RDP scanner.

## Example Module Workflow

```text
search rdp_scanner
use auxiliary/scanner/rdp/rdp_scanner
set rhosts 172.16.5.19
run
```

### Example Result

```text
[*] 172.16.5.19:3389 - Detected RDP on 172.16.5.19:3389
(name:DC01)
(domain:DC01)
(os_version:10.0.17763)
(Requires NLA: No)
```

## Why this matters

This confirms:

- the service is reachable through the pivot
- the target is a Windows host
- RDP is available
- NLA is not required in this example

That can significantly shape your next steps.

# RDP Access Through the SOCKS Tunnel

Once you have valid credentials, you can RDP through the tunnel.

Example:

```bash
proxychains xfreerdp /v:172.16.5.19 /u:victor /p:pass@123
```

### What happens

- `xfreerdp` connects to the target through proxychains
- proxychains sends traffic to the local SOCKS listener
- SSH carries that traffic to the pivot host
- the pivot host reaches the internal Windows host
- you receive an RDP session over the tunnel

### Practical note

You will usually need to accept the RDP certificate on first connection.

---

# Why Dynamic SOCKS Pivoting Is Powerful

Dynamic SSH + SOCKS is powerful because it lets you:

- scan internal hosts
- enumerate services
- use offensive tools through the pivot
- connect to RDP, SMB, web apps, and more
- avoid dropping additional tunnelling tools on disk
- route traffic through an already allowed SSH channel

This makes it a very clean and flexible pivoting method.

---

# Key Limitations

## 1. TCP-focused

This method primarily supports **TCP** traffic well.

## 2. Slower Scanning

Scanning through a proxy is slower than direct scanning.

## 3. Full Connect Scans Only

Use `-sT`, not raw scan types.

## 4. Host Discovery Can Be Unreliable

Especially with Windows targets that block ICMP.

## 5. Tool Compatibility Varies

Not every tool behaves perfectly through proxychains.

---

# Choosing Between `-L` and `-D`

## Use `ssh -L` when:

- you know the exact target port
- you want one specific service
- you want a stable direct mapping

Example:

- remote MySQL on `3306`
- remote HTTP on `80`

## Use `ssh -D` when:

- you want flexible pivoting
- you need to scan an internal subnet
- you do not yet know which ports are interesting
- you want multiple tools to route through the pivot

# Practical Workflow

A common workflow looks like this:

1. compromise a host with SSH access
2. inspect its interfaces
3. identify a second internal network
4. decide whether you need:
   - a specific port forward (`-L`)
   - or flexible pivoting (`-D`)
5. configure proxychains if using SOCKS
6. scan internal targets carefully with `-Pn -sT`
7. enumerate services
8. access internal services using your forwarded channel

---

# Key Takeaways

- **Local port forwarding** is service-specific.
- **Dynamic port forwarding** creates a local SOCKS proxy for broader pivoting.
- **Proxychains** is what makes many tools use that SOCKS tunnel.
- **Use `-sT` with proxychains**, not SYN scans.
- **Use `-Pn`** when host discovery may fail.
- **Dual-homed hosts** are prime pivot candidates.
- **SSH tunnelling** is often one of the cleanest pivot methods because it leverages a trusted, legitimate protocol.
