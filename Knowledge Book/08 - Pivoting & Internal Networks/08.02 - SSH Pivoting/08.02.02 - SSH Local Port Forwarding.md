## Overview

SSH local port forwarding creates a listening port on the machine running the SSH client.

Traffic sent to that listening port is:

1. accepted by the SSH client,
    
2. carried through the encrypted SSH connection,
    
3. received by the SSH server,
    
4. forwarded from the SSH server to the final destination.
    

General syntax:

```bash
ssh -L [LISTEN_IP:]LISTEN_PORT:DESTINATION_IP:DESTINATION_PORT user@SSH_SERVER
```

The most important rule is:

```text
The -L listener exists on the SSH client.
The destination connection is made from the SSH server.
```

---

# Core Mental Model

```text
Your tool
↓
Listening socket on SSH client
↓
SSH tunnel
↓
SSH server
↓
Final internal destination
```

For the command:

```bash
ssh -L 0.0.0.0:4455:172.16.50.217:445 database_admin@10.4.50.215
```

the roles are:

|Value|Meaning|
|---|---|
|`0.0.0.0`|Listen on every interface of the SSH client|
|`4455`|Port opened on the SSH client|
|`172.16.50.217`|Final destination reachable from the SSH server|
|`445`|Final destination service port|
|`database_admin`|SSH username|
|`10.4.50.215`|SSH server address|

# Example Network Layout

Assume the following hosts:

|Role|Host|IP|
|---|---|---|
|Attack host|Kali|`192.168.50.10`|
|First compromised host|CONFLUENCE01|`192.168.50.63`|
|SSH server / second pivot|PGDATABASE01|`10.4.50.215`|
|Internal SMB target|FILESERVER|`172.16.50.217`|

PGDATABASE01 has two interfaces:

```text
10.4.50.215
172.16.50.215
```

This means it can communicate with both:

```text
10.4.50.0/24
172.16.50.0/24
```

The internal SMB server is:

```text
172.16.50.217:445
```

Kali cannot reach `172.16.50.217` directly.

CONFLUENCE01 can SSH to PGDATABASE01.

PGDATABASE01 can reach `172.16.50.217`.

# Why This Is Called Local Forwarding

It is called local forwarding because the listening socket is created on the local side of the SSH connection.

In this scenario, the SSH command is executed on:

```text
CONFLUENCE01
```

Therefore, from SSH's perspective:

```text
CONFLUENCE01 is local
PGDATABASE01 is remote
```

This remains true even though CONFLUENCE01 is itself a compromised remote host from the tester's perspective.

> [!important]  
> “Local” and “remote” are relative to where the SSH command is executed.

# Identify Each Host Before Building the Tunnel

## SSH Client

The SSH client is the machine where the `ssh -L` command runs.

Here:

```text
CONFLUENCE01
192.168.50.63
```

It creates the listening port.

---

## SSH Server

The SSH server is the host after `user@` in the command.

Here:

```text
PGDATABASE01
10.4.50.215
```

It receives the tunnel and connects to the final destination.

---

## Final Destination

The final destination is the host and port in the middle of the `-L` rule.

Here:

```text
172.16.50.217:445
```

This destination must be reachable from the SSH server.

---

## External Client

The external client is the system that connects to the local forward.

Here:

```text
Kali
192.168.50.10
```

Kali connects to the SSH client's listener:

```text
192.168.50.63:4455
```

# 1. Enumerate the SSH Server's Networks

Before creating the forward, identify networks reachable from the SSH server.

SSH from CONFLUENCE01 to PGDATABASE01:

```bash
ssh database_admin@10.4.50.215
```

On PGDATABASE01:

```bash
ip addr
```

Example:

```text
ens192: 10.4.50.215/24
ens224: 172.16.50.215/24
```

Interpretation:

```text
PGDATABASE01 belongs to 10.4.50.0/24
PGDATABASE01 also belongs to 172.16.50.0/24
```

Check routes:

```bash
ip route
```

Example:

```text
10.4.50.0/24 dev ens192
172.16.50.0/24 dev ens224
```

This confirms PGDATABASE01 can route directly to hosts in both subnets.

---

# 2. Find the Internal Destination

From PGDATABASE01, search for hosts with SMB open:

```bash
for i in $(seq 1 254); do
    nc -zv -w 1 172.16.50.$i 445
done
```

Command breakdown:

|Component|Meaning|
|---|---|
|`seq 1 254`|Generates host addresses from `.1` to `.254`|
|`nc`|Netcat|
|`-z`|Port scan mode without sending application data|
|`-v`|Verbose output|
|`-w 1`|One-second timeout|
|`445`|SMB port|

Interesting result:

```text
Connection to 172.16.50.217 445 port succeeded
```

Final destination:

```text
172.16.50.217:445
```

---

# 3. Build the Local Forward Rule

Ensure TTY - `python3 -c 'import pty; pty.spawn("/bin/sh")'`

OpenSSH local-forwarding format:

```text
[LISTEN_IP:]LISTEN_PORT:DESTINATION_IP:DESTINATION_PORT
```

For this scenario:

```text
LISTEN_IP       = 0.0.0.0
LISTEN_PORT     = 4455
DESTINATION_IP  = 172.16.50.217
DESTINATION_PORT= 445
```

Result:

```text
0.0.0.0:4455:172.16.50.217:445
```

Add the SSH server:

```text
database_admin@10.4.50.215
```

Complete command:

```bash
ssh -N -L 0.0.0.0:4455:172.16.50.217:445 database_admin@10.4.50.215
```

---

# 4. Command Breakdown

```bash
ssh -N -L 0.0.0.0:4455:172.16.50.217:445 database_admin@10.4.50.215
```

|Component|Meaning|
|---|---|
|`ssh`|Start OpenSSH client|
|`-N`|Do not open a remote shell|
|`-L`|Create a local port forward|
|`0.0.0.0`|Bind listener on every interface of CONFLUENCE01|
|`4455`|Port opened on CONFLUENCE01|
|`172.16.50.217`|Final SMB server|
|`445`|SMB service port|
|`database_admin`|User on PGDATABASE01|
|`10.4.50.215`|SSH server, PGDATABASE01|

---

# 5. Where the Command Runs

Run the command on:

```text
CONFLUENCE01
```

Not on:

```text
Kali
PGDATABASE01
FILESERVER
```

Reason:

```text
CONFLUENCE01 must create a listener that Kali can reach
CONFLUENCE01 can SSH to PGDATABASE01
PGDATABASE01 can reach the final SMB server
```

---

# 6. Why Bind to 0.0.0.0?

This section:

```text
0.0.0.0:4455
```

means:

```text
Listen on port 4455 on all interfaces of CONFLUENCE01
```

This includes the interface Kali can reach:

```text
192.168.50.63
```

Kali therefore connects to:

```text
192.168.50.63:4455
```

> [!warning]  
> Binding to `0.0.0.0` may expose the forward to other hosts that can reach CONFLUENCE01.

A more restrictive command is:

```bash
ssh -N -L 192.168.50.63:4455:172.16.50.217:445 database_admin@10.4.50.215
```

This listens only on the interface using:

```text
192.168.50.63
```

---

# 7. Why Not Bind to 127.0.0.1?

This command:

```bash
ssh -N -L 127.0.0.1:4455:172.16.50.217:445 database_admin@10.4.50.215
```

would make the listener available only from CONFLUENCE01 itself.

Kali would not be able to connect directly to:

```text
192.168.50.63:4455
```

Use `127.0.0.1` only when the tool consuming the forward also runs on the SSH client.

---

# 8. Why Use Port 4455?

The real SMB service runs on:

```text
172.16.50.217:445
```

However, the listener on CONFLUENCE01 uses:

```text
4455
```

Ports below `1024` normally require elevated privileges on Linux.

The compromised user is unprivileged, so a higher port is selected.

The two ports do not need to match:

```text
CONFLUENCE01 listener: 4455
FILESERVER service:     445
```

---

# 9. Why the Destination Is Not PGDATABASE01

The SSH server address is:

```text
10.4.50.215
```

The final destination is:

```text
172.16.50.217
```

These are different roles.

PGDATABASE01 is only the tunnel endpoint and routing host.

It receives the tunneled connection and then opens a new connection to:

```text
172.16.50.217:445
```

Conceptually:

```text
SSH tunnel endpoint:
10.4.50.215

Final service:
172.16.50.217:445
```

---

# 10. Start the Tunnel

On CONFLUENCE01:

```bash
ssh -N -L 0.0.0.0:4455:172.16.50.217:445 database_admin@10.4.50.215
```

After entering the SSH password, no shell prompt appears.

This is expected because:

```text
-N = do not execute a remote command
```

The SSH process remains in the foreground maintaining the tunnel.

---

# 11. Verify the Listener

From another shell on CONFLUENCE01:

```bash
ss -lntp | grep 4455
```

Expected:

```text
0.0.0.0:4455
```

Example:

```text
tcp LISTEN 0 128 0.0.0.0:4455 0.0.0.0:* users:(("ssh",pid=59288,fd=4))
```

This confirms:

```text
SSH is listening on CONFLUENCE01 port 4455
```

It does not prove the final destination works, but it confirms the local side of the tunnel exists.

---

# 12. Connect from Kali

Kali connects to the SSH client's listening socket:

```text
192.168.50.63:4455
```

List SMB shares:

```bash
smbclient -p 4455 -L //192.168.50.63/ -U hr_admin --password='Welcome1234'
```

Command breakdown:

|Option|Meaning|
|---|---|
|`-p 4455`|Connect to forwarded SMB port|
|`-L`|List available shares|
|`//192.168.50.63/`|Connect to CONFLUENCE01's listener|
|`-U hr_admin`|SMB username for the final server|
|`--password`|Password for the final SMB server|

> [!important]  
> SSH credentials authenticate to PGDATABASE01. SMB credentials authenticate to `172.16.50.217`. They are separate authentication steps.

---

# Authentication Flow

## SSH Authentication

Used when creating the tunnel:

```bash
ssh database_admin@10.4.50.215
```

Credentials belong to:

```text
PGDATABASE01
```

## SMB Authentication

Used after the tunnel exists:

```bash
smbclient -p 4455 //192.168.50.63/scripts -U hr_admin
```

Credentials belong to:

```text
FILESERVER at 172.16.50.217
```

The pivot does not automatically provide authentication to the final service.

# Repeatable Methodology

## Step 1 – Identify the SSH Client

Ask:

```text
Where will I run the ssh -L command?
```

That host receives the listening port.

Record:

```text
SSH client: <client_host>
Client IP reachable by my tool: <client_ip>
```

---

## Step 2 – Identify the SSH Server

Ask:

```text
Which SSH server can the client log in to?
```

Record:

```text
SSH server: <ssh_server_ip>
SSH user: <ssh_user>
```

---

## Step 3 – Identify the Final Destination

Ask:

```text
What host and port can the SSH server reach that I cannot?
```

Record:

```text
Destination IP: <destination_ip>
Destination port: <destination_port>
```

---

## Step 4 – Select the Listening Socket

Ask:

```text
Who needs to connect to the listener?
```

If only the SSH client itself:

```text
127.0.0.1
```

If another host such as Kali must connect:

```text
0.0.0.0
```

or the SSH client's specific reachable IP.

Choose an unused port above `1024` where possible.

---

## Step 5 – Build the Command

Template:

```bash
ssh -N -L <listen_ip>:<listen_port>:<destination_ip>:<destination_port> <ssh_user>@<ssh_server_ip>
```

---

## Step 6 – Connect the Tool

Your tool connects to:

```text
<ssh_client_reachable_ip>:<listen_port>
```

The tool does not connect to:

```text
<ssh_server_ip>:<destination_port>
```

or:

```text
<destination_ip>:<destination_port>
```

# Useful SSH Options

## No Remote Shell

```text
-N
```

Use when the connection only exists for forwarding.

---

## Background the Tunnel

```bash
ssh -fN -L <rule> user@<ssh_server>
```

`-f` moves SSH into the background after authentication.

---

## Verbose Output

```bash
ssh -v -N -L <rule> user@<ssh_server>
```

More detail:

```bash
ssh -vvv -N -L <rule> user@<ssh_server>
```

---

## Fail If Forward Cannot Be Created

```bash
ssh -o ExitOnForwardFailure=yes -N -L <rule> user@<ssh_server>
```

Recommended:

```bash
ssh -o ExitOnForwardFailure=yes -N -L 0.0.0.0:4455:172.16.50.217:445 database_admin@10.4.50.215
```

---

## Keep the Tunnel Alive

```bash
ssh -o ServerAliveInterval=30 -o ServerAliveCountMax=3 -N -L <rule> user@<ssh_server>
```

---

# Recommended Full Command

```bash
ssh \
  -o ExitOnForwardFailure=yes \
  -o ServerAliveInterval=30 \
  -o ServerAliveCountMax=3 \
  -N \
  -L 0.0.0.0:4455:172.16.50.217:445 \
  database_admin@10.4.50.215
```

# Troubleshooting

## No Output After Entering Password

This is normal when using:

```text
-N
```

Verify the listener from another shell:

```bash
ss -lntp | grep <listen_port>
```

---

## Cannot Connect from Kali

Check the listener address:

```bash
ss -lntp
```

If it shows:

```text
127.0.0.1:4455
```

Kali cannot connect remotely.

Use:

```text
0.0.0.0:4455
```

or:

```text
<ssh_client_external_ip>:4455
```

---

## Listener Exists but Service Does Not Work

Test the final destination from the SSH server:

```bash
nc -zv <destination_ip> <destination_port>
```

The destination must be reachable from the SSH server, not necessarily from the SSH client.

---

## Wrong IP Used in Client Tool

The client tool should connect to:

```text
SSH client reachable IP
+
local listening port
```

Example:

```text
192.168.50.63:4455
```

Not:

```text
10.4.50.215:4455
172.16.50.217:4455
172.16.50.217:445
```

---

## Port Below 1024 Fails

Use a higher listening port.

Instead of:

```text
445
```

use:

```text
4455
```

The destination can still remain:

```text
445
```

---

## Address Already in Use

Check:

```bash
ss -lntp | grep <listen_port>
```

Choose another port or stop the existing listener.

---

## Connection Is Refused at Final Destination

Possible causes:

```text
wrong destination IP
wrong destination port
service not listening
firewall between SSH server and destination
destination only bound to localhost
```

Test from the SSH server.

---

## SSH Connection Drops

Use keepalive options:

```bash
ssh \
  -o ServerAliveInterval=30 \
  -o ServerAliveCountMax=3 \
  -N \
  -L <rule> \
  user@<ssh_server>
```

---

## Cannot Create Listener on 0.0.0.0

Possible causes:

```text
port already in use
local firewall
SSH client restrictions
system policy
```

Try binding to the SSH client's specific external IP:

```bash
ssh -N -L <ssh_client_ip>:<port>:<destination_ip>:<destination_port> user@<ssh_server>
```

# Key Takeaways

- `ssh -L` creates the listener on the host where the SSH command is run.
    
- The host after `user@` is the SSH server.
    
- The final destination is the IP and port inside the `-L` rule.
    
- The SSH server makes the connection to the final destination.
    
- External tools connect to the SSH client's reachable IP and listening port.
    
- “Local” means local to the SSH client, not necessarily local to Kali.
    
- Use `0.0.0.0` when another host must reach the listener.
    
- Use `127.0.0.1` when only the SSH client itself needs the listener.
    
- The local listening port and final destination port do not need to match.
    
- Authentication to the SSH server and authentication to the final service are separate.
    
- Confirm routing and service reachability from the SSH server before creating the tunnel.
