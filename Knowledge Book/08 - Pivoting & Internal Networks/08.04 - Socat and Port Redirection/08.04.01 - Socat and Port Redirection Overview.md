## Purpose

Socat and port redirection techniques are used to forward, redirect, relay, or bridge network connections between hosts and ports.

Use this when you need to expose an internal service, redirect traffic through a pivot host, or connect services that cannot directly communicate.

Socat is useful for:

- simple TCP forwarding
- redirecting internal ports
- pivoting through a compromised host
- exposing bind shells
- relaying reverse shell traffic
- creating ad-hoc network bridges

---

## Core Idea

Socat is a flexible networking relay.

It can listen on one address/port and forward traffic to another address/port.

The key question is:

> What connection do I need to bridge?

## When to Use Socat

Use Socat when:

- [ ]  SSH is unavailable
- [ ]  a simple TCP relay is needed
- [ ]  a reverse shell needs forwarding
- [ ]  a bind shell is only reachable from the pivot
- [ ]  an internal service needs exposing
- [ ]  traffic needs redirecting between interfaces
- [ ]  lightweight port forwarding is enough
- [ ]  Chisel/SSH/Meterpreter are not suitable

---

## Common Socat Use Cases

|Scenario|Use|
|---|---|
|Internal service reachable only from pivot|Relay service to attacker-accessible port|
|Bind shell only reachable internally|Redirect bind shell through pivot|
|Reverse shell needs forwarding|Relay reverse shell connection|
|Tool cannot route through SOCKS|Forward a specific port instead|
|Simple TCP bridge required|Use Socat as listener and connector|

---

## Port Redirection Workflow

1. Identify the service or connection to redirect.
2. Identify which host can reach the target.
3. Decide where the listener should run.
4. Decide where traffic should be forwarded.
5. Start Socat listener/relay.
6. Test with a simple connection.
7. Use the forwarded service.
8. Document the mapping.

---

## Common Mistakes

- [ ]  Listening on the wrong host
- [ ]  Forwarding to the wrong IP or port
- [ ]  Confusing attacker, pivot, and target roles
- [ ]  Forgetting firewall directionality
- [ ]  Binding only to localhost when remote access is needed
- [ ]  Exposing a service more broadly than intended
- [ ]  Not testing the relay before using tools
- [ ]  Losing the relay when the shell dies
