## Purpose

Tunnelling involves carrying traffic through another protocol or channel to bypass network restrictions, reach internal services, or maintain connectivity through constrained environments.

Use this when normal TCP pivoting, SSH, SOCKS, or direct connections are not available or are blocked.

Tunnelling may involve:

- DNS tunnelling
- ICMP tunnelling
- HTTP/HTTPS tunnelling
- SOCKS over restricted channels
- covert or constrained network paths

---

## Core Idea

Tunnelling wraps one type of traffic inside another.

The goal is to use an allowed protocol to carry traffic that would otherwise be blocked.

The key question is:

> What traffic is allowed out of the target network?

## When to Use Tunnelling

Use tunnelling when:

- [ ]  outbound TCP is restricted
- [ ]  only DNS is allowed
- [ ]  ICMP is allowed but other protocols are blocked
- [ ]  direct reverse shells fail
- [ ]  standard port forwards fail
- [ ]  proxying is required over an unusual channel
- [ ]  network egress is heavily restricted

---

## Common Tunnel Types

|Tunnel Type|Use Case|
|---|---|
|DNS tunnelling|When DNS egress is allowed|
|ICMP tunnelling|When ICMP is allowed through firewall|
|HTTP/HTTPS tunnelling|When web traffic is allowed|
|SOCKS tunnelling|Flexible tool routing through a tunnel|
|Reverse tunnelling|When target can call back but attacker cannot connect in|

---

## Tunnelling Workflow

1. Identify allowed outbound protocols.
2. Choose tunnel type.
3. Prepare server/listener side.
4. Transfer client/tool to target if needed.
5. Establish tunnel.
6. Confirm connectivity.
7. Route tools through tunnel.
8. Keep traffic controlled and stable.
9. Document tunnel setup.

---

## Common Constraints

Tunnels may be affected by:

- DNS filtering
- proxy requirements
- packet size limits
- latency
- blocked ICMP
- unstable sessions
- detection controls
- firewall directionality
- missing tools on target
- payload transfer restrictions

---

## Common Mistakes

- [ ]  Trying DNS tunnelling without confirming DNS egress
- [ ]  Trying ICMP tunnelling without confirming ICMP is allowed
- [ ]  Not testing basic callbacks first
- [ ]  Using tunnels for noisy scans too early
- [ ]  Not accounting for poor bandwidth or latency
- [ ]  Not documenting server/client configuration
- [ ]  Forgetting the tunnel depends on a running process
- [ ]  Confusing tunnel transport with proxied target traffic
