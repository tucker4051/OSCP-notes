## Overview

Modern Windows systems often include the OpenSSH client by default.

This means a Windows host can use:

```text
ssh.exe
scp.exe
sftp.exe
ssh-keygen.exe
ssh-agent.exe
ssh-add.exe
```

Default location:

```text
C:\Windows\System32\OpenSSH\
```

This allows Windows hosts to perform the same SSH tunnelling techniques commonly used from Linux, including:

```text
local port forwarding
remote port forwarding
dynamic port forwarding
remote dynamic port forwarding
```

This note focuses on using `ssh.exe` from Windows to create a **remote dynamic port forward** back to Kali.

---

## Why This Matters

In many environments, inbound access to internal Windows hosts is restricted.

Example:

```text
Kali can RDP to MULTISERVER03 on TCP/3389
Kali cannot connect to arbitrary ports on MULTISERVER03
MULTISERVER03 can make outbound SSH connections to Kali
MULTISERVER03 can reach internal services
```

This makes Windows `ssh.exe` useful for pivoting.

Instead of binding a listener on the Windows host, the Windows host connects outbound to Kali and creates a SOCKS proxy on Kali.

# Example Scenario

## Hosts

|Role|Host|IP|
|---|---|---|
|Attack host / SSH server|Kali|`192.168.118.4`|
|Windows pivot / SSH client|MULTISERVER03|`192.168.50.64`|
|Internal database target|PGDATABASE01|`10.4.50.215`|

## Goal

Use `ssh.exe` from Windows to create a SOCKS proxy on Kali.

Then use Proxychains from Kali to reach:

```text
10.4.50.215:5432
```

through the Windows host.

---

# Traffic Flow

```text
Kali
192.168.118.4
SSH SERVER
SOCKS proxy: 127.0.0.1:9998
    |
    | Proxychains sends SOCKS traffic here
    v
OpenSSH server on Kali
    |
    | SSH tunnel initiated from Windows
    v
MULTISERVER03
192.168.50.64
SSH CLIENT
ssh.exe
    |
    | Windows host opens final connection
    v
PGDATABASE01
10.4.50.215:5432
PostgreSQL
```

Final application path:

```text
psql on Kali
↓
Proxychains
↓
127.0.0.1:9998 on Kali
↓
SSH tunnel
↓
MULTISERVER03
↓
10.4.50.215:5432
```

> [!important]  
> The SOCKS proxy is on Kali, but the final internal connection is made by the Windows host.

# 6. Create Remote Dynamic Port Forward

Run on Windows:

```cmd
ssh -N -R 9998 kali@192.168.118.4
```

Command breakdown:

|Component|Meaning|
|---|---|
|`ssh`|Windows OpenSSH client|
|`-N`|Do not open a remote shell|
|`-R 9998`|Create a remote dynamic SOCKS proxy on Kali port `9998`|
|`kali`|SSH username on Kali|
|`192.168.118.4`|Kali SSH server IP|

After entering the Kali password, no prompt appears.

This is normal because:

```text
-N means no remote command or shell is opened.
```

The SSH process stays open to maintain the tunnel.

---

# Why `-R 9998` Creates a SOCKS Proxy

Classic remote port forwarding uses:

```cmd
ssh -R <remote_port>:<destination_ip>:<destination_port> user@ssh_server
```

Example:

```cmd
ssh -R 2345:10.4.50.215:5432 kali@192.168.118.4
```

This forwards one fixed destination:

```text
Kali:2345 → Windows SSH client → 10.4.50.215:5432
```

Remote dynamic port forwarding uses only one port:

```cmd
ssh -R 9998 kali@192.168.118.4
```

Because no fixed destination is specified, OpenSSH creates a SOCKS proxy on the SSH server.

Result:

```text
Kali gets a SOCKS proxy on 127.0.0.1:9998
Windows forwards requested connections to reachable internal hosts
```

---

# 7. Verify SOCKS Proxy on Kali

On Kali:

```bash
ss -lntp | grep 9998
```

Expected:

```text
127.0.0.1:9998
```

You may also see IPv6 loopback:

```text
[::1]:9998
```

Example:

```text
tcp LISTEN 0 128 127.0.0.1:9998 0.0.0.0:*
tcp LISTEN 0 128 [::1]:9998 [::]:*
```

This confirms the SOCKS proxy exists on Kali.

# 8. Configure Proxychains on Kali

Edit:

```bash
sudo nano /etc/proxychains4.conf
```

At the bottom:

```text
[ProxyList]
socks5 127.0.0.1 9998
```

Meaning:

|Value|Meaning|
|---|---|
|`socks5`|SOCKS protocol|
|`127.0.0.1`|SOCKS proxy is local to Kali|
|`9998`|Port created by `ssh -R 9998`|

Check:

```bash
tail /etc/proxychains4.conf
```

> [!important]  
> Proxychains points to Kali loopback because the SOCKS proxy is bound on Kali, not Windows.

---

# 9. Use Proxychains to Access Internal Services

With remote dynamic forwarding, tool commands use the real internal destination IP.

Do not connect tools to:

```text
127.0.0.1:9998
```

directly unless the tool is SOCKS-aware.

Instead, use Proxychains.

Example PostgreSQL connection:

```bash
proxychains psql -h 10.4.50.215 -U postgres
```

Proxychains sends this through:

```text
127.0.0.1:9998
```

The Windows host then connects to:

```text
10.4.50.215:5432
```

For the scenario:

```text
Kali:          192.168.118.4
```

## `192.168.118.4`

This is Kali.

It is:

```text

```

Proxychains points to:

```text

```

because the SOCKS listener is local on Kali.

---

## `192.168.50.64`

This is MULTISERVER03.

It is:

```text
the Windows host
the RDP target
the SSH client
the system running ssh.exe
the pivot that reaches internal services
```

The SSH command runs here.

---

## `10.4.50.215`

This is PGDATABASE01.

It is:

```text
the internal target
the PostgreSQL server
the final destination requested through SOCKS
```

It appears in the tool command:

```bash
proxychains psql -h 10.4.50.215 -U postgres
```

It does not appear in the `ssh -R 9998` command because the destination is chosen dynamically by SOCKS.

File:

```text

```

```text

socks5 127.0.0.1 9998
```

Useful settings:

```text
strict_chain

tcp_connect_time_out 8000
```

For faster scans:

```text

tcp_connect_time_out 1500
```

> [!warning]  
> Short timeouts can cause slow services to appear unavailable.

# Nmap Through Windows Remote Dynamic Tunnel

Recommended:

```bash
proxychains nmap -sT -Pn -n --top-ports 20 <target_ip>
```

Why:

|Option|Reason|
|---|---|
|`-sT`|Uses normal TCP connect calls that Proxychains can intercept|
|`-Pn`|Skips host discovery that may not work through SOCKS|
|`-n`|Avoids DNS resolution issues or leaks|
|`--top-ports 20`|Keeps scans smaller and faster|

Avoid:

```bash
proxychains nmap -sS <target_ip>
```

Reason:

```text
SYN scans use raw packets and do not work reliably through Proxychains SOCKS paths.
```

# Security and Detection Notes

Potential indicators:

```text
ssh.exe process on Windows
outbound SSH from a Windows server to Kali or external infrastructure
long-lived SSH connection
Kali sshd creating SOCKS listener on 127.0.0.1
many internal connections sourced from the Windows pivot
Proxychains-style scanning
RDP login followed by ssh.exe tunnelling
```

Defensive controls:

```text
restrict outbound SSH from servers
monitor Windows ssh.exe execution
alert on unusual ssh.exe command lines using -R
disable or remove OpenSSH client where unnecessary
restrict AllowTcpForwarding on SSH servers
monitor long-lived SSH sessions
segment internal networks
monitor egress from RDP-accessible servers
```

Example suspicious command line:

```cmd
ssh -N -R 9998 kali@192.168.118.4
```

# Decision Guide

Use Windows `ssh.exe` remote dynamic forwarding when:

```text
you have access to a Windows host
the Windows host can SSH outbound to Kali
the Windows host can reach internal targets
inbound access to the Windows host is restricted
OpenSSH client is installed and modern enough
you want SOCKS-style access to multiple internal services
```

Use classic remote port forwarding when:

```text
you only need one internal host and port
tools do not work well with SOCKS
you want a simple fixed mapping
```

Use other tools when:

```text
ssh.exe is missing
OpenSSH is too old
outbound SSH is blocked
Proxychains is not suitable
EDR blocks ssh.exe tunnelling
```

Alternatives:

```text
plink.exe
chisel
ligolo-ng
socat
netsh portproxy
PowerShell-based tunnelling tools
```

# Key Takeaways

- Modern Windows often includes `ssh.exe` by default.
    
- Windows OpenSSH can connect to Linux SSH servers.
    
- Remote dynamic forwarding from Windows creates a SOCKS proxy on Kali.
    
- The SSH command runs on the Windows pivot.
    
- Kali is the SSH server.
    
- The SOCKS listener appears on Kali, usually as `127.0.0.1:<port>`.
    
- Proxychains should point to `127.0.0.1:<port>` on Kali.
    
- Tool commands should target the real internal IPs.
    
- The Windows pivot makes the final internal network connections.
    
- This is useful when only RDP is exposed inbound but outbound SSH is allowed.
    
- Use `-sT`, `-Pn`, and `-n` for Nmap through Proxychains.
    
- Verify `ssh.exe` exists and OpenSSH is version `7.6` or newer.
