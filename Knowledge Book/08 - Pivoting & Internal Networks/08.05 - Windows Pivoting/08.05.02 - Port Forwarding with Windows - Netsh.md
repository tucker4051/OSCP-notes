## Overview

`netsh` is a native Windows command-line utility used to configure and manage network settings.

One useful pivoting feature is:

```text
netsh interface portproxy
```

`portproxy` allows a Windows host to listen on one TCP port and forward received traffic to another IP address and port.

This can turn a compromised Windows host into a TCP relay between networks.

Common uses:

```text
Expose an internal service through a compromised Windows host
Pivot into segmented networks
Forward RDP, SSH, WinRM, MSSQL, HTTP, or SMB
Use native Windows tooling without uploading extra binaries
```

> [!important]  
> `netsh interface portproxy` requires administrative privileges to create, modify, or delete forwarding rules.

---

## Core Mental Model

Instead of connecting directly to an internal host:

```text
Attacker
↓
Internal target
```

use the compromised Windows host as a relay:

```text
Attacker
↓
Windows pivot host listening port
↓
netsh portproxy rule
↓
Internal target service
```

Example:

```text
Kali
↓
192.168.50.64:2222
↓
MULTISERVER03 netsh portproxy
↓
10.4.50.215:22
```

From the attacker's perspective, connecting to:

```text
192.168.50.64:2222
```

behaves like connecting to:

```text
10.4.50.215:22
```

---

# Key Netsh Capabilities

|Function|Description|
|---|---|
|Routing inspection|View and troubleshoot Windows routing|
|Firewall configuration|Add, delete, and inspect firewall rules|
|Proxy configuration|Configure certain proxy settings|
|Port forwarding|Relay TCP traffic with `interface portproxy`|

---

# Portproxy Requirements

## Administrative Privileges

Portproxy changes require an elevated shell.

Useful checks:

```cmd
whoami /groups
whoami /priv
```

If running through RDP, open:

```text
cmd.exe as Administrator
```

> [!note]  
> UAC may block changes even when using an administrator account, depending on how the shell was obtained.

---

## IP Helper Service

Portproxy relies on the IP Helper service.

Check status:

```cmd
sc query iphlpsvc
```

Start it if needed:

```cmd
sc start iphlpsvc
```

---

## Windows Firewall

The portproxy rule may listen locally, but inbound traffic can still be blocked by Windows Firewall.

You may need a matching firewall rule for the listening port.

Example:

```cmd
netsh advfirewall firewall add rule name="port_forward_ssh_2222" protocol=TCP dir=in localip=192.168.50.64 localport=2222 action=allow
```

---

## TCP Only

`netsh portproxy` forwards TCP traffic.

It does not provide UDP forwarding.

# Scenario 1 – Forward RDP Through a Windows Pivot

## Network Layout

Compromised Windows host:

```text
Windows workstation
External-facing IP: 10.129.15.150
Internal-facing IP: 172.16.5.25
```

Internal target:

```text
172.16.5.25
RDP service: TCP/3389
```

Goal:

```text
Allow the attacker to access internal RDP by connecting to the compromised Windows host.
```

---

## Traffic Flow

```text
Attacker
    |
    | Connect to 10.129.15.150:8080
    v
Windows pivot
10.129.15.150
netsh listener: TCP/8080
    |
    | Forwards traffic
    v
Internal RDP service
172.16.5.25:3389
```

---

## Create RDP Forward

Run on the compromised Windows host:

```cmd
netsh interface portproxy add v4tov4 listenport=8080 listenaddress=10.129.15.150 connectport=3389 connectaddress=172.16.5.25
```

Multiline format:

```cmd
netsh interface portproxy add v4tov4 ^
listenport=8080 ^
listenaddress=10.129.15.150 ^
connectport=3389 ^
connectaddress=172.16.5.25
```

---

## Parameter Breakdown

|Parameter|Meaning|
|---|---|
|`v4tov4`|IPv4 listener forwarding to IPv4 destination|
|`listenport=8080`|Port exposed on the Windows pivot|
|`listenaddress=10.129.15.150`|Pivot IP that accepts attacker connections|
|`connectport=3389`|Destination service port|
|`connectaddress=172.16.5.25`|Internal target IP|

---

## Allow Firewall Port

```cmd
netsh advfirewall firewall add rule name="portforward_8080" protocol=TCP dir=in localport=8080 action=allow
```

More restrictive version:

```cmd
netsh advfirewall firewall add rule name="portforward_8080" protocol=TCP dir=in localip=10.129.15.150 localport=8080 action=allow
```

---

## Verify Forward

```cmd
netsh interface portproxy show v4tov4
```

or:

```cmd
netsh interface portproxy show all
```

Expected:

```text
Listen on ipv4:             Connect to ipv4:

Address         Port        Address         Port
--------------- ----------  --------------- ----------
10.129.15.150   8080        172.16.5.25     3389
```

Check listener:

```cmd
netstat -anp TCP | find "8080"
```

---

## Connect Through Pivot

From Kali:

```bash
xfreerdp /v:10.129.15.150:8080 /u:<username> /p:'<password>'
```

Connection path:

```text
xfreerdp
↓
10.129.15.150:8080
↓
netsh portproxy
↓
172.16.5.25:3389
```

## Network Layout

Attack host:

```text
Kali
Can reach MULTISERVER03
```

Windows pivot:

```text
MULTISERVER03
External-facing IP: 192.168.50.64
```

Internal target:

```text
PGDATABASE01
Internal IP: 10.4.50.215
SSH service: TCP/22
```

Goal:

```text
SSH from Kali to PGDATABASE01 by connecting to MULTISERVER03:2222.
```

---

## Traffic Flow

```text
Kali
    |
    | SSH to 192.168.50.64:2222
    v
MULTISERVER03
192.168.50.64
netsh listener: TCP/2222
    |
    | Forwards traffic
    v
PGDATABASE01
10.4.50.215:22
```

---

## Create SSH Forward

Run in elevated `cmd.exe` on MULTISERVER03:

```cmd
netsh interface portproxy add v4tov4 listenport=2222 listenaddress=192.168.50.64 connectport=22 connectaddress=10.4.50.215
```

This creates:

```text
Listener on MULTISERVER03:
192.168.50.64:2222

Forward destination:
10.4.50.215:22
```

---

## Verify Local Listener

```cmd
netstat -anp TCP | find "2222"
```

Expected:

```text
TCP    192.168.50.64:2222     0.0.0.0:0     LISTENING
```

Show saved portproxy rules:

```cmd
netsh interface portproxy show all
```

Expected:

```text
Listen on ipv4:             Connect to ipv4:

Address         Port        Address         Port
--------------- ----------  --------------- ----------
192.168.50.64   2222        10.4.50.215     22
```

---

## Test from Kali

Before opening the firewall, the port may appear filtered:

```bash
sudo nmap -sS 192.168.50.64 -Pn -n -p2222
```

Possible result:

```text
2222/tcp filtered
```

This usually means:

```text
netsh is listening
but
Windows Firewall is blocking inbound access
```

---

## Add Firewall Rule

Run on MULTISERVER03:

```cmd
netsh advfirewall firewall add rule name="port_forward_ssh_2222" protocol=TCP dir=in localip=192.168.50.64 localport=2222 action=allow
```

Expected:

```text
Ok.
```

Test again from Kali:

```bash
sudo nmap -sS 192.168.50.64 -Pn -n -p2222
```

Expected:

```text
2222/tcp open
```

---

## Connect to Internal SSH Service

From Kali:

```bash
ssh database_admin@192.168.50.64 -p 2222
```

Although the command connects to:

```text
192.168.50.64:2222
```

the SSH session terminates on:

```text
10.4.50.215:22
```

Expected clue:

```text
Welcome to Ubuntu
IPv4 address for ens192: 10.4.50.215
```

# Understanding the IP Addresses

## `listenaddress`

This is the IP address on the Windows pivot where the attacker connects.

Example:

```cmd
listenaddress=192.168.50.64
```

Meaning:

```text
Listen on MULTISERVER03's external-facing interface.
```

Kali connects to:

```text
192.168.50.64:<listenport>
```

---

## `listenport`

This is the port exposed on the Windows pivot.

Example:

```cmd
listenport=2222
```

Meaning:

```text
Open TCP/2222 on MULTISERVER03.
```

The listen port does not need to match the destination service port.

---

## `connectaddress`

This is the final destination IP from the Windows pivot's perspective.

Example:

```cmd
connectaddress=10.4.50.215
```

Meaning:

```text
Forward traffic to PGDATABASE01.
```

The destination must be reachable from the Windows pivot.

---

## `connectport`

This is the final destination service port.

Example:

```cmd
connectport=22
```

Meaning:

```text
Forward to SSH on the internal target.
```

# Delete Portproxy Rules

## Delete Specific Rule

```cmd
netsh interface portproxy delete v4tov4 listenport=<listen_port> listenaddress=<pivot_ip>
```

Example:

```cmd
netsh interface portproxy delete v4tov4 listenport=2222 listenaddress=192.168.50.64
```

Alternative shorthand:

```cmd
netsh interface portproxy del v4tov4 listenport=2222 listenaddress=192.168.50.64
```

> [!note]  
> Successful deletion may produce no output.

---

## Confirm Deletion

```cmd
netsh interface portproxy show all
```

Check that the rule no longer appears.

---

## Reset All Portproxy Rules

Use carefully:

```cmd
netsh interface portproxy reset
```

> [!warning]  
> This removes all configured portproxy rules, including legitimate admin-created rules.

# Advantages of Netsh Pivoting

## Native Windows Binary

No extra tunnelling tool is required.

## Useful in Restricted Environments

Works when uploading tools is restricted or blocked.

## Living-off-the-Land

Uses built-in administrative functionality.

## Persistent Configuration

Rules survive reboot unless removed.

## Works Without SSH

Useful when SSH, Plink, Socat, or Chisel are unavailable.

## Supports Multiple Static Forwards

Multiple services can be exposed simultaneously by creating multiple rules.

---

# Limitations

## Requires Administrator Privileges

Portproxy configuration needs elevated privileges.

## UAC Can Interfere

Being in the local Administrators group may not be enough if the shell is not elevated.

## TCP Only

No UDP forwarding support.

## Static One-Port Rules

Each service requires a separate rule.

## Firewall Dependency

Windows Firewall may block the listening port unless explicitly allowed.

## Leaves Artifacts

Rules persist until removed.

Artifacts include:

```text
portproxy rules
firewall rules
listening ports
event logs
admin shell activity
```

## Not SOCKS

Netsh is not dynamic proxying.

It does not provide:

```text
SOCKS routing
automatic multi-host scanning
dynamic destination selection
UDP tunnelling
```

# Comparison with Other Pivot Methods

|Method|Type|Extra Tool Required|Notes|
|---|---|--:|---|
|`netsh portproxy`|Static TCP forward|No|Native Windows, admin required|
|`ssh -L`|Static TCP forward|Yes|Requires SSH client/server|
|`ssh -D`|SOCKS proxy|Yes|Dynamic multi-host access|
|`ssh -R`|Reverse static TCP forward|Yes|Good when inbound blocked|
|Plink `-R`|Reverse static TCP forward|Yes|Useful when OpenSSH missing|
|Socat|Static TCP relay|Yes|Flexible, cross-platform|
|Chisel|HTTP/SOCKS tunnelling|Yes|Useful through restrictive egress|
|Meterpreter `portfwd`|Static TCP forward|Yes|C2-dependent|

# Troubleshooting

## Portproxy Rule Exists but Port Is Filtered

Likely cause:

```text
Windows Firewall is blocking inbound access.
```

Check from Kali:

```bash
sudo nmap -sS <pivot_ip> -Pn -n -p<listen_port>
```

Add firewall rule:

```cmd
netsh advfirewall firewall add rule name="<rule_name>" protocol=TCP dir=in localip=<pivot_ip> localport=<listen_port> action=allow
```

---

## Portproxy Rule Exists but No Listener Appears

Check IP Helper:

```cmd
sc query iphlpsvc
```

Start it:

```cmd
sc start iphlpsvc
```

Check bind address:

```cmd
ipconfig
```

Make sure `listenaddress` exists on the pivot.

---

## Cannot Add Portproxy Rule

Possible causes:

```text
not running as administrator
UAC token is not elevated
incorrect syntax
IP Helper service issue
unsupported address family
```

Confirm admin context:

```cmd
whoami /groups
```

Open elevated `cmd.exe` where possible.

---

## Attacker Connects but Service Does Not Respond

Test destination from Windows:

```powershell
Test-NetConnection <target_ip> -Port <target_port>
```

Possible causes:

```text
target service is down
target firewall blocks pivot host
wrong connectaddress
wrong connectport
routing issue between pivot and target
service only binds to localhost on target
```

---

## Wrong IP Used

Remember:

```text
listenaddress = IP on Windows pivot that attacker connects to
connectaddress = IP of final internal target from pivot perspective
```

Do not set `listenaddress` to the target IP unless that IP is assigned to the pivot.

---

## Port Already in Use

Check:

```cmd
netstat -ano | find "<listen_port>"
```

Choose another listen port if needed.

---

## Low Port Binding Issues

If forwarding SMB or another privileged/common service, use a high listen port.

Example:

```text
Use 1445 on the pivot
Forward to 445 on the target
```

---

## RDP Certificate Shows Internal Hostname

This is expected.

Example:

```text
You connect to <pivot_ip>:8080
The RDP certificate belongs to the internal target
```

The TLS or service banner usually belongs to the final destination.

# PowerShell Notes

Many firewall operations have PowerShell equivalents, such as:

```powershell
New-NetFirewallRule
Get-NetFirewallRule
Disable-NetFirewallRule
Remove-NetFirewallRule
```

However, the core `netsh interface portproxy` functionality does not have a direct simple PowerShell equivalent in the same way.

For pure Netsh workflows, use:

```cmd
netsh interface portproxy ...
netsh advfirewall firewall ...
```

# Detection and Defensive Notes

Blue teams may detect:

```text
unexpected listening ports
new portproxy rules
Windows Firewall rule modifications
unusual inbound access to high ports
RDP, SSH, SMB, or WinRM accessed through non-standard ports
administrative netsh usage
persistent portproxy configuration
traffic relayed through workstation or server hosts
```

Check portproxy rules:

```cmd
netsh interface portproxy show all
```

Check firewall rules:

```cmd
netsh advfirewall firewall show rule name=all
```

Look for suspicious commands:

```cmd

netsh advfirewall firewall add rule
```

Hardening ideas:

```text
monitor netsh execution
monitor changes to portproxy registry keys
restrict local administrator rights
control inbound Windows Firewall rules
alert on unusual listening ports
review IP Helper service usage
segment administrative access
disable unnecessary lateral paths
```

# Key Takeaways

- `netsh` is a native Windows tool.
    
- `interface portproxy` provides TCP port forwarding.
    
- The Windows host becomes a static TCP relay.
    
- `listenaddress` and `listenport` are where the attacker connects.
    
- `connectaddress` and `connectport` are the internal destination.
    
- The final target must be reachable from the Windows pivot.
    
- Portproxy requires administrative privileges.
    
- IP Helper service must be running.
    
- Windows Firewall may block the listening port.
    
- Add a firewall rule when the port appears filtered.
    
- Portproxy rules persist until removed.
    
- Always remove both the firewall rule and portproxy rule during cleanup.
    
- Netsh is useful when SSH, Plink, Socat, or other tooling is unavailable.
    
- Netsh is not SOCKS and does not provide dynamic pivoting.
