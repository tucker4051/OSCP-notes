## Purpose

Pivoting is the process of using access to one host to reach systems, services, or networks that were not directly accessible from the attacker machine.

Use this section after gaining access to a host that has additional network visibility.

Pivoting may involve:

- port forwarding
- SOCKS proxies
- SSH tunnels
- reverse tunnels
- dynamic port forwarding
- proxychains
- tunnelling over DNS or ICMP
- Windows port forwarding
- Meterpreter routing
- internal network enumeration

---

## Core Idea

A compromised host may have access to networks that the attacker cannot reach directly.

The pivot host becomes a bridge.

The key questions are:

- What networks can the pivot host reach?
- What services are listening internally?
- Which direction can connections flow?
- Can the attacker connect to the pivot?
- Can the pivot connect back to the attacker?
- Do we need a single port forward or a SOCKS proxy?
- Do we need TCP, UDP, DNS, ICMP, or HTTP-based tunnelling?
- How will tools be routed through the pivot?

## Pivoting Decision Guide

|Situation|Likely Technique|
|---|---|
|You have SSH access to pivot host|SSH local/dynamic/remote forwarding|
|You need to browse many internal ports|SOCKS proxy or dynamic port forwarding|
|You need one internal service exposed locally|Local port forward|
|Target can connect back but you cannot connect in|Remote/reverse port forward|
|Windows host with SSH unavailable|Plink, Netsh, Chisel, SocksOverRDP|
|You have Meterpreter session|Meterpreter routing/portfwd|
|Only DNS egress is allowed|DNS tunnelling|
|ICMP allowed but TCP restricted|ICMP tunnelling|
|Simple TCP redirection needed|Socat|

## Common Pivoting Workflow

1. Gain access to a host.
2. Enumerate network interfaces and routes.
3. Identify internal networks.
4. Check what the host can reach.
5. Decide whether a single port forward or SOCKS proxy is needed.
6. Set up the tunnel or forward.
7. Configure tools to use the tunnel.
8. Enumerate internal services.
9. Reuse credentials where appropriate.
10. Expand access or pivot deeper.

After gaining access, check:

- [ ]  IP addresses
- [ ]  Network interfaces
- [ ]  Routing table
- [ ]  DNS configuration
- [ ]  ARP cache
- [ ]  Listening ports
- [ ]  Established connections
- [ ]  Firewall rules
- [ ]  Reachable subnets
- [ ]  Internal hostnames
- [ ]  Domain context
- [ ]  Proxy settings
- [ ]  Credentials available for reuse

---

## Common Pivoting Mistakes

- [ ]  Not checking the pivot host routing table
- [ ]  Using local port forwarding when dynamic forwarding is better
- [ ]  Forgetting to configure proxychains or browser proxy settings
- [ ]  Confusing local, remote, and dynamic port forwarding
- [ ]  Using the wrong interface/IP
- [ ]  Forgetting firewall directionality
- [ ]  Not testing the tunnel with a simple connection first
- [ ]  Trying to scan too aggressively through unstable tunnels
- [ ]  Not documenting tunnel commands and mappings
- [ ]  Forgetting to tear down tunnels when finished
