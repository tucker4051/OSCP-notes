## Purpose

Credential and password attacks focus on finding, extracting, cracking, reusing, spraying, or abusing credentials.

This section covers credential-focused techniques across Linux, Windows, Active Directory, services, network shares, traffic captures, hashes, password stores, and authentication mechanisms.

Use this section when you have discovered:

- usernames
- passwords
- password hashes
- Net-NTLMv2 captures
- NTLM hashes
- SSH keys
- protected archives
- credential files
- password manager data
- Windows credential stores
- reusable credentials
- authenticated remote service access

---

## Core Idea

Credentials are often the fastest route to progress.

A credential can support:

- initial access
- privilege escalation
- lateral movement
- remote service access
- database access
- admin panel access
- Active Directory attacks
- password reuse
- post-exploitation

The key question is:

> What can this credential, hash, key, or token be used for?

## Credential Attack Workflow

A practical credential workflow is:

1. Find usernames.
2. Find or capture credentials.
3. Identify credential type.
4. Determine whether it is plaintext, hash, key, ticket, token, or protected file.
5. Crack or extract where needed.
6. Test against relevant services.
7. Check password reuse.
8. Use valid credentials for access, escalation, or movement.
9. Record the source and usage of each credential.
10. Avoid repeated testing that could cause lockouts.

## Credential Types

|Credential Type|Possible Use|
|---|---|
|Plaintext password|Direct login, reuse, spraying|
|Password hash|Cracking, pass-the-hash where applicable|
|Net-NTLMv2|Offline cracking|
|SSH private key|SSH access, lateral movement|
|Passphrase-protected key|Crack passphrase then use key|
|Kerberos ticket|Pass-the-ticket|
|Certificate|Pass-the-certificate or authentication abuse|
|API token|API access or application access|
|Session cookie|Web session access|
|Password manager data|Extract stored credentials|
|Protected archive|Crack/extract sensitive contents|

## Credential Reuse

Always consider reuse.

A password found in one location may work for:

- SSH
- SMB
- RDP
- WinRM
- database login
- CMS admin login
- application admin panel
- local user account
- domain account
- email account
- VPN or remote access service

Record each credential and where it has been tested.

## Prioritisation

A practical credential priority order is:

1. Plaintext credentials.
2. SSH keys or private keys.
3. Valid web/app sessions.
4. Local admin or privileged credentials.
5. Domain credentials.
6. Service account credentials.
7. Password hashes that can be passed directly.
8. Password hashes that need cracking.
9. Protected files that may contain credentials.
10. Default credentials and spray candidates.

## Common Mistakes

- [ ]  Finding credentials but not testing reuse
- [ ]  Not recording credential source
- [ ]  Mixing usernames and passwords without context
- [ ]  Cracking hashes before checking whether they can be passed
- [ ]  Ignoring SSH private keys
- [ ]  Ignoring config files
- [ ]  Ignoring network shares
- [ ]  Ignoring PowerShell or shell history
- [ ]  Ignoring password managers
- [ ]  Spraying without considering lockout
- [ ]  Not linking credentials back to the attack path
