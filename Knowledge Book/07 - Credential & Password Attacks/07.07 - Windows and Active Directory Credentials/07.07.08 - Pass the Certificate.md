## Overview

**Pass the Certificate** is an authentication abuse technique where an attacker uses an **X.509 certificate** instead of a plaintext password or NTLM hash to obtain Kerberos access.

In practice, this usually means:

- obtaining a certificate and private key for a target identity
- using that certificate to request a **Kerberos TGT**
- then reusing the resulting Kerberos ticket for lateral movement or privilege escalation

This technique is most commonly associated with:

- **Active Directory Certificate Services (AD CS) abuse**
- **Shadow Credentials attacks**
- **PKINIT-enabled Kerberos environments**

A good way to think about it is:

- **Pass the Hash** = log in with the password hash  
- **Pass the Ticket** = log in with the Kerberos ticket  
- **Pass the Certificate** = log in with the certificate to *get* the Kerberos ticket

---

# PKINIT refresher

## What is PKINIT?

**PKINIT** stands for **Public Key Cryptography for Initial Authentication in Kerberos**.

Normally, Kerberos initial authentication is password-based.  
With PKINIT, Kerberos allows the client to authenticate using:

- a certificate
- the associated private key

This is commonly used for:

- smart card logon
- certificate-based user authentication
- certificate-based machine authentication

If an attacker gains access to a certificate and private key that AD accepts for authentication, they may be able to request a valid **Ticket Granting Ticket (TGT)** as that user or machine.

---

# Why this matters

If you can successfully authenticate with a certificate, you may be able to:

- request a TGT without the victim’s password
- impersonate user or machine accounts
- move into **Pass the Ticket** territory
- access WinRM, SMB, LDAP, or other Kerberos-backed services
- escalate further, including to **DCSync** in the right scenarios

Certificates become another form of credential material — just like hashes, tickets, and keys.

---

# Two common Pass-the-Certificate paths

The two major paths covered here are:

1. **AD CS NTLM relay to certificate enrollment (ESC8)**
2. **Shadow Credentials via msDS-KeyCredentialLink**

---

# 1. AD CS NTLM Relay Attack (ESC8)

## Concept

This attack abuses **web enrollment** in Active Directory Certificate Services.

If AD CS web enrollment is enabled, a CA may expose a web application such as:

```text
/CertSrv
```

If that endpoint accepts NTLM authentication, an attacker may be able to:

1. relay an NTLM authentication attempt to the AD CS web enrollment endpoint
2. request a certificate on behalf of the relayed identity
3. receive a valid certificate and private key for that account
4. use it with PKINIT to obtain a TGT

This is commonly referred to as **ESC8**.

---

## Why ESC8 is dangerous

If the relayed identity is a **high-value machine account**, especially a **domain controller**, the resulting certificate may allow the attacker to authenticate as that machine account.

That can quickly lead to:

- Kerberos ticket issuance
- directory replication access
- DCSync
- full domain compromise

---

## High-level attack flow

### Step 1: Relay NTLM to AD CS web enrollment
Tool commonly used:
- `impacket-ntlmrelayx`

Goal:
- relay inbound NTLM auth to the AD CS HTTP endpoint
- request a certificate using a suitable template

### Step 2: Coerce a target to authenticate
The attacker may either:

- wait for inbound authentication passively, or
- actively trigger authentication from a target host

One common coercion vector is the **printer bug**, assuming the spooler service is running.

### Step 3: Receive issued certificate
If the relay succeeds, the attacker receives a certificate, often exported as:

- `.pfx` / PKCS#12 bundle

This contains:
- certificate
- private key

### Step 4: Use the certificate with PKINIT
A tool such as:
- `gettgtpkinit.py`

can use the `.pfx` to request a TGT and save it as:

- a Kerberos ccache file

### Step 5: Reuse the resulting Kerberos ticket
Once the TGT exists, this becomes a **Pass the Ticket** scenario.

Examples:
- use Kerberos-authenticated tooling
- perform DCSync
- access services as the relayed identity

---

## Important note on certificate templates

The value passed to certificate enrollment tooling often depends on the environment.

You must use a template that is:

- enabled
- enrollable
- accepted for authentication in that context

This is why template enumeration matters in AD CS assessments.

---

## Practical significance of machine certificates

A certificate issued for a machine account, especially a **DC machine account**, is extremely powerful.

If the machine account can authenticate and replicate directory data, it may be possible to dump sensitive secrets such as:

- domain user hashes
- krbtgt hash
- administrator account hashes

This is why AD CS web enrollment over HTTP is so risky when misconfigured.

# 2. Shadow Credentials (msDS-KeyCredentialLink)

## Concept

**Shadow Credentials** abuse the Active Directory attribute:

```text
msDS-KeyCredentialLink
```

This attribute stores public keys that can be used for **PKINIT authentication**.

If an attacker can write to this attribute on another account, they can effectively plant their own public key on that account and later authenticate as that user using the matching private key.

In BloodHound, this is commonly represented by the:

- **AddKeyCredentialLink** edge

---

## Why it works

If you can add your own key material to a victim account’s `msDS-KeyCredentialLink`, you create a certificate-based authentication path that AD will trust for that victim.

That means you can:

1. generate a certificate/private key pair
2. add the public key to the victim account
3. use the private key + certificate to request a TGT as the victim
4. impersonate the victim with Kerberos

No password reset is required.  
No hash cracking is required.  
You are effectively adding a new “accepted authenticator” to the account.

---

## High-level attack flow

### Step 1: Find write access over a victim’s key credential attribute
Typical requirement:
- write permission over `msDS-KeyCredentialLink`

### Step 2: Generate a certificate and attach the public key to the victim
Common tool:
- `pywhisker`

This typically creates:

- a `.pfx` file
- a randomly generated password for the PFX
- a new key credential entry on the victim object

### Step 3: Use the PFX with PKINIT
Tool:
- `gettgtpkinit.py`

Goal:
- request a TGT as the victim user
- save it as a ccache

### Step 4: Reuse ticket
Once the TGT is obtained, the attack becomes a standard **Pass the Ticket** workflow.

Examples:
- Kerberos-authenticated WinRM
- SMB access
- LDAP operations
- lateral movement as the victim

---

# After obtaining the TGT: back to Pass the Ticket

Once a TGT has been successfully obtained using a certificate, the “certificate” phase is over.

From that point onward, the workflow looks just like any other Kerberos ticket abuse scenario:

- export/use `KRB5CCNAME`
- verify with `klist`
- authenticate with Kerberos-aware tools
- access services permitted to that user or machine

So Pass the Certificate is best understood as:

> a way of **getting a Kerberos ticket without needing the password**

# Tools mentioned

## ntlmrelayx
Used to relay NTLM authentication to an AD CS HTTP enrollment endpoint.

Common role in this context:
- relay inbound auth
- request certificate from web enrollment endpoint

---

## printerbug.py
Used to coerce a target system into attempting authentication to an attacker-controlled host.

Common use in this context:
- trigger machine account authentication that can be relayed

---

## PKINITtools / gettgtpkinit.py
Used to request a Kerberos TGT using:

- certificate
- private key
- PKINIT

Typical output:
- `.ccache` file
- AS-REP key information

---

## pywhisker
Used for Shadow Credentials attacks.

Common role:
- generate certificate
- add key credential material to victim account
- output PFX bundle

---

## Evil-WinRM
Can be used with Kerberos after a TGT is obtained, assuming:

- the target user has WinRM access
- Kerberos is configured correctly on the attacking machine

---

## Impacket secretsdump
Useful after obtaining a machine TGT with sufficient replication rights.

Common role:
- perform DCSync-style operations
- retrieve domain credential data

---

## certipy
Useful for AD CS enumeration.

Common role:
- identify certificate templates
- identify enrollment paths
- assess AD CS attack surface

# Typical attack chains

## Chain A: ESC8 → certificate → TGT → DCSync

1. relay coerced NTLM auth to AD CS web enrollment
2. obtain certificate for machine account
3. request TGT via PKINIT
4. export ticket cache
5. use Kerberos-authenticated tooling
6. perform DCSync or similar post-auth operations

This is one of the most dangerous AD CS attack paths because it can move very quickly from a single relay to full domain compromise.

---

## Chain B: Shadow Credentials → victim user → TGT → lateral movement

1. identify control over `msDS-KeyCredentialLink`
2. add a key credential to victim user
3. generate PFX and private key
4. request TGT via PKINIT
5. pass the ticket
6. access systems the victim can access

This is especially useful when the target user has:
- WinRM rights
- local admin rights
- delegated privileges
- privileged group membership

---

# Key indicators to understand in output

## `.pfx`
PKCS#12 bundle containing:
- certificate
- private key

This is usually the most important artifact for Pass-the-Certificate.

---

## `.ccache`
Kerberos cache file.

Once generated, this is what tools will use for Kerberos-authenticated actions.

---

## `KRB5CCNAME`
Environment variable pointing tools to the active Kerberos cache.

Example:
```bash
export KRB5CCNAME=/tmp/jpinkman.ccache
```

---

## `klist`
Used to verify:
- which principal the cache belongs to
- validity period
- available tickets

---

## AS-REP encryption key
Some PKINIT tooling outputs the AS-REP encryption key as well.

This can be useful for follow-on operations depending on the workflow and tooling being used.

# Requirements / prerequisites

## For ESC8
Common prerequisites include:

- AD CS web enrollment enabled
- NTLM accepted on that endpoint
- relay path available
- coercion path or inbound authentication source
- suitable certificate template

---

## For Shadow Credentials
Common prerequisites include:

- write access to victim object’s `msDS-KeyCredentialLink`
- PKINIT usable in the environment
- victim account not blocked from that auth path

---

# No PKINIT?

In some environments, you may obtain a certificate but still be unable to perform PKINIT because:

- the KDC does not accept the certificate for that account
- the EKU is insufficient
- the cert is not usable for Kerberos pre-authentication in that context

In those cases, certificate-authenticated LDAP over LDAPS may still be possible.

A tool specifically mentioned for these edge cases is:

- **PassTheCert**

This is useful when the certificate is still valid for authentication, but not for the PKINIT path you were expecting.

# Operational considerations

## Name resolution matters
Kerberos is sensitive to hostnames and SPNs.  
When using Kerberos-authenticated tooling, ensure:

- DNS works, or
- `/etc/hosts` is populated correctly

---

## Kerberos config matters
If using Linux tooling such as Evil-WinRM with Kerberos, confirm:

- `/etc/krb5.conf` is correct
- realm name is correct
- KDC is resolvable

---

## Certificates are credentials
Treat `.pfx` files as highly sensitive.  
If you have the PFX and password, you may effectively have the target identity.

---

## Machine accounts are high value
A certificate for a domain controller machine account can be more dangerous than a certificate for a normal user, because of what that machine can access and do.

---

# Detection / defensive relevance

Pass-the-Certificate techniques are especially important from a blue-team perspective because they often bypass expectations built around:

- password theft
- password spraying
- hash dumping

A defender may focus heavily on passwords and NTLM, while certificate abuse slips through a weaker AD CS monitoring posture.

Key defensive themes include:

- locking down AD CS templates
- disabling insecure web enrollment exposure
- requiring HTTPS where appropriate
- preventing NTLM relay paths
- auditing `msDS-KeyCredentialLink` changes
- monitoring certificate issuance patterns
- monitoring suspicious PKINIT usage

---

# Quick mental model

- **Password** proves identity the usual way
- **Hash** can substitute for password in some protocols
- **Ticket** can substitute for repeated authentication
- **Certificate** can be used to obtain the ticket in the first place

So:

> **Pass the Certificate = use a certificate to get a TGT, then continue with Pass the Ticket**
