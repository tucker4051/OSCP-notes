## Overview

Not all password attacks involve trying many passwords against one account.

In real environments, it is often more effective to abuse:

- weak organizational password habits
- password reuse across services
- unchanged vendor default credentials

This note covers three common attack patterns:

- **password spraying**
- **credential stuffing**
- **default credential testing**

These attacks are especially useful when:

- lockout policies exist
- many usernames are known
- leaked credentials are available
- enterprise devices or services were poorly configured

---

# 1. Password Spraying

## What It Is

Password spraying is an authentication attack where **one password** is tried against **many usernames**.

Instead of:

- many passwords against one user

you do:

- one likely password against many users

This helps reduce the chance of triggering account lockouts.

---

## Why It Works

It is effective when organizations use:

- common onboarding passwords
- seasonal passwords
- simple company-wide patterns
- unchanged temporary passwords

Examples:

- `ChangeMe123!`
- `Welcome123!`
- `Summer2024!`
- `CompanyName123!`

---

## Typical Use Cases

Password spraying is especially useful against:

- Active Directory
- O365 / webmail
- VPN portals
- Citrix / SSO
- SMB / WinRM / LDAP
- internal admin panels

---

## Example with NetExec

```bash
netexec smb 10.100.38.0/24 -u usernames.list -p 'ChangeMe123!'
```

This attempts the same password against multiple users across SMB-enabled hosts.

---

## Common Tools

| Tool | Best For |
|------|----------|
| NetExec | SMB, WinRM, LDAP, MSSQL, SSH |
| Kerbrute | Active Directory / Kerberos |
| Burp Suite | web login portals |
| Hydra | some services, depending on target |
| CrackMapExec / NetExec style tools | domain environments |

---

## Key Considerations

Password spraying must be done carefully because of:

- account lockout thresholds
- rate limiting
- SIEM alerts
- MFA challenges
- smart lockout protections

Good practice:

- use low volume
- spread attempts over time
- understand lockout policy first
- test one password at a time

---

# 2. Credential Stuffing

## What It Is

Credential stuffing is the use of **known username:password pairs** from one source against another service.

This attack relies on password reuse.

For example:

- leaked forum credentials
- breached email/password combos
- passwords found in internal documents
- credentials recovered from config files
- old employee account dumps

---

## Why It Works

Many users reuse passwords across:

- email
- social media
- cloud portals
- VPN
- SSH
- internal business applications

A password from one breach may still work elsewhere.

---

## Example with Hydra

If you have a file of `username:password` pairs:

```bash
hydra -C user_pass.list ssh://10.100.38.23
```

This tests each full credential pair directly against SSH.

---

## Credential Pair Format

Typical file format:

```text
alice:Password123!
bob:Summer2024!
carol:Welcome1
```

---

## Common Sources of Credential Pairs

- public breach dumps
- client-provided password audits
- post-exploitation loot
- browser credential stores
- cracked hashes
- exposed configuration files
- old tickets / documentation

---

## Common Targets

Credential stuffing is often effective against:

- SSH
- VPN
- OWA / webmail
- RDP gateways
- cloud SSO
- Git services
- admin portals
- internal web apps

---

## Key Considerations

Credential stuffing differs from spraying because:

- spraying = one password, many users
- stuffing = many user/password pairs as-is

This makes stuffing useful when you already have **high-confidence pairs**.

---

# 3. Default Credentials

## What They Are

Many products ship with predefined usernames and passwords.

Examples include:

- routers
- firewalls
- printers
- IoT devices
- databases
- NAS devices
- hypervisors
- web admin panels
- appliances

If administrators fail to change them, they become easy entry points.

---

## Why They Matter

Default credentials are still found in:

- lab environments
- internal test systems
- misconfigured appliances
- legacy devices
- rushed deployments
- forgotten internal infrastructure

These are often high-value targets because they may expose:

- admin panels
- device configuration
- secrets
- backups
- network routes
- VPN settings

---

## Searching Default Credentials

One useful tool is:

```bash
pip3 install defaultcreds-cheat-sheet
```

Then search by vendor or product:

```bash
creds search linksys
```

Example output:

```text
| linksys       | admin         | admin    |
| linksys (ssh) | admin         | password |
| linksys       | root          | orion99  |
```

---

## Other Sources for Default Credentials

Default credentials can often be found in:

- official product documentation
- vendor quick start guides
- setup manuals
- KB articles
- community cheat sheets
- device stickers / labels
- embedded web UI hints

---

These should never be assumed, but they are worth checking in scope-approved assessments.

---

# 4. Choosing the Right Attack Type

## Use Password Spraying When

- you have many usernames
- you suspect a shared default / seasonal password
- you want to avoid lockouts
- AD / enterprise auth is in scope

## Use Credential Stuffing When

- you already have username:password pairs
- users are likely to reuse passwords
- you want high-confidence login attempts

## Use Default Credential Testing When

- dealing with appliances, routers, printers, or admin panels
- documentation suggests defaults exist
- service appears freshly deployed or neglected

# 6. Practical Workflow

## Password Spraying

1. gather usernames
2. identify one likely password
3. test carefully across service
4. validate success manually
5. check access level

## Credential Stuffing

1. collect `username:password` pairs
2. normalize pair format
3. test against target service
4. validate hits
5. pivot if reuse is confirmed

## Default Credentials

1. identify product/vendor
2. research known defaults
3. test cautiously
4. validate access
5. enumerate device/service

---

# 7. Risks and Limitations

These attacks can be noisy and may trigger:

- account lockouts
- MFA prompts
- SIEM alerts
- IP bans
- service rate limiting

Always consider:

- lockout thresholds
- user impact
- target criticality
- scope constraints
- sequencing and timing

---

# 8. Key Takeaways

- password spraying tests one password against many users
- credential stuffing tests known user/password pairs against new services
- default credential testing targets vendor-supplied logins
- these methods are often more effective than raw brute force
- context, policy knowledge, and user behavior matter more than volume
