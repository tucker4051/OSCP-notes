## Overview

During post-exploitation, encrypted or password-protected files can often be discovered on compromised systems.

These files may contain:

- credentials
- API keys
- SSH keys
- configuration secrets
- internal documentation
- network diagrams
- database connection strings
- private certificates
- backups
- financial data

Many encrypted files can be cracked offline by extracting password hashes and attempting dictionary or rule-based attacks.

John the Ripper provides numerous utilities (`*2john`) to convert protected files into crackable hash formats.

---

# Hunting for Encrypted Files

Encrypted files often use standard extensions.

Common examples:

| File Type | Extensions |
|----------|------------|
| Microsoft Office | .doc .docx .xls .xlsx .ppt .pptx |
| OpenOffice | .odt .ods .odp |
| PDF | .pdf |
| Archives | .zip .rar .7z |
| Certificates | .pfx |
| Key stores | .keychain |
| Backup containers | .bak |

Example search:

```bash
for ext in $(echo ".xls .xls* .xltx .od* .doc .doc* .pdf .pot .pot* .pp*"); do
    echo -e "\nFile extension:" $ext
    find / -name *$ext 2>/dev/null | grep -v "lib\|fonts\|share\|core"
done
```

Purpose:

- recursively search filesystem
- identify potentially sensitive encrypted documents
- exclude noisy system directories

# Hunting for SSH Private Keys

SSH keys often do not use file extensions.

Instead, they can be identified by header values.

Typical private key headers:

```
-----BEGIN RSA PRIVATE KEY-----
-----BEGIN OPENSSH PRIVATE KEY-----
-----BEGIN EC PRIVATE KEY-----
```

Recursive search:

```bash
grep -rnE '^\-{5}BEGIN [A-Z0-9]+ PRIVATE KEY\-{5}$' / 2>/dev/null
```

Example output:

```text
/home/jsmith/.ssh/id_ed25519
/home/jsmith/.ssh/id_rsa
/home/jsmith/Documents/backup.key
```

---

# Checking if SSH Keys Are Encrypted

Some keys are protected with passphrases.

Test using:

```bash
ssh-keygen -yf ~/.ssh/id_ed25519
```

If encrypted:

```text
Enter passphrase for "/home/jsmith/.ssh/id_rsa":
```

Older PEM format keys explicitly show encryption:

```
Proc-Type: 4,ENCRYPTED
DEK-Info: AES-128-CBC
```

Modern OpenSSH keys do not visibly indicate encryption.

# Cracking SSH Keys

Convert SSH key to hash:

```bash
ssh2john.py SSH.private > ssh.hash
```

Crack with John:

```bash
john --wordlist=rockyou.txt ssh.hash
```

Show cracked password:

```bash
john ssh.hash --show
```

Example result:

```text
SSH.private:1234
```

# General Cracking Workflow

## Step 1 — identify protected file

Examples:

- SSH keys
- PDF reports
- Office documents
- archives
- password managers

## Step 2 — convert file to hash

```bash
<tool> file > hash.txt
```

## Step 3 — run cracking attack

```bash
john --wordlist=rockyou.txt hash.txt
```

or:

```bash
hashcat -a 0 -m <mode> hash.txt wordlist.txt
```

## Step 4 — review results

```bash
john hash.txt --show
```

# Why Cracking Files Matters

Protected files often contain sensitive information:

Examples:

- VPN credentials
- database connection strings
- internal IP ranges
- SSH keys
- AWS keys
- API tokens
- internal architecture diagrams
- admin passwords
- password reuse opportunities

Even a single cracked file may enable:

- lateral movement
- privilege escalation
- persistence
- domain compromise

---

# Wordlist Importance

File passwords are often:

- reused across services
- based on company naming patterns
- based on user habits
- compliant with password policy but predictable

Custom wordlists dramatically improve success probability.

Combine:

- OSINT-derived words
- company keywords
- seasonal patterns
- rule-based mutations

---

# Performance Considerations

Password-protected files may use:

- strong encryption
- high iteration counts
- key stretching

This can significantly slow cracking attempts.

Factors affecting success:

- password complexity
- hash algorithm strength
- iteration count
- wordlist quality
- hardware performance
