## Purpose

Web shells are server-side scripts that allow command execution through a web request.

They are commonly used after gaining the ability to upload, write, edit, or include files on a web server.

Web shells can provide:

- command execution
- file browsing
- file upload/download
- payload staging
- reverse shell triggering
- persistence during a lab
- a fallback when reverse shells are unstable

---

## Core Idea

A web shell works by placing executable server-side code in a web-accessible location.

The attacker then interacts with the shell through the browser, curl, Burp, or another HTTP client.

Common web shell languages include:

|Platform|Web Shell Type|
|---|---|
|PHP|`.php`, `.phtml`, `.php5`|
|ASP.NET|`.aspx`, `.ashx`, `.asmx`|
|Classic ASP|`.asp`|
|Java/Tomcat|`.jsp`, `.jspx`, `.war`|
|CGI|`.cgi`, `.pl`, `.sh`|

---

## Common Web Shell Entry Points

Web shells usually become possible through:

- file upload vulnerabilities
- weak upload validation
- writable web directories
- SQL injection file write
- LFI plus file upload
- RFI
- CMS theme or plugin editors
- admin panel file managers
- exposed deployment functionality
- misconfigured FTP mapped to web root
- writable SMB/NFS share mapped to web root

## Typical Web Shell Flow

1. Identify upload, write, edit, or inclusion capability.
2. Confirm the target server-side language.
3. Prepare a matching web shell.
4. Upload or write the web shell.
5. Identify the uploaded file path.
6. Access the web shell over HTTP.
7. Confirm command execution.
8. Identify current user and host context.
9. Use the shell to enumerate, stage payloads, or trigger a reverse shell.
10. Document the path and cleanup requirements.

---

## Web Shell vs Reverse Shell

|Access Type|Description|Use Case|
|---|---|---|
|Web shell|Commands sent over HTTP|Useful when reverse shells fail or outbound traffic is blocked|
|Reverse shell|Target connects back to attacker|Useful when outbound traffic is allowed and interactive access is needed|
|Bind shell|Target listens for attacker connection|Useful when inbound access to target is allowed|

A web shell is often less interactive than a reverse shell, but it can be more reliable and repeatable.

## Web Shell Validation

After uploading or writing a web shell, confirm:

- [ ]  File uploaded successfully
- [ ]  File path identified
- [ ]  File is accessible over HTTP
- [ ]  File is interpreted by the server
- [ ]  Commands execute successfully
- [ ]  Current user identified
- [ ]  Current working directory identified
- [ ]  File read works
- [ ]  File write works
- [ ]  Reverse shell trigger tested if needed
- [ ]  Cleanup requirements noted

---

## Useful Questions

Ask:

- What server-side language is supported?
- Is the uploaded file executed or downloaded?
- Is the file extension accepted?
- Is the upload path web-accessible?
- Was the file renamed?
- Is authentication required to access the shell?
- Are commands executed by the web server user?
- Are dangerous functions disabled?
- Can the web shell write files?
- Can the web shell trigger a reverse shell?
- Is the web shell stable enough for enumeration?
- Can this access be regained if the shell dies?

---

## Common Web Shell Problems

|Problem|Likely Cause|
|---|---|
|File uploads but returns 404|Wrong path or file renamed|
|File downloads instead of executing|Server does not execute that extension|
|Source code is displayed|Server-side language not configured for that file type|
|403 Forbidden|Access controls or file permissions|
|Commands do not run|Disabled functions, sandboxing, wrong parameter|
|Reverse shell fails|Egress filtering, wrong callback IP, listener issue|
|Shell disappears|Cleanup process, temporary upload path, AV/EDR, app behaviour|
|Only limited commands work|Restricted web server user or command filtering|

## Common Web Shell Uses

Once confirmed, a web shell can be used to:

- identify the current user
- identify the hostname
- identify the operating system
- list directories
- read configuration files
- search for credentials
- upload additional payloads
- trigger a reverse shell
- check network connectivity
- inspect web root permissions
- enumerate local users
- prepare for privilege escalation

---

## Security Context

Web shells usually execute as the web server or application user.

Common examples:

|Platform|Common User Context|
|---|---|
|Linux Apache/Nginx/PHP|`www-data`, `apache`, `nginx`|
|Tomcat|`tomcat`|
|Windows IIS|`iis apppool\...`, `IUSR`, `NETWORK SERVICE`|
|XAMPP/WAMP-style installs|service-specific or local user context|

This context is often low-privileged, so local enumeration is required after access.

## Web Shell to Reverse Shell

A web shell can often be used to trigger a reverse shell.

Before doing this, confirm:

- [ ]  Command execution works
- [ ]  Target OS is known
- [ ]  Target can reach attacker IP
- [ ]  Correct callback IP selected
- [ ]  Correct callback port selected
- [ ]  Listener started
- [ ]  Payload syntax matches target OS/shell
- [ ]  URL encoding handled
- [ ]  Egress filtering considered

## Common Mistakes

- [ ]  Assuming upload success means execution
- [ ]  Not identifying the uploaded file path
- [ ]  Not checking whether the file was renamed
- [ ]  Using the wrong server-side language
- [ ]  Using the wrong file extension
- [ ]  Ignoring MIME/type restrictions
- [ ]  Not testing simple command execution first
- [ ]  Trying a reverse shell before validating the web shell
- [ ]  Using the wrong callback IP
- [ ]  Forgetting URL encoding
- [ ]  Forgetting to document how to regain access
- [ ]  Forgetting cleanup where appropriate
