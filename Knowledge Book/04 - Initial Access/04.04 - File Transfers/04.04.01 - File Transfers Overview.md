## Purpose

File transfer techniques are used to move files to or from a target after gaining initial access or partial command execution.

This may include transferring:

- payloads
- scripts
- enumeration tools
- exploit binaries
- web shells
- configuration files
- logs
- database dumps
- password hashes
- proof/evidence files

File transfers are often required when a basic shell or web shell does not provide enough functionality on its own.

---

## Core Idea

Once you have access, you often need to move files.

The key questions are:

- Am I uploading to the target or downloading from it?
- What operating system is the target?
- What tools are available on the target?
- Is outbound network access allowed?
- Is inbound network access allowed?
- Can I host files locally?
- Can I use HTTP, SMB, FTP, or another protocol?
- Are there restrictions or detection controls?
- Are file permissions blocking access?

---

## Upload vs Download

|Direction|Meaning|Example Use|
|---|---|---|
|Upload|Move a file from attacker machine to target|Transfer payloads, tools, scripts|
|Download|Move a file from target to attacker machine|Exfiltrate configs, hashes, logs, loot|

Both directions matter.

In labs, uploading tools is useful for enumeration and privilege escalation. Downloading files is useful for offline analysis, cracking hashes, reviewing source code, and preserving evidence.

## LOLBins

LOLBins are trusted native binaries that can be used to perform actions such as downloading, uploading, executing, or decoding files.

They are useful when:

- standard tools are missing
- direct transfer methods are blocked
- security controls are present
- native tooling is less suspicious
- the target has restricted tooling

## Protected Files

Protected files may require special handling.

Examples include:

- files owned by another user
- locked files
- files requiring elevated privileges
- Windows registry hives
- database files in use
- files in restricted directories
- sensitive configuration files
- files blocked by permissions

Important considerations:

- current user permissions
- whether the file is locked
- whether a copy can be made
- whether shadow copies/backups exist
- whether elevated privileges are required
- whether the file can be read indirectly

## Detection and Restrictions

File transfers may be blocked, logged, or detected.

Potential issues:

- outbound HTTP blocked
- SMB blocked
- PowerShell restricted
- AV/EDR removes tools
- uploaded binaries quarantined
- web proxy restrictions
- file extension restrictions
- execution blocked
- transfer method logged
- suspicious filenames
- known offensive tools detected

In CTFs this may matter less, but it is still useful to understand why transfers fail.

## File Transfer Flow

1. Identify the target operating system.
2. Identify available tools.
3. Decide upload or download direction.
4. Choose the simplest transfer method.
5. Host or prepare the file.
6. Transfer the file.
7. Verify the file arrived intact.
8. Set permissions if needed.
9. Execute or analyse the file.
10. Clean up where appropriate.

## Common Transfer Problems

|Problem|Likely Cause|
|---|---|
|File does not download|Wrong IP, blocked outbound, bad URL, hosting issue|
|File downloads as HTML|Wrong path or server returned error page|
|Permission denied|Destination not writable or file protected|
|File executes incorrectly|Wrong architecture, wrong format, missing runtime|
|AV deletes file|Signature detection or blocked tooling|
|SMB fails|Firewall, auth issue, SMB version/signing issue|
|PowerShell fails|Execution policy, quoting, proxy, AMSI/Defender|
|File corrupted|Binary transferred in text mode or encoding issue|

---

## Verification

After transfer, verify:

- [ ]  File exists
- [ ]  File size looks correct
- [ ]  File type looks correct
- [ ]  Hash matches, if needed
- [ ]  Permissions are correct
- [ ]  File is executable, if required
- [ ]  File runs as expected
- [ ]  Transfer did not create an error page instead of the real file

## Common Mistakes

- [ ]  Using the wrong attacker IP
- [ ]  Hosting files on the wrong interface
- [ ]  Downloading an error page instead of the intended file
- [ ]  Forgetting to make Linux files executable
- [ ]  Uploading a Windows binary to Linux

- [ ]  Assuming wget/curl are always installed
- [ ]  Ignoring PowerShell quoting issues

- [ ]  Not cleaning up obvious payloads or tools
