## Overview

Before a password spraying attack, you need a **clean list of valid domain users**.

A spray without a reliable user list is like trying keys at random in a huge office block. You waste time, create noise, and raise the chance of detection.

A strong target user list helps you:

- reduce failed authentication noise
- improve spray success rate
- avoid unnecessary lockouts
- track who has and has not been tested
- build a repeatable workflow

---

# Ways to Build a Target User List

There are several common ways to gather valid domain usernames:

- **SMB NULL session** to enumerate domain users from a DC
- **LDAP anonymous bind** to query LDAP without credentials
- **Kerbrute** username validation with a wordlist
- **Credentialed enumeration** using a valid domain account
- **External user generation** from LinkedIn, email formats, or naming conventions

Which method you use depends on what access you currently have.

---

# Why the Password Policy Still Matters

Before spraying, always consider the **domain password policy**.

Why it matters:

- **minimum password length** helps shape likely password candidates
- **complexity requirements** tell you what formats are acceptable
- **lockout threshold** tells you how many attempts are dangerous
- **reset timer** tells you how long to wait between rounds

If you do not know the policy:

- ask the client if appropriate
- stay extremely conservative
- consider a single targeted “hail mary” attempt only
- wait hours between attempts if unsure

Never assume the environment uses the “usual” settings.

---

# Logging Requirements

Always keep a detailed log of spraying activity.

Record at minimum:

- accounts targeted
- Domain Controller used
- date of the spray
- time of the spray
- password(s) attempted

## Why this matters

Good notes help you:

- avoid duplicate effort
- avoid accidental repeated attempts
- explain suspicious logon events to the client
- verify whether lockouts were caused by your activity or something else

Treat your spray log like a flight recorder.

---

# SMB NULL Session Enumeration

If you are internal and do **not** have valid domain credentials, check for:

- **SMB NULL sessions**
- **LDAP anonymous binds**

If either is available, you may be able to retrieve:

- full user lists
- password policy
- other domain data

If you already have:

- valid domain credentials, or
- SYSTEM on a Windows machine

then user enumeration becomes much easier.

---

## Why SYSTEM can work

A Windows computer account is treated similarly to a domain user account in many situations.

So if you have **SYSTEM** on a domain-joined host, you may be able to query AD using the machine’s trust relationship.

---

# Using enum4linux

`enum4linux` can pull domain users over SMB NULL session access.

## Command

```bash
enum4linux -U 172.16.5.5 | grep "user:" | cut -f2 -d"[" | cut -f1 -d"]"
```

## Example output

```text
administrator
guest
krbtgt
lab_adm
htb-student
avazquez
pfalcon
fanthony
wdillard
lbradford
sgage
asanchez
dbranch
ccruz
njohnson
mholliday
```

## What this does

- `enum4linux -U` requests user enumeration
- `grep "user:"` keeps only lines containing user entries
- the `cut` commands strip the extra formatting
- final result is one username per line

## Why it is useful

This is quick and effective when anonymous SMB access is enabled.

It gives you a cleaner list that can be saved directly into a spray wordlist.

---

# Using rpcclient

`rpcclient` is another reliable option for SMB NULL session enumeration.

## Connect anonymously

```bash
rpcclient -U "" -N 172.16.5.5
```

## Enumerate users

```bash
enumdomusers
```

## Example output

```text
user:[administrator] rid:[0x1f4]
user:[guest] rid:[0x1f5]
user:[krbtgt] rid:[0x1f6]
user:[lab_adm] rid:[0x3e9]
user:[htb-student] rid:[0x457]
user:[avazquez] rid:[0x458]
```

## Why this is useful

This method is direct and dependable.

You can also use `rpcclient` to retrieve:

- password policy
- domain info
- shares
- groups
- aliases

If NULL session access exists, `rpcclient` is one of the most valuable tools in your kit.

# Using CrackMapExec with NULL Session Access

CrackMapExec can enumerate users with the `--users` flag.

## Command

```bash
crackmapexec smb 172.16.5.5 --users
```

## Example output

```text
INLANEFREIGHT.LOCAL\administrator  badpwdcount: 0  baddpwdtime: 2022-01-10 13:23:09.463228
INLANEFREIGHT.LOCAL\guest          badpwdcount: 0  baddpwdtime: 1600-12-31 19:03:58
INLANEFREIGHT.LOCAL\lab_adm        badpwdcount: 0  baddpwdtime: 2021-12-21 14:10:56.859064
INLANEFREIGHT.LOCAL\krbtgt         badpwdcount: 0  baddpwdtime: 1600-12-31 19:03:58
INLANEFREIGHT.LOCAL\htb-student    badpwdcount: 0  baddpwdtime: 2022-02-22 14:48:26.653366
INLANEFREIGHT.LOCAL\avazquez       badpwdcount: 0  baddpwdtime: 2022-02-17 22:59:22.684613
```

## Why this is especially useful

CrackMapExec gives you more than just usernames.

It also shows:

- `badpwdcount` → current failed password count
- `baddpwdtime` → time of last bad password attempt

## Why that matters

This helps you filter out dangerous targets.

For example, if the lockout threshold is **5** and an account already has **4** failed attempts, spraying that account is reckless.

Think of this output as a **fuel gauge for lockout risk**.

---

## Important caveat: multiple Domain Controllers

In environments with multiple DCs:

- `badpwdcount` is tracked separately on each DC
- `baddpwdtime` may differ between DCs

For the most accurate picture, ideally query:

- each Domain Controller, or
- the **PDC Emulator**

The PDC Emulator is usually the safest place to check for authoritative password-related state.

# LDAP Anonymous Bind Enumeration

If LDAP anonymous bind is allowed, you can enumerate users without credentials.

Useful tools include:

- `ldapsearch`
- `windapsearch`
- `ad-ldapdomaindump.py`

---

## Using ldapsearch

```bash
ldapsearch -h 172.16.5.5 -x -b "DC=INLANEFREIGHT,DC=LOCAL" -s sub "(&(objectclass=user))" | grep sAMAccountName: | cut -f2 -d" "
```

## Example output

```text
guest
ACADEMY-EA-DC01$
ACADEMY-EA-MS01$
ACADEMY-EA-WEB01$
htb-student
avazquez
pfalcon
fanthony
wdillard
lbradford
sgage
asanchez
dbranch
```

## Important note

This output includes **computer accounts** too, which end with `$`.

Examples:

- `ACADEMY-EA-DC01$`
- `ACADEMY-EA-MS01$`

These should usually be removed from a human-user spray list unless you are specifically targeting machine accounts.

---

## Quick cleanup to remove computer accounts

```bash
ldapsearch -h 172.16.5.5 -x -b "DC=INLANEFREIGHT,DC=LOCAL" -s sub "(&(objectclass=user))" | grep sAMAccountName: | cut -f2 -d" " | grep -v '\$$'
```

This keeps only usernames that do **not** end in `$`.

---

# Using windapsearch

`windapsearch` is often easier and cleaner than raw `ldapsearch`.

## Command

```bash
./windapsearch.py --dc-ip 172.16.5.5 -u "" -U
```

## Example output

```text

[+] Found: DC=INLANEFREIGHT,DC=LOCAL
[+] Attempting bind
[+] ...success! Binded as:
[+] None

[+] Enumerating all AD users
[+] Found 2906 users:
```

Example entries:

```text
cn: Guest

cn: Htb Student
userPrincipalName: htb-student@inlanefreight.local

cn: Annie Vazquez
userPrincipalName: avazquez@inlanefreight.local

cn: Paul Falcon
userPrincipalName: pfalcon@inlanefreight.local
```

## Why this is useful

`windapsearch` gives cleaner, more readable output and saves you some of the filter-building pain of `ldapsearch`.

Still, it is worth understanding raw LDAP filters so you are not dependent on wrapper tools.

# Enumerating Users with Kerbrute

If you have **no credentials** and **no anonymous SMB/LDAP access**, Kerbrute is one of the best options for username enumeration.

Kerbrute works by abusing **Kerberos pre-authentication behavior**.

---

## How username enumeration works

The tool requests a TGT **without pre-authentication**.

The KDC response tells you a lot:

- if it returns **PRINCIPAL UNKNOWN**, the username is invalid
- if it asks for **pre-authentication**, the username exists

This is useful because **username enumeration itself does not cause a failed logon** and will not lock out accounts.

That makes it much safer than blindly attempting passwords.

---

## Kerbrute user enumeration command

```bash
kerbrute userenum -d inlanefreight.local --dc 172.16.5.5 /opt/jsmith.txt
```

## Example output

```text
[+] VALID USERNAME: jjones@inlanefreight.local
[+] VALID USERNAME: sbrown@inlanefreight.local
[+] VALID USERNAME: tjohnson@inlanefreight.local
[+] VALID USERNAME: jwilson@inlanefreight.local
[+] VALID USERNAME: bdavis@inlanefreight.local
[+] VALID USERNAME: njohnson@inlanefreight.local
[+] VALID USERNAME: asanchez@inlanefreight.local
[+] VALID USERNAME: dlewis@inlanefreight.local
[+] VALID USERNAME: ccruz@inlanefreight.local
```

## Why it is powerful

In the example, over **48,000** potential usernames were tested in a matter of seconds.

That makes Kerbrute excellent for validating naming conventions such as:

- `flast`
- `first.last`
- `firstlast`
- `f.lastname`

---

## Detection note

Kerbrute username enumeration does **not** trigger:

- Event ID **4625** (failed logon)

But it can generate:

- Event ID **4768** — a TGT was requested

This only appears if Kerberos logging is enabled.

### Why this matters

From a defender point of view, spikes in 4768 events may indicate username enumeration.

That makes successful Kerbrute use a good reporting point for detection improvement.

---

## Critical caution

Once you switch from **user enumeration** to **password spraying** with Kerbrute, failed pre-authentication attempts **do count** toward lockouts.

So:

- username enumeration is relatively safe
- password spraying is **not**

Do not confuse the two.

# External Username Generation

If internal enumeration fails, you can fall back to external recon.

Possible sources:

- public email addresses
- LinkedIn
- company naming conventions
- harvested names from documents / breaches / OSINT

Useful tools/resources:

- `linkedin2username`
- `statistically-likely-usernames`
- company email format guesses

This list will not be as complete or accurate as AD-based enumeration, but it may be enough to gain a foothold.

# Credentialed Enumeration

If you already have valid credentials, user enumeration becomes much simpler.

A fast method is CrackMapExec.

## Example

```bash
crackmapexec smb 172.16.5.5 -u htb-student -p Academy_student_AD! --users
```

## Example output highlights

```text
INLANEFREIGHT.LOCAL\administrator  badpwdcount: 1
INLANEFREIGHT.LOCAL\guest          badpwdcount: 0
INLANEFREIGHT.LOCAL\lab_adm        badpwdcount: 0
INLANEFREIGHT.LOCAL\krbtgt         badpwdcount: 0
INLANEFREIGHT.LOCAL\htb-student    badpwdcount: 0
INLANEFREIGHT.LOCAL\avazquez       badpwdcount: 20
INLANEFREIGHT.LOCAL\pfalcon        badpwdcount: 0
```

## Why this is valuable

This gives you:

- accurate domain usernames
- bad password counts
- better operational safety than blind spraying

### Important observation

If an account shows something like:

```text
avazquez  badpwdcount: 20
```

that is an immediate red flag.

This account may already be under attack, misconfigured, or heavily tested. It should be handled very carefully or excluded from early spray attempts.

# Building a Clean Spray List

Raw enumeration output is messy. Your goal is a **flat username list**, one entry per line.

Good cleanup steps:

- remove computer accounts ending in `$`
- remove built-in accounts if not useful
- remove disabled or obviously sensitive accounts if appropriate
- remove accounts close to lockout threshold
- normalize username format if needed

---

## Common accounts to think carefully about

Examples:

- `administrator`
- `guest`
- `krbtgt`
- service accounts
- admin-tier accounts
- highly privileged break-glass accounts

These may be:

- sensitive
- monitored
- protected
- poor initial spray targets

Sometimes the best early targets are ordinary user accounts, not crown-jewel accounts.

---

# Suggested Filtering Mindset

## Good early spray targets

- standard employee accounts
- active users with low `badpwdcount`
- users matching known naming convention
- users likely to choose seasonal / default / welcome-style passwords

## Accounts to avoid early

- accounts near threshold
- admin accounts
- service accounts
- machine accounts
- disabled accounts
- special built-ins such as `krbtgt`

# Practical Workflow

A sensible workflow often looks like this:

1. Check for SMB NULL session
2. Check for LDAP anonymous bind
3. Pull password policy if possible
4. Enumerate users with NULL session / LDAP if possible
5. If not, validate usernames with Kerbrute
6. If you have creds, enumerate with CrackMapExec
7. Clean the list
8. Remove risky targets
9. Log everything
10. Only then begin spraying

This keeps the process controlled instead of noisy.

---

# Key Takeaways

- A good spray starts with a good user list
- NULL sessions and anonymous binds are gold if available
- Kerbrute is excellent for username validation without immediate lockout risk
- CrackMapExec gives extra operational context with `badpwdcount`
- Always consider the password policy before spraying
- Always log what you did

The user list is not just a list of names. It is the **target map** for the attack.
