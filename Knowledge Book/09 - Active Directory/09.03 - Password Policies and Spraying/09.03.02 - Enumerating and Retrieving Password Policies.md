## Overview

Enumerating the **domain password policy** is one of the first things to do in an Active Directory assessment.

This tells you:

- minimum password length
- password complexity requirements
- account lockout threshold
- lockout duration
- password history
- whether passwords expire

Why this matters:

The password policy directly shapes how aggressive or cautious you can be with:

- password spraying
- brute force attempts
- credential validation
- lockout risk management

Think of the password policy as the **rules of the road** before you drive into a spraying attack. If you do not know the speed limit, you are much more likely to crash.

---

# Enumeration Paths

You can retrieve the password policy in several ways depending on:

- whether you have valid credentials
- whether SMB NULL sessions are enabled
- whether LDAP anonymous bind is enabled
- whether you are working from Linux or Windows

Common paths:

- **Linux, credentialed**
- **Linux, SMB NULL session**
- **Linux, LDAP anonymous bind**
- **Windows, built-in tools**
- **Windows, PowerView / other tooling**

---

# Linux - Credentialed Enumeration

If you have valid domain credentials, the easiest way is often through **CrackMapExec** or **rpcclient**.

## CrackMapExec

```bash
crackmapexec smb 172.16.5.5 -u avazquez -p Password123 --pass-pol
```

### Example output highlights

```text
Minimum password length: 8
Password history length: 24
Maximum password age: Not Set
Password Complexity Flags: 000001
Minimum password age: 1 day 4 minutes
Reset Account Lockout Counter: 30 minutes
Locked Account Duration: 30 minutes
Account Lockout Threshold: 5
Forced Log off Time: Not Set
```

## What this gives you

This is usually the fastest and cleanest credentialed approach from Linux because it clearly displays:

- minimum password length
- history length
- complexity
- lockout threshold
- lockout duration
- reset window

### Key interpretation from this example

- minimum password length is **8**
- complexity is **enabled**
- lockout threshold is **5**
- lockout reset window is **30 minutes**
- lockout duration is **30 minutes**
- maximum password age is **not set** → passwords may never expire

# Linux - SMB NULL Session Enumeration

## What is an SMB NULL session?

An **SMB NULL session** is an unauthenticated connection that allows anonymous access to certain SMB/RPC functionality.

If enabled, it may reveal:

- users
- groups
- shares
- domain info
- password policy

This is usually a **legacy misconfiguration**.

It often appears when:

- older Domain Controllers were upgraded in place
- legacy anonymous access settings were never cleaned up
- backward compatibility was prioritized over security

---

## Using rpcclient to test NULL session access

```bash
rpcclient -U "" -N 172.16.5.5
```

Once connected:

```bash
querydominfo
```

### Example

```text
Domain:         INLANEFREIGHT
Total Users:    3650
Server Role:    ROLE_DOMAIN_PDC
```

This confirms that anonymous RPC access is possible.

---

## Retrieving the password policy with rpcclient

```bash
getdompwinfo
```

### Example output

```text
min_password_length: 8
password_properties: 0x00000001
    DOMAIN_PASSWORD_COMPLEX
```

## What this tells you

Even a very limited anonymous result can still reveal two critical details:

- minimum password length
- whether complexity is enabled

That alone can be enough to tune a spraying strategy.

# Linux - enum4linux

`enum4linux` wraps several Samba-related tools and is useful for quickly checking NULL session access and pulling policy data.

## Tool/Port quick reference

| Tool | Ports |
|---|---|
| nmblookup | 137/UDP |
| nbtstat | 137/UDP |
| net | 139/TCP, 135/TCP, TCP and UDP 135 and 49152-65535 |
| rpcclient | 135/TCP |
| smbclient | 445/TCP |

## Using enum4linux

```bash
enum4linux -P 172.16.5.5
```

### Example output highlights

```text
Minimum password length: 8
Password history length: 24
Maximum password age: Not Set
Password Complexity Flags: 000001
Minimum password age: 1 day 4 minutes
Reset Account Lockout Counter: 30 minutes
Locked Account Duration: 30 minutes
Account Lockout Threshold: 5
Forced Log off Time: Not Set
```

### Why it is useful

`enum4linux` is convenient when you want:

- quick validation of NULL session access
- password policy output
- general SMB/RPC enumeration in one place

---

# Linux - enum4linux-ng

`enum4linux-ng` is a modern Python rewrite of enum4linux.

Advantages:

- cleaner output
- colored display
- JSON export
- YAML export
- easier downstream processing

## Usage

```bash
enum4linux-ng -P 172.16.5.5 -oA ilfreight
```

### Example output highlights

```text
[+] Server allows session using username '', password ''
[+] Domain: INLANEFREIGHT
[+] SID: S-1-5-21-3842939050-3880317879-2865463114
```

Policy output:

```yaml
domain_password_information:
  pw_history_length: 24
  min_pw_length: 8
  min_pw_age: 1 day 4 minutes
  max_pw_age: not set
  pw_properties:
  - DOMAIN_PASSWORD_COMPLEX: true
```

Lockout information:

```yaml
domain_lockout_information:
  lockout_observation_window: 30 minutes
  lockout_duration: 30 minutes
  lockout_threshold: 5
```

### Why this is useful

This format is easier to read and easier to feed into later analysis.

Exported JSON/YAML can be used for:

- note taking
- reporting
- automation
- parsing in custom scripts

# Linux - LDAP Anonymous Bind

## What is LDAP anonymous bind?

An **LDAP anonymous bind** allows unauthenticated queries to LDAP.

If enabled, it may expose:

- users
- groups
- computers
- attributes
- password policy

This is another **legacy or overly permissive configuration**.

Since Windows Server 2003 and later, anonymous LDAP access is much more restricted by default, but it still shows up from time to time.

---

## Using ldapsearch

```bash
ldapsearch -h 172.16.5.5 -x -b "DC=INLANEFREIGHT,DC=LOCAL" -s sub "*" | grep -m 1 -B 10 pwdHistoryLength
```

> In newer versions of `ldapsearch`, `-h` is deprecated in favor of `-H`.

### Example output

```text
forceLogoff: -9223372036854775808
lockoutDuration: -18000000000
lockOutObservationWindow: -18000000000
lockoutThreshold: 5
maxPwdAge: -9223372036854775808
minPwdAge: -864000000000
minPwdLength: 8
pwdProperties: 1
pwdHistoryLength: 24
```

## What this tells you

Important values here:

- `minPwdLength: 8`
- `lockoutThreshold: 5`
- `pwdProperties: 1` → complexity enabled
- `pwdHistoryLength: 24`

### Important note

LDAP values are often shown in raw AD format and may require interpretation.

For example:

- large negative numbers often represent time intervals
- `pwdProperties: 1` usually maps to password complexity enabled

# Windows - NULL Session Check

It is less common to do NULL session testing from Windows, but it is still possible.

## Create a NULL session

```cmd
net use \\DC01\ipc$ "" /u:""
```

### Example success

```text
The command completed successfully.
```

This confirms that anonymous access to IPC$ is allowed.

---

## Common authentication errors

### Disabled account

```cmd
net use \\DC01\ipc$ "" /u:guest
```

```text
System error 1331 has occurred.
This user can't sign in because this account is currently disabled.
```

### Wrong password

```cmd
net use \\DC01\ipc$ "password" /u:guest
```

```text
System error 1326 has occurred.
The user name or password is incorrect.
```

### Locked out account

```cmd
net use \\DC01\ipc$ "password" /u:guest
```

```text
System error 1909 has occurred.
The referenced account is currently locked out and may not be logged on to.
```

## Why these errors matter

These responses can help distinguish between:

- invalid credentials
- disabled accounts
- lockouts
- possible spraying impact

That makes them useful for both enumeration and operational safety.

# Windows - Built-in Password Policy Enumeration

If you are on a Windows host and authenticated to the domain, the simplest built-in command is:

```cmd
net accounts
```

### Example output

```text
Force user logoff how long after time expires?:       Never
Minimum password age (days):                          1
Maximum password age (days):                          Unlimited
Minimum password length:                              8
Length of password history maintained:                24
Lockout threshold:                                    5
Lockout duration (minutes):                           30
Lockout observation window (minutes):                 30
Computer role:                                        SERVER
```

## What this tells you

From this alone you can infer:

- passwords may never expire
- minimum password length is 8
- accounts lock out after 5 bad attempts
- lockout window resets after 30 minutes
- locked accounts auto-unlock after 30 minutes

### Why this matters

This is excellent spraying intelligence using only native Windows tooling.

It is especially useful when:

- you cannot upload tools
- AV/EDR is strict
- you need to live off the land

# Windows - PowerView

PowerView gives a more structured policy view and includes extra context such as the GPO path.

## Usage

```powershell
Import-Module .\PowerView.ps1
Get-DomainPolicy
```

### Example output highlights

```powershell
SystemAccess   : @{MinimumPasswordAge=1; MaximumPasswordAge=-1; MinimumPasswordLength=8; PasswordComplexity=1;
                 PasswordHistorySize=24; LockoutBadCount=5; ResetLockoutCount=30; LockoutDuration=30;
                 RequireLogonToChangePassword=0; ForceLogoffWhenHourExpire=0; ClearTextPassword=0;
                 LSAAnonymousNameLookup=0}
```

Also:

```powershell
Path           : \\INLANEFREIGHT.LOCAL\sysvol\INLANEFREIGHT.LOCAL\Policies\{31B2F340-016D-11D2-945F-00C04FB984F9}\MACHINE\Microsoft\Windows NT\SecEdit\GptTmpl.inf
GPODisplayName : Default Domain Policy
```

## What this gives you

PowerView confirms the same core details as `net accounts`, but also gives:

- direct policy object path
- GPO identifier
- policy source context
- Kerberos policy values too

This makes it excellent for AD-focused enumeration.

# Analysing the Policy

Using the example policy, we can break it down like this.

## Minimum password length: 8

This is common, but still weak by modern standards.

Why it matters:

- 8-character minimum allows many weak enterprise-style passwords
- examples like `Welcome1` or `Password1` still satisfy complexity

A longer minimum like 12–14 reduces low-effort password options, but does **not** eliminate password spraying.

---

## Password complexity: enabled

This means passwords generally must satisfy **3 of 4** character classes:

- uppercase
- lowercase
- number
- special character

### Important point

Complexity does **not** equal strong.

Examples like these may still be allowed:

- `Welcome1`
- `Summer2024!`
- `Password1`

These are “complex” but still weak and predictable.

---

## Lockout threshold: 5

This means the account locks after **5 bad password attempts**.

Why it matters:

This is the most important spraying safety number.

You should stay **well below** it.

A practical rule is often:

- **1 spray**
- maybe **2**
- sometimes **3 max**
- then wait beyond the reset window

---

## Lockout observation window: 30 minutes

This means failed attempts are counted inside a rolling 30-minute window.

Why it matters:

If you try too many passwords within that time, the failed count accumulates and triggers lockout.

---

## Lockout duration: 30 minutes

Locked accounts automatically unlock after 30 minutes.

Why it matters:

This reduces long-term damage, but you still do **not** want to lock out accounts.

Even auto-unlocking accounts can:

- alert defenders
- disrupt users
- impact business operations
- damage the assessment

---

## Maximum password age: Not Set / Unlimited

This means passwords may never expire.

Why it matters:

Old passwords may remain valid for a very long time.

That increases the chance that:

- old seasonal passwords still work
- common defaults persist
- stale passwords are reused elsewhere

---

# Why This Policy is Good for Password Spraying

Given:

- minimum length = 8
- complexity = enabled
- lockout threshold = 5
- reset window = 30 minutes

This policy is favorable for **careful password spraying**.

Examples of candidate passwords might include:

- `Welcome1`
- `Spring2024!`
- `Company01!`

### Safe operational thinking

Because the threshold is **5**, a cautious operator might do:

- 1 spray, then wait
- or at most 2–3 sprays spaced out safely
- wait **31+ minutes** between rounds

The goal is **zero lockouts**.

---

# Default AD Password Policy

When a new domain is created, the default policy is typically:

| Policy | Default Value |
|---|---|
| Enforce password history | 24 days |
| Maximum password age | 42 days |
| Minimum password age | 1 day |
| Minimum password length | 7 |
| Password must meet complexity requirements | Enabled |
| Store passwords using reversible encryption | Disabled |
| Account lockout duration | Not set |
| Account lockout threshold | 0 |
| Reset account lockout counter after | Not set |

## Why this matters

Some organizations never change the defaults.

That can mean:

- weak length requirements
- no lockout threshold at all
- predictable password behavior

A missing lockout threshold is especially dangerous because it may allow broad password attacks without account lockout protection.

---

# Operational Guidance

## If you know the policy

You can tune your spraying pace carefully and rationally.

## If you do not know the policy

Use extreme caution.

Recommended mindset:

- assume the threshold could be **3**
- assume lockouts may require **manual admin intervention**
- assume the organization may notice immediately

A conservative rule:

- perform **one** spray
- maybe **two max**
- wait **over an hour** between attempts if unsure

Never assume a lockout threshold of 5 unless you have verified it.

---

# Why Lockouts Are So Dangerous

You do **not** want to be the tester who locks out the organization.

Potential consequences:

- user disruption
- service account outages
- IT investigation
- blue team alerting
- broken trust with the client

A single successful spray is useful.

A mass lockout is memorable for all the wrong reasons.

# What to Record in Your Notes

Whenever you enumerate a password policy, capture:

- domain name
- target DC / source host
- min password length
- complexity enabled/disabled
- password history length
- max password age
- min password age
- lockout threshold
- reset window
- lockout duration
- whether accounts auto-unlock
- method used to retrieve it
- whether access was:
  - credentialed
  - NULL session
  - LDAP anonymous bind
  - built-in Windows tooling

This will make later spraying decisions much safer and much faster.

---

# Practical Takeaway

The password policy is not just “background info.”

It is your **risk map** for password attacks.

Without it, spraying is guesswork.  
With it, spraying becomes measured, controlled, and much safer.
