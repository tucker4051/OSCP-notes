## Overview

At this stage, we have already completed our initial enumeration of the domain.

We have:

- gathered basic user and group information
- identified important hosts and services
- learned the naming scheme in use
- built a rough picture of the domain

Now we move into a very common foothold technique:

- **LLMNR/NBT-NS poisoning**
- alongside **password spraying**

The goal here is simple:

- capture authentication material
- crack it offline if possible
- recover valid cleartext credentials
- gain a foothold in the domain

This is a classic internal attack path because it can work **without valid domain credentials** and often succeeds simply by waiting for user mistakes or name resolution failures.

---

# What Are LLMNR and NBT-NS?

## LLMNR

**LLMNR** stands for **Link-Local Multicast Name Resolution**.

It is a Microsoft Windows name resolution mechanism used when DNS fails.

- Uses **UDP 5355**
- Based on DNS-style name queries
- Allows hosts on the same local link to ask each other for name resolution

## NBT-NS

**NBT-NS** stands for **NetBIOS Name Service**.

It is another fallback name resolution mechanism used by Windows when DNS and/or LLMNR do not resolve a name.

- Uses **UDP 137**
- Resolves NetBIOS names on the local network

---

## Why they are dangerous

The key weakness is this:

> **Any host on the local network can answer these requests.**

So if a victim asks:

- “Does anyone know where `printer01` is?”

our attack box can simply say:

- “Yes, that’s me.”

This lets us impersonate the requested system and capture authentication attempts.

Think of it like shouting a question into a crowded room and trusting the first person who answers.

---

# High-Level Attack Flow

A simplified example looks like this:

1. A user tries to connect to:

   ```text
   \\print01.inlanefreight.local
   ```

2. They mistype it as:

   ```text
   \\printer01.inlanefreight.local
   ```

3. DNS fails to resolve the host.

4. The victim broadcasts to the local network asking if anyone knows the host.

5. Our machine, running **Responder**, replies pretending to be that host.

6. The victim trusts the response and sends an authentication attempt.

7. We capture:

   - **NetNTLMv1** or **NetNTLMv2** hash
   - and in some cases potentially other auth material

8. We then:

   - crack the hash offline
   - or possibly relay it in an SMB relay attack if conditions allow

---

# Why This Matters

This attack can provide:

- low-privileged domain credentials
- local admin hashes
- occasionally privileged user authentication material
- footholds for further AD enumeration
- material for relay attacks

Even a single cracked password can unlock:

- SMB access
- WinRM access
- RDP access
- LDAP enumeration
- password reuse across services
- password spraying validation

---

# What We Are Collecting

The main goal is to capture:

- **NTLMv1**
- **NTLMv2**
- other auth handshakes depending on protocol

These are useful because they may reveal:

- the user account
- the domain
- a crackable challenge-response pair

If cracked, we gain the user’s **cleartext password**.

# Responder Primer

## What Responder does

Responder is a Python-based tool commonly used from a Linux attack host to:

- listen for name resolution requests
- poison replies
- impersonate services
- capture NTLM challenge-response traffic
- log the captured hashes for offline cracking or later use

It can run in:

- **analyze mode** → listen only
- **poisoning mode** → actively answer requests and capture auth

Earlier in the module, analyze mode was used like a passive observer.  
Now we let Responder actively respond.

---

# Important Responder Options

## `-A` / `--analyze`

Passive mode.

- listens for traffic
- does **not** poison replies
- useful for safe recon before going active

## `-I`

Specify the network interface.

Example:

```bash
-I ens224
```

## `-w`

Enable the rogue **WPAD** proxy server.

This can be very effective in larger networks because systems with auto-proxy detection may start sending requests to us.

## `-f`

Fingerprint hosts issuing LLMNR/NBT-NS requests.

Useful for learning:

- hostname
- OS type/version
- possible client behavior

## `-r`

Enable answers for NetBIOS **wredir** suffix queries.

This may break things on the network and should be used carefully.

## `-F` and `-P`

Used to force authentication:

- force WPAD authentication
- force proxy authentication

These can be effective but are noisier and may create login prompts.

# Practical Workflow

A common workflow looks like this:

1. Start Responder in a tmux or screen session
2. Let it run in the background
3. Continue with other internal enumeration
4. Periodically review logs
5. Pull out interesting hashes
6. Crack them offline
7. Validate recovered credentials
8. Continue with credentialed enumeration

This is usually better than staring at the console waiting for activity.

# Why Cracking May Be Slow

NetNTLMv2 is not instant to crack.

Success depends on:

- password complexity
- password length
- wordlist quality
- rule tuning
- compute power

Weak domain policy, such as:

- 8 character minimum
- common human password patterns

makes success much more likely.

Strong passphrases may be impractical to crack during an engagement.

# Operational Notes

## Let Responder run while you work

This attack is opportunistic.

You do not always trigger it directly.  
Often the best move is simply to let Responder sit and wait while users generate traffic naturally.

## Cracking priority matters

Not every captured hash is worth spending time on.

Prioritize users that may help you move further:

- admins
- helpdesk
- IT staff
- service accounts
- users with lateral movement potential

## NetNTLMv2 is not PtH material

This is worth repeating:

- **NetNTLMv2 ≠ NTLM hash**
- you usually cannot pass it directly
- you need to crack it or relay it
