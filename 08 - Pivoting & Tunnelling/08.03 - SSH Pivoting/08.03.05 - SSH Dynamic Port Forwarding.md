## Overview

SSH dynamic port forwarding creates a SOCKS proxy on the machine running the SSH client.

Instead of forwarding one fixed local port to one fixed destination, a dynamic forward allows supported tools to connect to multiple hosts and ports through a single SSH tunnel.

General syntax:

```bash
ssh -N -D <listen_ip>:<listen_port> <ssh_user>@<ssh_server_ip>
```

Example:

```bash
ssh -N -D 0.0.0.0:9999 database_admin@10.4.50.215
```

The key rule is:

```text
The SOCKS listener is created on the SSH client.

The SSH server makes the final network connections requested through the proxy.
```

---

# Local Forwarding vs Dynamic Forwarding

## Local Port Forwarding

A local forward maps one listener to one destination.

```bash
ssh -L 0.0.0.0:4455:172.16.50.217:445 user@10.4.50.215
```

This mapping is fixed:

```text
SSH client port 4455
↓
172.16.50.217:445
```

To access another service, another forward is required.

---

## Dynamic Port Forwarding

A dynamic forward creates one SOCKS proxy that can reach many destinations.

```bash
ssh -D 0.0.0.0:9999 user@10.4.50.215
```

Supported tools can request connections to:

```text
172.16.50.217:445
172.16.50.217:3389
172.16.50.100:80
172.16.50.25:22
```

All requests use the same SOCKS listener:

```text
SSH client port 9999
```

---

# Core Mental Model

```text
Tool on Kali
↓
Proxychains adds SOCKS instructions
↓
SOCKS listener on SSH client
↓
Encrypted SSH tunnel
↓
SSH server
↓
Requested internal destination
```

Unlike a local forward, the destination is not specified in the SSH command.

The client tool specifies the destination through the SOCKS protocol.

# Understand Each IP Address

For this command:

```bash
ssh -N -D 0.0.0.0:9999 database_admin@10.4.50.215
```

## `0.0.0.0`

```text
Bind the SOCKS listener to every interface on the SSH client.
```

The SSH client is CONFLUENCE01 because the command runs there.

This allows Kali to reach the proxy through:

```text
192.168.50.63:9999
```

---

## `9999`

This is the SOCKS proxy port opened on CONFLUENCE01.

Kali sends proxy traffic to:

```text
192.168.50.63:9999
```

---

## `10.4.50.215`

This is the SSH server.

It is not the SOCKS listener and it is not necessarily the final target.

PGDATABASE01 receives the SSH tunnel and makes connections into networks it can reach.

---

## `172.16.50.217`

This is an internal destination.

It does not appear in the SSH dynamic-forwarding command.

Instead, it appears in commands run through Proxychains:

```bash
proxychains smbclient -L //172.16.50.217/
```

Proxychains tells the SOCKS proxy:

```text
Connect to 172.16.50.217 on the port requested by smbclient.
```

---

# Why This Is Called Dynamic Forwarding

The destination is selected dynamically for every connection.

One connection may request:

```text
172.16.50.217:445
```

The next may request:

```text
172.16.50.217:3389
```

Another may request:

```text
172.16.50.100:80
```

The SSH server attempts each destination using its own network access.

---

# SSH Client, SSH Server, and Final Target

## SSH Client

The machine running:

```bash
ssh -D
```

In this example:

```text
CONFLUENCE01
192.168.50.63
```

It owns the SOCKS listener.

---

## SSH Server

The host after `user@`:

```text
PGDATABASE01
10.4.50.215
```

It receives the SSH tunnel and opens requested destination connections.

---

## Final Target

The destination used by the proxied tool.

Examples:

```text
172.16.50.217:445
172.16.50.217:3389
```

The final target must be reachable from the SSH server.

---

## Proxy Client

The program using the SOCKS proxy.

Examples:

```text
smbclient
nmap
curl
xfreerdp
ssh
```

If the program does not support SOCKS directly, Proxychains may be used.

---

# Why Proxychains Is Required

A SOCKS listener does not behave like a normal raw port forward.

It expects SOCKS protocol information containing the requested destination.

A normal application connection contains only application traffic.

For example, raw SMB traffic does not tell the SOCKS proxy:

```text
I want to reach 172.16.50.217:445.
```

Proxychains wraps or redirects the application's connection so the SOCKS server receives the destination information.

Conceptually:

```text
smbclient wants 172.16.50.217:445
↓
Proxychains intercepts the connection
↓
Proxychains sends SOCKS request to 192.168.50.63:9999
↓
SOCKS proxy forwards through SSH
↓
PGDATABASE01 connects to 172.16.50.217:445
```

# 1. Enumerate Networks from the SSH Server

Before creating the dynamic forward, determine which networks the SSH server can reach.

From CONFLUENCE01:

```bash
ssh database_admin@10.4.50.215
```

On PGDATABASE01:

```bash
ip addr
```

Example:

```text
ens192: 10.4.50.215/24
ens224: 172.16.50.215/24
```

Check routes:

```bash
ip route
```

Example:

```text
10.4.50.0/24 dev ens192
172.16.50.0/24 dev ens224
```

This shows that PGDATABASE01 can directly access:

```text
172.16.50.0/24
```

---

# 2. Find Internal Targets

From PGDATABASE01, test a specific port across the subnet.

Example SMB sweep:

```bash
for i in $(seq 1 254); do
    nc -zv -w 1 172.16.50.$i 445
done
```

Interesting result:

```text
Connection to 172.16.50.217 445 port succeeded
```

Record:

```text
Internal target: 172.16.50.217
Known open port: 445
```

---

# 3. Create the Dynamic Forward

Run the command on CONFLUENCE01:

```bash
ssh -N -D 0.0.0.0:9999 database_admin@10.4.50.215
```

Command breakdown:

|Component|Meaning|
|---|---|
|`ssh`|Start OpenSSH client|
|`-N`|Do not open a remote shell|
|`-D`|Create a dynamic SOCKS proxy|
|`0.0.0.0`|Listen on every interface of CONFLUENCE01|
|`9999`|SOCKS listening port|
|`database_admin`|User on PGDATABASE01|
|`10.4.50.215`|SSH server|

After authentication, no shell prompt appears.

This is expected because:

```text
-N prevents remote command execution.
```

The foreground SSH process maintains the tunnel.

---

# 4. Verify the SOCKS Listener

From another shell on CONFLUENCE01:

```bash
ss -lntp | grep 9999
```

Expected:

```text
0.0.0.0:9999
```

Example:

```text
tcp LISTEN 0 128 0.0.0.0:9999 0.0.0.0:* users:(("ssh",pid=1234,fd=4))
```

This confirms that the SOCKS proxy is listening on CONFLUENCE01.

# 5. Configure Proxychains on Kali

Default configuration file:

```text
/etc/proxychains4.conf
```

Edit the proxy list:

```bash
sudo nano /etc/proxychains4.conf
```

At the bottom:

```text
[ProxyList]
socks5 192.168.50.63 9999
```

Meaning:

|Value|Role|
|---|---|
|`socks5`|Proxy protocol|
|`192.168.50.63`|CONFLUENCE01, where the SOCKS listener exists|
|`9999`|SSH dynamic-forwarding port|

> [!important]  
> Proxychains points to the SSH client and SOCKS port, not the SSH server and not the final destination.

Check:

```bash
tail /etc/proxychains4.conf
```

OpenSSH dynamic forwarding supports SOCKS4 and SOCKS5.

SOCKS5 supports additional features such as:

```text

UDP-related extensions
proxy-side DNS handling
```

Typical configuration:

```text
socks5 192.168.50.63 9999
```



# Dynamic vs Local Forward Client Syntax

## Local Forward

The client connects to the forwarded listener:

```bash
smbclient -p 4455 -L //192.168.50.63/
```

The client uses:

```text
pivot IP and forwarded port
```

---

## Dynamic Forward

The client refers to the real internal target:

```bash
proxychains smbclient -L //172.16.50.217/
```

Proxychains silently sends the connection through:

```text
192.168.50.63:9999
```

> [!important]  
> With a dynamic forward, application commands use the final destination IP. Proxychains handles the path to the SOCKS listener.

# 8. Scan Through the SOCKS Proxy

Use a TCP connect scan:

```bash
sudo proxychains nmap -sT -Pn -n --top-ports 20 172.16.50.217
```

Command breakdown:

|Option|Meaning|
|---|---|
|`proxychains`|Route supported connections through SOCKS|
|`nmap`|Port scanner|
|`-sT`|TCP connect scan|
|`-Pn`|Skip host discovery|
|`-n`|Disable DNS resolution|
|`--top-ports 20`|Scan the 20 most common ports|
|`172.16.50.217`|Real internal destination|

---

# Why `-sT` Is Required

Proxychains works by intercepting normal TCP connection functions.

A TCP connect scan uses the operating system's normal `connect()` function:

```text
Nmap
↓
connect()
↓
Proxychains intercepts it
↓
SOCKS proxy
```

A SYN scan uses raw packets rather than normal connections.

Avoid:

```bash
nmap -sS
```

through Proxychains.

Use:

```bash
nmap -sT
```

---

# Why `-Pn` Is Useful

Host discovery may rely on traffic such as:

```text
ICMP echo
ARP
raw TCP probes
```

These do not pass normally through a TCP SOCKS proxy.

Use:

```text
-Pn
```

to tell Nmap:

```text
Assume the target is online and scan its ports directly.
```

---

# Why `-n` Is Useful

DNS queries may not follow the same proxy route as TCP connections.

Use:

```text
-n
```

to disable reverse DNS lookups and avoid:

```text
DNS leaks
slow resolution
unreachable DNS requests
```

---

# Recommended Nmap Command

```bash
sudo proxychains nmap \
    -sT \
    -Pn \
    -n \
    --top-ports 20 \
    172.16.50.217
```

For one port:

```bash
proxychains nmap -sT -Pn -n -p 445 172.16.50.217
```

For selected ports:

```bash
proxychains nmap -sT -Pn -n -p 22,80,135,139,443,445,3389 172.16.50.217
```

# Binding Options

## Listen on Every Interface

```bash
ssh -N -D 0.0.0.0:9999 user@10.4.50.215
```

Use when another host such as Kali must reach the SOCKS listener.

Risk:

```text
Other systems may also be able to connect to the proxy.
```

---

## Listen on a Specific Interface

```bash
ssh -N -D 192.168.50.63:9999 user@10.4.50.215
```

This restricts the listener to the interface Kali can reach.

---

## Listen on Localhost

```bash
ssh -N -D 127.0.0.1:9999 user@10.4.50.215
```

Use only when Proxychains runs on the same host as the SSH client.

In this scenario, Kali would not be able to connect directly to the proxy.

# Repeatable Methodology

## Step 1 – Identify Where the SSH Command Runs

Record:

```text
SSH client: <host_running_ssh_D>
SSH client reachable IP: <client_ip>
```

The SOCKS listener appears here.

---

## Step 2 – Identify the SSH Server

Record:

```text
SSH server IP: <ssh_server_ip>
SSH username: <ssh_user>
```

The SSH server must be able to reach the target networks.

---

## Step 3 – Identify Reachable Internal Networks

On the SSH server:

```bash
ip addr
ip route
```

Record:

```text
Internal subnet: <subnet>
```

---

## Step 4 – Choose a SOCKS Listener

Record:

```text
SOCKS listen IP: <listen_ip>
SOCKS listen port: <listen_port>
```

If Kali needs direct access:

```text
Use 0.0.0.0 or the SSH client's external IP.
```

---

## Step 5 – Create the Dynamic Forward

```bash
ssh -N -D <listen_ip>:<listen_port> <ssh_user>@<ssh_server_ip>
```

---

## Step 6 – Configure Proxychains

```text
socks5 <ssh_client_reachable_ip> <listen_port>
```

---

## Step 7 – Address the Real Internal Target

Run commands using the internal destination:

```bash
proxychains <tool> <internal_target>
```

Do not replace the internal target with the SOCKS proxy IP inside the tool command.

# Speeding Up Proxychains Scans

Proxychains may use long default timeouts.

Look for:

```text
tcp_read_time_out
tcp_connect_time_out
```

Reducing them can speed up scans of filtered or non-responsive ports.

Example:

```text
tcp_read_time_out 3000
tcp_connect_time_out 1500
```

> [!warning]  
> Very short timeouts may cause slow but reachable services to appear closed or unavailable.

---

# Proxychains Limitations

Proxychains works best with dynamically linked applications using normal TCP socket functions.

It may fail with:

```text
statically linked binaries
raw packet tools
custom networking stacks
UDP-only applications
some Go applications
some Java applications
setuid binaries
```

It generally works well with:

```text
curl
ssh
smbclient
psql
mysql
TCP connect scans
```

---

# DNS Considerations

Proxychains can be configured with:

```text
proxy_dns
```

This attempts to prevent local DNS resolution.

When possible during internal enumeration, use IP addresses directly:

```bash
proxychains curl http://172.16.50.100/
```

This avoids relying on DNS that Kali may not be able to access.

# Troubleshooting

## No Output After SSH Authentication

This is normal with:

```text
-N
```

Verify from another shell:

```bash
ss -lntp | grep 9999
```

---

## Kali Cannot Reach the SOCKS Proxy

Check the bind address on the SSH client:

```bash
ss -lntp | grep 9999
```

If it shows:

```text
127.0.0.1:9999
```

Kali cannot connect remotely.

Use:

```bash
ssh -N -D 0.0.0.0:9999 user@<ssh_server>
```

or bind to the client’s external IP.

---

## Proxychains Says Connection Refused to the Proxy

Check:

```text
SSH tunnel is still running
Proxychains IP is the SSH client IP
Proxychains port matches the -D port
Firewall allows access to the listener
```

---

## Proxychains Reaches the Proxy but Not the Target

Test the destination from the SSH server:

```bash
nc -zv <target_ip> <target_port>
```

The target must be reachable from the SSH server.

---

## Nmap Results Are Incorrect

Ensure:

```text
-sT is used
-Pn is used
-n is used
```

Recommended:

```bash
proxychains nmap -sT -Pn -n -p <ports> <target_ip>
```

Do not use:

```bash
nmap -sS
```

through a standard SOCKS and Proxychains path.

---

## Scanning Is Very Slow

Possible causes:

```text
high Proxychains timeouts
filtered ports
large port ranges
serial proxy connections
slow pivot
high network latency
```

Reduce the port list and adjust Proxychains timeouts.

---

## Tool Ignores Proxychains

Possible causes:

```text
statically linked binary
raw socket use
custom network library
LD_PRELOAD protections
setuid execution
```

Try another client or create a fixed local forward for that service.

---

## DNS Names Do Not Resolve

Use the target IP directly or enable:

```text
proxy_dns
```

in `/etc/proxychains4.conf`.

# Dynamic Forwarding Decision Guide

Use local forwarding when:

```text
You need one known destination and port
The client tool does not work through SOCKS
You want simple, predictable forwarding
```

Use dynamic forwarding when:

```text
You need access to many hosts or ports
You want to perform broader enumeration
Your tools support SOCKS or work with Proxychains
You want one tunnel for multiple services
```

---

# Key Takeaways

- `ssh -D` creates a SOCKS proxy on the SSH client.
    
- The SOCKS listener belongs to the host where the SSH command runs.
    
- The host after `user@` is the SSH server and tunnel endpoint.
    
- The SSH server makes final connections to internal targets.
    
- Dynamic forwarding does not define a fixed destination in the SSH command.
    
- Proxy-aware tools specify the real internal destination.
    
- Proxychains points to the SOCKS listener on the SSH client.
    
- Application commands use the final internal target IP.
    
- Use Nmap TCP connect scans with `-sT`, `-Pn`, and `-n`.
    
- One dynamic tunnel can access multiple hosts and services.
    
- Authentication to the SSH server and authentication to the internal service are separate.
    
- Test destination reachability from the SSH server when troubleshooting.
