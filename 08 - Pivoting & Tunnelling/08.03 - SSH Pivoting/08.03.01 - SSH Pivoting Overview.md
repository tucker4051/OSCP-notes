## Purpose

SSH pivoting uses SSH access to a host as a route into networks or services that are not directly reachable from the attacker machine.

Use this when you have valid SSH access to a pivot host and need to reach internal services through it.

SSH pivoting can support:

- local port forwarding
- remote port forwarding
- dynamic SOCKS proxying
- reverse port forwarding
- access to internal web apps
- access to internal databases
- access to restricted network services
- proxy-based internal enumeration

---

## Core Idea

SSH can create tunnels between the attacker machine, the pivot host, and internal targets.

The key question is:

> Which side needs to connect to which service?

Different SSH forwarding modes solve different routing problems.

---

## SSH Pivoting Techniques

|Technique|Use Case|
|---|---|
|Local Port Forwarding|Access an internal service through a local port on attacker machine|
|Remote Port Forwarding|Expose an attacker-side service to the pivot/remote side|
|Reverse Port Forwarding|Allow remote/internal systems to reach a service through the SSH tunnel|
|Dynamic Port Forwarding|Create a local SOCKS proxy for flexible internal access|
|Dynamic Remote Port Forwarding|Create a SOCKS proxy on the remote side|
|SSH SOCKS Pivoting|Route tools through SSH-created SOCKS proxy|
|Sshuttle|Route internal subnets through SSH like a lightweight VPN|

## When to Use SSH Pivoting

Use SSH pivoting when:

- [ ]  SSH credentials are available
- [ ]  SSH key access is available
- [ ]  the pivot host can reach internal services
- [ ]  direct access to internal hosts is blocked
- [ ]  a single internal port needs forwarding
- [ ]  multiple internal services need proxy access
- [ ]  browser or proxychains access is needed
- [ ]  a stable tunnel is preferred over a fragile shell

---

## Choosing the Right SSH Tunnel

|Situation|Best Option|
|---|---|
|Need one internal web service|Local port forwarding|
|Need one internal database port|Local port forwarding|
|Need flexible access to many internal services|Dynamic port forwarding|
|Need tools to use a SOCKS proxy|Dynamic port forwarding|
|Need remote side to access attacker service|Remote/reverse port forwarding|
|Need subnet-style routing over SSH|Sshuttle|
|Need SOCKS proxy on remote side|Dynamic remote port forwarding|

---

## Common Mistakes

- [ ]  Confusing local and remote port forwarding
- [ ]  Binding the tunnel to the wrong interface
- [ ]  Forwarding the wrong remote host or port
- [ ]  Forgetting that `127.0.0.1` is relative to the host where it is used
- [ ]  Forgetting to configure proxychains or browser proxy settings
- [ ]  Trying to scan too aggressively through SSH tunnels
- [ ]  Not documenting the tunnel mapping
- [ ]  Closing the SSH session and breaking the tunnel
