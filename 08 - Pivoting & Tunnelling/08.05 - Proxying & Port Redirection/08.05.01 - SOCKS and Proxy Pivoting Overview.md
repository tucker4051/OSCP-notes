## Purpose

SOCKS and proxy pivoting allows tools and traffic to be routed through a compromised host or tunnel into internal networks.

Use this when you need flexible access to multiple internal hosts or ports rather than forwarding one specific service.

SOCKS/proxy pivoting is useful for:

- internal web browsing
- proxychains-based scanning
- accessing multiple internal services
- routing tools through a pivot
- dynamic internal enumeration
- working through Chisel, SSH, Meterpreter, or SocksOverRDP

---

## Core Idea

A SOCKS proxy acts as an intermediary.

Instead of creating a separate port forward for every target service, tools send traffic to the SOCKS proxy, and the proxy forwards it into the internal network.

The key question is:

> Can I route my tools through a proxy that has access to the target network?

## When to Use SOCKS Pivoting

Use SOCKS pivoting when:

- [ ]  multiple internal hosts need testing
- [ ]  multiple ports need testing
- [ ]  internal web apps need browsing
- [ ]  proxychains can be used
- [ ]  single-port forwarding would be inefficient
- [ ]  the pivot host has broad internal visibility
- [ ]  you need to route tools through the pivot

|Tool / Method|Use Case|
|---|---|
|SSH dynamic forwarding|Quick SOCKS proxy over SSH|
|Chisel|Reverse or forward SOCKS tunnelling|
|SocksOverRDP|SOCKS tunnelling through RDP|
|Proxychains|Route command-line tools through SOCKS|
|Browser proxy settings|Browse internal web apps through SOCKS|
|Meterpreter routing|Framework-based route/proxy access|

---

## SOCKS Pivoting Workflow

1. Identify internal networks reachable from pivot host.
2. Choose SOCKS-capable tool.
3. Start SOCKS tunnel/proxy.
4. Confirm local SOCKS port.
5. Configure browser or proxychains.
6. Test access to a simple internal service.
7. Enumerate carefully through the tunnel.
8. Document tunnel and proxy configuration.

---

## Tool Configuration Checks

Check:

- [ ]  SOCKS proxy started
- [ ]  Correct local SOCKS port used
- [ ]  Proxychains configured correctly
- [ ]  Browser proxy configured correctly
- [ ]  DNS handling considered
- [ ]  Target service reachable from pivot
- [ ]  Tool supports proxying
- [ ]  Scanning rate kept reasonable
- [ ]  Tunnel stability monitored

---

## Common Mistakes

- [ ]  Forgetting to configure tools to use the proxy
- [ ]  Using a tool that does not support SOCKS properly
- [ ]  Forgetting DNS may not go through the proxy
- [ ]  Scanning too aggressively through the tunnel
- [ ]  Not testing with a simple service first
- [ ]  Using the wrong SOCKS version
- [ ]  Confusing proxy port with target port
- [ ]  Not documenting the proxy setup
