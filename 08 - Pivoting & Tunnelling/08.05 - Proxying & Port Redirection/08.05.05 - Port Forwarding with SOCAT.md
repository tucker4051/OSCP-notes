## Overview

`socat` is a general-purpose networking tool that can relay traffic between two network endpoints.

For pivoting, the most useful pattern is:

```text
Attack host
↓
Connects to a listening port on the compromised pivot
↓
Socat receives the connection
↓
Socat forwards the traffic to an internal host and port
```

General command:

```bash
socat TCP-LISTEN:<pivot_port>,fork TCP:<internal_ip>:<internal_port>
```

This creates a TCP listener on the pivot host and forwards all traffic received on that port to the internal destination.

---

## Example Network Layout

Assume the following systems:

|Role|Host|IP address|
|---|---|---|
|Attack host|Kali|`192.168.50.10`|
|Compromised pivot|CONFLUENCE01|`192.168.50.63`|
|Internal target|PGDATABASE01|`10.4.50.215`|

The network situation is:

```text
Kali can reach CONFLUENCE01
CONFLUENCE01 can reach PGDATABASE01
Kali cannot directly reach PGDATABASE01
```

Traffic will flow as follows:

```text
Kali
192.168.50.10
    |
    | Connect to 192.168.50.63:2345
    v
CONFLUENCE01
192.168.50.63
    |
    | Socat forwards the connection
    v
PGDATABASE01
10.4.50.215:5432
```

> [!important]  
> The client connects to the pivot host's reachable IP and listening port. Socat then connects from the pivot to the internal host and its real service port.

---

# Core Syntax

```bash
socat TCP-LISTEN:<pivot_port>,fork TCP:<internal_ip>:<internal_port>
```

Example:

```bash
socat TCP-LISTEN:2345,fork TCP:10.4.50.215:5432
```

|Component|Meaning|
|---|---|
|`socat`|Starts Socat|
|`TCP-LISTEN:2345`|Listens on TCP port `2345` on the pivot|
|`fork`|Creates a child process for each incoming connection|
|`TCP:10.4.50.215:5432`|Forwards traffic to the internal target|

---

# Understanding the IP Addresses

For this command:

```bash
socat TCP-LISTEN:2345,fork TCP:10.4.50.215:5432
```

the addresses are used as follows.

## Pivot IP

```text
192.168.50.63
```

This is the IP address of the compromised host that Kali can reach.

Kali connects to:

```text
192.168.50.63:2345
```

The pivot IP is not shown in the basic Socat command because:

```bash
TCP-LISTEN:2345
```

usually listens on all IPv4 interfaces:

```text
0.0.0.0:2345
```

## Internal Destination IP

```text
10.4.50.215
```

This is the internal target that only the pivot can reach.

Socat forwards traffic to:

```text
10.4.50.215:5432
```

## Final Traffic Path

```text
Kali connects to:
192.168.50.63:2345

Socat forwards to:
10.4.50.215:5432
```

# 1. Confirm Connectivity from the Pivot

Before creating the forward, verify that the pivot can reach the internal target.

From the pivot:

```bash
ping -c 2 10.4.50.215
```

Test the destination port:

```bash
nc -nvz 10.4.50.215 5432
```

Alternative Bash TCP check:

```bash
timeout 3 bash -c '</dev/tcp/10.4.50.215/5432' && echo open
```

> [!note]  
> ICMP may be blocked even when the service is reachable. A direct TCP port test is more useful than relying only on `ping`.

---

# 2. Check Whether Socat Is Installed

```bash
which socat
```

Check the version:

```bash
socat -V
```

Expected path:

```text
/usr/bin/socat
```

Socat is not installed by default on many Linux systems. A statically linked binary can be transferred to the pivot when necessary.

# 3. Forward PostgreSQL

## Objective

Expose the internal PostgreSQL service:

```text
10.4.50.215:5432
```

through the pivot at:

```text
192.168.50.63:2345
```

## Run on the Pivot

```bash
socat -ddd TCP-LISTEN:2345,fork TCP:10.4.50.215:5432
```

|Argument|Purpose|
|---|---|
|`-ddd`|Displays verbose diagnostic output|
|`TCP-LISTEN:2345`|Opens TCP port `2345` on the pivot|
|`fork`|Allows multiple or repeated connections|
|`TCP:10.4.50.215:5432`|Sends traffic to the internal PostgreSQL service|

Expected output:

```text
listening on AF=2 0.0.0.0:2345
```

This means Socat is listening on all IPv4 interfaces on port `2345`.

---

# 4. Connect from Kali

Connect to the pivot's external IP and Socat listening port:

```bash
psql -h 192.168.50.63 -p 2345 -U postgres
```

|Option|Meaning|
|---|---|
|`-h 192.168.50.63`|Connect to the pivot host|
|`-p 2345`|Use the Socat listener|
|`-U postgres`|Authenticate as PostgreSQL user `postgres`|

Traffic flow:

```text
psql
↓
192.168.50.63:2345
↓
Socat on CONFLUENCE01
↓
10.4.50.215:5432
```

From the PostgreSQL client’s perspective:

```text
192.168.50.63:2345
```

behaves like:

```text
10.4.50.215:5432
```

# 6. Connect to the Internal SSH Server

From Kali:

```bash
ssh database_admin@192.168.50.63 -p 2222
```

|Component|Meaning|
|---|---|
|`database_admin`|Username on the internal SSH server|
|`192.168.50.63`|Reachable pivot IP|
|`-p 2222`|Socat listening port|

Although the SSH command references the pivot IP, the SSH session terminates on:

```text
10.4.50.215:22
```

The SSH host key and remote hostname will belong to the internal target.

# Binding to a Specific Pivot Interface

By default:

```bash
TCP-LISTEN:2345
```

usually listens on:

```text
0.0.0.0:2345
```

This exposes the listener on every interface.

If the pivot has multiple interfaces, bind only to the interface reachable from Kali:

```bash
socat TCP-LISTEN:2345,bind=192.168.50.63,fork TCP:10.4.50.215:5432
```

Example interfaces:

```text
ens192: 192.168.50.63
ens224: 10.4.50.63
```

Use:

```text
192.168.50.63
```

because Kali can reach that interface.

---

# Binding to Localhost

```bash
socat TCP-LISTEN:2345,bind=127.0.0.1,fork TCP:10.4.50.215:5432
```

This listener is only reachable from the pivot itself.

Do not use this for direct access from Kali unless another tunnel exposes the localhost listener.

---

# Reuse the Listening Port

After stopping Socat, the port may remain temporarily unavailable.

Use:

```bash
socat TCP-LISTEN:2345,reuseaddr,fork TCP:10.4.50.215:5432
```

Recommended command:

```bash
socat -ddd TCP-LISTEN:2345,bind=192.168.50.63,reuseaddr,fork TCP:10.4.50.215:5432
```

|Option|Purpose|
|---|---|
|`bind=`|Selects the pivot interface|
|`reuseaddr`|Allows the port to be reused sooner|
|`fork`|Supports multiple connections|

---

# Running Socat in the Background

Run in the background:

```bash
socat TCP-LISTEN:2345,reuseaddr,fork TCP:10.4.50.215:5432 &
```

Keep it running after the shell exits:

```bash
nohup socat TCP-LISTEN:2345,reuseaddr,fork TCP:10.4.50.215:5432 >/tmp/socat.log 2>&1 &
```

Check the process:

```bash
ps aux | grep '[s]ocat'
```

Read the log:

```bash
cat /tmp/socat.log
```

# Multiple Port Forwards

Run a separate Socat process for each service.

PostgreSQL:

```bash
socat TCP-LISTEN:2345,reuseaddr,fork TCP:10.4.50.215:5432 &
```

SSH:

```bash
socat TCP-LISTEN:2222,reuseaddr,fork TCP:10.4.50.215:22 &
```

HTTP:

```bash
socat TCP-LISTEN:8080,reuseaddr,fork TCP:10.4.50.215:80 &
```

Result:

|Pivot listener|Internal destination|
|---|---|
|`192.168.50.63:2345`|`10.4.50.215:5432`|
|`192.168.50.63:2222`|`10.4.50.215:22`|
|`192.168.50.63:8080`|`10.4.50.215:80`|

# Choosing a Listening Port

Ports below `1024` usually require root privileges.

Avoid low ports when running as a normal user:

```text
22
80
443
445
```

Use high ports instead:

```text
2222
2345
5433
8000
8080
9001
```

The listening port does not need to match the real service port.

Example:

```text
Pivot listener:       2222
Internal SSH service: 22
```

# Troubleshooting

## Connection Refused on the Pivot Port

Check the listener:

```bash
ss -lntp | grep <pivot_port>
```

Check the process:

```bash
ps aux | grep '[s]ocat'
```

Possible causes:

```text
Socat is not running
wrong pivot port
listener bound to the wrong interface
local firewall blocking the listener
port already in use
```

---

## Pivot Cannot Reach the Internal Service

Test from the pivot:

```bash
nc -nvz <internal_ip> <internal_port>
```

Possible causes:

```text
incorrect internal IP
incorrect destination port
routing problem
internal firewall
service is not listening
```

---

## Port Already in Use

Check:

```bash
ss -lntp | grep <pivot_port>
```

Stop the existing process:

```bash
kill <pid>
```

Or select another listening port.

Use `reuseaddr` when restarting:

```bash
socat TCP-LISTEN:<pivot_port>,reuseaddr,fork TCP:<internal_ip>:<internal_port>
```

---

## Forward Works Once and Stops

Ensure `fork` is present:

```bash
socat TCP-LISTEN:2345,fork TCP:10.4.50.215:5432
```

Without `fork`, Socat may terminate after one connection.

---

## Kali Cannot Reach the Listener

Check the bind address:

```bash
ss -lntp
```

If Socat is bound to:

```text
127.0.0.1
```

it cannot be reached directly from Kali.

Bind to the externally reachable pivot IP:

```bash
socat TCP-LISTEN:2345,bind=<pivot_ip>,fork TCP:<internal_ip>:<internal_port>
```

---

## Unexpected SSH Host Key

When SSH is forwarded, Kali connects to the pivot's IP but receives the internal server's host key.

This is expected:

```text
Connection address:
<pivot_ip>:<pivot_port>

Actual SSH endpoint:
<internal_ip>:22
```

# Process Management

Find Socat processes:

```bash
ps aux | grep '[s]ocat'
```

Stop one process:

```bash
kill <pid>
```

Force termination:

```bash
kill -9 <pid>
```

Stop all Socat processes owned by the current user:

```bash
pkill socat
```

> [!warning]  
> `pkill socat` may stop every active Socat forward and listener owned by the user.

---

# Cleanup

Stop the Socat process:

```bash
kill <pid>
```

Remove log files:

```bash
rm -f /tmp/socat.log
```

Remove a transferred static binary:

```bash
rm -f /tmp/socat
```

Verify the listener has stopped:

```bash
ss -lntp | grep <pivot_port>
```

Document:

```text
pivot IP
pivot listening port
internal destination IP
internal destination port
Socat PID
cleanup performed
```

# Security and Detection Notes

Simple Socat forwards are easy to identify.

Possible indicators include:

```text
unexpected listening ports
Socat running as an application or service user
outbound connections from the pivot to internal services
long-running TCP connections
listeners on unusual ports such as 2222 or 2345
```

Potential defensive detections:

```text
new listening socket on a compromised host
unexpected application-server-to-database connections
execution of a Socat binary
unusual child processes or network relays
```

# Alternative Tools

Other port-forwarding options include:

```text
SSH local forwarding
SSH remote forwarding
rinetd
Netcat with a FIFO
iptables
Chisel
Ligolo-ng
```

`iptables` forwarding generally requires root privileges and IP forwarding may need to be enabled:

```bash
echo 1 > /proc/sys/net/ipv4/ip_forward
```

Per-interface example:

```bash
echo 1 > /proc/sys/net/ipv4/conf/<interface>/forwarding
```

# Key Takeaways

- Run Socat on the pivot host.
    
- Kali connects to the pivot's reachable IP.
    
- Socat forwards traffic to the internal destination.
    
- The pivot listening port can be any unused high port.
    
- `fork` allows repeated and concurrent connections.
    
- `reuseaddr` helps when restarting a listener.
    
- Use `bind=` when the pivot has multiple interfaces.
    
- Confirm the pivot can reach the internal service before troubleshooting Kali.
    
- The client behaves as though it is directly connected to the internal service.
    
- Stop Socat and remove temporary files after use.
