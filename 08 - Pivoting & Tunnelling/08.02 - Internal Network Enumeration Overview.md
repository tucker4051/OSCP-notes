## Purpose

Internal network enumeration is the process of identifying networks, hosts, services, routes, and trust relationships visible from a compromised host.

Use this after gaining access to a host that may have visibility into internal networks.

The goal is to identify what can be reached from the pivot host and decide whether pivoting is required.

---

## Core Idea

A compromised host may have a different network view from the attacker machine.

Internal enumeration helps answer:

- What networks is this host connected to?
- What routes exist?
- What hosts has it communicated with?
- What DNS servers does it use?
- What internal names can be resolved?
- What services are reachable internally?
- Are there domain resources?
- Are there credentials that can be reused internally?

---

## Key Areas to Check

Check:

- [ ]  Network interfaces
- [ ]  IP addresses
- [ ]  Routing table
- [ ]  ARP cache
- [ ]  DNS configuration
- [ ]  Hosts file
- [ ]  Listening ports
- [ ]  Established connections
- [ ]  Firewall rules
- [ ]  Proxy settings
- [ ]  Domain/workgroup context
- [ ]  Internal hostnames
- [ ]  Mounted shares
- [ ]  Local credentials
- [ ]  Service accounts
- [ ]  Recent connections

---

## Internal Enumeration Workflow

1. Identify local interfaces and IP ranges.
2. Review routing table.
3. Check ARP cache and neighbour information.
4. Review DNS settings and internal names.
5. Check current connections.
6. Identify reachable subnets.
7. Test access to internal hosts carefully.
8. Identify interesting services.
9. Decide pivoting technique.
10. Document internal network map.

---

## Useful Findings

High-value internal findings include:

- internal web apps
- database servers
- domain controllers
- file shares
- management interfaces
- backup servers
- monitoring systems
- admin panels
- RDP/WinRM hosts
- SSH hosts
- internal APIs
- service credentials
- reused passwords

---

## Pivoting Decision Points

|Finding|Possible Next Step|
|---|---|
|One internal service reachable|Use local port forwarding|
|Many internal services reachable|Use SOCKS/dynamic proxy|
|Windows pivot host|Consider Netsh, Plink, Chisel, SocksOverRDP|
|SSH access available|Use SSH forwarding or Sshuttle|
|Meterpreter session available|Add route or port forward|
|DNS/ICMP only egress|Consider tunnelling|
|Internal AD environment|Move toward AD enumeration|

---

## Common Mistakes

- [ ]  Only checking the attacker-visible network
- [ ]  Missing secondary interfaces
- [ ]  Ignoring routing table
- [ ]  Ignoring ARP cache
- [ ]  Ignoring DNS settings
- [ ]  Scanning too aggressively from a fragile shell
- [ ]  Not checking internal hostnames
- [ ]  Not documenting discovered subnets
- [ ]  Not testing credentials against internal services
- [ ]  Moving to pivoting before understanding the network view
