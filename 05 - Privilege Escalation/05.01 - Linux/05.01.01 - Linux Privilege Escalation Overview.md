## Purpose

Linux privilege escalation is the process of moving from a low-privileged Linux user context to a higher-privileged account, usually `root`.

Use this section after gaining an initial shell, web shell, SSH session, or command execution on a Linux target.

The goal is to identify and abuse local weaknesses such as:

- weak permissions
- exposed credentials
- sudo rights
- privileged group membership
- SUID/SGID binaries
- Linux capabilities
- cron jobs
- writable scripts
- service misconfigurations
- PATH/library hijacking
- container access
- kernel vulnerabilities

---

## Core Idea

Linux privilege escalation is a structured local enumeration process.

The workflow should be:

1. Understand the current user context.
2. Enumerate the host.
3. Identify credentials and privilege boundaries.
4. Check sudo and group memberships.
5. Check filesystem permissions and special bits.
6. Check scheduled tasks and services.
7. Check hijacking opportunities.
8. Check containers and mounted filesystems.
9. Consider kernel/system exploits later.
10. Validate and document the escalation path.

Do not jump straight to kernel exploits unless simpler paths have been ruled out.

## Suggested Prioritisation

A practical order is:

1. Check current user and groups.
2. Check sudo rights.
3. Hunt for credentials.
4. Check privileged groups.
5. Check writable files and directories.
6. Check SUID/SGID binaries.
7. Check capabilities.
8. Check cron jobs and automation.
9. Check services and local processes.
10. Check PATH and library hijacking.
11. Check containers and orchestration.
12. Check mounted filesystems and NFS.
13. Check kernel/system exploits.

## Common Escalation Categories

### Credentials

Look for passwords, SSH keys, database credentials, config files, tokens, history files, and reused credentials.

Relevant note:

- [[05.01.03 - Credential Hunting]]

---

### Sudo and Privileged Groups

Check whether the current user can run commands as another user or belongs to a group with elevated access.

Relevant note:

- [[05.01.04 - Users Groups and Sudo]]

---

### File and Permission Abuse

Check for special permissions, writable files, writable directories, and file ownership issues.

Relevant note:

- [[05.01.05 - File and Permission Abuse]]

---

### Scheduled Tasks and Automation

Review cron jobs, recurring scripts, automation, and privileged processes that run files you can modify.

Relevant note:

- [[05.01.06 - Scheduled Tasks and Automation]]

---

### Services and Internals

Review running services, local ports, debug interfaces, vulnerable daemons, and internal application behaviour.

Relevant note:

- [[05.01.07 - Services Processes and Internals]]

---

### Path and Library Hijacking

Look for privileged processes that load binaries, libraries, Python modules, or scripts from locations you can control.

Relevant note:

- [[05.01.08 - Library and Path Hijacking]]

---

### Containers

Check whether the current user has access to Docker, LXD, Kubernetes, or other container-related privilege boundaries.

Relevant note:

- [[05.01.09 - Containers and Orchestration]]

---

### Kernel and System Exploits

Kernel exploits should usually be considered after safer and more reliable escalation paths have been checked.

Relevant note:

- [[05.01.11 - Kernel and System Exploits]]

## Common Mistakes

- [ ]  Running kernel exploits too early
- [ ]  Ignoring sudo rights
- [ ]  Ignoring group memberships
- [ ]  Missing credentials in config files
- [ ]  Missing writable scripts used by cron
- [ ]  Missing SUID binaries
- [ ]  Missing capabilities
- [ ]  Ignoring Docker or LXD group membership
- [ ]  Not checking local services
- [ ]  Not checking PATH/library hijacking
