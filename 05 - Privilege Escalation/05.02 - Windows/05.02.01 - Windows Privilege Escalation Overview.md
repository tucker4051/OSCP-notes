## Purpose

Windows privilege escalation is the process of moving from a low-privileged Windows user context to a higher-privileged account, such as local administrator, `NT AUTHORITY\SYSTEM`, or a more privileged domain context.

Use this section after gaining an initial shell, RDP session, WinRM session, web shell, or command execution on a Windows target.

The goal is to identify and abuse local weaknesses such as:

- weak permissions
- exposed credentials
- token privileges
- privileged group membership
- vulnerable services
- writable service binaries
- unquoted service paths
- DLL hijacking
- scheduled task abuse
- UAC bypasses
- credential stores
- kernel vulnerabilities

---

## Core Idea

Windows privilege escalation depends heavily on local enumeration.

The workflow should be:

1. Understand the current user context.
2. Enumerate the system and security controls.
3. Check privileges and group memberships.
4. Hunt for credentials.
5. Review services and processes.
6. Check weak permissions and hijackable paths.
7. Check token abuse opportunities.
8. Check UAC and desktop/session constraints.
9. Consider legacy and kernel exploits later.
10. Validate and document the escalation path.

## Suggested Prioritisation

A practical order is:

1. Check current user, groups, and privileges.
2. Check stored credentials and PowerShell history.
3. Check privileged groups and built-in roles.
4. Check token privileges.
5. Check services and weak service permissions.
6. Check unquoted service paths.
7. Check DLL hijacking opportunities.
8. Check scheduled tasks and weak file permissions.
9. Check UAC/session breakout paths.
10. Check vulnerable installed software.
11. Check legacy OS/kernel exploit paths.
12. Use tools to validate findings where useful.

## Common Escalation Categories

### Credentials

Look for passwords, saved credentials, command history, config files, registry entries, scripts, unattend files, and credential stores.

Relevant note:

- [[05.02.03 - Credentials and Loot]]

---

### Privileges and Tokens

Windows privileges can grant powerful escalation paths even when the user is not a local administrator.

Relevant note:

- [[05.02.04 - Privileges and Token Abuse]]

---

### Groups and Built-in Roles

Some local or domain groups grant dangerous rights over services, DNS, backup operations, event logs, print services, or virtualisation.

Relevant note:

- [[05.02.05 - Groups and Built-in Roles]]

---

### Services and Processes

Review running services, process owners, named pipes, local ports, and vulnerable service behaviour.

Relevant note:

- [[05.02.06 - Services Processes and Named Pipes]]

---

### Service and DLL Hijacking

Check whether services or applications load binaries or DLLs from writable locations or unquoted paths.

Relevant note:

- [[05.02.07 - Service and DLL Hijacking]]

---

### Scheduled Tasks and Weak Permissions

Look for tasks, scripts, or directories that are writable by the current user but executed by a privileged account.

Relevant note:

- [[05.02.08 - Scheduled Tasks and Weak Permissions]]

---

### UAC and Desktop Breakouts

If working in a desktop, RDP, or restricted session, check UAC behaviour and breakout paths.

Relevant note:

- [[05.02.09 - UAC and Desktop Breakouts]]

---

### Legacy and Kernel Exploits

Kernel exploits and legacy OS paths should usually be considered after safer misconfiguration paths have been checked.

Relevant note:

- [[05.02.10 - Legacy Systems and Kernel Exploits]]
