## Overview

In addition to dumping the SAM database, a high-value Windows credential target is **LSASS**.

**LSASS** (Local Security Authority Subsystem Service) is a core Windows process responsible for:

- enforcing security policy
- handling authentication
- creating access tokens
- writing security events
- caching credential material in memory

Because of this, dumping LSASS memory can expose:

- NT hashes
- SHA1 hashes
- cleartext credentials (in some cases)
- Kerberos tickets / keys
- DPAPI masterkeys
- other in-memory authentication artifacts

This is one of the most valuable credential access techniques on Windows.

---

# Why LSASS Matters

When a user logs in, LSASS may cache credential-related material in memory.

This makes it useful for:

- offline credential extraction
- password cracking
- pass-the-hash follow-on attacks
- Kerberos abuse
- DPAPI decryption
- lateral movement

Compared with SAM dumping:

- **SAM** gives local account hashes
- **LSASS** may give much richer live credential material

---

# What LSASS Can Contain

Depending on OS version, configuration, and active sessions, LSASS may contain:

- local account NT hashes
- domain logon material
- WDIGEST plaintext credentials
- Kerberos tickets and keys
- DPAPI masterkeys
- session metadata
- machine account material

What is present depends heavily on:

- who is logged in
- how they authenticated
- OS version
- patch level
- security settings

---

# General Workflow

Typical workflow:

1. gain admin / high privileges
2. dump LSASS memory
3. transfer dump file to attack host
4. parse dump offline
5. extract credential material
6. crack reusable hashes if needed
7. pivot using recovered credentials

---

# Dumping LSASS Memory

A common goal is to create an **LSASS memory dump** for offline analysis.

This is useful because:

- analysis can happen off-target
- reduces time spent interacting with host
- supports safer post-processing
- allows repeated parsing later

---

# Method 1 — Task Manager

## Requirements

- interactive GUI access
- sufficient privileges

## Steps

1. open **Task Manager**
2. go to **Processes**
3. locate **Local Security Authority Process**
4. right-click
5. choose **Create dump file**

This usually creates:

```text
lsass.DMP
```

Location:

```text
%temp%
```

This file can then be transferred off-host for offline analysis.

---

# Method 2 — rundll32 + comsvcs.dll

This is a command-line method for generating an LSASS dump.

Useful when:

- you only have terminal access
- no interactive GUI is available
- you want a faster method

Important:

- often flagged by AV / EDR
- widely recognized as suspicious behavior

---

## Step 1 — Identify LSASS PID

### In cmd

```cmd
tasklist /svc
```

Find:

```text
lsass.exe
```

Example:

```text
lsass.exe                      672
```

---

### In PowerShell

```powershell
Get-Process lsass
```

Example:

```text
Id 672
```

---

## Step 2 — Create Dump

```powershell
rundll32 C:\windows\system32\comsvcs.dll, MiniDump 672 C:\lsass.dmp full
```

Breakdown:

- `rundll32` loads DLL function
- `comsvcs.dll` exposes dump functionality
- `MiniDump` triggers memory dump
- `672` = LSASS PID
- `C:\lsass.dmp` = output file
- `full` = full dump

If successful, transfer `lsass.dmp` to attack host.

---

# Parsing LSASS Dumps with pypykatz

## Why pypykatz

`pypykatz` is a Python implementation of Mimikatz-style parsing.

Useful because:

- runs on Linux attack hosts
- works offline
- avoids needing to run Mimikatz on target
- parses minidumps directly

---

## Command

```bash
pypykatz lsa minidump /home/peter/Documents/lsass.dmp
```

This parses the dump and extracts credential material.

---

# Understanding pypykatz Output

The dump is usually organized by **LogonSession**.

Each session may contain multiple credential packages.

Important sections include:

- MSV
- WDIGEST
- Kerberos
- DPAPI

---

# MSV

Example:

```text
== MSV ==
Username: bob
Domain: DESKTOP-33E7O54
LM: NA
NT: 64f12cddaa88057e06a81b54e73b949b
SHA1: cba4e545b7ec918129725154b29f055e4cd5aea8
```

## What it means

MSV is the Windows authentication package used against local SAM / NTLM-style logons.

It may provide:

- username
- domain / hostname
- NT hash
- SHA1 hash

This is often immediately useful for:

- cracking with Hashcat
- pass-the-hash workflows
- password reuse analysis

---

# WDIGEST

Example:

```text
== WDIGEST ==
username bob
domainname DESKTOP-33E7O54
password None
```

## What it means

WDIGEST is an older authentication mechanism.

Historically, it could cache **cleartext passwords** in LSASS.

Important notes:

- older Windows versions often exposed cleartext here
- modern Windows usually has WDIGEST disabled by default
- patched systems often do not expose plaintext

If plaintext is present, this is extremely high value.

---

# Kerberos

Example:

```text
== Kerberos ==
Username: bob
Domain: DESKTOP-33E7O54
```

## What it means

In domain environments, LSASS may store Kerberos material such as:

- tickets
- keys
- session data
- authentication artifacts

This can support:

- ticket abuse
- lateral movement
- access to other domain resources

Even if the output shown is minimal, Kerberos-related material can still be highly valuable.

---

# DPAPI

Example:

```text
== DPAPI ==
masterkey e8bc2faf77e7bd18...
sha1_masterkey 52e758b612038989...
```

## What it means

DPAPI masterkeys can be used to decrypt user secrets protected by Windows applications.

Common DPAPI targets include:

- Chrome saved passwords
- Internet Explorer saved creds
- Outlook credentials
- RDP saved credentials
- Credential Manager entries
- VPN / wireless secrets

This makes DPAPI material very valuable in post-exploitation.

---

# Cracking the NT Hash

If LSASS yields NT hashes, they can be cracked with Hashcat.

## Hashcat Mode

```text
-m 1000
```

## Example

```bash
hashcat -m 1000 64f12cddaa88057e06a81b54e73b949b /usr/share/wordlists/rockyou.txt
```

Example result:

```text
64f12cddaa88057e06a81b54e73b949b:Password1
```

Once cracked, the plaintext password can be tested across:

- WinRM
- SMB
- RDP
- SSH
- local admin reuse
- domain services

---

# Why Offline Parsing Is Useful

Dumping LSASS to file and parsing offline is usually preferable because it:

- reduces target-side activity
- allows multiple parsing attempts later
- separates collection from analysis
- enables Linux-side tooling like pypykatz
- avoids running heavier tooling on victim

# Operational Notes

- LSASS dumping is high signal activity for defenders
- many AV/EDR products detect common dump methods
- `rundll32 + comsvcs.dll` is especially well known
- GUI dump methods may also be monitored
- offline analysis is safer than live parsing on target

---

# Key Takeaways

- LSASS is one of the most valuable Windows credential targets
- it may contain hashes, plaintext creds, Kerberos material, and DPAPI keys
- dumping LSASS to file allows offline analysis
- `pypykatz` is an excellent Linux-side parser
- recovered NT hashes can often be cracked quickly
- DPAPI and Kerberos material can enable deeper compromise
