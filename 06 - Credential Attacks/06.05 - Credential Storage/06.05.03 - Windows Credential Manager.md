## Overview

Windows includes a built-in feature called **Credential Manager** that allows users and applications to store credentials for later use.

These stored credentials may include:

- web credentials
- domain credentials
- service credentials
- shared resource logins
- OneDrive / Microsoft account tokens
- remote access credentials

From an attacker perspective, this is valuable because stored credentials may be:

- reusable for lateral movement
- accessible via built-in Windows tools
- recoverable from memory
- decryptable using DPAPI-related techniques
- exposed through tools like Mimikatz

---

# Credential Manager / Vault Basics

Microsoft has used several related terms over time:

- **Credential Manager** = user-facing feature / API
- **Windows Vault / Credential Locker** = protected encrypted storage locations

Credentials are stored in encrypted folders under user and system profiles.

Common paths:

```text
%UserProfile%\AppData\Local\Microsoft\Vault\
%UserProfile%\AppData\Local\Microsoft\Credentials\
%UserProfile%\AppData\Roaming\Microsoft\Vault\
%ProgramData%\Microsoft\Vault\
%SystemRoot%\System32\config\systemprofile\AppData\Roaming\Microsoft\Vault\
```

These stores are protected using:

- AES keys
- DPAPI
- newer protections such as Credential Guard / VBS on supported systems

---

# What Is Stored

Windows commonly stores two main categories of credentials.

| Type | Description |
|------|-------------|
| Web Credentials | website / browser / online account credentials |
| Windows Credentials | domain creds, shared resources, network logons, services, OneDrive, etc. |

---

# Policy.vpol

Each vault folder may contain a file called:

```text
Policy.vpol
```

This contains AES keys used to encrypt the stored credentials.

Important point:

- the AES keys are themselves protected by **DPAPI**

So Credential Manager extraction is closely related to **DPAPI attacks**.

---

# Why This Matters

Stored credentials can provide:

- access to network shares
- impersonation of another user
- reusable passwords for domain or local accounts
- access to cloud-linked services
- credentials for later lateral movement

This makes Credential Manager a high-value post-exploitation target.

---

# Exporting Vaults

Windows allows vaults to be exported to `.crd` files.

GUI / command method:

```cmd
rundll32 keymgr.dll,KRShowKeyMgr
```

This opens the Credential Manager backup interface.

Notes:

- exported backups are password protected
- backups can be imported on another Windows host
- useful for legitimate backup, but also relevant during investigations / offline analysis

# Enumerating Stored Credentials with cmdkey

A quick built-in way to enumerate stored credentials is:

```cmd
cmdkey /list
```

Example:

```cmd
C:\Users\sadams>whoami
srv01\sadams

C:\Users\sadams>cmdkey /list
```

Possible output:

```text
Currently stored credentials:

    Target: WindowsLive:target=virtualapp/didlogical
    Type: Generic
    User: 02hejubrtyqjrkfi
    Local machine persistence

    Target: Domain:interactive=SRV01\mcharles
    Type: Domain Password
    User: SRV01\mcharles
```

---

# Understanding cmdkey Output

| Field | Meaning |
|------|---------|
| Target | resource / service / account the credential belongs to |
| Type | credential type |
| User | associated username |
| Persistence | whether credential survives reboot / is stored persistently |

## Domain Password / Interactive

Example:

```text
Target: Domain:interactive=SRV01\mcharles
Type: Domain Password
User: SRV01\mcharles
```

This is especially interesting because it suggests a stored credential for another user.

---

# Using Stored Credentials with runas

If a credential is stored and usable, Windows may allow impersonation with:

```cmd
runas /savecred /user:SRV01\mcharles cmd
```

This can start a new process using the stored credential.

Example:

```cmd
Attempting to start cmd as user "SRV01\mcharles" ...
```

Why this matters:

- you may not need to know the plaintext password
- Windows may use the saved credential automatically
- useful for privilege escalation or lateral movement opportunities

# Extracting Credentials with Mimikatz

Several tools can target Credential Manager / Vault material.

Common options include:

- Mimikatz
- SharpDPAPI
- LaZagne
- DonPAPI

One simple approach with Mimikatz is to inspect **credman data from LSASS memory**.

---

## Mimikatz Example

Start Mimikatz:

```cmd
mimikatz.exe
```

Enable debug privileges:

```cmd
privilege::debug
```

Extract CredMan-related data:

```cmd
sekurlsa::credman
```

---

## Example Output

```text
Authentication Id : 0 ; 630472
Session           : RemoteInteractive from 3
User Name         : mcharles
Domain            : SRV01

credman :
 * Username : mcharles@inlanefreight.local
 * Domain   : onedrive.live.com
 * Password : ...SNIP...
```

This may reveal:

- stored usernames
- associated domains / targets
- plaintext passwords in some cases

---

# Why sekurlsa::credman Works

Credential Manager data may be accessible because:

- user has logged on
- relevant secrets are present in LSASS-related memory structures
- stored credentials are loaded or referenced during session activity

This ties Credential Manager attacks closely to **LSASS credential access**.

# Practical Workflow

## Quick Enumeration Workflow

1. check current user
2. run `cmdkey /list`
3. identify interesting stored credentials
4. test `runas /savecred` where relevant
5. investigate vault / credential paths
6. if high privilege available, use Mimikatz / DPAPI tooling

---

## High-Privilege Extraction Workflow

1. gain administrative access
2. inspect LSASS / CredMan data with Mimikatz
3. identify stored usernames / targets
4. recover passwords where possible
5. pivot using exposed credentials

# High-Value Targets to Look For

Particularly interesting stored credentials include:

- file share credentials
- RDP credentials
- browser-stored business app logins
- cloud admin accounts
- backup system credentials
- VPN / remote access secrets

---

# Limitations

Credential Manager attacks may be limited by:

- Credential Guard
- VBS protections
- hardened LSASS settings
- lack of privilege
- absent / unloaded credentials
- restricted interactive use of stored creds

Not all listed credentials immediately expose plaintext passwords.

Sometimes you get:

- metadata only
- targets and usernames
- reusable stored auth via Windows
- secrets only recoverable via DPAPI tooling

---

# Key Takeaways

- Windows Credential Manager stores reusable authentication material
- stored credentials may exist under user and system vault paths
- `cmdkey /list` is a fast built-in enumeration method
- `runas /savecred` may allow impersonation without knowing plaintext password
- Mimikatz can expose CredMan data from memory
- Credential Manager is strongly tied to DPAPI and LSASS credential access
