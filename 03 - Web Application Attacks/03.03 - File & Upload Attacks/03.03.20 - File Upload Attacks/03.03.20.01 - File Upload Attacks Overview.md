## Purpose

File upload vulnerabilities occur when an application allows users to upload files without properly validating the file name, extension, content, type, location, or execution behaviour.

This can lead to:

- web shell upload
- remote code execution
- stored payloads
- file overwrite
- path traversal
- client-side attacks
- malicious document upload
- denial of service
- sensitive file exposure
- bypass of intended application controls

---

## Core Idea

File upload exploitation is not just about uploading a file.

The key questions are:

- Can I upload a file?
- Can I control the file name?
- Can I control the extension?
- Can I control the content?
- Where is the file stored?
- Can I access the uploaded file?
- Can the server execute the uploaded file?
- Can I bypass validation?
- Can I use the upload to gain command execution?

---

## Typical Attack Flow

1. Identify upload functionality.
2. Upload a harmless test file.
3. Observe how the application handles the file.
4. Identify client-side and server-side validation.
5. Test extension restrictions.
6. Test content-type restrictions.
7. Test file signature or magic byte checks.
8. Identify the upload path.
9. Check whether the file is accessible.
10. Check whether the file can execute.
11. Bypass restrictions where appropriate.
12. Convert upload into command execution, file disclosure, or access.

## Validation Types

|Validation Type|Description|
|---|---|
|Client-side validation|Checks performed in the browser|
|Server-side validation|Checks performed by the backend|
|Extension validation|Allows or blocks based on file extension|
|MIME/type validation|Allows or blocks based on declared content type|
|Magic byte validation|Checks file signature/header bytes|
|File size validation|Limits upload size|
|Content validation|Checks actual file content|
|Filename validation|Controls names, special characters, and paths|
|Whitelist validation|Only allows specific approved extensions|
|Blacklist validation|Blocks known dangerous extensions|

---

## Common Upload Weaknesses

Common weaknesses include:

- no validation
- client-side only validation
- weak blacklist filtering
- incomplete whitelist filtering
- trusting MIME type headers
- trusting file extensions only
- poor file renaming
- predictable upload paths
- upload directory inside web root
- executable scripts allowed
- double extensions allowed
- case-sensitive extension bypasses
- null byte or encoding issues
- path traversal in filenames
- unrestricted file size
- insecure image processing
- archive extraction issues

## Useful Testing Questions

Ask:

- Is validation performed client-side, server-side, or both?
- Are only extensions checked?
- Are MIME types checked?
- Are magic bytes checked?
- Are filenames sanitised?
- Are files renamed?
- Is the upload path predictable?
- Can uploaded files be accessed directly?
- Are uploaded files stored in the web root?
- Are uploaded files executable?
- Can double extensions be used?
- Can case changes bypass filtering?
- Can encoded characters bypass filtering?
- Can a valid file type contain executable code?
- Can uploaded files be included through LFI?
- Can uploaded files overwrite existing files?

## Common Bypass Areas

File upload restrictions may be bypassed using:

- client-side validation bypass
- extension manipulation
- double extensions
- mixed-case extensions
- trailing characters
- content-type header changes
- magic bytes
- polyglot files
- filename encoding
- path traversal
- alternate script extensions
- server misconfiguration
- upload plus file inclusion
- archive extraction behaviour

---

## Moving From Upload to Access

A successful upload becomes useful when it can be converted into impact.

Possible outcomes:

|Upload Result|Possible Impact|
|---|---|
|Web shell uploaded and executed|Command execution|
|File uploaded to web root|Direct access to payload|
|File uploaded but not executable|May still support LFI to RCE|
|File overwrite possible|Application manipulation or credential overwrite|
|Image upload accepts script content|Filter bypass or polyglot attack|
|Archive upload extracts files|Path traversal or file write|
|Document upload accepted|Stored payload or client-side impact|
|Upload path disclosed|Direct file access|
