## Purpose

SQL injection occurs when an application includes user-controlled input in a database query without properly validating, sanitising, or parameterising it.

This can allow an attacker to interfere with the query and potentially access, modify, or extract database data.

SQL injection can lead to:

- authentication bypass
- data disclosure
- database enumeration
- credential extraction
- password hash disclosure
- file read
- file write
- command execution
- application compromise
- initial access

---

## Core Idea

The application sends SQL queries to a database.

If user input is inserted into the query unsafely, an attacker may be able to alter the query structure.

Example vulnerable logic:

```
SELECT * FROM users WHERE username = '<input>' AND password = '<input>';
```

If input is not handled safely, an attacker may be able to change the query logic and bypass authentication or extract data.

## SQL Injection Types

|Type|Description|
|---|---|
|Authentication bypass|Alters login query logic|
|UNION-based SQL injection|Uses UNION to combine attacker-controlled SELECT output|
|Error-based SQL injection|Uses database errors to disclose information|
|Boolean-based blind SQL injection|Infers data from true/false application behaviour|
|Time-based blind SQL injection|Infers data using database delays|
|Stacked queries|Executes multiple SQL statements where supported|
|Out-of-band SQL injection|Exfiltrates data via DNS/HTTP callbacks|
|File read/write|Uses database permissions to access the filesystem|
|SQLi to command execution|Converts database control into OS command execution|

---

## Typical Attack Flow

1. Identify a possible injection point.
2. Test whether input changes query behaviour.
3. Determine whether output is visible or blind.
4. Use comments to control the remaining query.
5. Determine the number of columns, if using UNION.
6. Identify which columns are reflected in the response.
7. Enumerate database type and version.
8. Enumerate databases, tables, and columns.
9. Extract useful data.
10. Look for credentials, hashes, or tokens.
11. Check whether file read or file write is possible.
12. Check whether command execution is possible.
13. Convert the finding into access where possible.

---

## Basic Behavioural Signs

Possible signs of SQL injection include:

- SQL error messages
- different response content
- different response length
- successful authentication bypass
- empty results after payload input
- delayed response
- HTTP 500 errors
- database syntax errors
- application crashes
- inconsistent filtering behaviour

## Query Logic Abuse

Subverting query logic is often the first practical step.

Common goals:

- bypass login checks
- force a true condition
- force a false condition
- control operator precedence
- comment out the rest of the query
- break out of quoted strings
- identify whether the input is numeric or string-based

---

## UNION-Based SQL Injection

UNION-based SQL injection is useful when query results are reflected in the response.

The goal is to:

- determine the column count
- identify reflected columns
- match compatible data types
- extract database information
- extract table and column data

---

## Blind SQL Injection

Blind SQL injection is used when query output is not directly visible.

Common blind techniques:

- boolean-based checks
- time-based delays
- conditional responses
- response length differences
- out-of-band callbacks

---

## Database Enumeration

Once injection is confirmed, enumerate the database.

Useful targets:

- database type
- database version
- current database
- current user
- table names
- column names
- user tables
- password hashes
- application configuration tables
- admin accounts
- API keys or tokens

---

## Moving From SQL Injection to Access

SQL injection becomes more valuable when it leads beyond simple data disclosure.

Possible paths include:

|SQLi Outcome|Possible Next Step|
|---|---|
|Admin credentials found|Log into application admin panel|
|Password hashes found|Crack hashes and test reuse|
|Database credentials found|Test database and system reuse|
|File read possible|Read config files or keys|
|File write possible|Write web shell if path is executable|
|Command execution possible|Attempt shell access|
|Admin session bypassed|Re-enumerate authenticated functionality|

---

## File Read and File Write

Some databases and configurations allow filesystem interaction.

This may enable:

- reading application configuration files
- reading system files
- writing web shells
- writing files into web-accessible directories
- staging payloads

---

## SQL Injection to Command Execution

In some cases, SQL injection can lead to operating system command execution.

This depends heavily on:

- database type
- database privileges
- enabled features
- filesystem permissions
- stacked query support
- writable web paths
- dangerous database procedures
- service account privileges

## Common Mistakes

- [ ]  Assuming no error means no SQL injection
- [ ]  Ignoring blind SQL injection
- [ ]  Not checking both numeric and string contexts
- [ ]  Forgetting to use comments correctly
- [ ]  Not determining column count before UNION injection
- [ ]  Not checking reflected columns
- [ ]  Not identifying the database type
- [ ]  Extracting data without looking for credentials
- [ ]  Forgetting password reuse
- [ ]  Not checking file read or file write possibilities
- [ ]  Trying command execution before confirming privileges
- [ ]  Not documenting the working payload
