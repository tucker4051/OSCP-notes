## Purpose

Command injection occurs when an application passes user-controlled input into an operating system command without properly validating or sanitising it.

This can allow an attacker to execute arbitrary system commands on the target server.

Command injection is high impact because it can often lead directly to:

- command execution
- file read/write
- credential discovery
- reverse shell access
- web shell placement
- privilege escalation opportunities
- full system compromise

---

## Core Idea

The application is already running a system command in the background.

The attacker’s goal is to break out of the intended command structure and append, chain, or inject additional commands.

Example vulnerable behaviour:

```
Application input:
127.0.0.1

Backend command:
ping 127.0.0.1
```

If the input is not handled safely, an attacker may be able to submit something like:

```
127.0.0.1; whoami
```

The backend may then execute:

```
ping 127.0.0.1; whoami
```

Command injection is commonly found in functionality that interacts with the operating system.

Examples:

- log viewing tools
- admin panels
- monitoring tools
- device management interfaces

## Basic Attack Flow

1. Identify input that may interact with the operating system.
2. Test for command execution using safe commands.
3. Identify which injection operators work.
4. Confirm output behaviour.
5. Identify filtering or blocked characters.
6. Bypass filters if required.
7. Determine the target operating system.
8. Move from proof of concept to useful execution.
9. Attempt file read, file write, or reverse shell where appropriate.
10. Document the working payload and impact.

## Detection

Start with simple, low-impact tests.

Useful checks include:

- time delay
- command output
- error output
- DNS or HTTP callback
- change in response length
- change in response time
- application error message
- unexpected command result

Common safe commands:

|Platform|Commands|
|---|---|
|Linux|`whoami`, `id`, `hostname`, `pwd`, `uname -a`|
|Windows|`whoami`, `hostname`, `ver`, `cd`, `ipconfig`|

## Injection Operators

Command injection often depends on whether the application allows command chaining operators.

Common operators include:

|Operator|Purpose|
|---|---|
|`;`|Run another command after the first|
|`&&`|Run next command if previous command succeeds|
|`||
|`&`|Run command in background or chain command|
|`|`|
|`` `command` ``|Command substitution on Linux|
|`$(command)`|Command substitution on Linux|
|`%0a`|Newline injection|
|`%26`|URL-encoded ampersand|

## Output-Based vs Blind Command Injection

### Output-Based Command Injection

The command output is returned in the application response.

This is easier to exploit because the result can be seen directly.

Example signs:

- command output appears on the page
- error messages are returned
- response changes based on command output

### Blind Command Injection

The command executes, but output is not shown.

This requires indirect validation.

Common techniques:

- time delays
- DNS callbacks
- HTTP callbacks
- writing files
- triggering outbound connections
- using side effects

---

## Filter Identification

Applications may block specific characters, commands, spaces, or payload patterns.

Common filters include:

- blocked command separators
- blocked spaces
- blocked slashes
- blocked keywords
- blocked shell metacharacters
- blocked encoded characters
- command allowlists
- input length limits
- server-side validation
- web application firewall rules

---

## Common Bypass Areas

If basic injection is blocked, test for bypasses carefully.

Common bypass categories:

- blacklisted character bypasses
- blacklisted command bypasses
- space filter bypasses
- encoding
- environment variable expansion
- command substitution
- path manipulation
- quote escaping
- case changes
- wildcard usage

---

## Moving From Command Execution to Access

Once command execution is confirmed, decide what to do next.

Possible next steps:

- identify current user
- identify operating system
- identify working directory
- check available tools
- check outbound connectivity
- read useful files
- write a test file
- attempt reverse shell
- upload a payload
- enumerate local users
- look for credentials
- move to initial access

Do not jump straight to a reverse shell until basic command execution is confirmed.

---

## Useful Impact Questions

Ask:

- What user is the command running as?
- Is the command running on Linux or Windows?
- Is command output visible?
- Can I use command separators?
- Are spaces filtered?
- Are common commands filtered?
- Can the target reach my machine?
- Can I write files?
- Can I execute uploaded files?
- Can I get a reverse shell?
- Can I use this to find credentials?
- Can I use this for privilege escalation?

## Common Outcomes

|Outcome|Meaning|
|---|---|
|Command output visible|Direct command execution confirmed|
|Delayed response|Blind command injection likely|
|DNS/HTTP callback received|Blind command execution confirmed|
|File created|File write possible|
|Reverse shell received|Initial access achieved|
|Command blocked|Filtering or validation likely|
|No response change|Injection failed or blind testing needed|
