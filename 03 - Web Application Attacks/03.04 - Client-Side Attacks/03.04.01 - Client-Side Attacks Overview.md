## Purpose

Client-side attacks target the behaviour of the user’s browser or client-side application logic.

This section currently focuses on Cross-Site Scripting, with room to expand later into other browser-side web vulnerabilities.

---

## Core Idea

Client-side vulnerabilities occur when unsafe data reaches the browser and is processed in a way that affects the user’s session, page, data, or actions.

Unlike server-side vulnerabilities, the immediate target is usually the user’s browser context.

The impact depends on:

- who triggers the payload
- where the payload executes
- what data is accessible
- what actions the victim can perform
- whether the victim is authenticated
- whether a privileged user can be targeted

## Client-Side Testing Flow

1. Identify user-controlled input.
2. Check whether input is reflected, stored, or processed client-side.
3. Identify the browser context.
4. Check encoding and sanitisation.
5. Confirm execution or client-side impact.
6. Determine whether another user can trigger the issue.
7. Determine whether privileged users are affected.
8. Demonstrate realistic impact safely.

---

## Common Impact

Client-side attacks may lead to:

- session hijacking
- credential phishing
- account actions as another user
- page defacement
- sensitive data exposure
- token theft where accessible
- admin-user compromise
- user impersonation
