## Purpose

Cross-site scripting occurs when an application includes user-controlled input in a web page without safely encoding, sanitising, or validating it.

This can allow attacker-controlled JavaScript to execute in another user’s browser.

XSS can lead to:

- session hijacking
- account takeover
- phishing
- page defacement
- credential capture
- browser-based attacks
- user impersonation
- sensitive data theft
- actions performed as another user

---

## Core Idea

XSS targets the **user’s browser**, not the server directly.

The application reflects, stores, or processes input in a way that allows JavaScript execution.

The key questions are:

- Where is user input reflected?
- Is the input HTML-encoded?
- Is JavaScript execution possible?
- Is the payload executed in another user’s context?
- Is the vulnerable page accessible to privileged users?
- What impact can be demonstrated?

---

## Main XSS Types

|Type|Description|
|---|---|
|Reflected XSS|Payload is included in the request and reflected immediately in the response|
|Stored XSS|Payload is saved by the application and served to users later|
|DOM-Based XSS|Client-side JavaScript processes unsafe input and updates the page insecurely|

---

## Common Injection Points

XSS can appear in:

- search boxes
- comment fields
- profile fields
- usernames
- display names
- contact forms
- support tickets
- message boards
- product reviews
- URL parameters
- error messages
- redirect parameters
- hidden fields
- HTTP headers
- file names
- metadata fields
- admin panels
- API responses rendered by the frontend

---

## Common Contexts

The payload depends heavily on where the input lands.

|Context|Example Risk|
|---|---|
|HTML body|Script or tag injection|
|HTML attribute|Attribute breakout|
|JavaScript block|String breakout or JS execution|
|URL context|JavaScript URI or redirect abuse|
|CSS context|Style injection or browser-specific behaviour|
|DOM sink|Client-side execution through unsafe JavaScript|

Understanding the context is more important than memorising payloads.

---

## Typical Attack Flow

1. Identify user-controlled input.
2. Check where the input appears in the response or DOM.
3. Identify the output context.
4. Test with harmless characters.
5. Check whether input is encoded, stripped, or modified.
6. Build a context-appropriate payload.
7. Confirm JavaScript execution.
8. Determine whether the issue is reflected, stored, or DOM-based.
9. Assess who can trigger the payload.
10. Demonstrate realistic impact.

---

## Discovery Approach

Start with simple markers and observe reflection behaviour.

Check:

- [ ]  Input reflected in response
- [ ]  Input reflected in DOM
- [ ]  Special characters encoded
- [ ]  Tags stripped
- [ ]  Quotes escaped
- [ ]  Event handlers allowed
- [ ]  JavaScript URLs allowed
- [ ]  Payload stored and replayed
- [ ]  Payload executes for another user
- [ ]  Payload executes in admin context

---

## Impact Areas

XSS impact depends on context, user role, and application behaviour.

Possible impact includes:

|Impact|Description|
|---|---|
|Session hijacking|Stealing or abusing session tokens where possible|
|Phishing|Presenting fake login prompts or malicious forms|
|Defacement|Modifying page content in the user’s browser|
|Account actions|Performing actions as the victim user|
|Data theft|Reading sensitive page content or tokens available to JavaScript|
|Admin compromise|Triggering payloads in privileged user sessions|

---

## Stored vs Reflected Impact

|Type|Typical Impact|
|---|---|
|Reflected XSS|Requires victim interaction with crafted link/request|
|Stored XSS|Executes automatically when affected content is viewed|
|DOM XSS|Depends on client-side JavaScript and browser-side input handling|

Stored XSS is often higher impact because the payload persists and may trigger for multiple users.

---

## Useful Testing Questions

Ask:

- Is the payload reflected?
- Is the payload stored?
- Is execution happening in the browser?
- Is the vulnerable context authenticated?
- Can another user trigger the payload?
- Can an admin trigger the payload?
- Are cookies accessible to JavaScript?
- Is there a Content Security Policy?
- Can the payload perform authenticated actions?
- Can the payload read sensitive page content?
- Can the payload support a phishing or session attack?

---

## Moving From XSS to Impact

Once XSS is confirmed, demonstrate meaningful impact safely.

Possible demonstrations:

- JavaScript execution proof
- page content modification
- reading non-sensitive page data
- showing current user context
- demonstrating stored execution
- demonstrating execution in a higher-privileged context
- showing how actions could be performed as the victim

Avoid focusing only on alert boxes. Prove why the issue matters.

## Common Mistakes

- [ ]  Treating every reflection as XSS
- [ ]  Not checking the output context
- [ ]  Not testing stored behaviour
- [ ]  Not checking DOM sinks
- [ ]  Using payloads before understanding filtering
- [ ]  Ignoring encoding and escaping
- [ ]  Ignoring Content Security Policy
- [ ]  Only proving alert execution without impact
- [ ]  Not checking whether cookies are HttpOnly
- [ ]  Not considering privileged-user impact
- [ ]  Not documenting the exact vulnerable request or field
