## Overview

**IDOR (Insecure Direct Object Reference)** is a common access control vulnerability that occurs when an application exposes a direct reference to an internal object and does not properly verify whether the requesting user is authorised to access it.

Examples of objects:

- files
- database records
- user profiles
- invoices
- API resources
- messages
- account settings

If the application allows a user to manipulate the reference and access data belonging to other users, the application is vulnerable.

Think of IDOR like hotel room key cards labelled **Room 101, 102, 103** where the lock does not actually check which guest owns the card.

---

# 1. Basic Example

Application provides file download link:

```http
download.php?file_id=123
```

User accesses their file successfully.

Attacker modifies parameter:

```http
download.php?file_id=124
```

If the application does not verify ownership, the attacker may retrieve another user's file.

---

# 2. What Actually Causes IDOR

The vulnerability is **not simply exposing an ID**.

The vulnerability exists when:

- the application fails to validate authorisation
- the backend does not verify ownership
- access control logic is missing or weak

Example of missing check:

```php
SELECT * FROM files WHERE id = $_GET['file_id']
```

Secure version would also verify:

```php
SELECT * FROM files 
WHERE id = $_GET['file_id']
AND owner_id = $_SESSION['user_id']
```

---

# 3. Why IDOR Is So Common

Building secure access control systems is difficult.

Developers must ensure that:

- every object
- every endpoint
- every API
- every request

performs correct authorisation checks.

Common mistakes:

- relying only on frontend controls
- assuming users cannot modify requests
- trusting hidden form fields
- exposing predictable object identifiers

Because access control logic is complex, IDOR vulnerabilities are frequently found in production systems.

Even large platforms have historically suffered from IDOR issues.

---

# 4. Potential Impact

IDOR vulnerabilities can allow attackers to:

## Information disclosure

Access:

- personal files
- invoices
- emails
- internal documents
- credit card data
- medical records

---

## Data modification

Modify:

- account settings
- user profile data
- payment information
- roles
- configuration

---

## Account takeover

Change:

- passwords
- email addresses
- MFA settings

---

## Privilege escalation

Access admin-only functions.

Example:

```http
POST /admin/change_role
```

If backend does not validate role permissions, attackers may gain admin privileges.

# 5. Where IDORs Are Found

IDOR vulnerabilities commonly appear in:

## URL parameters

```http
/profile?uid=10
/invoice?id=2001
/download?file=report.pdf
```

---

## API endpoints

```http
GET /api/users/15
GET /api/orders/7781
```

---

## POST parameters

```http
POST /update
user_id=45
```

---

## hidden form fields

```html
<input type="hidden" name="user_id" value="12">
```

---

## cookies

```http
Cookie: user_id=15
```

---

## JSON requests

```json
{
  "user_id": 5
}
```

# 6. Identifying IDOR Vulnerabilities

First step is locating **direct object references**.

Look for parameters such as:

| Parameter | Meaning |
|----------|--------|
| id | numeric identifier |
| uid | user id |
| file | filename |
| document | document reference |
| account | account id |
| order | order id |

---

## Example

```http
GET /download?file_id=100
```

Test:

```http
GET /download?file_id=101
GET /download?file_id=102
GET /download?file_id=103
```

If other users' data is returned → IDOR exists.

---

# 7. Sequential ID Testing

Many applications use incremental IDs:

```text
1
2
3
4
5
```

This makes enumeration straightforward.

Tools such as:

- Burp Intruder
- ffuf
- wfuzz

can automate testing many IDs.

---

# 8. AJAX Endpoint Discovery

Modern applications often use JavaScript frameworks.

Important API calls may exist in frontend code even if hidden from UI.

Example AJAX request:

```javascript
function changeUserPassword() {
    $.ajax({
        url:"change_password.php",
        type: "post",
        dataType: "json",
        data: {uid: user.uid, password: user.password, is_admin: is_admin}
    });
}
```

Even if non-admin users cannot see this functionality in the interface, the endpoint may still exist.

If backend validation is weak, attackers may call:

```http
POST /change_password.php
uid=1
password=newpass
is_admin=true
```

Potential result:

privilege escalation.

---

# 9. Encoded References

Applications may encode object identifiers.

Example:

```http
download?filename=ZmlsZV8xMjMucGRm
```

Base64 decoding:

```text
file_123.pdf
```

Modify reference:

```text
file_124.pdf
```

Encode:

```text
ZmlsZV8xMjQucGRm
```

Test:

```http
download?filename=ZmlsZV8xMjQucGRm
```

If file downloads → IDOR present.

---

# 10. Hashed References

Applications sometimes hash object names.

Example:

```http
download.php?filename=c81e728d9d4c2f636f067f89cc14862c
```

Frontend code reveals:

```javascript
CryptoJS.MD5('file_1.pdf')
```

Attacker calculates:

```text
MD5(file_2.pdf)
```

Requests:

```http
download.php?filename=<calculated hash>
```

If successful → IDOR vulnerability.

---

# 11. Comparing Multiple Users

Creating multiple accounts helps identify patterns.

Example:

User1 API response:

```json
{
  "type": "salary",
  "url": "/services/data/salaries/users/1"
}
```

Logged in as User2:

Attempt:

```http
GET /services/data/salaries/users/1
```

If data returned → IDOR.

# 12. IDOR Testing Workflow

## Step 1 – identify object references

Look for:

- numeric IDs
- filenames
- UUIDs
- encoded values
- hashes

---

## Step 2 – manipulate values

Test:

- increment numbers
- decrement numbers
- change filenames
- decode and modify values
- re-hash guessed values

---

## Step 3 – observe responses

Indicators of IDOR:

- data from other users returned
- successful access to restricted content
- ability to modify resources

---

## Step 4 – automate testing

Use:

Burp Intruder

Example positions:

```text
file_id=§123§
```

Payload list:

```text
1
2
3
4
5
```

---

# 13. IDOR vs Authentication

Authentication verifies identity.

Access control verifies permissions.

IDOR occurs when:

authentication exists

but authorisation checks are missing.

---

# 14. Secure Implementation

Always verify user ownership server-side.

Example:

```php
if($file.owner_id !== $_SESSION['user_id']){
    deny_access();
}
```

Never rely solely on:

- hidden fields
- frontend controls
- UI restrictions

---

# 15. Key Takeaways

- IDOR occurs when object references are exposed without proper access control
- modifying identifiers may allow access to other users' data
- sequential IDs are especially vulnerable
- encoded or hashed references do not guarantee security
- AJAX endpoints often expose hidden functionality
- always validate authorisation on the backend
- IDOR vulnerabilities can lead to data exposure, privilege escalation, and account takeover
