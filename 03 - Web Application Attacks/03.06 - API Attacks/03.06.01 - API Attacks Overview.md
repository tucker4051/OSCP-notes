## Purpose

API attacks target weaknesses in application programming interfaces, including REST APIs, JSON endpoints, GraphQL APIs, mobile app APIs, and backend service endpoints.

This section currently contains API-specific IDOR material and can expand later as more API notes are added.

---

## Core Idea

APIs often expose application logic directly.

They may rely heavily on:

- object identifiers
- tokens
- JSON bodies
- hidden fields
- role claims
- client-side enforcement
- predictable IDs
- backend object relationships

A secure frontend does not mean the API is secure.

## Common API Vulnerability Areas

Potential future notes include:

- BOLA / IDOR
- broken object property level authorisation
- mass assignment
- excessive data exposure
- weak rate limiting
- JWT attacks
- API token leakage
- GraphQL introspection abuse

---

## API Testing Flow

1. Identify API endpoints.
2. Capture requests through browser, Burp, or client app.
3. Identify authentication method.
4. Map object identifiers.
5. Test horizontal access control.
6. Test vertical access control.
7. Modify JSON fields and hidden parameters.
8. Test mass assignment.
9. Test rate limits.
10. Review response data for excessive exposure.
11. Test API-specific IDOR/BOLA paths.

## Common API Data Sources

Review:

- browser developer tools
- Burp history
- JavaScript files
- mobile app traffic
- Swagger/OpenAPI docs
- GraphQL endpoints
- API error messages
- frontend route files
- network requests
- old API versions

## Common Impact

API vulnerabilities can lead to:

- unauthorised object access
- mass data extraction
- account takeover
- role modification
- user impersonation
- sensitive data disclosure
- unauthorised admin actions
- bypass of frontend controls
